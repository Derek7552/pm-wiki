# Heuristic Learning 范式假设——翁家翌让 Codex"养"策略代码与梯度之外的学习路径

> 来源：原文《OpenAI 后训练工程师翁家翌，给 Agentic AI 提出了新范式假设》（用户直接提供正文）；原始博客《Learning Beyond Gradients》，作者翁家翌，2026-05-08
> 提炼日期：2026-05-09
> 分类：AI方法论

## TL;DR

OpenAI 后训练工程师翁家翌在 2026-05-08 博客《Learning Beyond Gradients》中提出 **Heuristic Learning（HL，启发式学习）** 范式假设：当 coding agent（Codex / gpt-5.4）足够强，AI 不必把经验压进神经网络权重，而是可以通过自主写代码、跑测试、看回放、改逻辑的循环，"养"出一套**Heuristic System（HS，启发式系统）**——纯 Python 程序化策略。实验在 Atari Breakout 拿到 864 分理论满分，在 Atari57 全集上中位 HNS 达 0.83，与 PPO 基线（0.80–0.98）打成平手；但在 Montezuma's Revenge 上退化为 86 步硬编码动作序列，暴露 HL 在长视野规划上的表达力上限。这条路径的真正价值不是"AI 写出完美策略"，而是把"如何更新参数"的数学问题，改写成"如何维护一套不断吸收反馈的软件系统"的工程问题——可解释、可审计、可回归测试、不灾难性遗忘。

## 核心观点

### 观点一：当 coding agent 足够强，学习不必只发生在权重里

过去十年深度学习证明"经验可以被压缩进权重"，但留下了"为什么成功、为什么失败难以解释和修正"的难题。翁家翌这次的核心命题是：在明确目标、可运行环境和反馈闭环中，AI 既可以通过训练模型变强，也可以通过 coding agent **自主改代码**变强——经验被写进代码、测试、日志和回放，变成可阅读、可修改、可复查、可审计的软件系统。这不是"AI 替代程序员"，是"AI 替代部分梯度下降"。

### 观点二：HL 的实质贡献不是程序化策略本身，而是"coding agent 作为维护通道"

学术界对 Programmatic RL（程序化强化学习）已讨论多年——PROPEL（2019, Rice & Caltech）、LEAPS（2021）、HPRL（ICML 2023）、LLM-GS（2024, 台大 × 微软）等都在尝试用程序而非神经网络做策略。共识是程序化策略具备更好的可解释性、可形式化验证性和泛化能力。**翁家翌的实质贡献，是把 coding agent 视为维护启发式系统的工程通道**：过去依赖手工 DSL 或受限程序空间搜索，现在借助 Codex 把代码、日志、测试、视频回放、参数调整都纳入同一个 agent 的工作流，让程序策略的迭代成本被一次性压低。"维护成本太高"的启发式策略，第一次重新变得划算。

### 观点三：HL 把灾难性遗忘从结构问题改写为工程问题

神经网络的灾难性遗忘是结构性的——学了新东西旧能力会被冲掉，工程师只能补数据、重训、回归测试，然后赌新模型没引入新问题。HL 也会遗忘，但形式工程化：一条新规则修复了一个失败模式却破坏了旧场景、一条新记忆把 agent 引向错误方向、一次补丁改了共享接口、旧调用路径悄悄失效。这些问题没有自动消失，但它们都是软件工程已经处理几十年的问题，有现成的工具链——回归测试、版本 diff、固定种子回放、golden trace、显式失败方向记录。**"如何更新参数"被改写成"如何维护一套不断吸收反馈的软件系统"——后者未必更容易，但更接近人类已有的能力边界。**

## 关键知识点

### 1. Heuristic Learning（HL）与 Heuristic System（HS）的定义

**HL**：由程序代码构成的学习过程；和深度强化学习一样有"状态、动作、反馈、更新"循环；不同的是被更新对象是软件结构，反馈被 coding agent 消化，更新不使用反向传播——coding agent 直接编辑策略、状态检测器、测试、配置或记忆。

**HS**：HL 维护出的对象，即一套不断进化的程序化策略 + 状态检测器 + 测试集 + 日志 + 记忆。

### 2. HL vs Deep RL 对照表（翁家翌博客原表）

| 维度 | Heuristic Learning | Deep RL |
|------|---------------------|---------|
| 策略形式 | 规则、状态机、控制器、MPC、宏动作组成的代码 | 神经网络参数 |
| 状态形式 | 显式变量、检测器、缓存 | 网络可读的观测向量 |
| 反馈形式 | 测试、日志、回放都是有效信号 | 主要依赖固定奖励函数 |
| 记忆形式 | 显式存储试验、摘要、失败原因、版本 diff | on-policy 几乎无；off-policy 依赖 replay buffer |
| 更新方式 | coding agent 编辑代码 / 配置 / 记忆 | 反向传播 + 梯度更新 |

### 3. 由这个对照导出的工程性质

- **可解释**：策略可被翻译成自然语言描述。
- **样本效率**：单位是"一次有效的代码改动"而非缓慢的梯度更新。
- **旧能力固化**：可变成回归测试、固定种子回放、golden case。
- **过拟合可控**：通过简化、回归检查、多种子评估约束。
- **抗灾难性遗忘**：旧能力存在于规则集和测试中，不必只存在于权重。

### 4. Breakout 实验的工程闭环

**触发动机**：翁家翌业余维护 EnvPool，需要比"每次都跑神经网络"更便宜的 CI 测试方式——把神经网络放进 CI 太贵。

**关键转折**：第一版提示词"写一个能解决 Breakout 的策略"失败——低分本身没提供任何信息（动作语义、状态检测、评估流程、策略结构都可能是问题）。**翁家翌不再要求 Codex 直接交 policy.py，而是要求它维护一整套循环**：探测动作和观测 → 写状态检测器 → 写策略 → 跑完整 episode → 记录 trials.jsonl 和 summary.csv → 生成视频或曲线 → 检查失败模式 → 修改策略 → 简化代码 → 跑回归。

**RAM 版本分数演进**：

| 阶段 | 分数 | 关键改动 |
|------|------|---------|
| baseline | 99 | 从 RGB 帧识别球/挡板/砖块颜色 + 扫描 128 字节 RAM |
| 加隧道偏移 | 387 | 隧道偏移逻辑 |
| 加循环打破 | 507 | 长时间无奖励时给落点预测加偏移量，打出局部循环 |
| 加 fast_low_ball_lead_steps=3 | 839 | 修复"快速低球过度领先"漂走问题 |
| 后期偏移释放 | **864 满分** | "第一面砖墙打完后，只在球离挡板较远时启用卡住偏移，球靠近时则逐步释放" |

**关键诊断**：387 分是局部高分陷阱，策略稳定接球但球路困入周期性循环——人类工程师可能会继续微调"接球准确度"，**Codex 看视频和最近几十步轨迹后定位到"球路缺乏扰动"**。

**RAM 版本最终成绩**：第 81 个 episode、累计 150 万环境步首次满分；三轮 episode 稳定 864/864/864。

**视觉版本迁移**：把同一套几何控制器迁移到纯 RGB 输入版，第 1 次 310 分 → 第 2 次 428 分 → 第 7 次 episode 后 864 分（对应 14504 步）。⚠️**这不是从零起步 14.5K 步打满分**——结构在 RAM 版本上已经稳定，14.5K 是状态读取层从 RAM 切到 RGB 的"迁移预算"。

### 5. Atari57 批量验证的实验设计

**严苛约束**：
- 57 款游戏 × 2 种观测模式（RAM / Vision）× 3 次重复 = **342 条无人值守搜索轨迹**
- 同一份提示模板，无人工干预
- 不准训练神经网络、不准读游戏源代码、不准利用任何隐藏信息
- 调试和试错的步数必须计入总开销（防止"偷看答案"作弊）

**HNS（Human-Normalized Score）**：把得分按"人类玩家平均水平 = 1"标准化。

**关键数据点**：

| 阶段 | Codex Vision HNS | Codex RAM HNS | PPO 对比 |
|------|------------------|----------------|----------|
| 100 万环境步 | 0.32 | 0.26 | 明显高于 PPO 同期 |
| 970 万环境步 | 0.81 | 0.59 | 接近 PPO 1000 万步水平（约 0.88–0.92）|
| 最佳输入聚合 | 中位 0.83 | — | OpenAI Baselines PPO2: 0.80; CleanRL EnvPool PPO: 0.98 |

**翁家翌冷静划界**：这只是环境交互效率比较，**没有把 Codex 阅读日志、写代码、看视频的成本折算进去**——"跑得快不等于总成本低"，后者目前是黑箱。

### 6. Codex 在 Atari57 上的能力分布（不均匀）

| 表现 | 游戏特征 | 代表 |
|------|---------|------|
| HL 与 Deep RL 都明显超过人类 | 几何结构清晰 | Breakout、Boxing、Krull |
| HL 反而更强 | 有明确规则 | Asterix、Jamesbond、Tennis |
| PPO 仍碾压 HL | 节奏快、模式复杂 | Atlantis、VideoPinball、RoadRunner、StarGunner |

### 7. Montezuma's Revenge 反例——HL 表达力上限

**现象**：Codex 在 Montezuma's Revenge 拿到 400 分，但策略文件不是真正的策略，而是**86 个动作的硬编码序列，对应 1769 个环境步**——背了一条固定路线，没学会走迷宫。

**翁家翌原话**："这是一个边界案例，不应该被理解为通用的 Montezuma 策略。"

**暴露的问题**：普通程序策略本质是"看到什么状态就做什么动作"的反应式逻辑，难以处理：
- 严格动作时序
- 从中间状态接续计划
- 长视野规划

**这类任务需要的不是更多 if-else，而是"宏动作组合 + 可恢复搜索状态 + 长期记忆"的程序结构**——哪怕 coding agent 再强，有些问题也不是普通代码能装下的。

### 8. 神经网络 + HL 的结合：System 1 / System 2 分工

翁家翌借用 Kahneman 的 System 1 / System 2 语言：

| 层 | 角色 | 承担方 |
|----|------|--------|
| System 1（部分）| 快速感知、分类、物体状态估计 | 专门化的浅层神经网络 |
| System 1（部分）| 新鲜数据处理、规则、测试、回放、记忆、安全边界、局部恢复 | HL |
| System 2 | 反馈提供、数据改进；周期性从 HL 生成数据中提取信息更新自身 | LLM agent |

**翁家翌坦诚边界**：HL 不能完成神经网络能做的所有事情——尤其在复杂感知和长视野泛化上；以今天的认知，**他无法想象一个 agent 用纯 Python 代码、不借助任何神经网络去解决 ImageNet**。

### 9. 学术谱系（HL 不是凭空冒出）

| 年份 | 工作 | 团队 | 核心思想 |
|------|------|------|---------|
| 2019 | PROPEL | Rice + Caltech | 把策略表示为符号语言中的短程序 |
| 2021 | LEAPS | — | 学习程序嵌入空间，可微程序策略 + RL 训练结合 |
| 2023 | HPRL（ICML）| — | 分层程序化强化学习，meta-policy 组合多个程序 |
| 2024 | LLM-GS | 台大 × 微软 | 用 LLM 编程能力 + 常识推理引导程序化 RL 策略搜索 |

共识：**相比神经策略，程序化策略具备更好的可解释性、可形式化验证性、对未见场景的泛化能力**。

## 启发与思考

### 与个人工作的关联（云起无垠产品总监 / 安全 + AI 方向）

1. **HL 思想可直接迁移到漏洞挖掘 / 红队场景**：把"漏洞规则、payload 模板、测试用例、扫描脚本、误报抑制"看作 Heuristic System，用 Codex/Claude 维护——这与上一篇 GigaOm 报告里 Exaforce "多模型融合、不押注单一 LLM"的思路同构。云起的安全规则库、扫描器配置、payload 集合，都是天然的 HS 候选。
2. **可审计性是安全产品的杀手级卖点**：在合规、监管、责任追溯敏感的场景（金融、医疗、政企），客户最大的痛点不是"模型不够强"，而是"出问题不知道是哪个权重"。HL 让安全决策能进入正常的代码评审 + CI 拦截 + SRE 值班响应流程——这是云起做政企客户时可以直接讲的故事。
3. **Agent 能力的代码资产化**：Agent 的失败经验沉淀为可跨会话、跨用户、跨任务复用的代码资产，意味着不同公司、不同团队的 Agent 可以**共享 heuristic 但不共享模型**——这正是企业客户最希望的形态。云起 SaaS 化时，"客户私有 heuristic 仓库 + 公共底座模型"是合理的产品架构。
4. **HL 的边界提醒：不要把所有问题塞进代码**：Montezuma 反例说明长视野规划、稀疏奖励、严格时序的任务无法用普通 if-else 解决。漏洞挖掘里的复杂利用链构造、APT 行为时序分析、跨主机攻击路径推理——这类问题大概率仍需神经网络承担，HL 只能做"局部反应式策略 + 长期记忆 + 安全边界"。
5. **CI 嵌入式 AI 测试是被低估的机会**：翁家翌起点其实是"把神经网络放进 CI 太贵"。这个工程痛点对所有做 AI 驱动产品的团队都成立——能用 HL 做出"轻量、可复现、强于随机"的测试策略，意味着 CI 阶段就能拦截大部分回归。

### 可落地的行动项

- **画一张内部 HS 蓝图**：把云起当前的安全规则、payload 模板、误报抑制规则、扫描器配置作为初始 HS，明确状态检测器、反馈源（环境奖励 / 测试 / 日志 / 回放 / 人工反馈）、更新触发器。
- **设计 Codex/Claude 维护通道的 PoC**：选一个低风险的子模块（如 Web 漏洞规则集），让 coding agent 跑一次"读日志 → 改规则 → 跑回归 → 提交 PR"的完整闭环，复现翁家翌的工作流。
- **借用 System 1 / System 2 框架做产品分层**：浅层神经网络（流量分类、异常检测）做 System 1；HL 做规则、测试、回放、安全边界；LLM agent 做 System 2 的反馈与改进——把这个分层写进产品架构图。
- **把 HL 的 5 个工程性质（可解释、样本效率、旧能力固化、过拟合可控、抗灾难性遗忘）做成销售话术对照表**：每条对应客户痛点（合规、可审计、可回归、可监控），形成"为什么选我们而非纯 LLM 厂商"的论据。

### 值得进一步探索

- **HL 与漏洞知识图谱融合**：HL 的"显式变量、检测器、缓存"和"显式存储试验、摘要、失败原因"天然适合接入漏洞知识图谱——能否做一套"漏洞规则 HS + 漏洞知识图谱"的双向同步机制？
- **Codex 维护 HS 的总成本测算**：翁家翌承认"跑得快不等于总成本低"。如果团队真做内部 PoC，必须建立 token 消耗 + 工程师审查时长 + 失败迭代次数的完整成本模型，否则会被"看起来很美"误导。
- **HL 在多 Agent 协作场景下的表达力**：Montezuma 暴露的"长视野规划 + 失败恢复"问题，在多 Agent 协作（任务分解、错误恢复、上下文交接）里是常态——HL 是否需要新的"宏动作 + 可恢复状态机 + 长期记忆"程序范式？这与上一篇 CSA Agentic Red Teaming 12 类威胁里的"Goal/Instruction Manipulation""Multi-agent Exploitation"形成有趣的攻防对照。
