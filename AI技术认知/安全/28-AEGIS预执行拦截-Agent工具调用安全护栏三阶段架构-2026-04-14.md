## 核心命题

当前 AI Agent 框架的工具调用路径：

```
模型生成工具调用 → 框架解析参数 → 直接执行
```

中间**没有框架无关的控制点**。一旦模型决定删除文件、发送邮件、访问数据库，没有任何机制能在副作用发生前阻止它。事后审计记录不了已删的文件、已发的邮件。

> Agent 的安全边界不在模型训练阶段，也不在提示词工程层，**而在工具执行路径上**。

---

## 当前框架的结构性缺陷

各主流框架（LangChain、AutoGPT 等）把安全能力内置在各自实现里，导致三个问题：

1. 无法在不修改框架代码的情况下统一管控所有工具调用
2. 换框架/升级版本时安全策略要重新适配
3. 跨语言、跨框架的 Agent 应用无法共享同一套防护机制

---

## AEGIS 架构：框架无关的预执行拦截层

```
[14个Agent框架 SDK层]
        ↓ 拦截 tool_use
[AEGIS 网关 - 三阶段管道]
        ↓
  ┌─────────────────────────┐
  │ 阶段1: 深度字符串提取    │  → 遍历嵌套结构/Base64/序列化对象
  │ 阶段2: 内容优先风险扫描  │  → SQL注入/命令注入/路径遍历/数据泄露
  │ 阶段3: 可组合策略验证    │  → 允许 / 拦截 / 升级人工审批
  └─────────────────────────┘
        ↓
[合规驾驶舱 - 人工审核待定调用]
        ↓
[防篡改审计追踪 - Ed25519签名 + SHA-256哈希链]
```

支持 14 个框架（Python、JavaScript、Go），集成方式是在工具执行前拦截，不修改框架内部逻辑。

---

## 三阶段管道详解

### 阶段一：深度字符串提取

从工具参数中提取**所有**字符串内容：
- JSON 深层嵌套字段
- Base64 编码后的数据
- 序列化对象里的属性

**为什么重要**：攻击者故意将恶意内容藏在深层结构，只检查表层参数会直接穿透。

### 阶段二：内容优先的风险扫描

检测目标：SQL 注入、命令注入、路径遍历、敏感信息泄露

**内容优先 vs 规则优先**：
- 传统 WAF 维护规则库，新型攻击总能绕过已知规则
- AEGIS 理解内容的**语义风险**，而不是匹配固定特征

### 阶段三：可组合策略验证

策略示例（可任意组合）：
- "删除操作必须在白名单目录内"
- "涉及用户数据的操作必须人工审批"
- "单次调用金额超过 1000 元必须暂停"

**设计哲学**：前两阶段做**技术判断**（可自动化）；第三阶段做**业务判断**（必要时才触发人工介入）。

这就是 8.3ms 延迟的来源：大部分调用在前两阶段完成，只有高风险操作进入人工审批。

## 性能指标

| 指标 | 数值 |
|------|------|
| 中位延迟增加 | **8.3ms**（1000 次连续拦截） |
| 误报率 | **1.2%**（500 个良性工具调用） |
| 攻击拦截率 | **100%**（48 个精心设计的攻击实例） |

## 防篡改审计追踪

普通日志可以被修改、删除、伪造。安全事故发生后需要证明：
1. 调用确实发生过（不是事后伪造）
2. 调用参数和结果未被篡改
3. 审批流程（如果有）真实有效

**AEGIS 的方案**：Ed25519 签名 + SHA-256 哈希链

```
记录N = {决策内容, 时间戳, 上下文} + 私钥签名(记录N) + hash(记录N-1)
```

- 每条记录包含前一条记录的哈希值 → 无法删除中间某条记录而不被发现
- 高风险操作的人工审批决策也记录在链上 → 可追溯谁在什么时间批准了什么

## 部署前自检清单

已有 Agent 应用必须评估：

- [ ] 模型生成的工具调用是否会经过框架无关的控制点？
- [ ] 这个控制点能否在执行前拦截（而不只是事后记录）？
- [ ] 拦截策略是否可以跨框架、跨语言统一管理？

三个答案都是"否" → Agent 应用处于**裸奔状态**。
