# OWASP MCP 十大安全风险全景

## 背景定位

OWASP 发布 MCP Top 10，专门针对 Model Context Protocol 这一 Agent 调用外部工具的标准协议。

**MCP Top 10 vs Skill Top 10 的核心区别：**
- Skill Top 10：关注"装进 Agent 里的能力包本身有没有问题"（打包、分发、权限声明、加载、隔离）
- MCP Top 10：关注"模型通过协议连接外部工具和上下文时，调用链会不会失控"（token、身份、工具契约、上下文、执行链路、运行时审计）

---

## 风险详解

### 1. 令牌管理不当与密钥暴露

**问题根源：** 开发者把 secret 放入配置文件、环境变量、prompt 模板，或残留在上下文记忆和日志里。MCP 系统长会话+状态持久化，凭证可能在后续交互中被召回或通过提示注入套出。

**危害放大点：** 泄露的不是"只读字符串"，而是能直接调用仓库、流水线、云 API、文件系统的**执行令牌**。一旦泄露 = 整个连接环境被接管。

**防护方向：**
- 短期 token（按会话绑定，用后即失效）
- 最小权限（只给工具需要的最小 scope）
- 避免把 secret 存入上下文和日志
- 统一凭证生命周期治理

---

### 2. 权限范围蔓延导致提权

**问题根源：** "慢性风险"——临时权限因为方便、历史兼容或配置漂移，一点点扩大。MCP 同时连接代码仓、云 API、工单系统、CI/CD，小幅权限膨胀迅速变成高风险入口。

**Agent 场景放大效应：** 传统系统里权限大不一定马上出事，但过权的 agent 会**自动行动**——无需人工复核就能改代码、发起部署、读取敏感数据，甚至创建新凭证和身份资源。

**防护方向：** 最小权限原则 + 自动 scope 过期 + 严格访问评审

---

### 3. 工具投毒（MCP 特有高危风险）

**问题根源：** 不是普通恶意插件，而是 schema、manifest、tool descriptor 以及工具输出本身被篡改，让模型在"表面合法"的情况下做错事。典型案例：一个无害操作名实际映射到破坏性动作。

**为什么难防：** 日志里看到的仍是"合法调用"，agent 是按照 schema 行动的，但 schema 本身已被污染。

**防护方向：**
- 对 schema 和 manifest 做签名校验
- Registry 做成不可变 + 可追溯
- Schema 变更实施多人审批
- 把语义约束写成 policy（如"archive 不能映射为 DELETE"）

---

### 4. 软件供应链攻击与依赖篡改

**问题根源：** MCP 高度依赖第三方组件（SDK、连接器、协议服务、向量库客户端、插件）。这些组件运行在受信执行路径里，一旦依赖被污染，攻击者能悄悄改变 agent 行为、插入后门、修改协议语义。

**比传统供应链更严重：** 被污染的不只是一个程序模块，而是"**模型如何理解世界、如何操作外部系统**"的整个能力边界。

**防护方向：** 组件签名 + 来源校验 + SBOM/CBOM 可见性 + 版本固定 + 内部镜像仓库 + 第三方插件隔离运行

---

### 5. 命令注入与执行

**问题根源：** Agent 使用不可信输入构造并执行系统命令、shell 脚本、API 调用或代码片段。不可信输入来源：用户 prompt、检索上下文、第三方数据源、工具返回结果。

**比传统命令注入更隐蔽：** 中间多了一层模型解释——攻击者把诱导内容藏在文档或上下文里，让模型"主动"生成看起来语法正确、语义合理的执行命令。

**高危入口：** 工具封装了 shell/数据库/文件系统访问，且直接把 agent 输出交给解释器执行。

---

### 6. 意图流劫持（Intent Flow Hijacking）

**概念：** Intent Flow = agent 把用户高层目标翻译成工具调用和执行动作的关键路径。

**攻击方式：** 攻击者把恶意指令藏进被 agent 信任的上下文里（资源文档、schema 定义、工具输出），让 agent 在流程内部被带偏，从用户目标转向攻击者目标，表面上仍像在完成原任务。

**典型场景：**
- 让 agent 总结日志 → 日志里的隐藏指令诱导 agent 外传日志
- 让 agent 查仓库状态 → 仓库文本里的恶意内容触发额外动作

**防护方向：**
- 锚定用户原始目标（全程追踪 intent）
- 外部取回的自然语言上下文统一视为不可信
- 检测执行动作是否偏离原目标
- 高风险偏移时强制人工确认

---

### 7. 认证与授权不足

**典型表现：** API key 校验可选、共享静态密钥、配置文件硬编码凭证、token 无过期机制、仅依赖客户端声明身份。

**多 agent 场景放大：** 参与方是多个 agent + 多工具 + 多系统协同，身份绑定不严会导致"谁在以谁的名义做事"彻底失真。

**防护方向：** 强身份验证 + 短期 scoped 凭证 + 服务端强校验（不依赖前端/客户端自觉）

---

### 8. 审计与遥测缺失

**危险特性：静默性** —— 没有被监控的 agent 可能连续数周执行敏感操作或外传数据，无人知晓。

**需要记录的内容：** 工具调用、上下文变化、用户与 agent 交互，并形成**不可篡改的审计链**。

**核心思路：** MCP 安全不只是拦截能力，还包括"**出事后能不能把链路讲清楚**"。

---

### 9. 影子 MCP 服务器（Shadow MCP）

**类比 Shadow IT：** 未经批准、未经监管的 MCP 实例，被开发/研究/数据团队临时搭起来测试，常带着默认凭证、宽松配置或未加固 API，运行在正式治理体系之外。

**危害：** 直接暴露数据检索、工具执行甚至模型控制能力，却绕开了集中认证、监控和数据治理。

**企业应对第一步：** 不是"防攻击"，而是"**先把所有活跃的 MCP 实例找出来**"。

---

### 10. 上下文注入与过度共享

**Context 定义：** Agent 的工作记忆，包含 prompt、检索文档、中间结果和交互历史。

**两类风险：**
1. **上下文过度共享** → 一个用户/会话/agent 的敏感信息泄露给另一个（多租户 session bleed）
2. **上下文注入** → 恶意内容被写入共享记忆，持续影响后续任务

**MCP 放大因素：** 上下文是持久化的、可复用的，还会反过来影响模型行为 → 风险更快更隐蔽扩散。

**防护方向：** namespace 严格隔离 + 限制上下文生命周期 + 避免跨任务复用 + 进入上下文的数据做分类/脱敏/访问审计

---

## 风险分组视角

按攻击面归类：

**身份与权限类（1、2、7）**
- 核心是 token 管理、权限边界、身份绑定

**工具与协议完整性类（3、4）**
- 核心是 schema/manifest 可信度、依赖链完整性

**执行链路类（5、6）**
- 核心是输入净化、意图锚定、上下文信任边界

**运行时治理类（8、9、10）**
- 核心是可见性、审计、影子实例发现、上下文隔离

---

## 可复用决策规则

1. **任何 secret 都不应该进入 context 或 log**——包括临时凭证
2. **外部取回的自然语言内容统一视为不可信**——无论来自哪个受信来源
3. **schema 变更 = 代码变更**——需要同等级别的审查和签名
4. **先做 MCP 资产盘点**——找出所有活跃实例，再谈加固
5. **审计链是安全基线**——没有不可篡改审计链的 MCP 系统不应上生产
