## 核心命题

AI 智能体不同于传统软件——它们自主行动、非确定性、能适应上下文。这使得**传统 IAM 框架在以下场景下失效**：跨域操作、异步执行、递归委托子智能体、代表多名用户行动。

> 智能体身份管理的核心转变：**从模拟用户（Impersonation）→ 委托授权（Delegated Authority）**

---

## 智能体 IAM 的两层认证挑战

代表用户行动的智能体必须同时解决两个认证问题：

| 层次 | 问题 | 方案 |
|------|------|------|
| **智能体认证** | 智能体软件本身是否可信 | 工作负载身份（SPIFFE/SPIRE、客户端ID元数据） |
| **用户委托认证** | 用户授权意图是否被可验证地捕获 | OAuth 2.1 + PKCE + OBO 流程 |

---

## 当前框架的能力边界

**OAuth 2.1 + MCP 适用场景（有效）**：
- 单一信任域内的同步智能体
- 企业智能体访问内部多个工具（共享 IdP、统一授权策略）

**失效场景**：
- 跨域操作（企业内部 + 外部 API）
- 异步长时间运行任务（令牌超期）
- 递归委托（主智能体 → 子智能体）
- 代表多名用户（权限交集问题）
- 浏览器/Computer-Use 智能体（绕过 API 授权层）

---

## 关键概念与解决方案

### 1. 委托授权（OBO 流程）

**问题**：智能体直接使用用户凭证 = 模拟 = 问责黑洞

**解决**：On-Behalf-Of (OBO) 流程 — 访问令牌中包含**两个不同身份**：
```
JWT {
  sub: "user_alice"         # 授权的用户
  act: "agent_assistant_v2" # 执行操作的智能体
}
```
PEP 可据此生成区分"谁授权"与"谁执行"的审计日志。

### 2. 异步授权（CIBA）

**问题**：长时间运行的任务中，敏感操作需要用户批准，但用户不在线

**解决**：Client Initiated Backchannel Authentication (CIBA)
- 智能体无需阻塞工作流，通过带外机制请求用户授权
- 三种交付模式：轮询（批处理）/ 探测���节省流量）/ 推送（最快响应）
- 用户在受信任设备上收到通知，随时批准/拒绝

### 3. 递归委托与范围衰减（Scope Attenuation）

**问题**：主智能体 → 子智能体 → 孙智能体，权限如何逐级收缩？

**两种机制**：

| 机制 | 适用场景 | 原理 |
|------|---------|------|
| OAuth 2.0 令牌交换 | 集中管理/中心辐射型 | 在线向授权服务器请求权限缩减的新令牌，集中策略控制 |
| Biscuits / Macaroons | 去中心化动态网络 | 令牌持有者可离线创建更受限版本，权力嵌入凭证本身（OCap模型） |

**核心原则**：链末端的资源服务器必须能加密验证回溯到原始用户的整个委托路径。

### 4. 撤销 vs 取消配置

| 操作 | 含义 | 机制 |
|------|------|------|
| **撤销 (Revocation)** | 立即让活动令牌失效，终止当前会话 | 令牌过期 + OpenID 提供商命令 + 共享信号框架(SSF) |
| **取消配置 (De-provisioning)** | 永久移除智能体身份及所有相关权利 | SCIM DELETE → 广播到所有联合域 → 清除所有 ACL |

> 关键：单个受损智能体身份可通过递归委托引发**级联故障**，因此取消配置必须是不可逆、全系统传播的。

### 5. 智能体身份的四种架构模型

| 模型 | 适用场景 | 特点 |
|------|---------|------|
| **增强型服务账户** | 企业近期部署 | 扩展现有工作负载身份，token 中富含 agent_model/version 元数据 |
| **委托用户子身份** | 直接代表用户行事 | OBO 流程，身份不同但与用户权力绑定 |
| **联合信任与互操作** | 跨域无中央 IdP | OpenID Federation + X.509 证书跨信任域验证 |
| **主权可移植身份** | 去中心化生态 | DID + 自管理加密密钥，点对点断言身份 |

## 企业部署：SCIM 扩展与生命周期管理

- 提议的 **AgenticIdentity 资源类型**（SCIM 扩展草案）将智能体作为一等公民管理
- 智能体的配置/权限分配/退役使用与人类员工相同的自动化策略驱动工作流
- **SSO** 使用企业已有凭证访问智能体平台

---

## PEP/PDP 外部化架构

```
[智能体请求]
    ↓
[PEP - 策略执行点]         ← API 网关 / 服务网格 sidecar
    ↓ 查询
[PDP - 策略决策点]         ← 集中式授权服务（OpenID AuthZEN 工作组标准化中）
    ↓ 允许/拒绝
[资源服务器]
```

PEP 同时承担：解析委托凭证、区分用户与智能体、兼容遗留系统（将丰富智能体令牌转换为传统 API Key）。

---

## 可扩展同意：解决同意疲劳

单个用户可能有数十个智能体每天做出数千决策。三种新模式取代"每操作点一次批准"：

1. **策略即代码**：预定义运营范围（预算上限、数据访问层级），IAM 程序化执行
2. **基于意图的授权**：用户用自然语言批准高级意图，系统转化为最小权限许可集
3. **基于风险的动态授权**：常规操作自动批准，异常操作触发 CIBA 带外人类审批

---

## 六大失效场景（用例）

| 场景 | 核心挑战 | 解法方向 |
|------|---------|---------|
| 高速智能体 | 同意疲劳 | 预授权 + 策略即代码 |
| 异步长任务 | 令牌超期、持久委托 | CIBA + 委托身份作一等公民 |
| 跨域联合 | 无统一 IdP | OAuth 令牌交换 + 可验证凭证 |
| 递归委托网络 | 权限无法衰减 | Biscuits/Macaroons 离线衰减令牌 |
| 信息物理系统 | 操作不可逆 | 机器可读操作范围策略 + 强制升级人类审批 |
| 代表多名用户 | 权限交集问题 | ABAC 细粒度授权（当前无完美方案） |

## 经济层：智能体支付协议

| 协议 | 解决问题 | 核心机制 |
|------|---------|---------|
| **FAPI 2.0** | 高风险 API（银行转账等） | 发送者受限令牌（mTLS/DPoP）+ 严格同意记录 |
| **AP2（Google）** | 自主商业交易意图验证 | 签名授权书（Intent Mandate + Cart Mandate）构成不可否认审计链 |
| **KYAPay** | 智能体首次与新服务交互冷启动 | "了解你的智能体"(KYA) + JWT 绑定身份声明与支付授权 |

---

## 关键风险与未解难题

- **浏览器/Computer-Use 智能体**：直接操作 UI 绕过所有 API 授权控制（Web Bot Auth 协议方向）
- **撤销传播**：去中心化离线令牌无法即时撤销（次数限制凭证是当前缓解手段）
- **隐私 vs 问责**：可追溯性创建行为档案；零知识证明是方向但与现有标准集成困难
- **碎片化风险**：供应商专有智能体身份系统，互操作性待 IPSIE 等工作组推动标准化

---

## 最佳实践清单

- 使用 OAuth 2.1 认证，SCIM 做生命周期管理，避免自定义机制
- 严格实施最小权限原则（与委托用户权限对齐）
- 认证必须覆盖智能体对资源、智能体对智能体的所有交互
- 维护所有认证事件、授权决策、智能体操作的清晰审计线索
- 为互操作性设计（对齐 IPSIE / 未��� MCP 规范）
