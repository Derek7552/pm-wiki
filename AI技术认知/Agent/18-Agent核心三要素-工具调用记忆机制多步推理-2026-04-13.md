# AI Agent 核心三要素：工具调用、记忆机制、多步推理

> 来源：微信公众号「小林 coding」（美团面试经分享）
> 提炼日期：2026-04-13
> 分类：AI方法论

## TL;DR

AI Agent 与大模型的本质区别在于"自主性"和"能行动"。Agent 背后有三件核心支撑：① 工具调用（Tool Use）——让模型从"说话"变成"做事"，模型只决策"调什么"，代码真正执行；② 记忆机制——短期记忆（当前任务中间状态）+ 长期记忆（跨任务偏好记录）；③ 多步推理与自我纠错——感知失败→分析原因→换方式重试，而非直接崩掉。

## 核心观点

**1. 工具调用是 Agent 从"说话"变"做事"的关键**

Agent 能调用外部工具（搜索引擎、代码执行器、数据库、API），但核心认知是"决策与执行分离"：模型只是大脑，告诉代码"该调什么工具、参数填什么"，真正执行的是你的代码，结果再反馈给模型。模型始终不是手脚，而是决策中枢。工具定义文件（JSON Schema）本质是一份"技能说明书"，模型读了说明书自主决定调用策略。

**2. Agent 的运作闭环：感知 → 规划 → 行动 → 再感知**

传统 LLM 是"问答机器"——输入→输出→结束，不会主动做事，不知道上一步做了什么、下一步该做什么。Agent 则完全不同：接收高阶目标后，先拆解任务（搜索哪些关键词、访问哪些网站、组织内容方式），然后一步步执行，每步结果反馈指导下一步，直到完成目标。

**3. 三要素缺一不可，共同支撑 Agent 的自主性**

工具调用赋予执行能力，记忆机制保持任务连贯性（不做一半忘目标），多步推理+自我纠错保障复杂任务韧性。三者组合，才让 Agent 从"被动响应"进化为"主动完成目标"。

## 关键知识点

### 一、工具调用（Tool Use）

**核心原理：决策与执行分离**

```
tools = [
    {
        "name": "get_weather",
        "description": "获取指定城市的当前天气",
        "parameters": {
            "type": "object",
            "properties": {
                "city": {"type": "string", "description": "城市名称"}
            },
            "required": ["city"]
        }
    },
    {
        "name": "send_email",
        "description": "发送邮件给指定收件人",
        "parameters": {
            "type": "object",
            "properties": {
                "to": {"type": "string"},
                "subject": {"type": "string"},
                "body": {"type": "string"}
            },
            "required": ["to", "subject", "body"]
        }
    }
]
```

- 工具定义里**没有一行真正执行的逻辑**，只有"名字、描述、需要哪些参数"
- 模型读了说明书 → 决定该调哪个工具、参数填什么 → 以 JSON 格式返回决策
- 真正执行的是**你的代码**，不是模型

**实际执行流程示例**：
用户说"帮我查北京天气，发邮件给 boss@company.com"
- 第一步：`get_weather(city="北京")` → 返回 "晴天 15°C"
- 第二步：`send_email(to="boss@company.com", subject="今日天气", body="北京今天晴天 15°C")`
- 每一步都是真实发生的，不是"假装回答"

### 二、记忆机制

| 类型 | 作用 | 示例 |
|------|------|------|
| 短期记忆 | 当前任务的中间状态 | Agent 写报告时，记得已经搜索了哪些关键词、访问了哪些网站 |
| 长期记忆 | 跨任务的用户偏好、历史记录 | 用户偏好少糖、上次选择了哪家酒店，跨任务保持一致 |

**解决的问题**：传统 LLM 每次对话都是"失忆"的，除非手动传上下文。Agent 系统设计双层记忆，在执行复杂任务时保持连贯性，不会做到一半忘了目标是什么。

### 三、多步推理与自我纠错

Agent 在执行过程中如果某一步失败了：
1. **感知失败**——识别当前步骤没有达到预期
2. **分析原因**——判断是工具参数问题、API 问题还是逻辑问题
3. **换方式重试**——调整参数或切换备用工具，而不是直接崩溃

这让 Agent 像一个真正在"思考"的执行者，而不是只会背答案的系统。

### Agent vs LLM 核心对比

| 对比维度 | 普通 LLM（GPT chat 接口） | AI Agent |
|----------|--------------------------|----------|
| 本质角色 | 问答机器 | 自主完成目标的 AI 系统 |
| 输入输出 | 输入→输出→结束 | 感知→规划→行动→再感知 循环闭环 |
| 任务连贯性 | 每次独立，不知道上一步做了什么 | 记得中间状态，知道下一步该做什么 |
| 执行能力 | 只生成文字，给出建议 | 真的调 API、发邮件、执行操作 |
| 失败处理 | 直接输出错误答案或停止 | 感知失败→分析原因→重试/换方案 |
| 用户交互 | 你问一句它答一句 | 你给一个目标，它自主执行到底 |

## 数据与案例

**天气邮件场景对比**（直观看差距）：

| 执行者 | 行为 |
|--------|------|
| 普通 LLM | 告诉你"可以这样写代码来实现……"，然后结束 |
| AI Agent | 调天气 API → 拿数据 → 组织邮件内容 → 调邮件发送接口 → 全程自动完成 |

**核心区别**：从生成文字，到执行任务。

## 启发与思考

**与当前工作的关联**：
- 设计 Agent 功能时，工具定义（JSON Schema）是最核心的交付物之一——它决定了 Agent 能做什么、参数怎么传
- "决策与执行分离"原则是技术架构层面的关键：模型层负责推理决策，执行层负责真正调用，不要混在一起

**可落地的行动项**：
- 为自己的 AI 产品功能，设计一张"工具能力清单"：列出所有工具的名字、描述、必填参数、选填参数
- 在 Agent 执行链路中设计"失败重试"机制：明确每种工具调用失败时的备用方案

**值得进一步探索的方向**：
- 短期记忆的实现方案：用向量数据库 vs 用结构化状态存储，哪种更适合复杂任务？
- 多步推理的"反思深度"如何设置？是否需要限制重试次数防止无限循环？

## 原文精华

> 不是模型自己执行，而是模型「告诉你该调什么」，你的代码去真正执行，结果再反馈给模型。模型始终只是大脑，不是手脚。

> Agent 就完全不一样了。它有一个核心的运作闭环：感知 -> 规划 -> 行动 -> 再感知。你给它一个目标，它不是直接输出一段文字了事，而是先拆解任务，然后一步一步去执行，每一步的结果又反馈回来，指导下一步怎么做。

> 普通 LLM 能做的只是告诉你「你可以这样写代码……」；而一个 Agent，它会真的去调天气 API、拿到数据、组织邮件内容、再调邮件发送接口，整个过程自动完成。这就是本质区别：从生成文字，到执行任务。
