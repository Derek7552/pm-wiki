# Agent 记忆模块设计：四类记忆与三层架构

> 来源：小红书面试题解析（秀才的学习网站 golangstar.cn）
> 提炼日期：2026-04-03
> 分类：技术趋势

## 核心观点

1. **记忆模块决定 Agent 从"能用"到"好用"的跨越**。没有记忆的 Agent 记不住用户偏好、会反复犯同一个错误、每次对话都像第一次见面。面试考察的不是"记忆有哪些类型"，而是面对具体项目时的设计框架和工程取舍能力。

2. **四类记忆分类是架构设计的起点**。工作记忆（当前上下文）→ 情景记忆（具体事件）→ 语义记忆（抽象规则）→ 程序记忆（操作流程），四类之间存在明确的流转和提升关系，分类清楚了架构自然就出来了。

3. **好的记忆系统不只是存取，还要有反思能力**。定期让 LLM 从情景记忆中提炼语义记忆，是 Agent 真正从经验中"学到东西"的关键机制，直接来自斯坦福 Generative Agents 论文的验证。

## 关键知识点

### 四类记忆模型

| 类型 | 内容 | 特点 | 技术映射 |
|------|------|------|----------|
| **工作记忆** | 当前任务上下文、推理中间状态、工具返回结果 | 高频读写、生命周期短 | LLM Context Window |
| **情景记忆** | 过去经历的具体事件（带时间戳和场景） | 按相似场景匹配检索 | 向量数据库 |
| **语义记忆** | 从经验提炼的通用知识和规则 | 脱离具体事件的抽象认知 | 向量 + 关系型 DB |
| **程序记忆** | 学会的操作流程和技能 | 结构化 SOP/workflow | 结构化存储 |

**流转关系**：工作记忆 → 情景记忆 → 语义记忆 → 程序记忆（逐层沉淀提升）

### 记忆写入流水线

**反模式**：全量记录（每轮对话原封不动扔进数据库）→ 记忆库膨胀、检索噪音大

**正确做法**：感知 → 判断 → 提炼 → 存储
- 只有包含**新信息**的对话才触发写入
- 用 LLM 提炼成**结构化的记忆条目**
- 写入前和已有记忆做**冲突检查**，防止自相矛盾

### 三维检索评分模型（来自 Generative Agents）

```
score = α × recency + β × relevance + γ × importance
```

| 维度 | 含义 | 实现方式 |
|------|------|----------|
| **Recency（时近性）** | 越近的记忆分数越高 | 指数衰减函数 |
| **Relevance（相关性）** | 语义匹配度 | Embedding + 余弦相似度 |
| **Importance（重要性）** | 记忆固有重要程度 | LLM 写入时打分 + 引用频率动态调整 |

**权重调参场景**：客服场景 → recency 权重高；知识问答 → relevance 权重高

**工程优化**：
- 元数据预过滤：按用户 ID、记忆类型、时间范围先筛掉不相关条目
- 二阶段检索：向量粗召回 Top-50 → Cross-Encoder 精排 → Top-5 注入上下文

### 反思机制（Reflection）

- **触发条件**：近期情景记忆的重要性总分累计超过阈值
- **执行过程**：LLM 对最近 N 条记忆提问"从这些经历中能得出什么更高层次的结论？"
- **输出**：新的语义记忆（重要性评分较高）
- **价值**：50 条退货记忆 → 提炼成 1 条"退货主因是商品描述不符"的语义规则

**记忆整合操作**：
- **合并去重**：定期用 LLM 对高度相似条目做聚类和合并
- **主动遗忘**：衰减系数 + LLM 定期审视标记过时记忆

### 三层落地架构

| 层级 | 名称 | 存储引擎 | 职责 |
|------|------|----------|------|
| **L1** | 工作记忆层 | Context Window | 摘要 + Buffer（保留最近 3-5 轮原始对话 + 更早历史摘要压缩） |
| **L2** | 近期记忆层 | Redis（sorted set） | 当前会话完整历史 + 近 7 天高频记忆，快速范围查询 |
| **L3** | 长期记忆层 | 向量 DB + PostgreSQL + Neo4j（可选） | 持久化存储 + 三维评分检索 + 反思机制 |

**数据流**：双向流动——重要信息 L1→L2→L3 逐层沉淀；新任务时 L3→L1 按需加载

## 数据与案例

- **客服 Agent 反思案例**：过去一周处理 50 个退货请求，其中 30 个是"商品描述和实物不符"。有反思机制的 Agent 早就提炼出"优先询问描述不符"的规则，无需逐条检索
- **记忆可信度分级**：LLM 自动提炼 → "低置信"；用户确认 → "高置信"；检索时高置信记忆获得更高权重

## 启发与思考

1. **Claude Code 本身就是活案例**。它的 auto memory 系统正是这套理论的简化实践——user/feedback/project/reference 四类记忆对应了语义记忆的细分，MEMORY.md 索引 + 独立文件是 L3 的文件系统版实现，"写入前检查是否已有可更新的记忆"就是冲突检测

2. **三维评分可直接复用到 RAG 系统**。很多 RAG 只做了 relevance 维度，加上 recency 和 importance 后检索质量会有质的提升，尤其适合对话历史长、知识库大的场景

3. **面试回答框架**：需求分类（四类记忆）→ 写入流水线 → 检索评分模型 → 反思机制 → 三层架构 → 工程取舍，这个结构本身就是一个通用的系统设计回答模板

4. **记忆粒度的两级存储策略值得借鉴**：一级摘要条目用于日常检索，二级原始记录用于细节追溯。避免了全量存储的噪音和过度压缩的信息丢失

## 原文精华

> 大多数 Agent 的 Demo 都不需要记忆——跑一段 ReAct 循环，调几个工具，输出结果，完事。但一旦进入真实业务场景，你会发现没有记忆的 Agent 几乎不可用：它记不住用户五分钟前说过的偏好，会反复犯同一个错误，每次对话都像第一次见面。

> 设计记忆模块的第一步，不是急着选向量数据库或者翻 LangChain 文档，而是先想清楚：这个 Agent 到底需要记住什么？

> 纯粹的事件记录（情景记忆）会让记忆库越来越庞大，但 Agent 的认知水平却没有真正提升。就像一个人经历了一百次客户投诉但从来不总结规律，他处理第一百零一次投诉时不会比第一次强多少。

> LLM 提炼的记忆条目不一定准确——它可能误解了用户意图，也可能在摘要过程中丢失了关键限定条件。在高风险场景（比如医疗、金融），需要对 LLM 生成的记忆条目做额外的校验，或者允许用户确认和修正。
