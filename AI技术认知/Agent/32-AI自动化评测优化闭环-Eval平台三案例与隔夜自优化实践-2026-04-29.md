# AI自动化评测优化闭环：Eval平台三案例与隔夜自优化实践

> 来源：阿里技术公众号（个人实践分享）
> 日期：2026-04-29
> 领域：AI方法论 / 评测工程 / Harness Engineering

## 一句话总结

搭建一个**只允许AI操作的评测平台**，让Agent自主生成评测集→执行评测→提交报告→基于报告优化代码→再评测，形成"评测-优化"闭环，实现隔夜自动优化三轮、分数从90.7→97.4→99.1的稳步上升。

## 核心知识点

### 1. AI First评测平台设计理念

**关键洞察：从入口层面杜绝人干苦力活。**

传统评测流程的痛点：
- 人收集评测集苦且累
- 评测执行烧时间
- 评测同学意愿不强

AI First解法：平台**只暴露API/Skill接口给AI Agent操作**，人无法直接操作UI，强制所有评测工作由AI完成。

平台三个核心实体：
| 实体 | 说明 | 关系 |
|------|------|------|
| **评测任务** | 明确评测目标+验收标准 | 1个任务 → N个评测集 |
| **评测集** | 具体评测步骤+预期结果 | 绑定到任务 |
| **评测报告** | 基于评测集的结果+打分 | 1个任务 → N个报告 |

### 2. 评测集的两种类型

- **标准型（Standard）**：有明确的成功/失败状态，适用于功能性测试
- **Rubrics型（等级评分）**：无法简单判断对错时，生成不同等级的评测用例

Rubrics关键价值：把"查到了/没查到"的二元判断升级为多维度等级评估。例如OKR查询场景，不止是"查没查出来"，而是从数据准确性、格式规范、响应速度等多维度分级评分。

### 3. 三个递进案例

#### Case 1：基础API/MCP自动评测
- 工具：QoderWork + 评测平台
- 对象：钉钉文档MCP
- 流程：Agent读取Skill说明 → 自动生成13个测试用例 → 逐个执行 → 提交报告（95分）
- 亮点：用例之间有**连贯性和衔接**（创建文档→查询→修改→删除的完整链路）

#### Case 2：UI + 内容质量评测
- 对象：绘报（汇报文稿生成工具）的5个PPT项目
- 关键突破：不仅测UI功能，还**评测AI生成内容的质量**
- Agent连接浏览器，共享登录态，打开系统逐个评测
- 耗时约20分钟，报告85分，覆盖功能+质量双维度

#### Case 3：隔夜自动优化闭环（核心案例）
- 工具：Cursor + 评测平台
- 流程：发任务 → 生成评测集 → 等待系统AI跑完 → 评测 → **基于报告自动修改代码** → 再评测，至少三轮
- 结果：v1=90.7 → v2=97.4 → v3=99.1，五个维度稳步提升
- 耗时：每轮约1小时，三轮共3-4小时，人直接睡觉

### 4. 闭环自优化的核心机制

```
┌────────────────────────────────────────────┐
│            评测-优化闭环                     │
│                                            │
│  Agent生成评测集 → Agent执行评测 → 评测报告  │
│       ↑                              ↓     │
│  Agent再次评测  ← Agent修改代码 ← 读取报告  │
│       ↓                                    │
│  版本递增(v1→v2→v3)，分数稳步上升            │
└────────────────────────────────────────────┘
```

关键：Coding Agent（Cursor/CC/Codex）读评测报告 → 理解失分项 → 针对性修改代码 → 重新评测验证，形成**自动化的红绿灯循环**。

### 5. 两个先决条件（实践教训）

**条件一：系统UI规范和基础设施要达标**
- AI在UI里"迷路"=用户也会迷路，这本身是质量问题的信号
- 不规范的UI导致自动化测试断链

**条件二：系统本身的AI Coding含量要高**
- 手工系统"约定大于配置"太多，AI难以跑通和优化
- 跑得好的案例都是AI Coding含量高的系统
- 老系统没有本地开发环境、到处断头路，无法支撑自动化

## 可复用的方法论

### 评测平台的最小可行设计
1. 工作空间隔离
2. Skill说明文件（让Agent知道如何操作平台）
3. 任务-评测集-报告三层数据模型
4. API接口（非UI）作为唯一操作入口

### "隔夜优化"的启动模板
给Coding Agent的指令结构：
1. 读取评测平台Skill说明
2. 对指定功能创建评测任务
3. 生成评测集并执行评测
4. 基于评测报告自动优化代码
5. 优化后重新评测，至少N轮

### 推广方向
- Skill包优化：生成评测用例 → 判断触发准确性 → 自动优化Skill
- AI功能质量提升：内容生成质量的Rubrics评测 + 自动调优
- MCP工具验证：全功能覆盖测试 + 回归测试

## 个人思考

1. **"只允许AI操作"是个精妙的设计约束**——不是做不到让人操作，而是故意不让人操作，倒逼全流程AI化，这比"也支持手动"要彻底得多
2. **评测集质量决定优化天花板**——AI生成的评测集如果维度不够或覆盖不全，优化也只能在已覆盖的维度上提升，盲区依然是盲区
3. **"AI Coding含量高"作为先决条件很关键**——这暗示了一个趋势：未来系统从一开始就应该为AI可维护性而设计（AI-native系统），而不是事后补救
4. **分数稳步上升（90.7→97.4→99.1）说明递减效应明显**——三轮就接近天花板，后续轮次的边际收益会急剧下降，说明这个方法适合快速提升到90+而非精细打磨到100
