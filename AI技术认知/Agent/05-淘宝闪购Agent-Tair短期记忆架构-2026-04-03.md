# AI Agent 短期记忆架构：淘宝闪购 Tair 实践

> 来源：Tair 短期记忆架构实践：淘宝闪购 AI Agent 的秒级响应记忆系统
> 提炼日期：2026-04-03
> 分类：技术趋势

## 核心观点

1. **AI Agent 的记忆系统对延迟极度敏感，是系统能否稳定扩展的分水岭**：根据 Little 定律（并发数 ≈ QPS × 延迟），记忆访问延迟从 5ms 上升到 50ms，在途请求数膨胀 10 倍，可能引发排队、超时乃至雪崩。Agent 每轮对话涉及多次记忆读写，延迟会被反复叠加放大。

2. **Agent 短期记忆需要按"模型消费"和"业务消费"两条线分别建模**：对话历史（供 LLM 推理）用 List 有序追加，业务上下文（供工具层查询）用 Hash 按领域拆分为独立字段。两类记忆的读写模式、消费者、更新频率完全不同，用同一数据结构会带来竞争问题。

3. **Agent 场景的流量特征是"读多写少 + 不可预测的尖峰"**：读写比 5:1 到 10:1，春节活动期间承受超 10 倍预估峰值。这要求存储层具备读写分离、无感扩缩容和弹性突发带宽能力。

## 关键知识点

### Agent 记忆双层数据模型

| 记忆类型 | 数据结构 | Key 设计 | 核心操作 | 消费者 |
|----------|----------|----------|----------|--------|
| **模型记忆**（对话历史） | List | `memory:model:{sessionId}` | RPUSH 追加、LRANGE 范围读取 | LLM 推理上下文 |
| **业务上下文** | Hash（6个field） | `memory:context:{sessionId}` | HSET/HGET 字段级读写 | 工具层、意图处理器 |

**业务上下文 Hash 的 6 个子模块**：
- `session`：会话元信息（用户ID、渠道、阶段）
- `search`：搜索状态（query、结果、推荐列表）
- `order`：订单状态（购物车、SKU、数量）
- `conversation`：对话状态（当前意图、意图切换标记）
- `coupon`：优惠信息
- `bizState`：业务状态（地址、配送、支付）

**选 Hash 而非 JSON String 的关键原因**：field 级独立读写，各模块互不干扰——搜索更新推荐结果时不影响订单模块写入购物车。

### 并发控制方案：会话级分布式锁

```
SET lock:memory:{sessionId} {requestId} NX EX 3
```

- 锁粒度：会话级（非全局），不同用户无锁竞争
- 锁超时：3秒，防止异常阻塞
- 释放：Lua 脚本确保只释放自己持有的锁
- 场景：用户快速连续发消息、流式响应中用户再次输入

### 弹性扩展策略

| 维度 | 日常 | 峰值（10倍） | 机制 |
|------|------|-------------|------|
| **计算** | 8分片 × 1只读副本 | 16分片 × 3只读副本 | 无感扩缩容（Slot原子迁移） |
| **带宽** | 固定带宽 | 单节点弹性突发至288MB/s | 分片粒度独立生效，按量计费 |
| **内存** | TTL 30分钟自动清理 | 高峰扩、低谷缩 | 全自动资源管理 |

### Little 定律在 Agent 场景的应用

> 并发数 ≈ QPS × 延迟

- 延迟 5ms → 50ms：在途请求膨胀 10 倍
- Agent 每轮对话触发数十次存储操作
- 延迟叠加放大效应：单次 IO 延迟增加 → 单轮对话延迟增加 → 会话占用时间增加 → 并发会话数膨胀 → 系统资源耗尽

## 数据与案例

- **性能目标**：点单耗时从传统 3-5 分钟压缩至 **30 秒以内**
- **P99 延迟**：春节流量洪峰期间始终控制在**毫秒级**
- **流量峰值**：千问春节红包活动，超过 **10 倍预估峰值**
- **读写比**：Agent 对话链路中读写比 **5:1 到 10:1**
- **Tair 性能**：多线程内核，读写性能达同规格 Redis 开源版的 **3 倍**
- **TTL 设置**：会话过期时间 **30 分钟**
- **每次对话**触发**数十次** Tair 操作（读历史、更新状态、锁操作等）
- **写入优化**：对话记录写入前将富媒体内容转为自然语言格式，减少 Token 消耗

## 启发与思考

- **Agent 记忆是被严重低估的基础设施问题**：大多数 Agent 框架关注的是 Prompt 编排和工具调用，但在生产环境中，记忆层的延迟和一致性才是决定用户体验的关键瓶颈。这篇文章是少见的从存储视角审视 Agent 架构的实战案例。
- **"模型记忆"和"业务记忆"的分离设计值得借鉴**：LLM 消费的是自然语言格式的对话历史（关注语义连贯），业务逻辑消费的是结构化状态（关注字段精确）。两者的读写模式、数据格式、更新频率完全不同，用同一数据结构存储会带来不必要的耦合。
- **与之前淘宝语音点餐文章形成完整拼图**：产品设计层面的 GUI→VUI 转变 + 技术架构层面的记忆系统设计，完整展现了一个 AI Agent 产品从交互到基础设施的全貌。
- **记忆从"对话级"到"用户级"的演进方向**：当前是短期记忆（会话内），下一步是长期记忆（用户偏好、历史行为）。这对所有 AI Agent 产品都是值得关注的演进路线。

## 原文精华

> 5ms 和 50ms 的差别，不是体验上的优化，而是系统能否稳定扩展的分水岭。

> 使用 Hash 的 field 级读写能力，各业务模块可以独立更新，互不干扰，避免了读取完整 JSON 再回写的竞争问题。

> AI Agent 的记忆系统，正在从"对话级"走向"用户级"，让 Agent 不仅记住"这次对话说了什么"，更记住"这个用户是谁、喜欢什么"。

> 对话记录在写入前，会将原始数据（包含文本和卡片等富媒体内容）转换为模型更易理解的自然语言格式，减少 Token 消耗。
