# 拷问模式替代Plan Mode：AI编程中的颗粒度对齐方法论

> 来源：Matt Pocock（TypeScript大神、AI Hero创始人）的Claude Code Skill实践分享
> 提炼日期：2026-04-07
> 分类：AI方法论

## 核心观点

1. **Plan Mode的根本缺陷是"伪对齐"**：默认Plan Mode让AI太急于生成计划文档，但它既不主动提问关键假设，也不深挖需求细节。产出的长文档看似完备，实则双方根本没有对齐颗粒度。糟糕的计划 = 糟糕的输出（Bad plans = Bad outputs）。

2. **规划的本质不是产出文档，而是建立深度共识**：真正有效的规划应该反转主动权——不是AI生成计划让你审核，而是AI疯狂提问逼你把模糊想法想清楚、把隐藏假设暴露出来。前期多花时间"被拷问"，后期编码和调试时间大幅减少。

3. **三步Skill链实现从模糊意图到可执行任务的完整闭环**：/grill-me（拷问对齐）→ /write-a-prd（生成PRD）→ /prd-to-issues（拆解任务），完成后AI已有足够清晰的上下文，可以AFK自主推进编码。

## 关键知识点

### Plan Mode的三个问题

| 问题 | 表现 | 后果 |
|------|------|------|
| 太急于生成 | AI还没理解意图就吐出长文档 | 计划与真实需求脱节 |
| 很少主动提问 | 不问关键假设、不挖需求细节 | 隐藏的复杂度在编码阶段才暴露 |
| 计划变废话 | 又臭又长，扫一眼就跳过 | 双方没有真正对齐，返工不可避免 |

### Matt Pocock的三步Skill链

```
/grill-me（拷问对齐）
  ↓ AI连续问20-30个问题，遍历设计决策树所有分支
  ↓ 逼出模糊想法和隐藏假设
/write-a-prd（生成PRD）
  ↓ 基于深度对话生成结构化需求文档
  ↓ 此时AI已充分理解需求，PRD质量远超Plan Mode
/prd-to-issues（拆解任务）
  ↓ 将PRD转化为GitHub Issues
  ↓ 具体、可执行、可跟踪
→ 进入编码阶段（AI可AFK自主推进）
```

### /grill-me 的设计精髓

- **极简指令**：只有三句话，却威力巨大——指令越短，AI的自由度越高，提问越深入
- **核心机制**：让AI "relentlessly"（无情地）采访你关于计划的每一个方面
- **遍历决策树**：走遍设计决策树的所有分支，一步步解决依赖关系
- **终止条件**：直到双方对齐颗粒度才结束

### 安装方式

```bash
npx skills@latest add mattpocock/skills/grill-me
```

## 数据与案例

- Matt Pocock实践反馈：前期被拷问20-30个问题（约15-20分钟），后期编码和调试时间大幅减少
- /grill-me Skill仅三句话指令，证明了"Skill不需要长"的设计原则——与"Skills详解"中渐进式披露的理念一致
- 开源可复用：通过npx直接安装，社区化分享

## 启发与思考

1. **与"提问能力>回答能力"的呼应**：吴军说AI时代核心竞争力是"提问"，Matt Pocock的方法是让AI反过来拷问你——本质上是用AI的提问能力来补人类思考的盲区。二者互补：人要学会提问，也要学会被AI拷问。

2. **与PRD已死原型万岁的连接**：Anthropic PM说PRD已死，但Matt Pocock的流程是先拷问再生成PRD。差异在于：此处的PRD不是人写的长文档，而是深度对话后AI自动生成的精炼共识记录。PRD没死，死的是人手写的PRD。

3. **对PM工作的直接启发**：PM做需求分析时，可以先让AI拷问自己的需求描述，暴露隐藏假设和模糊地带，再生成PRD。这比直接写PRD更高效，因为很多需求细节在写文档时想不到，但被反复追问时能想到。

4. **Skill设计的极简原则**：三句话的Skill比长篇大论的Prompt更有效，因为它给AI足够的发挥空间。这验证了"Skill ≠ 长指令"的设计理念。

## 原文精华

> 糟糕的计划 = 糟糕的输出（Bad plans = Bad outputs）。

> 规划的本质不是产出一个文档，而是和 AI 建立深度共识。只有双方对齐颗粒度，后面的代码生成才会又快又准。

> 前期多花时间"被拷问"，后期编码和调试的时间大幅减少。
