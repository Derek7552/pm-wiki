# Agent 工程实践全景：控制流、上下文、工具、记忆、评测、安全十二维落地指南

> 来源：阿里技术公众号 - 你不知道的 Agent：原理、架构与工程实践
> 提炼日期：2026-04-28
> 分类：技术趋势

## 核心观点

1. **Harness 和验证测试质量对 Agent 成功率的影响，远大于换更贵的模型**。任务清晰度 x 验证自动化程度构成四象限：右上角（目标明确+自动验证）是 Agent 最佳发挥区，Harness 的目标就是把任务推向右上角。

2. **调试 Agent 时应优先检查工具定义，而不是先怀疑模型能力**。多数工具选择错误出在描述不准确——没说明"何时该用/何时不该用"，而不是模型不够聪明。

3. **评测系统本身的问题，比 Agent 出问题更难发现**。评测环境资源不足、评分器 bug、测试用例与生产脱节，这些在表现上和模型退化一模一样，但基于失真信号去改 Agent 会越改越坏。

## 关键知识点

### 一、Agent Loop 核心稳定性

Agent Loop 核心就是感知→决策→行动→反馈的循环，抽象后不到 20 行代码。关键工程判断：

- **新能力只通过三种方式接入**：扩展工具集、调整系统提示结构、状态外化到文件/数据库
- **不要让循环体变成巨大状态机**：模型负责推理，外部系统负责状态和边界
- 从最小实现到支持子 Agent、上下文压缩、Skills 加载，主循环基本不变

### 二、Workflow vs Agent 精确区分

| 维度 | Workflow | Agent |
|------|----------|-------|
| 控制权 | 代码预定义，同输入必走同路径 | LLM 动态决策 |
| 状态 | 显式状态机，节点跳转清晰 | 隐式上下文，状态在对话历史中累积 |
| 维护 | 改流程需修改代码重新部署 | 调整系统提示即可 |
| 可观测性 | 日志定位节点，延迟可预估 | 需完整执行记录，轮数不固定 |

**核心区别**：执行路径由代码写死 = Workflow，由 LLM 动态决定 = Agent。现实中很多标着 Agent 的产品深入看更接近 Workflow，两者无高下之分。

### 三、五种控制模式

大多数 AI 系统都是这五种模式的组合：

1. **提示链 Prompt Chaining**：顺序步骤，中间可加代码检查点（线性流程）
2. **路由 Routing**：输入分类后走对应专用流程（简单/复杂分流）
3. **并行 Parallelization**：分段法（拆子任务并发）+ 投票法（同任务多次取共识）
4. **编排器-工作者 Orchestrator-Workers**：中央 LLM 动态分解，委派给工作者
5. **评估器-优化器 Evaluator-Optimizer**：生成→评估→反馈循环，直到达标

### 四、上下文工程——防 Context Rot

**Context Rot**：无关内容占上下文大头后，Agent 决策质量明显下滑。很多看起来像模型能力不足的问题，追溯后是上下文组织不当。

**分层管理策略**：

| 层 | 内容 | 策略 |
|----|------|------|
| 常驻层 | 身份、约定、禁止项 | 短、硬、可执行 |
| 按需加载 | Skills/领域知识 | 描述符常驻，完整内容触发时注入 |
| 运行时注入 | 时间、渠道ID、用户偏好 | 每轮按需拼入 |
| 记忆层 | 跨会话经验 | 写入 MEMORY.md，需要时读取 |
| 系统层 | Hooks/代码规则 | 完全不进上下文 |

**压缩时的保留优先级**（这条非常实用）：
1. 架构决策——不得摘要
2. 已修改文件和关键变更
3. 验证状态 pass/fail
4. 未解决的 TODO 和回滚笔记
5. 工具输出——可删，只保留结论

**压缩的关键陷阱**：不要改动标识符（UUID、hash、IP、端口、URL、文件名），改错一位后续工具调用直接失效。

### 五、Prompt Caching 的反直觉结论

- **稳定的大系统提示，比频繁变动的小提示实际成本更低**——写入成本只付一次，后续读取折扣可达 90%
- 常驻层越稳定，前缀命中率越高
- Skills 延迟加载不破坏系统提示前缀，追加在稳定前缀之后
- MCP 工具集频繁变动会导致缓存命中不断失效

### 六、Skills 按需加载的工程细节

**Skill 描述符的关键**："何时该用我"比"我能做什么"重要得多。

```
# 低效（约 45 tokens）
description: |
  This skill handles the complete deployment process...
# 高效（约 9 tokens）  
description: Use when deploying to production or rolling back.
```

**反例的量化影响**：
- 无反例：准确率从基准 73% 掉到 53%
- 加反例：升到 85%，响应时间还降 18.1%

**三个典型反模式**：
1. 几百行工作手册全塞 Skill 正文，不拆 supporting files
2. 一个 Skill 试图覆盖 review/deploy/debug/incident 五件事
3. 有副作用的 Skill 没有显式限制调用时机

**Skills vs MCP**：MCP 把完整结果返回给模型，更容易吃掉上下文预算；CLI + 单句描述的 Skill 在数据读取任务里更简洁。MCP 适合需要维护状态的任务（如 Playwright）。

### 七、ACI 工具设计三代演进

| 代际 | 思路 | 问题 |
|------|------|------|
| 第一代 API 封装 | 每个 Endpoint = 一个工具 | 粒度过细，Agent 需协调多工具 |
| 第二代 ACI | 工具对应 Agent 目标而非 API 操作 | 面向目标，一次把动作说完整 |
| 第三代 Advanced Tool Use | 动态发现 + 代码编排 + 示例驱动 | 进一步优化发现、调用和描述 |

**第三代的三个方向**：
- **Tool Search**（动态发现）：Agent 按需发现工具定义，上下文保留率 95%，准确率 49%→74%
- **Programmatic Tool Calling**（代码编排）：中间结果不经过 LLM，token 消耗从 ~150,000 降到 ~2,000
- **Tool Use Examples**（示例驱动）：附 1-5 个真实调用示例，准确率 72%→90%

**ACI 好工具 vs 差工具**：

| 维度 | 好工具 | 差工具 |
|------|--------|--------|
| 粒度 | 对应 Agent 目标 | 对应 API 操作 |
| 返回 | 与下一步决策直接相关 | 完整原始数据 |
| 错误 | 结构化 + 修正建议 | 通用字符串 "Error" |
| 描述 | 何时用、何时不用 | 只写功能说明 |

### 八、记忆系统四类分层

按 Agent 要解决的问题分（不是按存储介质）：

| 类型 | 存储 | 生命周期 |
|------|------|----------|
| 工作记忆 | 上下文窗口 messages[] | 会话级，主动管理 |
| 程序性记忆 | Skills 文件 | 持久化，按需加载 |
| 情景记忆 | JSONL 会话历史 | 磁盘持久化，支持检索 |
| 语义记忆 | MEMORY.md | Agent 主动写入，启动时注入 |

**记忆整合的关键设计**：50% token 阈值触发整合，系统只移动指针不删除原始消息，整合失败时原始消息写入 archive/ 保留完整历史。**流程必须可回退**。

**ChatGPT 四层记忆**（比预期更简洁，没用向量数据库也没用 RAG）：Session Metadata（会话级）→ User Memory（约 33 条，每次注入）→ Conversation Summary（约 15 个摘要）→ Current Session（滑动窗口）。

### 九、Agent 自主度三个基础设施

**1. 跨 session 续跑**：Initializer Agent 只跑一次，生成 feature-list.json + init.sh + claude-progress.txt；后续 Coding Agent 从文件系统恢复状态，每次实现一个功能、跑测试、更新 passes、提交代码后退出。进度放文件里不放上下文里，功能清单用 JSON 不用 Markdown。

**2. 显式任务状态**：同一时间只能有一个 in_progress，每完成一步先更新状态再继续。连续多轮未更新时自动注入 reminder。

**3. 后台 I/O**：慢速 subprocess 放后台线程，通过通知队列在下一轮 LLM 调用前注入结果，比把整个 loop 改成复杂 async runtime 更稳。

### 十、多 Agent 组织——先隔离再并行

**两种工作模式**：
- **指挥者模式**（同步协作）：人每轮调整决策，session 结束 context 就没了
- **统筹者模式**（异步委派）：人在起点终点出现，中间产出变成分支/PR 等可持久化工件

**建设顺序**：协议先定 → 隔离先做 → 再谈协作和并行

**多 Agent 幻觉放大**：Agent A 带偏 → B 强化 → C 叠加 → 所有 Agent 收敛到同一个高置信度的错误结论。交叉验证能打断这条链。

**子 Agent 两个基本限制**：
1. 深度限制（防无限递归生成孙 Agent）
2. 最小系统提示（只给 Tooling/Workspace/Runtime，不带 Skills 和 Memory）

### 十一、评测体系——最被低估的工程模块

**Pass@k vs Pass^k 不能混用**：
- **Pass@k**（k 次至少一次正确）：验证能力边界，开发阶段用
- **Pass^k**（k 次全部正确）：保证上线质量，每次变更跑回归

**三类评分器选择顺序**：
1. **代码评分器**（确定性最高）：有明确正确答案就优先用
2. **模型评分器**：语义质量、推理过程
3. **人工评分器**：建立基准、校准自动 judge

**transcript vs outcome 必须都覆盖**：Agent 说"订票已完成"是 transcript，数据库真的生成订单才是 outcome。只看 transcript 会漏掉"说了但没做到"，只看 outcome 看不出中间步骤走歪。

**从零搭评测的关键判断**：如果两个领域专家拿同一案例独立判断结论不一致，这个案例的验收标准就还没写清楚——先解决定义，再收集数据。

**先修评测再改 Agent**：评测分数下降时，先查环境（资源不足/评分器 bug/测试用例脱节），再动 Agent。基于失真信号调 Agent 会越改越坏。

### 十二、可观测性两层架构

- **第一层**：人工抽样标注（错误案例/长对话/负反馈），摸清失败模式
- **第二层**：LLM 自动评估，以第一层标注结果校准，做全量覆盖

**在线评测采样策略**（10%-20% Trace）：
- 负反馈：100% 进队列
- 高 token 消耗（Agent 绕圈子）：优先审查
- 模型/Prompt 变更后：头 48 小时全量审查

**事件流做底座**：tool_start/tool_end/turn_end 三个节点 emit 事件，一次发布多路消费（日志/UI/评测/审查），主循环不需要为下游改代码。

### 十三、安全——先于功能的三道关

1. **白名单授权**：只有授权用户可触发 Agent
2. **工作空间隔离**：shell 工具强制路径检查，用 `execFile` 而非 `exec` 避免注入
3. **操作审计日志**：每次执行记一笔 JSONL

**Prompt Injection 防御**（source-sink 模型）：
- 最小权限：没有 sink，source 侧注入无法落地
- 敏感操作显式确认：不能静默执行
- 标注外部内容边界：`<untrusted_content>` 标签隔离
- 关键路径加独立 LLM 验证

### 十四、OpenClaw 落地五层架构

| 层 | 实现 | 关键设计决策 |
|----|------|-------------|
| Gateway | WebSocket:18789 | Channel 和 Agent 统一走 Gateway |
| Channel 适配器 | 23+ 渠道 | 新增渠道不修改 Agent 代码 |
| Pi Agent | 工具调用支持流式 | Agent 核心循环和渠道完全解耦 |
| 工具集 | shell/fs/web/browser/MCP | ACI 原则，面向目标 |
| 上下文+记忆 | Skills 延迟加载 + MEMORY.md | 50% token 阈值自动整合 |

**工程实现顺序**（这个顺序本身就是知识）：
1. 单渠道先跑通（不要第一版就抽象多渠道）
2. 安全边界先于功能
3. 记忆整合要早做（不加整合，第 20 轮对话就垮）
4. Skills 先于新工具
5. 第一个失败就建评测

### 十五、常见反模式速查表

| 反模式 | 问题 | 修法 |
|--------|------|------|
| 系统提示当知识库 | 关键规则被忽略 | 约定留系统提示，知识移 Skills |
| 工具数量失控 | 频繁选错工具 | 合并重叠，明确命名空间 |
| 缺少验证 | 说完成但没法验证 | 每类任务绑可执行验收标准 |
| 多 Agent 无边界 | 状态漂移，归因困难 | 角色权限 + worktree 隔离 |
| 记忆不整合 | 第 20 轮后质量下降 | 超阈值自动触发整合 |
| 没有评测 | 不知是否引入回归 | 每个真实失败案例转测试用例 |
| 过早多 Agent | 协调开销超并行收益 | 先验证单 Agent 上限 |
| 约束靠期望 | Agent 选择性遵守 | 编码进 Linter/Hook/工具验证 |

## 个人思考

这篇文章的核心判断——Harness > 模型、先修评测再改 Agent、调试先查工具定义——都指向同一个方向：**Agent 的稳定性问题，本质上是工程问题而不是模型问题**。这和我之前整理的"Prompt ⊂ Context ⊂ Harness"包含关系一脉相承，但本文给出了更具操作性的落地路径：工程实现顺序（单渠道→安全→记忆→Skills→评测）和压缩保留优先级，这两个清单在实际搭建 Agent 系统时可以直接拿来用。

评测部分的"先修评测再改 Agent"这个判断最反直觉也最有价值——当你看到 Agent 表现下降时，本能反应是去改 Agent，但正确做法是先验证评测本身有没有出问题。这和软件工程里"先检查测试用例再检查代码"的原则完全一致。
