# Agent 记忆基建化：OpenChronicle 本地优先记忆层与 Proactive Agent

> 一句话总结：**记忆正在从"某个 AI 产品的差异化功能"演变为"Agent 时代的基础设施"——OpenChronicle 用 AX Tree 优先 + 截图兜底的混合方案采集屏幕上下文，按 7 类扁平存储为 Markdown，通过 MCP 协议对外暴露，实现本地优先、模型无关的记忆层，为 Proactive Agent（主动式智能体）提供"我知道你怎么做事"的基座。**

## TL;DR

- OpenAI 于 2026-04-21 上线 Chronicle（Codex 记忆功能），仅 Pro 用户 + Mac
- 次日 OpenChronicle 开源上线，冲上 X trending 第一
- 核心论点：模型能力趋同后，**记忆与 Context 成为体验差异的关键变量**
- 技术方案：AX Tree 优先（便宜、准、结构化），截图兜底，事件触发式采集
- Token 成本：轻度 ~$0.50/天，重度 $3-5/天
- 记忆存储：7 类扁平 Markdown（人/项目/工具/话题/联系人/组织/事件）
- 对外接口：MCP 协议，已支持 Claude、Codex、OpenCode 等一键接入
- 终极方向：Proactive Agent — 从 reactive（你问它才动）到 proactive（主动判断、建议、行动）

## 核心观点

### 1. 记忆竞争力假说：模型趋同，Context 为王

> "以前模型能力太差，记不记得住影响都不大。但现在模型差距逐步缩小，真正影响用户使用体验的将是模型拥有的关于用户的记忆与 Context 多少。"

这是一个值得产品经理重视的判断：

- **2024 年**：模型能力是第一竞争力，记忆是锦上添花
- **2026 年**：模型趋同，记忆 + Context 成为核心差异化壁垒
- **推论**：各家厂商争夺用户上下文，用户用得越多→Memory 越完整→迁移成本越高

### 2. 记忆的两层：认知层 vs 行动层

| 层级 | 含义 | 难度 | 举例 |
|------|------|------|------|
| **认知层** | 记住你说过什么 | 相对简单 | "用户喜欢用 TypeScript" |
| **行动层** | 记住你怎么做事 | 极其困难 | "用户说'加日程'时，默认放飞书而非 Apple Calendar" |

行动层需要理解用户在不同场景下的**偏好、惯性、潜规则**——这不是简单的 key-value 存储能解决的。

### 3. 记忆所有权：归用户，不归厂商

> "Agent 记忆的所有权，应该归属用户，而不是模型厂商。记忆该是设备里的一层基础设施，跟谁的模型协作都行。"

这是 OpenChronicle 的核心产品哲学，也是开源的根本动机：
- Chronicle（OpenAI）：记忆绑在 Codex 里 → 提高迁移成本
- OpenChronicle：记忆在本地 → 可接任何 Agent / 设备

## 技术方案

### AX Tree 优先 + 截图兜底

**AX Tree**（Accessibility Tree）：macOS 系统层辅助技术接口，原为残障读屏设计。

| 维度 | AX Tree | 截图 OCR |
|------|---------|----------|
| **成本** | 极低（纯文本） | 高（图片处理） |
| **准确度** | 高（结构化意图信号） | 中（OCR 误差） |
| **存储/检索** | 结构化轻量，便于数据库 | 非结构化，需额外处理 |
| **覆盖范围** | 部分应用受限（Word、飞书等绕开辅助通道） | 全屏幕覆盖 |

**短板**：Word、飞书等应用内部渲染绕开系统辅助通道，AX Tree 只能读到顶栏菜单和文档标题 → 因此采用**混合方案**：AX Tree 优先，截图兜底。

**采集策略**：基于触发器（光标运动、页面滚动、应用切换），非定时截图。看电影时不会触发记忆采集 → 避免 Token 浪费和记忆污染。

### 7 分类扁平存储

OpenChronicle 将所有应用操作汇入**同一个本地记忆池**，按 7 类扁平存为 Markdown：

| 分类 | 对应 |
|------|------|
| **Person** | 你这个人（身份、偏好） |
| **Project** | 你的项目 |
| **Tool** | 你常用的工具 |
| **Topic** | 你关心的话题 |
| **Contact** | 你联系的人 |
| **Organization** | 你工作的组织 |
| **Event** | 你经历的事件 |

### MCP 协议对外暴露

记忆层通过 MCP（Model Context Protocol）标准协议暴露为工具调用：
- Claude Desktop、Codex、OpenCode 已有一键集成配置
- 实现**跨产品记忆连贯**：Cursor 里的设计决策 → 飞书里的架构讨论 → Claude Desktop 里直接承接

### Token 经济

| 使用模式 | 日均成本 | 说明 |
|---------|---------|------|
| 轻度（静默记录） | ~$0.50 | 8 小时工作日，不主动调用 |
| 重度（高频交互） | $3-5 | 深度交互、频繁调用 |
| 本地模型 | 电费 | 需本地部署 gpt-oss-120b 级别模型 |

### 记忆基建化趋势

```
2024: 记忆 = 产品差异化功能（各自为政）
         ↓
2025: Chronicle 等推动记忆叙事升温
         ↓
2026: 记忆 → Agent 基础设施层（跨产品、跨设备、归用户所有）
```

## Proactive Agent：记忆基建之上的终极方向

Calvin/Vida 团队真正研究的方向：

| 维度 | Reactive Agent（当前主流） | Proactive Agent（目标） |
|------|------------------------|----------------------|
| 触发 | 你问它才动 | 主动判断、建议、行动 |
| 前提 | 当前对话上下文 | 全局记忆 + 行为模式理解 |
| 场景 | 单次任务 | 代码中断恢复、跨文档协作、设计稿延续、知识沉淀 |

**核心依赖**：Agent 真正进入日常作息，需要"我知道你怎么做事"这一层记忆支撑。

## 可复用的知识

### 记忆系统设计 Checklist

1. **采集层**：优先结构化数据（AX Tree > OCR），混合方案兜底
2. **触发策略**：事件驱动优于定时轮询（省 Token、防污染）
3. **存储结构**：扁平分类 + Markdown（人/项目/工具/话题/联系人/组织/事件）
4. **对外接口**：MCP 协议标准化，不绑定特定模型
5. **所有权设计**：本地优先 → 用户可迁移 → 降低厂商锁定
6. **成本控制**：轻度 $0.50/天是可接受的基线

### 记忆产品的三个判断维度

1. **记了什么**：认知层（说过什么）还是行动层（怎么做事）
2. **归谁所有**：厂商锁定 vs 用户自主
3. **能接谁**：单产品封闭 vs MCP 开放协议

## 个人批注

这篇访谈最有价值的洞察不是技术方案（AX Tree 等），而是**趋势判断**：记忆从产品功能→基础设施的转变。

当记忆成为基础设施，意味着：
1. **产品层面**：不再是每个 AI 产品自己造记忆模块，而是接入统一记忆层
2. **竞争层面**：厂商的护城河从"我的记忆功能更好"变成"我在统一记忆之上的应用体验更好"
3. **用户层面**：记忆跟着人走，不跟着产品走 — 这是用户主权的一种体现

Proactive Agent 方向值得持续关注：当 Agent 知道"你怎么做事"，它就不再是工具，而是协作者。这条路的关键瓶颈不是技术（AX Tree / MCP 都成熟了），而是**怎么从行为流中提炼出可泛化的做事模式** — 这可能需要一种新的记忆抽象层。
