# Single-Agent 与 Multi-Agent 架构选型：Orchestrator 模式实战

> 来源：京东面试官：单体 Agent 遇到瓶颈，Multi-Agent 方案怎么设计？（微信公众号文章）
> 提炼日期：2026-04-13
> 分类：AI方法论

## TL;DR

Single-Agent 适合任务清晰、复杂度适中的场景，优势是链路透明、好排查；当任务出现 context 溢出、多专业分工需求或需要并行执行时，才值得引入 Multi-Agent。Multi-Agent 工程上几乎只用中心化的 Orchestrator 模式——Orchestrator 负责拆解任务和调度，Worker 只做自己那块，去中心化方案因为不可控基本停留在学术研究。

## 核心观点

**1. Single-Agent 的核心优势不是"简单"，而是"链路完全在你掌控之内"**

所有任务逻辑在一个地方写清楚，出了问题链路短，好排查。如果任务不属于三类瓶颈（context 溢出/多专业分工/需要并行），用 Single-Agent 就够了——不要为了"用新技术"强行引入 Multi-Agent，系统变复杂变难维护却没有对应收益。

**2. Multi-Agent 的中心化模式本质是"职责分离"——Orchestrator 只管调度，Worker 只做执行**

Orchestrator 不做任何具体工作，只做三件事：拆解任务、分配 Worker、汇总结果。Worker 的 context 是干净的，只装自己职责相关的信息。这种分离让每个环节出问题时能精准定位——不用猜，顺着 Orchestrator 的调度记录追就能找到根源。

**3. 去中心化方案听起来更灵活，但生产环境几乎不用**

无总调度的多 Agent 系统会同时出现四类工程问题：任务分配无协调（重复工作）、执行顺序无保证（依赖关系混乱）、失败无感知（错误静默扩散）、无人确认整体完成。这些问题让系统行为不可预测，生产环境无法接受，去中心化方案主要用于学术探索。

## 关键知识点

### Single-Agent 力不从心的三类场景（触发 Multi-Agent 的信号）

1. **Context 溢出**：任务太长、信息量太大，context 撑爆，Agent 开始遗忘关键信息
2. **多专业分工需求**：不同步骤需要完全不同的专业能力，塞进一个 Agent 每件事都做得不够专注
3. **可并行子任务**：任务中有多个独立子任务理论上可以并行，但单 Agent 只能串行执行

**强调**：不属于这三类的任务，Single-Agent 就够了，不要过度设计。

---

### Orchestrator-Worker 中心化架构

**Orchestrator（总调度员/项目经理）**的三个职责：
1. 读懂用户大目标，拆解为子任务列表
2. 判断每个子任务交给哪个 Worker Agent
3. 收集各 Worker 产出，合并为最终答案

**Worker Agent（执行者）**的特点：
- 只关注自己那块，不需要知道整体任务或其他 Worker
- 收到指令 → 执行 → 返回结果 → 退出
- Context 干净，只装和自己职责相关的信息

**竞品分析任务的完整流程示例**：
```
用户："帮我写一份 AI 行业竞品分析"
  ↓
Orchestrator 拆解：
  → 任务1：市场信息搜集（分配给 Researcher Agent）
  → 任务2：竞品对比分析（分配给 Analyst Agent）
  → 任务3：报告撰写（分配给 Writer Agent）
  ↓
Orchestrator 汇总 → 最终报告
```

**调试优势**：报告内容不准 → 查 Researcher；分析逻辑有问题 → 查 Analyst；格式不对 → 查 Writer。每个 Agent 职责清晰，沿调度记录追踪，不需要猜。

---

### 去中心化方案的四类工程问题

以三个 Agent 协作场景为例（A、B 搜索信息，C 汇总）：

| 问题类型 | 具体表现 |
|---------|---------|
| 任务分配无协调 | A 和 B 搜了大量重叠内容，重复工作 |
| 执行顺序无保证 | C 不知道 A 和 B 何时搜完，不知道该等多久 |
| 失败无感知 | A 中途出错，B 和 C 继续运行，最终产出不完整但系统不知道 |
| 无整体完成确认 | 没有人宣告"整个任务完成了" |

**类比**：没有项目经理的团队——每个人都能干，但没人协调接口和时间节点，交出来的是互不兼容的结果。

### 三方案选型对比表

| 维度 | Single-Agent | Multi-Agent（中心化） | Multi-Agent（去中心化） |
|------|-------------|---------------------|----------------------|
| 架构复杂度 | 低 | 中 | 高 |
| Context 压力 | 全压在一个 Agent | 各 Agent 独立管理 | 各 Agent 独立管理 |
| 专业能力 | 泛才，什么都做 | 专才分工 | 专才分工 |
| 并行能力 | 不支持 | 支持子任务并行 | 支持并行 |
| 可控性 | 高 | 高（Orchestrator 统管） | 低，难以统一调度 |
| 调试难度 | 容易 | 中（按调度链路追踪） | 难，行为不可预测 |
| 工程实用性 | 高 | 高 | 低（主要用于学术研究） |
| 适用场景 | 任务清晰、复杂度适中 | 需要分工或并行的复杂任务 | 学术探索 |

### 两步选型决策框架

```
问题1：Single-Agent 能搞定吗？
  → 任务流程明确、不太长、不需要多种专业分工？
  → 是 → 用 Single-Agent，不要引入 Multi-Agent

  → 否 → 进入问题2

问题2：能接受系统行为不可控的风险吗？
  → 生产环境答案几乎一定是"不能"
  → 用 Orchestrator 模式（中心化 Multi-Agent）
```

## 启发与思考

**与 Agent 架构系列的关联**：本文是架构级的选型判断，与 `21-Agent反思机制` 是互补的——反思机制是单个 Agent 内部的质量提升手段，而 Orchestrator 模式是多 Agent 间的协作组织方式。两者可以结合：Orchestrator 调度 Worker，每个关键 Worker 内部启用反思机制。

**与安全产品设计的关联**：云起无垠的安全场景中，典型的 Multi-Agent 场景包括：漏洞扫描（扫描 Agent）+ 漏洞分析（分析 Agent）+ 报告生成（写作 Agent），由 Orchestrator 统一调度，既能并行扫描提速，又能保证分析和报告的专注度。

**面试回答框架**：
1. 先说 Single-Agent 的本质和优势（链路可控、维护简单）
2. 说清楚触发 Multi-Agent 的三个信号（context 溢出/多分工/需并行）
3. 说 Orchestrator 模式：Orchestrator 拆解+调度，Worker 专注执行，调度记录可追踪
4. 点明去中心化方案的工程问题，说明为什么生产环境选中心化

## 原文精华

> Single-Agent 最大的优势不只是「架构简单」，更核心的是「整条任务链路完全在你掌控之内」。任务怎么走、用什么工具、什么时候结束，所有逻辑都是你在一个地方写清楚的，出了问题链路短，好排查。

> 不要为了「用新技术」而强行引入 Multi-Agent，系统会变复杂、变难维护，但没有带来对应的收益。

> Orchestrator 不做任何具体工作，它只负责三件事：读懂用户的大目标把它拆成一个个子任务；判断每个子任务该交给哪个 Worker Agent 去做；收集每个 Worker 的产出，把它们拼成最终答案。

> 去中心化系统里这几类问题会频繁出现：任务分配没有协调、执行顺序没有保证、失败没有感知、没有人来确认「任务整体完成了」。
