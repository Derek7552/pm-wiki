# Anthropic 构建高效 AI 智能体：五种工作流模式与克制的工程哲学

> 来源：Anthropic《Building Effective Agents》技术博客解读（公众号文章直接粘贴）
> 提炼日期：2026-04-03
> 分类：AI方法论

## 核心观点

1. **最成功的 Agent 往往是最简单的**：Anthropic 与数十个行业团队实战合作得出的反直觉结论。先从最简单方案开始，只有简单方案确实不够用时才增加复杂度。一个精心调优的单次大模型调用 + RAG + 上下文示例，就足以解决大多数问题。

2. **工作流 vs 智能体的关键区分**：工作流（Workflow）= 预先编排好的流程，大模型只是执行节点；智能体（Agent）= 大模型自己决定下一步做什么、用什么工具。前者像流水线，后者像独立思考的员工。这个区分直接决定架构选型。

3. **工具设计比提示词设计更重要**：Anthropic 在 SWE-bench 上花在优化工具的时间比优化提示词还多。要像重视 HCI（人机界面）一样重视 ACI（智能体-计算机界面）。

## 关键知识点

### 五种工作流模式（从简单到复杂）

| # | 模式 | 核心逻辑 | 隐喻 | 典型场景 |
|---|------|---------|------|---------|
| 1 | **提示链** Prompt Chaining | 大任务拆小步骤，前步输出→后步输入，中间插检查点 | 接力赛 | 先写文案→再翻译；先生成大纲→校验→写正文 |
| 2 | **路由** Routing | 输入分类→导向不同专用处理流程 | 医院分诊台 | 客服：咨询/退款/技术支持走不同路径；简单→小模型，复杂→大模型 |
| 3 | **并行化** Parallelization | 多模型同时工作，汇总结果。变体：分段（拆子任务）/ 投票（跑多次取多数） | 分头行动 | 一个处理请求+一个做安全审查；多模型分别审查代码漏洞 |
| 4 | **编排者-执行者** Orchestrator-Workers | 中央模型动态拆解任务+分配，工作模型执行。与并行化区别：子任务是动态决定的 | 项目经理+工人 | 同时修改多个代码文件；多源信息收集分析 |
| 5 | **评估者-优化者** Evaluator-Optimizer | 一个生成+一个评估反馈，迭代循环 | 作家+编辑 | 文学翻译迭代；搜索任务判断信息是否充分 |

### 复杂度递进决策树

```
单次调用 + RAG 够用吗？
  ├─ 够 → 停在这里
  └─ 不够 → 需要多步骤吗？
       ├─ 步骤固定 → 用工作流（提示链/路由/并行化）
       └─ 步骤动态 → 需要自主决策吗？
            ├─ 部分自主 → 编排者-执行者
            └─ 完全自主 → 真正的 Agent（循环：任务→规划→执行→观察→调整）
```

### 真正的 Agent 运行循环
接收任务 → 规划 → 执行工具调用 → 观察结果 → 调整计划 → 继续执行 → 直到任务完成

实现上就是一个大模型在循环中根据环境反馈不断调用工具。

### ACI（Agent-Computer Interface）设计原则
- 站在模型角度想：这个工具好不好用？
- 参数命名是否清晰？
- 有没有使用示例？
- 实例：相对路径→绝对路径，错误率骤降

### 框架使用建议
- 框架帮你快速起步，但增加抽象层，看不清底层发生了什么
- 建议直接用大模型 API 写代码
- 很多「高级模式」其实几行代码就能实现

### 两个已验证的落地场景
| 场景 | 为什么适合 Agent |
|------|----------------|
| **客户服务** | 对话式交互 + 外部数据调用（查订单/退款）+ 成功标准清晰可衡量；有公司已采用按解决量计费 |
| **代码开发** | 结果可验证（跑测试就知道对错）→ 天然反馈闭环；Agent 可根据测试结果不断迭代 |

## 启发与思考

1. **与之前 Agent Harness 六层架构笔记的对照**：Anthropic 的五种模式 + Agent 循环，恰好映射到 Harness 架构的 Loop（循环执行）+ Tools（工具调用）+ Verification（评估反馈）层。两篇文章互为补充：Harness 讲「有什么」，Anthropic 讲「怎么选」。

2. **「克制比技巧更重要」= 奥卡姆剃刀的工程化应用**：与刚整理的「世界观三大基石」笔记直接呼应。如无必要，勿增实体——每增加一层复杂度，都应该有可衡量的收益来支撑。

3. **ACI 概念对 PM 的启示**：产品经理设计 AI 功能时，不仅要想用户界面，还要想 Agent 界面。当你的产品需要被 AI Agent 调用时，API 设计的「对 Agent 友好度」会成为新的竞争维度。

4. **客服场景与沃丰科技面试的关联**：Anthropic 确认客服是 Agent 最成熟的落地场景之一——对话式、可衡量、有明确操作。这直接支持沃丰科技 AI 客服方向的战略判断，面试中可引用。

5. **路由模式在实际产品中的普适性**：几乎所有 AI 产品都需要路由——简单请求用小模型（省钱快速），复杂请求用大模型（高质量）。这是成本控制和用户体验的平衡点。

## 原文精华

> 最成功的AI智能体，往往不是最复杂的那个，而是最简单的那个。

> 先从最简单的方案开始，只有在简单方案确实不够用的时候，才增加复杂度。

> 工具设计比提示词设计更重要。他们在SWE-bench基准测试中花在优化工具上的时间，比优化提示词还多。

> 要像重视人机界面（HCI）一样重视"智能体-计算机界面"（ACI）——站在模型的角度想，这个工具好不好用？

> 在AI时代，克制比技巧更重要。先把单次调用优化到极致，再考虑工作流；先用预设流程解决问题，再考虑自主智能体。每增加一层复杂度，都应该有可衡量的收益来支撑。

> 简单、透明、精心设计的工具接口——这三条原则，值得每一个AI开发者反复品味。
