# Agent 反思机制：生成-评估-改进循环与工程权衡

> 来源：小红书面试官：说说 Agent 的反思机制？为什么要用反思？具体怎么实现？（微信公众号文章）
> 提炼日期：2026-04-13
> 分类：AI方法论

## TL;DR

反思机制（Self-Refine）是给 LLM 加一个"回头检查"的环节：生成输出后，用评估 prompt 找问题，再用改进 prompt 修正，循环直到通过或达到最大轮次。代价是每轮至少多一次 LLM 调用，3 轮反思 = 至少 3 倍调用量。工程上只在质量要求高的关键节点启用，必须设最大轮次（通常 2-3 轮）防止死循环。

## 核心观点

**1. LLM 第一次输出是"一口气完成"的，缺乏自检环节**

LLM 生成时无法停下来检查，常见问题有三类：逻辑跳跃（推理步骤不完整）、遗漏细节（任务要求没全覆盖）、表达含糊（意思到了但说得不清晰）。给它一个"回头检查"的机会，它自己有能力发现并修正这些问题——反思机制就是把这个环节工程化。

**2. 评估和改进需要各自独立的 prompt，缺任何一个输入都会让改进变盲目**

改进 prompt 必须同时传入原始任务 + 原始输出 + 评估意见三件套：只有任务没有原始输出，不知道在什么基础上改；只有原始输出没有评估意见，不知道改哪里；只有评估意见没有任务，可能改着改着偏离目标。三者缺一都会导致 LLM 盲目重写而非针对性修改。

**3. 独立的 Critic Agent 比自我反思效果更好，但成本也更高**

单 Agent 自我反思时，评估者和生成者是同一个模型，生成时形成的"内部逻辑"会在评估时沿用，对自己的错误不够敏感，容易"自洽"。独立 Critic Agent 唯一职责是找问题，视角客观，类似代码 review 比自查发现的问题质量更高。代价是系统复杂度和调用成本进一步增加。

## 关键知识点

### Self-Refine 核心循环

来源：Self-Refine 论文（Madaan et al.）

```
生成（Generate）→ 评估（Evaluate）→ 改进（Refine）→ 循环
直到：LLM 回复 PASS 或 达到最大轮次
```

类比：草稿 → 批阅 → 修改 → 再审阅

---

### 评估 Prompt 模板

```
任务：{task}

当前输出：
{current_output}

请评估以上输出：
1. 有没有事实错误或逻辑问题？
2. 有没有遗漏重要内容？
3. 表达是否清晰准确？

如果输出已经足够好，回复「PASS」；
否则指出具体问题并给出改进建议。
```

**设计要点**：
- 给出明确检查维度（事实/逻辑/完整性/表达），不让 LLM 自由发挥，否则容易流于表面
- `PASS` 机制是必须的——没有明确出口，LLM 会为了反思而反思，把对的东西改错

---

### 改进 Prompt 模板

```
原始任务：{task}
当前输出：{current_output}
评估意见：{reflection}

请根据评估意见改进输出：
```

**三要素缺一不可**：

| 缺少的输入 | 导致的问题 |
|-----------|-----------|
| 缺原始任务 | 改着改着偏离目标 |
| 缺原始输出 | 不知道在什么基础上改 |
| 缺评估意见 | 不知道改哪里，可能全部重写 |

---

### 两种反思粒度对比

| 维度 | 步骤级反思 | 任务级反思 |
|------|-----------|-----------|
| 触发时机 | 每个工具调用/推理步骤后立即检查 | 整个任务执行完后做一次整体评估 |
| 核心优势 | 早发现早纠正，错误不会在后续步骤中放大 | 开销小（只多1次调用），能发现整体视角的矛盾 |
| 核心缺陷 | 10步任务可能实际调用20次LLM | 中途出大问题到最后才发现，前面执行全浪费 |
| 适用场景 | 步骤之间强依赖，前一步错了后面全错 | 步骤相对独立，最终整体质量更重要（如生成报告） |

---

### 多 Agent 互评（Critic Agent）

**流程**：执行 Agent 生成输出 → Critic Agent 审查并给出批注 → 执行 Agent 根据批注修改 → Critic Agent 再次确认

**为什么比自我反思更好**：自我反思时评估者和生成者是同一个模型，形成的"内部逻辑"导致对自身错误不敏感，容易"自洽"。Critic Agent 的唯一职责是找问题，无包袱，更客观。

**适用场景**：质量要求极高的场景——生成代码后让独立测试 Agent 验证、生成分析报告后让事实核查 Agent 交叉验证

**代价**：系统复杂度进一步上升，调用成本更高，普通场景用自我反思就够。

---

### 工程权衡决策框架

**值得开反思的场景**：
- 输出质量要求高、错误代价大的关键节点（最终报告、重要决策推理）
- 任务复杂、LLM 容易遗漏细节

**不值得开的场景**：
- 简单直接的任务（格式转换、简单问答）——纯浪费
- 实时性要求高的场景——一次反思延迟从 1 秒涨到 3 秒

**防死循环（工程必须项）**：
- 必须设最大轮次，通常 2-3 轮，**绝对不能依赖 LLM 自己判断停止**
- 原因：LLM 有时陷入"为了改而改"循环，每轮改动微小但实质无进步，系统一直转圈
- 硬性轮次上限是唯一可靠的退出机制

**成本意识**：3 轮反思 = 至少 3 倍 LLM 调用，延迟和成本线性增加

## 启发与思考

**与 Agent 架构设计的关联**：反思机制是 Agent 自主性提升的关键模块，直接影响输出可靠性。在云起无垠的安全 Agent 设计中，错误代价高（误报/漏报影响安全决策），关键推理节点适合引入反思；但实时告警场景对延迟敏感，不适合加步骤级反思。

**与 notes/AI方法论/18-Agent核心三要素 的关联**：反思机制是 Agent "多步推理"能力的重要实现路径——不只是一次推理，而是推理+评估+再推理的迭代。

**可落地的行动项**：
- 在 Agent 设计时明确标注"高质量关键节点"（适合开反思）vs "低敏感节点"（不适合）
- 任何使用反思的地方必须有最大轮次硬限制，这是工程必须项
- 评估 prompt 设计时给出具体维度列表，不能让 LLM 自由发挥

## 原文精华

> 反思机制就是给它加上这个环节：LLM 一口气生成完，没有机会停下来检查。给它一个「回头检查」的机会，它自己是有能力发现并修正的。

> 改进 prompt 同时传入了原始任务、原始输出、评估意见这三样东西，缺任何一个都会让改进变得盲目。三者都在，它才能有针对性地修改，而不是把内容全部重写一遍。

> 单 Agent 自我反思时，评估者和生成者是同一个模型，它在生成输出时形成的一套「内部逻辑」，做评估时也会沿用这套逻辑，对自己输出的错误不够敏感，容易陷入「自洽」。

> 必须设最大轮次，通常设 2-3 轮，绝对不能依赖 LLM 自己判断停止。硬性的轮次上限是唯一可靠的退出机制。
