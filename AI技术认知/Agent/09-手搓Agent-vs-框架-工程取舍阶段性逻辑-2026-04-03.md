# 手搓 Agent vs 框架：工程取舍的阶段性逻辑

> 来源：用户提供文本（美团面试题解析）
> 提炼日期：2026-04-03
> 分类：AI方法论

## 核心观点

1. **框架的痛点不是一开始暴露的，而是随项目推进在不同阶段逐渐浮出**：探索期框架很爽（POC 几十行跑通）；第一个奇怪 bug 出现后感觉变了（50 行代码但 40 层 stack trace）；版本升级踩坑（breaking change 导致十几处修改）；规模化阶段发现隐性开销（框架每次调用做你不需要的序列化/callback/日志，累积成可见延迟）。

2. **手搓的本质优势是"完全掌控"，体现在三个层面**：链路透明/可观测性好（任意位置加日志打断点）、精确裁剪/无多余开销（只写需要的逻辑）、稳定可控/不受框架升级影响（依赖仅底层 LLM SDK）。

3. **最务实的策略是"核心手写，周边借用"**：工具调用循环、对话历史管理、错误重试、任务状态维护——这些是 Agent 的"心脏"，必须手写掌控。LangSmith tracing、LlamaIndex 文档解析、向量库客户端——这些是"工具性"周边功能，用外部工具完全值得。

## 关键知识点

### 框架三大痛点及出现阶段

| 阶段 | 痛点 | 具体表现 |
|------|------|---------|
| **排查期** | 抽象层太多，调试穿透困难 | 50 行代码 + 40 层 stack trace，不知道问题在自己代码还是框架内部 |
| **运维期** | 版本升级 breaking change | LangChain 早期频繁改接口，线上稳定性受制于第三方 |
| **规模化期** | 隐性性能开销 | 框架通用性设计：每次调用做不需要的序列化/callback/日志，高流量下累积成真实延迟 |

> 类比：老式车打开引擎盖能看到漏油的管子；豪华车打开引擎盖只能送 4S 店扫诊断仪。

### 手搓三层优势

| 层面 | 说明 | 生产价值 |
|------|------|---------|
| **链路透明** | 每行代码都知道在干什么，任意位置插监控 | 线上故障定位最快 |
| **精确裁剪** | 只写需要的逻辑，无通用性包袱 | 优化空间完全在自己手里 |
| **稳定可控** | 不受外部 breaking change 影响 | 生产环境长期稳定运行 |

> 类比：框架是"租房"（方便但改不了结构，房东随时调政策）；手搓是"自建"（建起来慢但住着踏实）。

### 选型决策矩阵

| 场景 | 推荐 | 原因 |
|------|------|------|
| POC/快速验证 | **框架** | 两周工作缩短到两天，跑通即可 |
| 团队刚接触 Agent | **框架** | 少踩基础坑 |
| 周边工具依赖生态 | **框架** | 文档解析/向量检索等 |
| 准备上生产 | **手搓核心** | 稳定性成为核心关切 |
| 流量上来 | **手搓核心** | 性能和成本敏感 |
| 高度定制业务 | **手搓核心** | 和框架通用设计偏差大 |
| 需要高可观测性 | **手搓核心** | 链路随时监控回溯 |

### 折中方案：核心手写 + 周边借用

**手写（Agent 心脏）**：
- 工具调用循环（ReAct loop）
- 对话历史管理
- 错误处理和重试
- 任务状态维护

**借用（工具性周边）**：
- LangSmith tracing（调用链追踪）
- LlamaIndex 文档解析
- 向量库 Python 客户端
- 可观测性工具

> 盖房子类比：自己设计承重墙和布局，但门锁/插座/水龙头买现成的。

### 典型项目演进轨迹

```
1. 框架快速跑通 POC → 验证方向
2. 遇到线上 bug → 排查困难的关键部分替换为手写
3. 流量上来 → 性能敏感的核心逻辑全部手写
4. 最终态 → 框架只保留做得好的周边工具
```

### 一个判断信号

> 如果你能说出"框架在某个地方替我做了什么"——有掌控感，可以用。
> 如果你完全不知道方法内部发生了什么——黑盒，需要警惕。
> 框架本身不是问题，"不理解就依赖"才是。

## 数据与案例

- **LangChain 早期**：@tool 装饰器 + AgentExecutor 封装 ReAct loop + 内置 tracing/callback/记忆，POC 阶段极高效
- **框架速度优势**：两周工作缩短到两天（探索期）
- **隐性开销**：高流量下框架内部的序列化/callback/日志累积成可见延迟和费用浪费

## 启发与思考

1. **与 Anthropic 构建高效 Agent（notes 06）的"克制哲学"完全一致**：Anthropic 建议"能用简单 prompt 链解决的不要上 Agent 框架"，本文从工程实践角度验证了同样的判断——框架的复杂性在生产环境中是真实的成本。

2. **与 Harness Engineering 系列的关系**：Harness（06/07/12）强调 Agent 外层执行框架的重要性。但这篇提醒我们：Harness 的"核心心脏"（工具调用/状态管理/错误处理）应该手写掌控，只有"周边工具"可以借用框架。这为 Harness 的实现层面提供了清晰的控制边界。

3. **对 AI 客服产品的架构决策参考**：沃丰等 AI 客服如果用 LangChain 做核心 Agent 逻辑，上线后可能面临同样的痛点——调试困难、版本升级风险、隐性开销。核心的意图识别→知识检索→回答生成→转人工判断链路应该手写，LangSmith/向量库客户端等可以借用。

## 原文精华

> 框架的抽象层太多，排查问题需要穿透那些你没写、也不完全懂的层，这是实实在在的认知负担。

> 框架是"租房"，装修好直接住，方便，但结构改不了，房东随时可能调整政策；手搓是"自建"，建起来慢，但所有结构都熟悉，改什么都能改，住着踏实。

> 手搓不是"比框架更好"，而是"在特定阶段有特定的价值"。

> 框架本身不是问题，"不理解就依赖"才是。
