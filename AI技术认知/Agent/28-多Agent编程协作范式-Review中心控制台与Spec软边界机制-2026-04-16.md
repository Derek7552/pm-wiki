# 多 Agent 编程协作范式转移：Review 为中心的控制台设计

> 来源：[从聊天窗口到多 Agent 控制台：一次 AI 编程协作范式的转移]（阿里技术，2026-04）
> 提炼日期：2026-04-16
> 分类：技术趋势

## TL;DR

单 Agent 协作模式让人始终困在"等待-检查-确认"的循环里，无法真正从执行流程中抽身。作者基于实践设计了 Mexus：一个以 Review 为工作流中心、支持多 Agent 并行的 WebUI 控制台，核心机制是 Spec→Plan 两段式人工确认 + allowedPaths 软边界 + Observer Agent 运行时协调。"约束不是限制，而是放大器"——这是多 Agent 协作从不可控变为可管理的关键。

## 核心观点

**1. 单 Agent 的瓶颈不是能力，而是协作模式**

单 Agent 工作流的本质是串行：人提需求 → Agent 执行 → 人确认 → 下一轮。只要任务还在这条链上推进，人就没有真正抽身。解法不是让 Agent 更强，而是换范式：多个 Agent 并行执行，人的工作重心从"参与执行"转移到"Review + 决策"。

**2. 可观测是多 Agent 协作的基础设施**

单 Agent 时盯着输出就够了；多个 Agent 并行时需要的是"态势感知"而不是更多终端窗口。更重要的是：给人看的观测数据和给 Observer Agent 用的决策数据应该建立在同一层事实基础上，避免"UI 显示一种状态，Agent 依据另一种状态行动"的分裂。今天由人眼完成的判断（发现冲突、判断 Agent 是否卡住），未来可以交给 Observer Agent 自动完成。

**3. 约束不是限制，而是放大器**

引自 Harness Engineering 的洞察：在人类工作流里，严格的架构约束显得教条；但在 Agent 场景下，一旦约束被编码，它就能同时作用于所有 Agent 和所有任务。通过 prompt 告诉 Agent"请注意不要越界"是不够的，需要用结构化机制让越界行为可被检测、可被拦截。

## 关键知识点

### 多 Agent 编排的核心机制

**Spec-first 两段式确认**
1. 用户提出需求
2. 规划 Agent 生成结构化 spec（目标、范围、验收标准、影响范围、任务拆解建议）
3. **人工确认点 1**：用户审阅并批准 spec
4. 系统基于 spec 生成执行计划（每个 Agent 负责什么任务、可改动哪些路径、任务依赖关系）
5. **人工确认点 2**：用户批准执行计划
6. 多 Agent 开始并行执行

**allowedPaths 软边界 + claim 系统**
- 每个 Agent 在执行计划中获得 `allowedPaths`，明确可改动的文件路径范围
- 执行期间维护轻量的 claim 记录，标记哪个 Agent 正在编辑哪些文件
- claim 是**软边界，不是强锁**——不阻塞执行，但让占用情况对所有参与者可见，提供冲突发现的事实信号

**Observer Agent 运行时协调**
- Observer 不写业务代码，只做观测和协调
- 发现轻量冲突：直接提示相关 Agent 避让
- 遇到更大风险：向用户上报，由人决策
- 约束条件：频率限制 + 审计日志 + 用户可随时冻结其控制权

**方案本质**：用结构化 spec 定义边界，用 claim 暴露事实，用 Observer 做运行时仲裁，用人工确认点兜底关键决策。不依赖 Agent 的"自觉"，而是通过机制设计让多 Agent 协作可观测、可约束、可干预。

### 四种可观测视角

| 视角 | 内容 | 解决的问题 |
|------|------|-----------|
| Agent 仪表盘 | 每个 Agent 的改动行数、触及文件数、context 使用率、工作节奏曲线 | 谁在持续产出、谁卡住了、任务分配是否均衡 |
| 模块视图 | 按功能模块聚合各 Agent 的工作分区 | 发现多个 Agent 挤在同一模块的协调需求 |
| 依赖视图 | 被触及文件的依赖链，改动自动高亮下游影响 | 识别牵一发动全身的高风险改动 |
| 冲突面板 | 直接冲突、依赖冲突、模块冲突集中呈现 | 快速回答：现在有没有问题？在哪？我需要介入吗？ |

### 共享工作区 vs git worktree 的设计选择

**默认共享工作区（非默认 git worktree）的理由：**
- 实际多 Agent 场景中任务往往有依赖、共享接口和类型
- 默认隔离 = 把协作问题后置到合并阶段，而合并阶段的冲突解决成本远高于执行中的及时纠偏
- 共享 + 机制管控（allowedPaths + claim + Observer）比隔离 + 最后合并更高效

**git worktree 仍有适用场景：**
- 高风险重构
- 实验性改动
- 需要独立验证的任务

设计上支持在同一执行计划中混用 `shared` 和 `isolated` 模式。

### Review 在工作流中的角色升级

| 单 Agent 模式 | 多 Agent 模式 |
|-------------|-------------|
| Review 在终点做验收 | Review 是持续进行的判断主界面 |
| "Agent 做完了检查一遍" | "一边看 Diff，一边决定继续/返工/收下" |
| Review 后结束 | Review 的反馈可直接发回对应 Agent，形成闭环 |

**结论：Review 取代代码编辑器，成为日常工作的主操作面。**

### 混用不同 Agent 的策略

- 核心模块用 Claude Code（强模型）
- 测试和文档用更轻量的 Agent（节省 Token）
- 某个 Agent 达到使用限额后，其他 Agent 可接力（能力有降级，但好过中断）

## 数据与案例

- OpenAI Harness Engineering：用 Codex Agent 构建超过百万行代码产品，全程无人类直接写代码。早期进展慢的原因不是 Agent 能力不够，而是**环境没有准备好**（缺少工具、抽象和结构）
- 核心论断：**"Humans steer. Agents execute."** 工程师的工作从写代码 → 设计环境、定义意图、构建反馈循环

## 启发与思考

**对安全产品的直接映射**：安全运营的多任务场景（告警分析、漏洞处置、情报调研并行）与多 Agent 并行协作高度同构。allowedPaths 的授权边界设计可以直接对应安全 Agent 的最小权限原则；Observer Agent 的角色类似安全运营中心的 SOAR 编排引擎。

**"约束是放大器"的工程哲学**：安全产品中合规检查、访问控制等约束往往被视为负担。但如果将约束编码为机制（而不是依赖人工自觉遵守），它就能无边界地扩展到每个 Agent 和每个任务。这是安全 AI 产品设计的核心价值点之一。

**Review 中心化的工作流值得借鉴**：不只是 AI 编程，任何多智能体协作系统都应该把"人的判断点"设计为工作流中心，而非末端的验收动作。这也是安全 SOC 平台设计的核心原则。

**待深入的方向**：claim 系统在大规模并发时的性能边界；Observer Agent 的"频率限制"如何设计才能平衡实时性和可控性。
