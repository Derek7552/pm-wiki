# Agent 链路推理评估七大方法

## 为什么 Agent 推理评估比传统 NLP 难？

传统 NLP 评估只看最终输出，但 Agent 的特殊性决定了这不够：

| 难点 | 具体表现 |
|------|---------|
| 链路很长 | 一个任务涉及 5-20 步推理 + 工具调用，结果正确不代表每步合理 |
| 过程不透明 | CoT 和工具选择背后的"思考"难以直接度量 |
| 路径不唯一 | 同一任务可能有多条合理执行路径，不能简单比对标准答案 |
| 错误会级联 | 早期一步失误导致后续整链偏离，但最终结果可能碰巧正确（歪打正着）|

---

## 七大评估方法全景

### 方法 1：最终结果评估（Outcome-based）

**思路：** 不管中间过程，只看最终任务是否完成。

**常见指标：**
- 任务成功率（Success Rate）：是否达成用户目标
- 答案准确率：最终输出与标准答案的匹配度
- 完成度（Completion Rate）：多目标任务完成了几个子目标

**适用：** 通用场景，和用户体验最直接对齐。

**局限：** 无法区分"歪打正着"和"正确推理"，无法定位具体哪步出问题。

---

### 方法 2：工具调用准确率（Tool Call Accuracy）

**思路：** 将每一步 function call 选择与预定义标准路径对比。

**常见指标：**
- 函数选择准确率（Tool Selection Accuracy）：每步调用函数是否正确
- 参数准确率（Argument Accuracy）：传入参数是否正确
- 调用顺序一致性：工具调用先后顺序是否合理
- F1 Score：标准路径工具集合 vs 实际调用集合的 precision/recall

**适用：** 路径明确的任务（如标准化业务流程）。

**局限：** 需要预先标注"标准执行路径"；过于严格的对比会误判合理的替代路径。

---

### 方法 3：过程奖励模型（Process Reward Model, PRM）

**思路：** 训练一个专门的模型，对链路中每一步推理打分，而不只看最终结果。

**做法：**
1. 收集人工标注的逐步推理质量数据（每步标记"正确/中性/错误"）
2. 训练 PRM 学习给中间步骤打分
3. 评估时，PRM 对 Agent 链路每步输出置信度分数

**代表工作：** OpenAI "Let's Verify Step by Step" 论文，证明 PRM 在数学推理中显著优于只看结果的 ORM（Outcome Reward Model）。

**适用：** 长链路推理，精确定位薄弱环节。

**局限：** 需要大量逐步标注数据，标注成本极高。

---

### 方法 4：LLM-as-Judge 评估推理过程

**思路：** 用另一个强大 LLM 审查 Agent 的完整推理轨迹（trajectory）。

**实现方式：**
1. 将 Agent 完整轨迹（CoT + tool call + tool response）输入 Judge 模型
2. Judge 从多维度打分：
   - 推理逻辑是否连贯
   - 工具选择是否合理
   - 是否存在冗余步骤
   - 是否遗漏关键信息
3. 设计结构化评分 rubric，让 Judge 按维度逐项评估

**适用：** 开放式任务，不需要预先标注标准路径。

**局限：** Judge 本身可能有偏差；对超长轨迹的评估能力会下降。

---

### 方法 5：轨迹对比与偏好排序

**思路：** 给同一任务生成多条轨迹，通过对比评估质量。

**做法：**
1. 针对同一 query，让 Agent 跑多次，产出多条轨迹
2. 用人工或 LLM-as-Judge 对轨迹排序（哪条推理过程更好）
3. 计算轨迹间一致性（多次执行是否趋同于同一路径）

**适用：** 天然适合 GRPO 等基于 group 的训练范式（评估与训练闭环）。

**局限：** 多次采样成本高；排序标准可能主观。

---

### 方法 6：基于状态的评估（State-based Evaluation）

**思路：** 不看 Agent 选了什么工具，而是看每步执行后"世界状态"是否向目标靠近。

**做法：**
1. 定义任务的关键状态节点（key milestones / checkpoints）
2. 检查 Agent 是否依次达到这些中间状态

**示例（订机票任务）：**
```
是否获取出发地、目的地、日期？
→ 是否搜索了航班？
→ 是否选择了符合条件的航班？
→ 是否完成支付流程？
```

**适用：** 流程型任务，路径无关，容忍多种合理路径。

**局限：** 需要针对每类任务定义状态空间，泛化成本高。

---

### 方法 7：效率与成本指标

**思路：** 即使推理正确，也要评估效率。

**常见指标：**
- 步数（Step Count）：完成任务用了多少步，是否有冗余
- Token 消耗：推理过程总 token 数
- 工具调用次数：是否有不必要的重复调用
- 延迟（Latency）：端到端完成时间
- 纠错能力（Recovery Rate）：遇到工具调用失败后，能否自我修正并继续

## 七种方法对比总表

| 评估方法 | 评估对象 | 是否需要标注 | 适用场景 | 关键局限 |
|---------|---------|------------|---------|---------|
| 最终结果 | 输出 | 需要标准答案 | 通用 | 无法定位问题 |
| 工具调用准确率 | 每步调用 | 需要标准路径 | 路径明确的任务 | 不容忍替代路径 |
| PRM | 每步推理 | 需要逐步标注 | 长链路推理 | 标注成本高 |
| LLM-as-Judge | 完整轨迹 | 不需要 | 开放式任务 | Judge 有偏差 |
| 轨迹对比 | 多条轨迹 | 可选 | 配合 RL 训练 | 采样成本高 |
| 状态检查 | 中间状态 | 需定义状态 | 流程型任务 | 泛化成本高 |
| 效率指标 | 资源消耗 | 不需要 | 通用 | 不反映推理质量 |

**实际系统中的组合：** 最终结果 + 中间步骤 + 效率指标

## 理想的面试回答框架

```
"Agent 链路的推理评估可以从几个层次来看：
1. 最基础：看最终任务是否成功（Outcome）
2. 往细了：评估每步工具选择和参数是否正确（和标准轨迹对比）
3. 更灵活：用 LLM-as-Judge 审查完整轨迹（不依赖预定义路径）
4. 或者：定义中间状态检查点来看推理进展（State-based）
5. 学术前沿：Process Reward Model 训练模型对每步推理打分
实际系统一般是最终结果 + 中间步骤 + 效率指标的组合。"
```

## 延伸追问方向

1. **如何构建 Agent 评估 Benchmark？** → AgentBench、WebArena、SWE-bench 的设计思路和评估维度有何不同？

2. **评估与训练的闭环：** PRM 分数能否直接作为 GRPO 的 reward 信号？

3. **多 Agent 协作评估：** 多个 Agent 协作完成任务时，如何归因哪个 Agent 的推理出了问题？

4. **在线 vs 离线评估：** 生产环境中如何实时监控 Agent 推理质量？可以用哪些 proxy 指标？

5. **Reward Hacking：** Agent 是否会学到"走捷径"完成任务但实际推理质量很差的策略？
