# DeepResearch Agent：ReAct 框架实现与上下文溢出工程实践

> 来源：粘贴文本（作者：吴师兄，DeepResearch Agent 从0到1系列第一篇）
> 提炼日期：2026-04-14
> 分类：AI 方法论

## TL;DR

从工程实战角度讲清楚三件事：① RAG 和 Deep Research Agent 的本质差距（词条型 vs 研究型问题）；② ReAct 框架最小可运行实现的每个设计决定（含工具失败处理，成功率78%→91%）；③ ReAct 的天然上限——上下文线性膨胀导致注意力稀释，解法是 IterResearch（固定大小研究摘要）。

## 核心观点

**RAG 和 Deep Research Agent 的本质差距不是技术栈，而是问题类型匹配。** 研究型问题的关键特征是"后面的搜索依赖前面的结果"——你不知道要搜什么，直到你看到前一步找到的信息。这种动态决策 RAG 的一次检索做不到，不是工程优化问题，是设计前提问题。

**ReAct 框架的核心价值在错误处理，不在主路径。** 工具调用在生产环境里一定会失败（网络超时/404/API 限流），不捕获错误会导致整个 Agent 崩掉。正确做法是把错误写回 Observation，让模型自己决定换策略继续——成功率从78%提升到91%，代价只是多写几行 try/except。

**上下文膨胀是 ReAct 的物理天花板，不是工程 bug。** 每步平均 2500 token，10 步就是 25000 token。超过阈值后模型对早期信息的注意力稀释，靠后内容权重更高。这不是模型的 bug，是注意力机制的物理特性。"上下文越长模型越聪明"是错的。

## 关键知识点

### RAG vs Deep Research Agent 问题类型对照

| 问题类型 | 典型例子 | 需要检索轮次 | RAG 能做吗 |
|---------|---------|------------|-----------|
| 单跳问答 | "等待期是多少天" | 1 轮 | 完全够用 |
| 两跳推理 | "A 公司 CEO 毕业于哪所大学" | 2-3 轮 | 勉强应付 |
| 研究型问题 | "五大车企固态电池路线对比" | 10-20 轮 | 无法处理 |
| 复杂分析 | "某政策对三个行业的差异化影响" | 20+ 轮 | 完全失效 |

**判断标准**：先想清楚问题是"词条型"还是"研究型"。能靠一次检索回答的，用 RAG 就够，不需要引入 Agent 复杂度。

### DeepResearch Agent 四根柱子

1. **训练数据**：从问题出发，一步步搜索判断调整直到拿到答案的完整轨迹样本（需专门构造）
2. **Agent 框架**：模型的行动逻辑——输出格式、工具调用方式、结果反馈机制（今天讲 ReAct）
3. **训练系统**：SFT（监督微调）学格式 + RL（强化学习）学策略选择
4. **工具集**：核心四件套 Search / Visit（网页提取）/ Scholar（学术检索）/ Python（代码执行）

### ReAct 框架三步循环

```
① Thought：我现在知道什么？还缺什么？下一步该做什么？
② Action：调用工具（搜索关键词 / 打开网址等）
③ Observation：把工具结果追加进对话历史
→ 回到 ① 继续，直到信息足够或达到 max_steps
```

### ReAct 最小实现关键设计决定

**为什么保留完整对话历史而不截断？**
ReAct 依赖模型看到完整推理链——切掉早期搜索结果，模型不知道"这个信息已经找到了"，会重复搜索相同内容，浪费步数和 token。

**为什么用 try/except 捕获工具失败？**
搜索引擎超时、网页404、网络抖动在生产里每天发生。失败时把错误信息写回 Observation（而不是抛异常），让模型自己决定换策略继续。内测数据：加了 try/except 后成功率从 78% 提升到 91%。

**tool 消息的顺序约束**：
OpenAI 格式要求 `tool` 消息必须紧跟触发它的 `assistant` 消息，通过 `tool_call_id` 配对。顺序错了 API 直接报错。

**max_steps 超限时的处理**：
不返回 None 或报错——返回模型当前最后一条 assistant 输出，附上步数超限说明。"用户等了这么久，哪怕不完整的答案也比什么都没有强"。

**核心代码结构**（精简版）：
```python
def react_loop(user_query, llm, tools, max_steps=10):
    messages = [{"role": "system", ...}, {"role": "user", "content": user_query}]

    for step in range(max_steps):
        response = llm.chat(messages=messages, tools=tools, tool_choice="auto")

        if response.tool_calls:
            tool_call = response.tool_calls[0]
            try:
                observation = tools[tool_call.function.name](**args)
            except Exception as e:
                observation = f"工具调用失败（{type(e).__name__}）：{e}。请尝试换一种方式。"

            messages.append(response.to_message())          # Thought + Action
            messages.append({"role": "tool", "content": str(observation)})  # Observation
        else:
            return response.content  # 信息足够，给出最终答案

    # 超限时返回最后一条 assistant 输出
    return f"[已达 {max_steps} 步上限] {last_assistant_content}"
```

### ReAct 的天然上限：上下文膨胀

**膨胀速率**：每步平均 2500 token → 10 步 = 25000 token，15 步 = 37500 token

**症状**：
- 注意力稀释：模型对第1步找到的信息注意力下降，靠后内容权重更高，导致信息混淆
- 策略停滞：模型用几乎相同的关键词反复搜索，或在同一子问题上绕圈

**真实案例**：新能源政策比较问题，第7步 messages 超 28000 token，模型把欧盟补贴数据安到了美国政策里，排查两小时才发现根因。

**解法：IterResearch（迭代研究框架）**
不把原始搜索结果堆进上下文，而是每轮执行完更新一份固定大小的结构化"研究摘要"，记录"目前已知信息"和"还没解决的子问题"。上下文大小保持常量，无论跑多少轮都不会溢出。

**监控建议**：在 Agent 执行完后打印 messages 的 token 总数，当某类问题到第5步就超过 2 万 token，就是上 IterResearch 的时机。

### 系统提示的三件事

系统提示在 Agent 里不是"介绍你是谁"，而是承担三个角色：
1. **定义角色**：研究型助手 vs 对话助手，行为模式完全不同
2. **规定输出格式**：每次调工具前先说 Thought，工具结果回来后先评估质量
3. **设置行为边界**：针对真实测试中发现的问题加约束

**关键边界规则（有效）**：
- "如果信息来源之间有矛盾，明确指出，不要自己融合矛盾的信息"
- "如果找不到答案，直接说找不到，不要编造"

加这两条后，信息混淆率从 13% 降到 6%。

**系统提示写法原则**：先跑起来让 Agent 出真实问题，然后针对性地加约束。一开始就写得很长，模型反而"读不进去"。

## 数据与案例

- **工具失败处理前后**：成功率 78% → 91%（加 try/except 并写回 Observation）
- **系统提示边界规则**：信息混淆率 13% → 6%（加矛盾标注 + 禁止编造规则）
- **上下文临界点**：28000 token 时出现明显注意力稀释，信息混淆问题复现
- **典型失败案例**：新能源政策多市场比较，第7步超 28000 token，欧盟数据被混入美国政策报告

## 启发与思考

**"工具失败时不崩"是 Agent 工程的第一优先级。** 这个原则可推广到所有 Agent 系统设计：任何外部依赖调用都要有降级处理，失败信息写回 Agent 上下文，让 Agent 自主决策换策略，而不是直接让系统崩掉。

**上下文膨胀问题本质上是"原始数据 vs 提炼摘要"的设计选择。** ReAct 默认把原始 Observation 全部堆进去（信息全但膨胀快），IterResearch 改为每轮更新结构化摘要（信息有损但大小稳定）。这个取舍思路可以复用到任何需要多轮积累的 Agent 场景。

**系统提示是测试驱动的，不是一次性设计的。** "先跑起来让 Agent 出问题，再针对性加约束"这个工作方式与传统软件开发的防御性编程逻辑一致，但很多人搭 Agent 时习惯性先把系统提示写得很完整，反而效果不如迭代式补充。

**面试答 Agent 相关问题的核心策略**：不背定义，而是讲"你见过什么真实失败，为什么会这样，你怎么处理的，有没有数据"。能说出具体数字（成功率从78%到91%）的候选人，远比能说出"上下文管理很重要"的候选人有说服力。

## 原文精华

> 很多人做 Agent 做到"能跑"就停了，从没让它跑到出问题的地方，也从没思考过为什么会出问题。真正能回答"搜崩了怎么办"的人，一定是把系统压测过的人。

> 很多人以为上下文越长模型越聪明，实际上超过某个阈值之后，结果反而会变差。这不是模型的 bug，是注意力机制的物理特性。

> 一个好的系统提示不是写越多越好，是每一条都针对你在测试里看到的真实问题写的。
