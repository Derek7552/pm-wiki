# AI 工作流三层架构：Workflow、Graph、Loop 概念与实现

> **来源**：公众号技术文章（粘贴内容）
> **日期**：2026-04-21
> **领域**：AI 工程 / 工作流架构
> **关键词**：Workflow, Graph, Loop, Node, Edge, State, LangGraph, Spring AI Alibaba

---

## 一句话总结

AI 工作流的本质是三层嵌套——**Workflow 是目标与过程，Graph 是结构与载体，Loop 是图上的控制模式**——核心抽象是 Node（执行单元）、Edge（流转规则）、State（共享上下文），框架会换代但这层抽象不会变。

---

## 核心知识点

### 1. 为什么 AI 系统需要工作流

传统工作流的核心假设是**确定性**：相同输入 → 相同输出。AI 工作流面对的是**不确定性**环境，引出三个结构性需求：

| 需求 | 说明 |
|------|------|
| 动态决策 | 下一步不唯一，需根据运行时生成内容的质量评估选择路径 |
| 自动修正 | 结果不理想时系统自动迭代，而非直接失败 |
| 状态可追踪 | 中间状态必须记录，否则无法调试、追踪与恢复 |

**关键区别**：传统工作流分支条件在设计时确定（金额>10000走高级审批）；AI 工作流的"是否达标"本身需要运行时评估，且评估结论可能驱使流程回退。

### 2. 三层架构关系

```
Workflow（目标层）── "先生成，再审核，不达标就修改，直到达标输出"
    ↓ 结构化表达为
Graph（结构层）── Node + Edge + State 构成的有向图
    ↓ 其中包含
Loop（控制层）── 图上的回边（Back Edge），实现可控回溯
```

### 3. Graph 三要素：Node、Edge、State

**Node（节点）= 执行单元**
- 三大功能：读取 State → 执行逻辑 → 写回 State
- 抽象原则：按**职责边界**划分（产出什么结果），不按"调了哪个 API"

**Edge（边）= 流转规则**，六种类型：

| 类型 | 特征 | 典型场景 |
|------|------|----------|
| 顺序边 | 固定顺序，无条件 | 预处理→生成 |
| 条件边 | 候选集设计时确定，运行时选择 | 评分>=80走输出，否则走修改 |
| 动态路由 | 候选集运行时才确定 | LangGraph Send API，map-reduce |
| 循环边 | 回指先前节点 | 审核→修改→审核 |
| 终止边 | 引导至结束 | 输出最终结果 |
| 并行边 | 同时分发多个后续节点 | RAG/工具并发 |

> 条件边 vs 动态路由是**连续谱系**：多数场景条件边够用，动态路由适用于运行时决定并行分支数量的场景。

**State（状态）= 共享工作记忆**
- 本质是键值对（Map/dict/Record）
- 三种更新语义：

| 语义 | 适用场景 | Spring AI Alibaba | LangGraph |
|------|----------|-------------------|-----------|
| 覆盖（Replace） | 单值字段（评分、当前草稿） | ReplaceStrategy | 默认 |
| 追加（Append） | 累积字段（对话历史） | AppendStrategy | `Annotated[list, operator.add]` |
| 自定义合并 | 复杂逻辑（按ID去重追加） | - | `add_messages` |

> **并发陷阱**：多个并行节点写同一个覆盖语义字段会竞态，LangGraph 会抛 `INVALID_CONCURRENT_GRAPH_UPDATE`。

### 4. Loop 设计三要素

Loop ≠ "多跑几次"，而是**图结构上的可控回溯机制**：

| 要素 | 说明 |
|------|------|
| 继续条件 | 为什么还要再来一轮（评分<80） |
| 退出条件 | 什么时候足够好（评分>=80） |
| 安全边界 | 最大轮次 + 超时 + Token预算 + 熔断 |

两种 Loop 类型：
- **固定次数**（for）：最多重试 N 次
- **条件驱动**（while）：评分低于阈值就继续

> 实际必须**两者结合**——条件驱动决定何时满意，固定次数兜底防死循环。

**嵌套循环**：外层=质量迭代（生成→审核→修改），内层=工具重试（指数退避）。两层的作用域、终止条件、计数器**必须独立**。

**区分 Graph Loop vs Agent Loop**：Agent Loop 是顶层"推理→行动→观察"循环，Graph Loop 是图内部特定节点子集的迭代修正，后者嵌套在前者内部。

### 5. 工作流抽象原则（分水岭）

低抽象：每步写成具体动作（检查标题长度、检查语气、判断是否补资料……）→ 碎片化，复用性差。

高抽象：按**稳定的职责边界**划分：

| 节点 | 职责 |
|------|------|
| DraftNode | 产出当前版本内容 |
| ReviewNode | 评估是否达标 |
| ReviseNode | 根据反馈修正 |
| ExitNode | 满足条件时输出 |

**三层抽象原则**：
- **Node** 抽象职责边界（产出什么结果），不抽象调了哪个 API
- **Edge** 抽象流转规则（什么状态允许去哪），不在图外写 if-else
- **State** 抽象推进任务必须记住的信息（工单快照、审核结论、重试次数）

### 6. 落地注意事项

| 坑 | 建议 |
|----|------|
| State 粒度 | 按业务含义分块：用户输入/生成结果/审核结论/流程控制 |
| 死循环 | 必须明确：最大轮次 + 评分阈值 + 超时 + fallback |
| 错误处理 | 不能只靠外围 try-catch，图上要有明确的重试/降级/转人工边 |
| Token 成本 | Loop 放大消耗，先粗筛再精修，"足够好"即停 |
| 数据格式 | 节点间传递的 schema 尽早统一（JSON Schema/Pydantic） |

**四类错误处理策略**（Spring AI Alibaba）：

| 错误类型 | 策略 |
|----------|------|
| 瞬时错误（超时/限流） | 指数退避重试 |
| LLM 可恢复错误（格式异常） | 错误存 State，循环让 LLM 调整 |
| 用户可修复错误（信息缺失） | interruptBefore 暂停等人工输入 |
| 意外错误 | 冒泡给开发者 |

可映射分布式弹性模式：指数退避、熔断器（连续 N 次失败→降级）、舱壁隔离（独立并发上限）、补偿事务（Saga 回滚）。

### 7. 安全考量

工作流特有的两类攻击面：

| 威胁 | 机制 | 防御 |
|------|------|------|
| State 污染 | 恶意输入写入路由控制字段（next_node），跳过审核直达输出 | 路由字段白名单校验 |
| Loop 放大攻击 | 构造输入使审核永远低分，耗尽 Token | iteration_count + Token 预算双重边界 |

### 8. 框架映射速查

| 概念 | Spring AI Alibaba | LangGraph |
|------|-------------------|-----------|
| State | `OverAllState` + `KeyStrategyFactory` | `TypedDict` + `Annotated` |
| Node | `NodeAction` 接口 | 函数/Runnable |
| 条件边 | `addConditionalEdges(src, fn, map)` | `add_conditional_edges(src, fn)` |
| 持久化 | `MemorySaver`/`RedisSaver` | `MemorySaver`/`SqliteSaver` |
| 人机协同 | `interruptBefore()` + `updateState()` | `interrupt_before` + `update_state` |

## 我的思考

1. **"图+状态+可控循环"是不变量**——这个判断与之前记录的 [Harness 三层结构](18-Harness三层结构-知识约束反馈-2026-04-03.md) 一致：Harness 的约束层和反馈运行时层本质上就是 Edge 规则和 Loop 机制的上层封装。
2. **State 更新策略是被低估的设计决策**——并发写入覆盖字段的竞态问题在多 Agent 协作场景下会放大，与 [Agent 记忆覆盖问题](14-Agent记忆覆盖问题-三类故障四种解法-2026-04-03.md) 中的直接冲突故障同构。
3. **抽象能力是分水岭**——这与 Anthropic 的克制哲学（[构建高效 Agent](../AI方法论/06-Anthropic构建高效Agent-五种工作流与克制哲学-2026-04-03.md)）呼应：不是节点越多越好，而是职责边界越清晰越好。
4. **Loop 安全边界 = Token 预算 + 轮次上限**——单纯的轮次上限不够，Token 预算作为独立维度防御 Loop 放大攻击，这个思路值得在实际项目中采用。
