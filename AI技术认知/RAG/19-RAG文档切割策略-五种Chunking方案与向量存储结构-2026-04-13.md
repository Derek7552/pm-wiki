# RAG 文档切割策略：五种 Chunking 方案与向量存储结构

> 来源：京东面试官：RAG 里的文档你是怎么切割的？（微信公众号文章，原文文字整理）
> 提炼日期：2026-04-13
> 分类：AI方法论

## TL;DR

RAG 文档不能整篇存入向量库，必须切分为 chunk 分别向量化。每条记录包含向量（检索用）、原文（LLM 阅读用）、metadata（过滤溯源用）三部分缺一不可。切割粒度没有银弹，500～1000 token 是起点，更关键的是按文档类型选对策略：普通文本用固定大小+重叠兜底，结构化文档用语义边界切割，代码按函数切，追求高质量用父子切割。

## 核心观点

**1. 整篇存入是两个错误的叠加**

向量模型有输入长度限制（通常几百到几千 token），整篇文档根本塞不进去。即便模型支持超长输入，把整篇内容压缩成一个向量会让细节信息被"平均掉"——你想找"退款政策"，但向量里还混着"配送时效""积分规则"，检索到的是笼统文档而非精确段落。切分是 RAG 精度的基础前提，不是可选优化。

**2. 向量和原文各司其职，缺一不可**

向量是一串浮点数，LLM 看不懂，只有检索系统才用；原文是纯文字，没有向量就没法做相似度检索。向量负责"找到"，原文负责"阅读"。很多初学者误以为向量就是内容本身，这是根本性的认知错误。

**3. 粒度取舍的本质是精度与完整度的博弈**

Chunk 太大 → 语义笼统、检索噪音高、超模型输入限制；Chunk 太小 → 语义不完整、上下文丢失、LLM 无法理解碎片。没有全局最优解，只有针对文档类型的最优策略选择。

## 关键知识点

### 向量库每条记录的三元结构

| 组件 | 作用 | 类比 |
|------|------|------|
| 向量（embedding） | 语义空间位置，用于相似度检索 | 索引卡 |
| 原文（text） | LLM 实际阅读的内容，命中后塞进 prompt | 书页 |
| metadata | 来源文件名、页码、章节等，用于过滤和溯源 | 书签 |

查询完整链路：`用户问题 → 向量化 → 向量库相似度检索 → 取出原文 → 塞进 prompt → LLM 生成答案`

---

### 策略一：固定大小切割 + 重叠（Fixed Size + Overlap）

- **做法**：按固定 token 数切割，相邻 chunk 有重叠区域（如 chunk_size=500，overlap=100）
- **重叠的意义**：前一个 chunk 末尾与后一个 chunk 开头共享 100 个字符，保证边界处的语义至少完整出现在某个 chunk 里
- **适用**：纯文本、无明显结构的文档
- **优缺点**：实现最简单、chunk 大小可控；但可能在语义中间截断
- **定位**：兜底策略，几乎不单独使用于结构化内容

---

### 策略二：语义边界切割（Semantic/Structure Based Chunking）

- **做法**：按段落、句子、标题层级等天然断点切割，维护分隔符优先级列表（段落 → 句子 → 标点），逐级降级直到满足 chunk_size
- **标题层级变体**：对 Markdown/HTML 文档按标题层级切，每个 chunk 对应完整章节，metadata 自动带上标题路径（如"产品手册 > 退款政策 > 申请流程"）
- **适用**：段落分明的文章、有清晰标题结构的文档
- **优缺点**：语义完整，召回质量好；实现稍复杂，chunk 大小不均

---

### 策略三：特殊内容专项处理

**代码文件**：
- 以函数或类为单位切割（用 AST 解析工具识别边界，如 Python 的 `ast` 模块）
- 原因：函数是最小语义完整单元，中间截断则参数定义和返回值分离，单独 chunk 毫无意义

**表格**：
- 整块保留，转成 Markdown 格式，不能按行截断
- 原因：每一行数据依赖表头才有意义，"2 小时"单独来看无法理解，"响应时间：2 小时"才是完整信息

---

### 策略四：父子切割（Parent-Child Chunking）

- **核心思路**：检索时用小块（精准定位），返回时用大块（上下文完整）
- **存储方式**：同一内容存两份——小 chunk（约 200 token，用于向量检索）+ 大 chunk（约 1000 token，通过 ID 关联，供 LLM 阅读）
- **检索流程**：小 chunk 命中 → 根据 ID 取出关联的大 chunk → 大 chunk 塞进 prompt
- **类比**：用目录卡（小 chunk）定位章节，拿出来读的是完整那一章（大 chunk）
- **适用**：对召回质量要求高的场景
- **代价**：存储量翻倍，索引构建更复杂

---

### 策略选型对照表

| 策略 | 适用文档类型 | 优点 | 缺点 |
|------|-------------|------|------|
| 固定大小 + 重叠 | 纯文本、无明显结构 | 实现简单、chunk 大小可控 | 可能在语义中间截断 |
| 语义边界切割 | 段落分明的文章 | 语义完整，召回质量好 | 实现稍复杂，chunk 大小不均 |
| 标题层级切割 | Markdown、HTML 文档 | 天然语义独立，带结构 metadata | 依赖文档有清晰的标题结构 |
| 代码按函数切割 | 代码文件 | 保留代码逻辑完整性 | 需要 AST 解析，限定语言 |
| 父子切割 | 各类文档（追求高质量） | 检索精准 + 上下文完整两全 | 存储量翻倍，索引构建复杂 |

## 启发与思考

**与 notes/AI方法论/04 的关联**：`04-RAG-Chunk切分工程-三代方案演进` 从工程演进角度讲了三代方案，本文从实战面试角度系统梳理了五种策略，两者互补——前者有历史脉络，后者有选型判断。

**面试回答模板**：被问到 RAG Chunking 策略时，先说"为什么不能整篇存"（两个原因），再说"怎么切"（四种策略从简到复），最后说"怎么选"（根据文档类型）。落地说法："固定大小加重叠做兜底，结构化文档用语义边界切割，高质量要求用父子切割"。

**工程实践优先级**：快速起步用固定大小 + 重叠；有结构化文档就切换语义边界；对质量有明确要求时引入父子切割，存储成本翻倍但值得。

**产品视角**：在设计 RAG 类产品时，文档类型的多样性（合同 PDF、代码仓库、表格数据）意味着 Chunking 策略必须是可配置的，而不是一套固定流水线。
