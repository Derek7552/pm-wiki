# RAG 检索失败排查：四层管道逐层定位方法论

> 来源：用户提供文本（小米大模型二面题解析）
> 提炼日期：2026-04-03
> 分类：AI方法论

## 核心观点

1. **RAG 检索链路是多环节串联管道，任何一环出问题表现都一样——"问了答不上来"**：排查核心思路是沿数据流方向从前往后逐层定位，像查水管漏水一样一段一段检查。四层管道依次为：文档处理 → 向量化 → 检索召回 → 重排序/后处理。

2. **最致命也最常被忽视的问题往往在源头**：文档没正确入库（脚本跑挂/PDF 解析静默失败/中途异常只进一半）、Chunk 切分破坏语义完整性、元数据缺失导致静默过滤——这些"低级"问题在生产环境中比想象的常见得多。

3. **查询与文档的表述差异是最隐蔽的向量化问题**：口语化短问句 vs 正式书面长段落，语义相关但向量距离大。解法包括查询改写、HyDE（假想文档嵌入）、入库时生成假想问题。

## 关键知识点

### 四层排查管道

```
第1层：文档处理 → 第2层：向量化 → 第3层：检索召回 → 第4层：重排序/后处理
```

### 第1层：文档处理排查

| 检查点 | 排查方法 | 常见根因 |
|--------|---------|---------|
| **文档是否入库** | 向量库查文档总数，与预期对比 | 入库脚本挂了/PDF 解析静默跳过/中途异常 |
| **Chunk 切分质量** | 随机抽几个 Chunk 看内容连贯性 | 切太大(多主题混杂)/切太小(上下文丢失)/关键信息切断(表格劈两半) |
| **元数据完整性** | 检查元数据字段是否缺失或格式错误 | 时间戳为空导致按时间过滤时静默丢弃 |

### 第2层：向量化排查

| 检查点 | 排查方法 | 解决方案 |
|--------|---------|---------|
| **模型一致性** | 确认入库和查询用同一模型同一版本 | 模型升级后存量文档必须重新向量化 |
| **模型场景匹配** | 专业领域检查通用模型表现 | 换 bge/m3e 系列，或用领域数据 fine-tune |
| **查询-文档表述差异** | 对比口语查询和书面文档的向量距离 | Query Rewriting / HyDE / 假想问题生成 |

**三种缓解表述差异的策略**：

| 策略 | 原理 | 匹配方式 |
|------|------|---------|
| **Query Rewriting** | LLM 把口语查询改写为文档风格 | 改写问题 → 文档 |
| **HyDE** | LLM 生成假想答案文档，用其向量检索 | 假想文档 → 文档 |
| **假想问题生成** | 入库时对每个 Chunk 生成假想问题 | 问题 → 假想问题 |

### 第3层：检索召回排查

逐个放宽限制，定位哪个参数卡住了召回：

| 参数 | 排查方法 | 判断标准 |
|------|---------|---------|
| **相似度阈值** | 临时调低到 0.5-0.6 | 能检索到 → 阈值过严 |
| **TopK** | 调到 20 看相关内容是否在后面位置 | 排第5但TopK=3 → 排序问题非检索问题 |
| **元数据过滤** | 临时去掉所有过滤做纯向量检索 | 能找到 → 过滤条件排除了它 |
| **索引参数** | 对比精确检索(暴力遍历) vs 索引检索 | 精确能找到索引找不到 → HNSW ef_search / IVF nprobe 过小 |

### 第4层：重排序/后处理排查

| 检查点 | 排查方法 |
|--------|---------|
| **Reranker 误判** | 对比重排序前后结果列表，看排名变化 |
| **去重过于激进** | 同文档不同段落的多个相关 Chunk 被只保留一个 |
| **上下文截断** | 检索到10个但只送3个给LLM，答案在第7个——检索到了但LLM没看到 |

### 排查工具三件套

1. **Trace 系统**（LangSmith/LangFuse）：记录完整链路——原始查询→改写查询→TopK结果(含分数和内容)→重排序结果→最终送入LLM的上下文
2. **对比实验**：每次只改一个变量（换 Embedding/换切分策略），精确定位
3. **人工标注测试集**：典型查询+标注应检索到的文档，跑召回率和准确率做客观对比

## 数据与案例

- **元数据静默过滤案例**：用户问"2024年销售数据"，文档时间戳字段为空 → 被时间过滤静默丢弃
- **Embedding 模型不一致案例**：入库用 ada-002，升级后查询用 bge-large-zh，存量未重新向量化 → 两个向量空间，相似度无意义
- **表述差异案例**：用户问"怎么退货"，文档写"退货流程：客户需在收到商品后7日内联系客服" → 口语vs书面，向量距离大
- **上下文截断案例**：检索到10个 Chunk，窗口限制只送3个，答案在第7个 → 用户以为"检索不到"

## 启发与思考

1. **与 RAG Chunk 切分（notes 04）和权限控制（notes 08）构成 RAG 工程三部曲**：04 讲"怎么切"，08 讲"谁能看"，本篇讲"出问题怎么排查"。三篇合在一起覆盖了 RAG 系统从建设到运维的完整工程知识。

2. **对 AI 客服知识库运维的直接价值**：沃丰的 AI 客服上线后，"为什么问不出来"是客户最常见的投诉。本文的四层排查管道可以直接作为客服知识库的排查 SOP——从文档入库到向量化到检索参数到后处理，逐层有明确检查点。

3. **Trace 系统是生产级 RAG 的基础设施**：没有 Trace 等于盲排查。每次检索记录完整链路（原始查询→改写→TopK→重排序→送入 LLM 的上下文），排查时直接看数据而非猜。

4. **面试答题结构**：四层管道 + 每层的检查点和验证方法 + 工具三件套，这个"分层+具体+工具"的结构是技术排查类面试题的通用答题框架。

## 原文精华

> 排查的核心思路是沿着数据流方向，从前往后逐层定位。像查水管漏水一样，从水源开始一段一段检查，哪段没水就是哪段出了问题。

> 很多时候根因出奇地简单——文档压根没有正确入库。这些情况在生产环境中比想象的常见得多。

> 检索到了 10 个相关 Chunk，但因为上下文窗口限制只送了前 3 个进去，真正回答问题的内容在第 7 个 Chunk 里——用户觉得"检索不到"，其实是检索到了但 LLM 没看到。

> 整体原则就是每层都有明确的检查点和验证方法，靠数据定位而不是靠猜。
