# RAG 扛 800 QPS 服务化压测：分段瓶颈定位、Token 维度限流、多级缓存三件套

> 来源：AI 落地面试深水区 · 第 11 课 · 服务化工程篇（字节跳动 AI 平台架构面真题）
> 提炼日期：2026-05-15
> 分类：AI技术认知 / RAG

## TL;DR

RAG 服务从 200 QPS 顺到 800 QPS 崩，瓶颈几乎一定不是"机器不够"而是 LLM 段——四段链路（embedding / 召回 / rerank / LLM）的吞吐能力差两个数量级。扛量的正确姿势是**分段定位瓶颈 + Token 维度限流挡在 LLM Gateway 前 + 按知识版本失效的多级缓存**，而不是堆 GPU。一个真实改造案例：GPU 没加，P95 从 30s 降到 1.8s，超时率从 12% 降到 0.3%，单次成本降 35%。

## 核心观点

### 观点 1：RAG 扛量不是扩容问题，是分段瓶颈识别问题

最容易翻车的回答是"加机器、上 GPU、换更大的模型"。这种回答的根本错误是**没把链路分段**——RAG 是四段串行管道，每段的吞吐能力差两个数量级以上。一刀切扩容会出现"最贵的 GPU 闲着、瓶颈段还是没救"的尴尬局面。真正的破题动作是先画出四段各自的 QPS 上限，再找出最先打满的那一段（几乎一定是 LLM），针对性优化。

### 观点 2：限流单位必须从"请求数（RPS）"切换到"token"

入口 nginx 看到的是 1 个请求，但到 LLM 那里这 1 个请求可能烧 200 token、也可能烧 5000 token——成本差 25 倍。按 RPS 限流的代价是：限严了便宜请求白白被拒、限松了贵请求照样烧爆 GPU。正确做法是把限流挪到 LLM Gateway 前面，按 input token（tokenizer 实算）+ output token（预估值）的总和做配额，按用户 × 模型维度配 TPM / RPM 双闸口。

### 观点 3：缓存命中率不是越高越好，要按"风险等级"分层启用

三种 RAG 缓存的安全性差一个数量级：query→embedding 缓存零风险（embedding 模型版本不变就一直有效）；检索结果缓存中等风险（知识库一更新就要失效）；query→answer 缓存高风险（旧答案是定时炸弹）。优先顺序应该是**先零风险、再可控风险、最后高风险**，而不是按命中率高低决定上线顺序。

## 关键知识点

### RAG 四段链路的吞吐能力分布

链路结构：**embedding → 向量召回 → rerank → LLM 生成**

| 段 | 资源 | 单实例吞吐 | 相对成本 |
|---|------|-----------|---------|
| ① embedding（文字转向量） | CPU | 几千 QPS / 节点 | 1x（基准） |
| ② 向量召回（找近邻，Faiss/Milvus） | 单机内存 | 几百~千 QPS | 5~10x |
| ③ rerank（cross-encoder 候选重排） | 单卡 GPU | 几十 QPS | 30~50x |
| ④ LLM 生成（写答案） | 单卡 GPU | 个位数 QPS（output 1k token ≈ 1~3 秒） | 100x+ |

**结论**：晚上活动崩的几乎一定是 LLM 段，先去看 LLM 排队队列长度，别先翻 nginx 日志。

### Token-based 限流的工程实现

**为什么必须按 token 限**：
- 短 query（input 500 + output 200 ≈ 700 token）vs 长 query（input 4000 + output 2000 ≈ 6000 token），实际 GPU 代价差近 10 倍
- 按 RPS 看不到这个差异，限流策略要么过严要么过松

**落地方案**（最简版本）：

```
位置：LLM Gateway 层（自研 / Portkey / LiteLLM）
存储：Redis 令牌桶
维度：user_id × model_name 二维配额
input token：tokenizer 精确计算
output token：预估上限（如 max_tokens=800）
扣额时机：调用 LLM 之前
超额处理：429 直接拒掉 或 进入排队队列
双闸口：TPM（每分钟 token 数）+ RPM（每分钟请求数）
```

### Continuous Batching 的吞吐 / 延迟权衡

**机制**：多条请求塞同一个 GPU 一起跑（vLLM 等主流推理框架的默认机制，2026-05 时点）。
**收益**：吞吐翻 3~5 倍。
**代价**：你的请求必须等"队友"——同 batch 里只要有人写超长答案，整条 batch 都要等他写完，你的延迟被拖到他的水平。

**场景选择矩阵**：

| 场景 | 是否开 continuous batching | 原因 |
|------|--------------------------|------|
| 在线客服 | 开，但盯紧 P99 延迟 | 吞吐重要，但用户感知延迟不能爆 |
| 离线批处理 | 无脑开 | 吞吐唯一目标，延迟无所谓 |
| 强 SLA（金融/医疗等延迟可预测场景） | 反而用固定 batch | 长尾延迟不可接受 |

**长尾被放大的诊断方法**：
- 分桶看 P50 / P95 / P99 延迟随 batch size 的变化
- 同时把 input / output token 长度分布画出来
- 如果 **P99 / P50 > 5 倍**，说明长尾被 batch 放大
- 修复路径：① 用 `max_tokens` 强约束输出长度 ② 把长输入用户分到独立 pool

### 三种缓存的安全性与启用优先级

| 缓存类型 | 命中省什么 | 失效条件 | 风险 | 启用顺序 |
|---------|-----------|---------|------|---------|
| ① query→embedding | 一次 embedding 推理 | embedding 模型版本变 | 零风险 | 第一优先 |
| ② 检索结果（query→docs） | 召回 + rerank 两段 | 知识库 (KB) 版本变 | 中等：旧 KB 命中导致答案过期 | 第二优先（按 KB 版本失效） |
| ③ query→answer | 整条链路 | KB 变 + query 语义相似度 | 高：旧答案误命中是定时炸弹 | 仅在 FAQ 类白名单场景启用 |

**query→answer 缓存的三道防线**（启用前必须配齐）：

1. **缓存 key 含 KB 版本号**：ETL 跑完 `KB_VER += 1`，旧 key 自然失效，无需手动 invalidate
2. **TTL 按知识稳定性分级**：FAQ 类 24h，政策类 1h，实时数据类 0（不缓存）
3. **命中后做轻量校验**：把当前 query 重新 embedding，与缓存中原 query embedding 做 cosine 相似度，**< 0.95 视为不同问法，不命中**

**反向操作建议**：先列"什么 query 不能缓存"的负面清单（含时间、含个人化、含计算），剩下的才进缓存——而不是先缓存了再补漏。

## 数据与案例

### 字节 AI 平台架构面真题（面试现场）

候选人答"加机器、换大显卡"，面试官追问"你算过单卡 token/s 折成 QPS 是多少吗？"候选人卡住——这道题表面问扩容，实际考的是能否区分"入口 RPS"和"LLM token 消耗"两个完全不同维度的容量。

### 电商客服 RAG 800 QPS 扛量改造（完整案例）

**场景**：白天 200 QPS 平稳，晚上整点活动推送瞬间冲到 800 QPS，LLM 段排队 30 秒，前端大面积超时。

**改造四步法**：

| 步骤 | 动作 | 关键设计 | 效果 |
|------|------|---------|------|
| 1 | embedding 缓存先上 | key = query 文本 hash；embedding 模型版本变才失效 | 热门问题命中率 60%+，embedding 段 CPU 占用降一半 |
| 2 | 检索结果缓存按 KB 版本失效 | key = (q_emb 量化, KB_VER)；KB ETL 跑完版本号 +1，旧 key 自然过期 | 向量库 + rerank QPS 降 35%，rerank GPU 不再排队 |
| 3 | token 配额挡在 LLM 前 | Redis 令牌桶，user_id × model 维度；input tokenizer 实算 + output 预估上限 800；超额 429 | LLM 队列长度从 200+ 降到 < 20，雪崩消失 |
| 4 | query→answer 缓存有限开 | 仅 FAQ 类白名单 intent，TTL 1h，命中前 cosine ≥ 0.95 二次校验 | FAQ 流量整链路命中率 45%，活动峰值 LLM 调用量降 35% |

**改造前后对比**（数据来源：内部回归集，2026-05）：

| 指标 | 改造前 | 改造后 | 变化 |
|------|-------|-------|------|
| 峰值 QPS | 800 | 800 | 不变 |
| LLM 实际调用 QPS | 800 | 520 | -35% |
| P95 延迟 | 30s | 1.8s | -94% |
| 超时率 | 12% | 0.3% | -97.5% |
| 单次请求成本 | 基准 | -35% | -35% |

**最关键的事**：**GPU 没加**，靠分段缓存 + token 限流救回来。

### 单卡 LLM 吞吐量级参考

- 单卡 LLM 生成 output 1k token 约 **1~3 秒**
- 折合单卡 QPS：**个位数到十几个**（取决于模型大小、output 长度、是否开 batching）
- 因此当入口 QPS 上千时，必须靠**限流 + 缓存 + batching** 把 LLM 实际调用量压下来

### 可落地的工程动作

1. **画一张分段瓶颈图**：把当前业务的 embedding / 召回 / rerank / LLM 单实例 QPS 写下来贴在墙上——任何 RAG 性能讨论的第一张图就是这个
2. **检查现有限流位置**：如果限流还在 nginx 入口、单位是 RPS，立刻规划迁移到 LLM Gateway 前 + 改成 token 维度
3. **建立缓存优先级矩阵**：embedding 缓存（零风险）→ 检索结果缓存（KB 版本号控制）→ FAQ 白名单的 answer 缓存，按此顺序灰度
4. **监控 P99/P50 比值**：如果 > 5 倍，说明 continuous batching 长尾被放大，需要 max_tokens 约束或独立长 query pool

### 开放性问题

- **多模型混合部署时的 token 配额怎么分配**：同一用户调用 Sonnet 和 Haiku 的 token 应不应该统一计费？还是按"成本权重"折算后再限？
- **缓存命中后的"鲜度感知"**：query→answer 缓存命中时，是否要在响应中带上 `cached_at` / `kb_version` 字段，让前端可以在 UI 上提示"该答案基于 X 日前数据"？
- **限流 vs 排队的产品决策**：超额时直接 429 还是进入排队（带预估等待时间）？两种体验对客服场景的转化率影响如何度量？

## 原文精华

> RAG 链路分四段，最贵的 LLM 段比最便宜的 embedding 段贵 100 倍以上。

> 入口看到的 1 个请求，到 LLM 那里可能要烧 2000 个 output token——按请求数限流根本对不上 GPU 的真实代价。

> continuous batching 就是多条请求塞同一个 GPU 一起跑，吞吐翻 3~5 倍——代价是你的请求要等队友。

> 同 batch 里只要有一个人写超长答案，整条 batch 都要等他写完，你的延迟就被拉到他的水平。

> 缓存命中率不是越高越好，要分类型看风险——有的命中零风险，有的命中就是踩雷。

> 先把"什么 query 不能缓存"的负面清单列出来（含时间、含个人化、含计算），剩下的才进缓存——而不是先缓存了再补漏。

> RAG 扛 QPS 不靠堆 GPU，靠分段定位瓶颈、token 维度限流挡在 LLM 前、按知识版本失效的多级缓存——三件套先到位，再谈扩容。

## 面试速查卡

**先说**：RAG 链路四段瓶颈差两个数量级，崩的几乎一定是 LLM；不分段堆机器是浪费。

**再说**：限流单位用 token 不用 RPS，挡在 LLM Gateway 前；continuous batching 加吞吐但拉长尾，延迟敏感场景必须监控 P99。

**最后补**：query→embedding 缓存先上（零风险）、检索结果缓存按 KB 版本失效、query→answer 缓存只在 FAQ 白名单开。

**别再踩的坑**：
- 加机器加 GPU 一刀切，不分段 — 最贵的 GPU 闲着瓶颈段照崩
- nginx RPS 限流当 LLM 限流 — 长 query 把 GPU 队列堆爆
- 全场景开 continuous batching 不看 P99 — 长尾延迟翻 4 倍
- query→answer 缓存所有 query 全开 — KB 更新后老答案误命中

**判断清单**：
- [ ] 是否分段画出 embedding / 召回 / rerank / LLM 各自的 QPS 上限？
- [ ] 限流单位是 token 还是 RPS？
- [ ] 限流位置是 nginx 入口还是 LLM Gateway 前？
- [ ] continuous batching 的 P99 / P50 是否监控了？
- [ ] 缓存 key 是否含 KB 版本号？
- [ ] query→answer 缓存是否限定在 FAQ 白名单？
