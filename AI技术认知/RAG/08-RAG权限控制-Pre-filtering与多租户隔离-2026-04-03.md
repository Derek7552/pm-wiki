# 企业级 RAG 权限控制：Pre-filtering、多租户隔离与动态权限

> 来源：吴师兄公众号文章（直接粘贴）
> 提炼日期：2026-04-03
> 分类：AI方法论

## 核心观点

1. **权限控制是数据访问层的问题，不是展示层的问题**：后置过滤（Post-filtering）看似能用，但数据访问已经发生——向量数据库已经检索出了无权文档，只是 UI 没展示。这在安全审计中不可接受，违反最小权限原则。正确做法是 Pre-filtering：在向量相似度计算之前，先用权限条件过滤掉无权文档。

2. **多租户隔离有三种粒度，选型取决于合规要求**：Metadata 过滤（最轻量但最不安全）→ Partition 隔离（中间方案）→ Collection 隔离（最彻底但成本最高）。生产环境推荐 Partition + Metadata 双重防护。

3. **动态权限变更是真正的难点**：权限数据不应写死在 Chunk 元数据里，应存在关系型数据库中检索时实时 JOIN，配合 Redis 短时缓存 + 敏感场景主动失效。

## 关键知识点

### 后置过滤的三个根本问题

| 问题 | 说明 | 影响 |
|------|------|------|
| 数据访问已发生 | 向量检索本身就是数据读取，不管是否展示 | 违反最小权限原则，过不了安全审计 |
| Top-K 名额被占用 | 用户只能访问 40% 文档时，Top-5 实际可用只剩 Top-2 | 检索质量严重下降 |
| 多租户信息泄漏 | 不同租户文档共享向量空间，检索时相互「可见」 | 合规场景不可接受 |

### Pre-filtering 实现架构

```
用户请求 → JWT Token 服务端验证（不信任前端参数）
        → 解析用户权限（dept/role/classification）
        → 构造 metadata filter 条件
        → 只在有权文档子集中做向量检索
        → 返回结果
```

### 四种权限元数据模型

| 模型 | 字段示例 | 适用场景 |
|------|---------|---------|
| 部门级 | `dept_id = ["finance", "hr"]` | 按部门划分知识库 |
| 角色级 | `required_role_level = 3` | 按职级分层访问 |
| 文档密级 | `classification = "confidential"` | 政府/军工/金融 |
| 用户白名单 | `allowed_user_ids = ["u001"]` | 精细化个人级控制 |

### 向量数据库 Pre-filtering 能力对比

| 向量数据库 | Filter 支持 | Pre-filtering 效率 |
|-----------|------------|-------------------|
| **Milvus** | 完整，复杂布尔表达式 | 高（索引级过滤） |
| **Qdrant** | 完整，payload filter | 高（与向量搜索完全集成） |
| **Weaviate** | 支持，where filter | 中高 |
| **Chroma** | 基础，仅 eq/in | 中 |
| **FAISS** | 不原生支持 | 低（需外部实现） |

### 多租户三种隔离粒度

| 策略 | 物理隔离 | 安全性 | 成本 | 适用场景 |
|------|---------|--------|------|---------|
| Metadata 过滤 | 否 | 低（bug 可暴露全量数据） | 低 | 内部低敏感知识库 |
| Partition 隔离 | 部分 | 中 | 中 | 中等安全的 SaaS |
| Collection 隔离 | 完全 | 高 | 高 | 金融/医疗强合规 |

**推荐方案**：Partition + Metadata 双重防护（纵深防御）

### 动态权限变更设计原则
- **不要**：把权限写死在 Chunk 元数据里（部门重组时要更新数千条记录）
- **要**：权限关系存关系型数据库，Chunk 只存 doc_id，检索时实时 JOIN
- **缓存优化**：Redis 短时缓存（30-60s）accessible_doc_ids
- **离职即时生效**：权限服务触发离职事件时，同步删除该用户所有缓存条目

### 面试回答框架（3 分钟）
1. **30s**：后置过滤三个问题（审计违规 + Top-K 占用 + 信息泄漏）
2. **1min**：Pre-filtering 实现（Chunk 权限元数据 + 服务端 JWT 验证 + 缓存 TTL）
3. **1min**：多租户三种隔离粒度 + 项目选型理由
4. **30s 加分**：动态权限（关系型数据库实时 JOIN + 离职主动失效缓存）

## 数据与案例

- **京东面试真题**：面试官追问后置过滤的安全审计问题，「你在错误的地方做了正确的事」
- **金融保险实战**：5000 份合同文档，按保险公司分租户，Qdrant Partition 隔离 + Metadata 双重过滤
- **缓存 TTL 权衡**：60 秒 TTL 意味着权限变更最多 60 秒延迟，离职场景需主动失效

## 启发与思考

1. **与之前 RAG Chunk 切分笔记的互补**：Chunk 切分解决「检索质量」问题，权限控制解决「检索安全」问题。两者共同构成企业级 RAG 的两大基座。Demo 级 RAG 只需要前者，生产级 RAG 两者缺一不可。

2. **对沃丰科技 AI 客服的直接关联**：沃丰做多租户 SaaS 客服平台，不同企业客户的知识库必须严格隔离。这篇文章的多租户隔离方案可以直接映射到沃丰的架构需求——面试时可作为技术深度的展示。

3. **「在错误的地方做了正确的事」是产品设计的常见陷阱**：不只是技术问题。PM 在设计功能时也常犯类似错误——在展示层做了应该在数据层做的事，比如用前端校验代替后端验证，用 UI 隐藏代替真正的权限控制。

4. **纵深防御（Defense in Depth）思维**：Partition + Metadata 双重防护的设计，本质上是安全领域的纵深防御原则——任何单一防线都可能失败，所以叠加多层。

## 原文精华

> 权限控制不是展示层的问题，是数据访问层的问题。你在错误的地方做了正确的事。

> 向量检索是一次数据读取操作。不管最后有没有展示给用户，被检索出来的文档已经被系统访问了。

> "先检索后过滤"是最容易踩的坑——逻辑直觉、实现简单，但安全模型是错的。

> 权限控制是企业级 RAG 和 demo 级 RAG 之间最明显的差距之一。
