# RAG Rerank 延迟优化：候选规模优先、分层精排与尾延迟控制

## 核心判断

Rerank 的难点不是"怎么让模型再快一点"，而是"有限的几十毫秒，到底该花在哪些样本上"。优化顺序：**先动候选规模，再动单条打分成本**，最后才考虑换模型。

---

## 一、为什么 Rerank 总是卡在这

Rerank 贵的原因是结构性的：

1. **成本随候选数线性放大**：召回 200 条 = 把同一件事做 200 次，单条多 1ms 乘上去就很可观
2. **难以离线化**：需要 query 与 doc 的现场交互（标题是否命中、query 附近段落是否答到点），等请求进来才能算
3. **链路末端、无犯错余地**：前面召回/过滤/特征拉取已消耗大半预算，留给 rerank 的时间天然紧张

真线上的组合拳通常是：候选太多 + 文本太长 + 特征组装夹着远程调用 + 高峰期 batch 抖动 → P99 立刻难看。

---

## 二、延迟公式与第一刀该砍哪里

```
总耗时 ≈ 候选数 × 单条打分成本 + 固定开销
```

**数值对比**（180 候选，单条 1.5ms，固定开销 20ms）：

| 优化方式 | 结果 |
|---------|------|
| 无优化（基线） | 180×1.5+20 = **290ms** |
| 模型提速 20%（1.2ms） | 180×1.2+20 = **236ms** |
| 候选砍半（90 条，模型不变） | 90×1.5+20 = **155ms** |

**结论**：候选数减半比模型提速 20% 节省的时间多近一倍。

### 减少候选的常用手段

- 多路召回先去重
- 前置轻量 pre-rank 过滤掉明显不满足条件的候选
- 给不同召回通道设配额，防止某一路撑爆候选池
- **按 query 意图动态分配预算**：
  - 窄意图 query（如"官网登录入口"）→ 候选少，预算紧
  - 发散 query（如"推荐系统效果不好怎么办"）→ 候选多，差异细，值得多给预算

---

## 三、模型之外才是真正的瓶颈

真实 Rerank 流程远不是"query + doc → 模型 → 分数"，前面还有：

- 文本清洗、字段拼接、长文本截断
- 统计特征读取、用户侧实时特征拉取
- 标签/类目/质量分补全

混进几个慢 IO 或远程 RPC，模型本身只跑 20ms，整个 rerank 也可能拖到 100ms+。

**工程优先级**：
1. 能离线算的特征提前离线，文档侧静态信息预展开
2. 减少 RPC 次数，能并发的 IO 并发跑
3. 控制文本输入长度

### 文本截断策略

交互型 rerank 模型的输入长度几乎等于成本，优先级：

```
标题 > 摘要 > 正文前几个高信息密度段落 > query 附近命中窗口
```

不是所有文本都同样值钱，决定排序的往往只是最能表达主题的那一小段。

---

## 四、分层 Rerank

单层方案的死局：模型轻了效果不够，模型重了延迟爆掉。

**分层结构**：
- 第一层：轻模型/规则，处理全部候选，快速淘汰后半段
- 第二层：中等模型，处理第一层筛选后的候选
- 第三层（可选）：重模型，只处理最终竞争 TopN 的候选

**核心逻辑**：收益分布不均匀。后半段大量候选大概率进不了最终结果，给它们全上重模型是算力浪费。越接近 TopN 的候选，越值得认真审。

---

## 五、权衡时该看什么指标

**不要只盯 mean（平均延迟）**：平均 80ms 没问题，不代表体验稳定。P95 180ms、P99 300ms，用户感知是"偶尔卡得很狠"。

**关注优先级**：
1. **TopN 质量**：前 3~10 条有没有被伤到（用户真正关注的区域）
2. **P95/P99**：尾延迟有没有变丑
3. **退化场景**：高峰期、长 query、长文档场景

### 减法该减哪里

| 优化动作 | 评估 |
|---------|------|
| 候选从 150→100，Top10 基本不动，长尾变化 | **值得做**：砍的是边缘候选，用户感知弱 |
| 强交互模型换轻模型，前 3~5 条开始乱 | **不值得**：用户只看第一页前几条 |

**原则**：优先动长尾，不动头部。第 80~120 名谁前谁后没人在意；前 10 名一旦晃，点击率、满意度、生成效果都跟着受影响。

---

## 六、线上防御机制

Rerank 在线上不能硬撑，高峰流量、下游特征服务抖动、长 query 都会让尾延迟飙升。**有没有退路比平均值好不好看更重要**。

基础防御清单（土但有效）：

| 机制 | 作用 |
|------|------|
| 硬超时 | 超时直接截断，防止单请求拖垮链路 |
| 回退排序 | 超时或异常时降级到简单规则排序 |
| 高负载时降配 | 少看候选 / 少喂文本（动态降级） |
| 热点 query 缓存 | 高频 query 结果缓存，省去重复计算 |

**核心原则**：稍微弱一点但能按时回来，比更准但老超时更实用。

---

## 核心结论

Rerank 踩在一个别扭的位置：对结果影响大、计算贵、还是实时链路的一部分。

> **怕的不是做减法，怕的是���预算减错了地方。**

优化路径：
1. 先收候选规模（pre-rank + 去重 + 通道配额）
2. 再清模型外的脏活（特征离线化 + 减 RPC + 文本截断）
3. 分层精排把重计算留在头部
4. 补齐线上防御（超时 + 回退 + 缓存）
5. 最后才考虑换更快的模型
