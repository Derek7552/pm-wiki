# Harness Engineering：AI Agent 系统设计方法论

> 来源：AI Engineer Guide - 《Harness Engineering 到底是什么？》
> 提炼日期：2026-04-13
> 分类：AI应用案例

## TL;DR

Harness Engineering 是 2026 年 4 月迅速走红的 AI Agent 工程概念：模型之外的一切基础设施（系统提示词、工具调用、沙箱、约束机制、反馈回路）统称 Harness，Agent = Model + Harness。核心洞察：决定 Agent 表现的天花板不是模型智能水平，而是 Harness 的基础设施质量——同一模型只换工具调用格式，编码基准分数从 6.7% 跳到 68.3%。实践入口：P0 先做 AGENTS.md + 自定义 Linter + 知识仓库化。

## 核心观点

### 1. 模型决定上限，Harness 决定底线——瓶颈不在智能而在基础设施

**论点**：同一个模型，配套不同的 Harness，性能差异可以达到 10 倍以上。多家一线团队的实验数据一致指向：Agent 表现的瓶颈是基础设施质量，而非模型智能水平。

**推理逻辑**：
- 模型本质是"文本进文本出"的函数，它不能记忆历史、不能执行代码、不能自我验证
- 这些能力缺口必须由 Harness 补齐，补的质量直接决定 Agent 的实际表现
- 团队遇到 Agent 表现差的第一反应是"换更强的模型"，但问题往往在 Harness

**关键支撑**：
- Can.ac 实验：同一模型只换文件编辑接口的调用方式，编码基准分数从 6.7% 跳到 68.3%（约 10 倍差距）
- LangChain 实验：只优化 Agent 运行环境（文档组织 + 验证回路 + 追踪系统），Terminal Bench 2.0 排名从第 30 升至第 5，得分从 52.8% 升至 66.5%；模型未变，Harness 变了
- Model-Harness 耦合陷阱：Claude Code、Codex 等产品的模型和 Harness 是一起训练的，Opus 在 Claude Code Harness 下的得分远低于在其他 Harness 下的得分 → "最好的 Harness 不一定是模型默认的那个"

### 2. 上下文不是越多越好——40% 阈值是 Agent 的质量红线

**论点**：Agent 的上下文利用率超过约 40% 时，输出质量开始显著下降（幻觉增多、兜圈子、代码质量差）。正确的做法不是压缩塞更多信息，而是让 Agent 始终运行在"干净、相关"的上下文里。

**推理逻辑**：
- 168K token 的上下文窗口，用到 40% 左右就进入"Dumb Zone"
- Anthropic 发现 Sonnet 4.5 会产生"上下文焦虑"：上下文快填满时倾向于提前收工，哪怕任务未完成
- 光靠压缩不够，Anthropic 的解法是 context resets：直接清空上下文，通过结构化交接文档保留关键状态

**关键支撑**：
- Dex Horthy 观察到的 Smart Zone（0-40%）vs Dumb Zone（>40%）现象
- 一线团队的实践都围绕"渐进式披露"和"分层管理"来控制上下文体积
- Anthropic 的 context resets 策略：类似"重启进程解决内存泄漏"，而非无限加内存

### 3. Harness 是嵌套的三层体系——不同层解决不同类型的问题

**论点**：Prompt Engineering、Context Engineering、Harness Engineering 不是并列关系，而是嵌套关系（Prompt ⊂ Context ⊂ Harness），分别解决"表达""信息""执行"三个完全不同层次的问题。在长链路、低容错的真实商业场景中，Harness 层才是决定成败的关键。

**推理逻辑**：
- 简单任务：提示词写清楚就行
- 依赖外部知识的任务：上下文管理是关键
- 长链路、可执行、低容错场景：必须有完整的 Harness 来保证持续正确和故障恢复

**三层对比**：

| 层级 | 解决的核心问题 | 关注点 | 典型工作 |
|------|---------|------|---------|
| Prompt Engineering | 表达——怎么写好指令 | 塑造局部概率空间 | 系统提示词、Few-shot、CoT |
| Context Engineering | 信息——给 Agent 看什么 | 正确时机拿到必要信息 | RAG、记忆注入、Token 优化 |
| Harness Engineering | 执行——系统怎么防崩、量化、持续运转 | 长链路持续正确、故障恢复 | 沙箱、约束、熵管理、反馈回路 |

## 关键知识点

### Harness 的定义与组件

**定义**：模型之外的一切系统组件。
- 系统提示词、工具调用接口
- 文件系统（持久化）
- Bash 执行环境（通用工具）
- 沙箱环境（安全隔离）
- 记忆机制（跨会话积累）
- 上下文压缩（对抗衰减）
- 约束机制、反馈回路、编排逻辑

**"模型做不到 → Harness 来补"对应表**：

| 模型做不到 | Harness 如何补 | 核心组件 |
|-----------|--------------|---------|
| 记住多轮对话历史 | 维护历史，每次请求时拼进上下文 | 记忆系统 |
| 执行代码、跑命令 | Bash + 代码执行环境 | 通用执行环境 |
| 获取实时信息 | Web Search、MCP 工具 | 外部知识获取 |
| 操作文件和环境 | 文件系统抽象 + Git 版本控制 | 文件系统 |
| 知道自己做对了没有 | 沙箱 + 测试工具 + 浏览器自动化 | 验证闭环 |
| 在长任务中保持连贯 | 上下文压缩、记忆文件、进度追踪 | 上下文管理 |

### Harness 六层架构

一个成熟 Harness 的完整层次结构，从"定义边界"到"兜底恢复"的闭环：

| 层级 | 名称 | 解决什么问题 | 关键设计 |
|------|------|------------|---------|
| L1 | 信息边界层 | Agent 该知道/不该知道什么 | 定义角色与目标，裁剪无关信息，结构化任务状态 |
| L2 | 工具系统层 | Agent 怎么跟外部世界交互 | 工具选拔、调用时机、结果提炼与反馈 |
| L3 | 执行编排层 | 多步骤任务怎么串起来 | 引导模型走完"理解→判断→分析→生成→检查"完整轨道 |
| L4 | 记忆与状态层 | 长任务中间结果怎么管 | 独立管理任务状态、中间产物和长期记忆 |
| L5 | 评估与观测层 | Agent 怎么知道自己做对了没有 | 建立独立于生成过程的验证机制 |
| L6 | 约束、校验与恢复层 | 出错了怎么办 | 预设规则拦截错误，失败时提供重试或回滚 |

**实践建议**：不要一开始搭齐六层。优先做 **L1（信息边界）和 L6（约束与恢复）**，投入产出比最高——L1 决定 Agent 知道该干什么，L6 决定搞砸了能不能拉回来。

### 搭建 Harness 的行动清单（P0/P1/P2）

**P0：立即可做，效果最显著**

| 行动 | 为什么 | 参考实践 |
|------|--------|---------|
| 创建 AGENTS.md 并持续维护 | Agent 每次启动自动加载，犯错就更新，形成反馈循环 | Hashimoto：每一行对应一个历史失败案例 |
| 构建自定义 Linter + 修复指令 | 错误消息直接告诉 Agent 怎么改，纠错同时在"教" | OpenAI：Linter 报错自带修复方法 |
| 把团队知识放进仓库 | 写在 Slack/Wiki/Docs 的知识对 Agent 等于不存在 | OpenAI：仓库为唯一事实源 |

> ⚠️ 误区：AGENTS.md 不要当"超级 System Prompt"，不要把所有规则塞一个文件。正确做法：AGENTS.md 只当目录用（约 100 行），详细规则放子文档按需加载。

**P1：P0 完成后考虑**

| 行动 | 为什么 | 参考实践 |
|------|--------|---------|
| 分层管理上下文 | 渐进式披露，不要一次性塞满 | OpenAI AGENTS.md 当目录 |
| 建立进度文件和功能列表 | JSON 格式追踪功能状态，Agent 不太会乱改结构化数据 | Anthropic 两阶段 Agent |
| 给 Agent 端到端验证能力 | 浏览器自动化让 Agent 能像用户一样验证功能 | Anthropic 用 Playwright/Puppeteer MCP |
| 控制上下文利用率 | 尽量不超过 40%，增量执行 | Dex Horthy 的 Smart/Dumb Zone |

**P2：有余力再考虑**

| 行动 | 为什么 | 参考实践 |
|------|--------|---------|
| Agent 专业化分工 | 每个 Agent 携带更少无关信息，留在 Smart Zone | Carlini 的去重/优化/文档 Agent |
| 定期垃圾回收 | 确保清理速度跟得上生成速度 | OpenAI 后台清理 Agent |
| 可观测性集成 | 把"性能优化"从玄学变成可度量 | OpenAI 接入 Chrome DevTools |

### Harness 成熟度五阶段

| 阶段 | 特征 | 工程师角色 |
|------|------|---------|
| Level 0：无 Harness | 直接给 Agent prompt，无结构化约束 | 手动写代码 + 偶尔用 AI |
| Level 1：基础约束 | AGENTS.md + 基础 Linter + 手动测试 | 主要写代码，AI 辅助 |
| Level 2：反馈回路 | CI/CD 集成 + 自动化测试 + 进度追踪 | 规划 + 审查为主 |
| Level 3：专业化 Agent | 多 Agent 分工 + 分层上下文 + 持久化记忆 | 环境设计 + 管理为主 |
| Level 4：自治循环 | 无人值守并行化 + 自动化熵管理 + 自修复 | 架构师 + 质量把关者 |

### 当前领域未解问题（面试可用）

| 问题 | 现状 |
|------|------|
| 棕地项目（存量代码库）怎么改造？ | 所有公开案例全是绿地项目，零方法论 |
| 怎么验证 Agent 真的做对了？ | 大家擅长"约束不做错"，"验证做对了"远未解决 |
| AI 生成代码的长期可维护性？ | LLM 代码经常重新实现已有功能，长期效果未知 |
| Harness 该做厚还是做薄？ | 场景决定：通用产品追求最小化，特定产品可高度定制 |
| 单 Agent 还是多 Agent？ | 规模决定：小项目单 Agent 够用，大项目几乎必然需要专业化 |

## 数据与案例

### 关键数据

- **Can.ac 实验**（2026年）：同一模型，只换文件编辑接口调用方式，编码基准分数从 **6.7% → 68.3%**（~10 倍）
- **LangChain Terminal Bench 2.0**：只优化运行环境（文档组织 + 验证 + 追踪），排名从 **全球第 30 → 第 5**，得分 **52.8% → 66.5%**；模型未变
- **上下文 40% 阈值**：168K token 窗口，超过约 40% 后幻觉增多、代码质量下降（Dex Horthy 观察，非硬性规律）

### 典型案例

#### OpenAI：三人五月百万行零手写（附录案例）
**背景**：OpenAI 内部团队 3 人，5 个月，完成约 100 万行代码，几乎不手写代码。

**五大方法论**：
1. **地图式文档**：AGENTS.md 只当目录（~100 行），详细规则按需加载（渐进式披露）
2. **机械化约束**：自定义 Linter，报错信息直接包含修复方法
3. **可观测性接入**：接入 Chrome DevTools，性能分析可度量
4. **熵管理**：专门的后台 Agent 做垃圾回收，确保清理速度 > 生成速度
5. **仓库即事实源**：所有知识必须在代码仓库，Slack/Wiki 对 Agent 不可见

#### Anthropic：GAN 式三智能体 + context resets 策略（附录案例）
**背景**：Anthropic 多智能体架构，通过专业化分工和上下文管理解决长任务质量问题。

**核心做法**：
- 初始化 Agent + 编码 Agent 两阶段分工
- **Context resets 策略**：上下文快满时，不压缩，而是启动全新"干净"的 Agent，通过结构化交接文档（JSON 格式功能列表 + 进度追踪）恢复状态 → 类似"重启进程解决内存泄漏"
- 端到端验证：用 Playwright/Puppeteer MCP 做浏览器自动化验证

**结果**：验证了随着模型变强，已有 Harness 应该定期简化（Anthropic 实测）。

#### Mitchell Hashimoto：AGENTS.md 驱动的单 Agent 模式（附录案例）
**核心做法**：
- AGENTS.md 的每一行对应一个历史失败案例
- 坚持单 Agent 路线（适合中小规模项目）
- 六步进阶路径：积累失败 → 更新 AGENTS.md → 形成持续反馈循环

> ⚠️ 待验证：文中提到的 Can.ac 实验数据（6.7%→68.3%）来源于第三方，具体实验条件（基准测试名称、模型型号、接口变化细节）未在原文中详述，数值合理但无法独立核实。

## 启发与思考

### 对 AI 产品经理的启发

1. **重新定义 AI 产品评估指标**：评估 AI 产品不能只看模型版本号，要看 Harness 质量——AGENTS.md 是否完善、Linter 是否智能、上下文管理是否合理。这是更实质性的竞争力指标。

2. **Agent 产品的护城河在 Harness，不在模型**：模型会被追平，但精心打磨的 Harness（积累了大量失败案例的 AGENTS.md、领域特定的约束规则）是真正难以复制的资产。

3. **上下文管理是 Agent 产品的首要工程问题**：在设计多轮、长任务的 AI 功能时，必须从立项阶段就规划上下文策略（分层管理、渐进式披露、context resets），而不是等出问题再补救。

4. **Model-Harness 解耦是选型考量**：使用 Claude Code、Codex 等 AI 产品时，要意识到它们的 Harness 是为通用场景设计的，不一定适合你的具体任务。对于特定领域，自建专用 Harness 可能有数量级的提升空间。

5. **棕地问题是最大的市场机会**：所有公开成功案例都是绿地项目。如何帮助企业把 Harness 接入存量代码库，目前没有公开方法论，是未被满足的真实需求。

### 可落地的行动项

1. **立即创建 AGENTS.md**：为当前项目的 AI 编码助手创建 AGENTS.md，从当前踩过的坑开始写，每次犯错后更新。从 5 条规则开始，不要一次写满。

2. **监控上下文利用率**：在 AI 辅助开发流程中，设置上下文使用量的观测点，当接近 40% 时主动触发总结/交接，而非让 Agent 在 Dumb Zone 里运行。

3. **把口头规范变成 Linter 规则**：团队的编码规范、命名约定、架构原则——凡是在 Code Review 里反复提的，都应该变成可执行的 Linter 规则，让 Agent 自动遵守。

4. **评估当前是哪个成熟度阶段**：用五阶段模型（Level 0-4）评估当前团队的 Harness 成熟度，明确下一步重点。大多数团队在 Level 0-1，优先升级到 Level 2（反馈回路）。

5. **Harness 专项复盘**：下次 AI 工具效果不好时，在换模型之前先做 Harness 诊断：AGENTS.md 有没有？Linter 够不够智能？上下文是否过载？

### 值得进一步探索的方向

1. **棕地项目的 Harness 改造方法论**：如何为有历史包袱的代码库设计 Harness？Böckeler 的"Ambient Affordances"（环境结构特性决定 Harness 能做多好）概念值得深入研究。

2. **Harness 与模型版本的适配策略**：随着模型能力提升，已有 Harness 应该如何简化？什么时候应该把 Harness 的约束交还给模型？

3. **Harness 即产品**：Harness 本身是否可以成为独立的产品形态？类似 CI/CD 工具，专注于特定领域的 Agent 运行时基础设施。

4. **验证闭环的设计**：Böckeler 指出"用 AI 测试验证 AI 代码"的循环问题。如何设计真正独立的、不被生成过程污染的验证机制？

## 原文精华

> "Agent = Model + Harness。你不是模型，那你就是 Harness。"

> "模型是 CPU，Harness 是操作系统。CPU 再强，OS 拉胯也白搭。"

> "Can.ac 的实验证明，同一模型只换了工具调用格式，效果就能差十倍。瓶颈大概率不在模型智能水平，而在 Harness 的基础设施质量。"

> "你的目标不是给 Agent 塞更多信息，而是让它在任何时候都运行在干净、相关的上下文里。"

> "the best harness for your task is not necessarily the one a model was post-trained with——为你的任务选择 Harness 时，不要被模型的默认 Harness 束缚。"

> "棕地项目（Brownfield）——OpenAI、Anthropic、Stripe、Hashimoto 这些成功案例，全部是在全新项目上从零搭 Harness。但现实中绝大多数团队面对的是已经跑了多年的代码库——目前没有任何公开方法论。"

> "模型决定了系统的上限，Harness 决定了系统的底线。与其纠结选哪个模型，不如先把 Harness 搭好。"
