# Harness Engineering 实战：OpenAI百万行代码、Agent失效模式与动态演化

> 来源：用户提供文本（综合分析文章）
> 提炼日期：2026-04-03
> 分类：技术趋势

## 核心观点

1. **Harness Engineering 是三层工程认知的第三跃迁**：Prompt Engineering 解决"和模型说什么"（单次对话质量），Context Engineering 解决"往窗口里放什么"（信息可见性），Harness Engineering 解决"给 Agent 造一个什么样的世界"（长任务自主运行的整个执行环境）。三者是同一批工程师随 Agent 能力提升必经的认知升级。

2. **Harness 的每个组件都编码了一个"模型做不到"的假设，会随模型升级而失效**：Sprint 分解在 Opus 4.5 是必要的，在 Opus 4.6 变成负担；Context Reset 是对 Context Anxiety 的补偿，模型消除焦虑后即可删除。新模型发布时，Harness Engineer 的第一件事是找哪些组件可以删。

3. **Agent 自我评估必然虚高，需要外部独立评判**：把 Generator 和 Evaluator 彻底拆开（GAN 思路），比让 Agent 自我批判有效得多。三 Agent 架构（Planner-Generator-Evaluator）成本是 Solo 的 20 倍，但实现了从"不能用"到"能用"的跨越。

## 关键知识点

### 三层工程的区分

| 工程层 | 核心问题 | 影响范围 |
|--------|---------|---------|
| Prompt Engineering | 和模型说什么 | 单次对话输出质量 |
| Context Engineering | 往 Context Window 放什么 | 模型能看见多少信息 |
| Harness Engineering | 给 Agent 造什么世界 | 长任务/复杂场景的稳定自主运行 |

### OpenAI 百万行代码实验：三个问题及 Harness 解法

**实验参数**：3 名工程师，5 个月，从空仓库开始，全部交给 Codex → 100 万行生产级代码，1,500 个 PR

**问题 1：Agent 不知道工作细节，只能猜**

巨型 AGENTS.md 失败的四个原因：
- Context 是稀缺资源，臃肿文件挤掉任务空间
- 指导过多反而失效（"一切都重要 = 没有重点"）
- 文档快速"腐烂"，旧规则无人更新
- 单文件无法做机械验证

**解法**：AGENTS.md 收缩到 100 行，只做"地图"指向 docs/ 分类文档 + linter/CI 验证链接完整性

> 核心逻辑：运行时没出现在 Context 里的内容，对 Agent 来说就等于不存在。

**问题 2：人工 QA 速度跟不上 Agent 代码生成速度**

**解法**：接入 Chrome DevTools 协议（截图+运行时事件）+ LogQL/PromQL 可观测性堆栈 + 明确验证标准（如服务启动 ≤800ms）

**问题 3：无架构约束，Agent 复现仓库中的坏模式**

**解法**：
- 严格分层架构 + 依赖方向单向验证（Types → Config → Repo → Service → Runtime → UI）
- 自定义 linter 机械强制规则，报错信息嵌修复指令
- 技术债务清理循环：定期 Codex 任务扫描代码偏差 → 发送重构 PR → 大多数一分钟自动合并

> 关键洞察：架构约束在传统公司要等团队扩到数百人才引入，但对编码 Agent 来说是早期先决条件——没有约束，速度越快，架构漂移越严重。

### Agent 两大失效模式

**失效模式 1：Context Anxiety（上下文焦虑）**

Agent 感觉 Context Window 快满了 → 提前收尾任务（不是因为完成了，而是"焦虑"了）

| 应对策略 | 机制 | 权衡 |
|---------|------|------|
| Compaction（压缩） | 摘要历史对话，同一 Agent 继续 | 保住连续性，但无法消除焦虑感 |
| Context Reset（重置） | 清空上下文，启动新实例 + handoff artifact | 消除焦虑，但交接文件必须完整 |

本质是在**连续性 vs 清醒度**之间选边。Claude Opus 4.5+ 已基本消除此行为。

**失效模式 2：自我评估偏差**

Agent 评价自己时倾向虚高分——先发现问题，然后说服自己"没那么严重"。

**解法**：GAN 思路——Generator/Evaluator 彻底拆开
- Evaluator 独立调教为"天生挑剔的审查者"
- 配备 Playwright MCP，像真实用户一样操作测试
- Sprint Contract：在写代码前先就"完成的定义"达成共识

### 三 Agent Harness 实验对比

| 维度 | Solo Agent | 三 Agent Harness (Planner+Generator+Evaluator) |
|------|-----------|------------------------------------------------|
| 结果 | 游戏能启动但连线断裂，界面无提示 | 正常游玩 + AI关卡生成 + 精灵动画 + 音效 |
| 成本 | 基准 | 20x |
| 本质 | 不能用 | 能用 |

### 混合检索是 Harness 的关键基础设施

Agent 在 Harness 中需要同时执行两类检索：
- **语义查询**：产品对用户会话的设计原则是什么？（文档可能用"会话管理""访问控制"等不同措辞）
- **精确匹配**：哪些文档提到 `validateToken` 函数？（函数名无语义，向量无法匹配）

Milvus 2.6 Sparse-BM25 解法：
- 入库时同时生成 dense embedding + TF 编码稀疏向量
- 全局 IDF 自动维护，随文档增删实时更新
- 查询时 Reciprocal Rank Fusion 融合排序
- 性能：BEIR 上吞吐量比 ES 高 3-4x，特定负载 QPS 7x

### Harness 动态演化规律

```
新模型发布 → 审视整套 Harness → 找出不再是承重墙的组件 → 删除

组件存在的前提 = 模型做不到的假设
假设失效 → 组件变成冗余 → 增加系统复杂度 → 降低效率
```

实例：
- Sprint 分解：Opus 4.5 必要 → Opus 4.6 可删
- Context Reset：早期模型必要 → Opus 4.5+ 可删
- Context Slice 精细管理：短窗口模型必要 → 长上下文模型可粗放

> 基础设施配置是和模型能力一起演化的变量。Harness 里任何一个必要的组件，都在等待被更聪明的模型证明为多余。

## 数据与案例

- **OpenAI Codex 实验**：3人 x 5个月 → 100万行代码 + 1,500 PR，全程不写一行代码
- **服务启动验证标准**：800ms 内完成，从"愿望清单"变成可执行的 prompt
- **单次 Codex 任务**：可连续工作超过 6 小时（人类睡觉时间）
- **三 Agent vs Solo**：成本 20x，但实现从不能用到能用的跨越
- **Milvus 2.6 性能**：BEIR 上比 ES 吞吐量 3-4x，特定负载 QPS 7x
- **AGENTS.md 最佳实践**：收缩到 100 行，只做地图不做规则库

## 启发与思考

1. **对 AI 客服产品架构的直接启示**：沃丰等 AI 客服系统本质上就是一个 Harness——定义 Agent 能调用哪些知识库、走什么流程、什么情况下转人工。文中的 Generator-Evaluator 分离思路可以直接应用：让一个 Agent 生成回答，另一个 Agent 评判回答质量，避免"自己给自己打高分"的问题。

2. **混合检索是 AI 客服的基础需求**：客服场景天然需要同时处理语义查询（"我的包裹怎么还没到"）和精确匹配（订单号、产品型号）。Milvus 2.6 的一体化方案值得关注。

3. **"AGENTS.md 地图模式"对知识库管理的启发**：不要试图把所有知识塞进一个巨型 prompt，而是建立结构化的知识索引，让 Agent 按需检索。这与之前 RAG Chunk 切分文章的结论一脉相承。

4. **Harness 动态演化思维要求产品经理持续关注模型能力边界**：每次模型升级都要重新评估哪些产品功能/限制/workaround 可以简化或删除，避免在"模型已经会了"的地方做多余的工程。
