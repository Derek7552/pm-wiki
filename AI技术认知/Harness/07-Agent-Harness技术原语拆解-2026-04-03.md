# Agent Harness 技术拆解：从框架原语到长期自主执行

> 来源：[智能体框架（Harness）到底是什么？一文拆透](https://x.com/Vtrivedy10/status/2031408954517971368)
> 提炼日期：2026-04-03
> 分类：技术趋势

## 核心观点

1. **智能体 = 模型 + 框架，如果你不是模型，你就是框架**：框架（Harness）是除模型之外的所有东西——系统提示、工具与技能（含MCP）、基础设施（文件系统/沙箱/浏览器）、编排逻辑（子代理/任务交接/模型路由）、确定性钩子和中间件。裸模型只是能输出文字的"大脑"，框架给它装上状态管理、工具执行、反馈循环和约束后，才变成能干活的智能体。

2. **文件系统是框架最底层的基石，不只是存储**：它解锁了工作区、增量卸载、跨会话持久化、多智能体协作。叠加 Git 后还有版本控制。后续所有能力（代码执行、记忆、长期任务）都建立在文件系统之上。

3. **模型训练与框架设计已深度耦合，形成反馈循环但也导致泛化下降**：有用的框架原语被发现 → 加进框架 → 训练下一代模型 → 模型更强 → 发现新原语。但副作用是模型过度拟合特定框架，换工具逻辑就性能下滑。同一个 Opus 4.6，换框架可以从 Top 30 升到 Top 5。

## 关键知识点

### 框架七大原语（按依赖关系自底向上）

| 原语 | 解决的问题 | 核心机制 |
|------|-----------|----------|
| **文件系统** | 持久存储、信息卸载、跨会话状态 | 文件读写工具 + Git 版本控制 |
| **Bash + 代码执行** | 避免工具集上限卡死能力 | 给模型一个 Bash 工具，让它自己写代码解决问题 |
| **沙箱** | 安全执行 + 横向扩展 | 隔离环境，按需创建/销毁，预装运行时和工具 |
| **记忆与搜索** | 跨会话知识 + 训练截止后信息 | AGENTS.md 记忆文件 + Web搜索/MCP工具 |
| **上下文管理** | 对抗上下文衰减 | 压缩机制 + 工具输出卸载 + 技能渐进式披露 |
| **长期自主执行** | 过早停止、跨窗口不一致 | Ralph循环 + 规划文件 + 自我验证 |
| **编排逻辑** | 复杂任务分解与并行 | 子代理生成、任务交接、模型路由 |

### 对抗上下文衰减的三种策略

- **压缩机制**：上下文快满时智能卸载并汇总，避免 API 报错或任务崩溃
- **工具调用卸载**：大型工具输出只保留头尾，完整内容卸载到文件系统按需取用
- **技能渐进式披露**：启动时只加载元数据，用到什么再展开什么，避免工具过载占满上下文

### Ralph 循环：让智能体不放弃

一种对抗"过早停止"的框架模式：
1. 钩子拦截模型的"退出"企图
2. 在干净的新上下文窗口里重新注入原始目标
3. 新上下文读取上一次的文件系统状态，继续推进
4. 每次迭代：新上下文 + 旧状态 = 持续执行

### Bash 工具：从固定工具集到通用解题

关键洞察：与其为每个操作都预写工具，不如给模型一个 Bash 工具让它自己写代码。这是让模型"拥有一台电脑"的关键一步——从受限于预配置工具集，到通过代码动态创造工具。代码执行已成为自主解决问题的默认通用策略。

### 模型-框架协同演化的双刃剑

**正循环**：框架原语 → 训练数据 → 模型能力提升 → 发现新原语

**副作用**：
- 模型过度拟合训练时用的框架，泛化能力变差
- 换一种补丁应用方式，模型就跟不上（Codex-5.3 案例）
- 最适合你任务的框架 ≠ 模型后训练时用的框架

## 数据与案例

- **Terminal Bench 2.0**：同一个 Opus 4.6，在 Claude Code 里的得分远低于其他框架。仅通过换框架，编码智能体排名从 **Top 30 → Top 5**
- **聊天产品的本质**：最基础的框架就是把模型包在 while 循环里，记录并追加历史消息
- **AGENTS.md 持续学习模式**：智能体更新记忆文件 → 框架下次启动注入最新版 → 实现跨会话学习
- **Codex-5.3 泛化问题**：换一种补丁应用方式，模型性能直接下滑

## 启发与思考

- **与上篇 Harness 六层架构的对照**：上篇（06）提出 Loop/Tools/Context/Persistence/Verification/Constraints 六层，本篇从实现层面给出了更具体的原语：文件系统(Persistence)、Bash(Tools)、沙箱(Constraints)、压缩机制(Context)、Ralph循环(Loop)、自我验证(Verification)。两篇是概念框架与技术实现的完美互补。
- **"文件系统是一切基石"的洞察值得深思**：无论是淘宝 Tair 记忆架构（用 Redis 做短期记忆）、ima 知识库（做持久化记忆），还是本文的文件系统+Git，本质都是在解决同一个问题——让 Agent 有"记忆"。不同场景选不同的存储形态，但架构位置一致。
- **"代码执行是通用工具"改变了工具设计思路**：不需要为每个场景预写工具，给模型一个 Bash 就够了。这对 AI 产品的工具设计有根本性启示——少做预定义工具，多给通用能力。
- **框架-模型耦合的风险对产品选型有指导意义**：不要假设"用官方框架就是最优"，针对具体任务测试不同框架组合，可能有巨大的性能提升空间。

## 原文精华

> 智能体 = 模型 + 框架。如果你不是模型，你就是框架。

> 文件系统不只是存储，它实际上是后面所有能力的地基。

> 与其让用户为每一个可能的操作都写工具，不如直接给智能体一个 Bash 工具，让它自己写代码解决问题。这是让模型"拥有一台电脑"的关键一步。

> 有用的框架原语被发现，加进框架，用来训练下一代模型，模型在这些框架里越来越强，然后继续发现新原语。但这种协同演化有个副作用——泛化能力会变差。

> 框架的价值不只是弥补模型的短板，它是围绕模型智能构建系统、让模型更高效发挥的那层"放大器"。

> 模型负责提供智能，框架负责把这种智能变成真正有用的东西。
