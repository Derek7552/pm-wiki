# Agent Harness 六层架构：从模型能力到系统化交付

> 来源：公众号文章 - 2025 是 Agent 年，2026 是 Harness 年：AI 竞争进入"交付时代"
> 提炼日期：2026-04-03
> 分类：技术趋势

## 核心观点

1. **AI 的上限由模型决定，但下限由 Harness 决定**：模型之间的基础能力差距正在收敛，真正决定 AI 产品体验好坏的，是模型之外的系统工程——即 Harness。大多数用户体验到的不稳定、不可靠，根源不在模型，在 Harness 的缺失。

2. **Harness 是把模型变成"能稳定干活"的完整系统工程**：它不是 Prompt，不是某一个工具，也不是一段系统指令。它是覆盖从任务输入到结果交付的六层架构：Loop（循环）、Tools（工具）、Context（上下文）、Persistence（持久化）、Verification（验证）、Constraints（约束）。

3. **AI 竞争维度正在从模型转移到系统化交付能力**：Prompt Engineering 是技巧，Harness Engineering 是工程。技巧会被模型进化消化掉，工程能力积累下来就是护城河。

## 关键知识点

### Agent Harness 六层架构

| 层级 | 名称 | 解决的问题 | 核心设计 |
|------|------|-----------|----------|
| **L1** | Loop 循环层 | AI 做完一步就停 | Observe → Decide → Act → Verify → Repeat 执行循环 |
| **L2** | Tools 工具层 | 模型只会"说"不会"做" | 文件读写、API调用、代码执行等能力接入（但不是越多越好） |
| **L3** | Context 上下文层 | 信息太少乱猜，太多漂移 | 精确的信息调度系统：何时注入、格式组织、优先级、压缩剔除 |
| **L4** | Persistence 持久化层 | 跨对话/跨步骤的"失忆" | 记住进度、历史决策、贯穿始终的约束 |
| **L5** | Verification 验证层 | AI 不知道自己做错了 | 自动测试、语法检查、回归验证、结果比对、自审机制 |
| **L6** | Constraints 约束层 | 行为不可预测、不可审计 | 权限控制、预算限制、行为边界、安全策略 |

### 三个关键误区澄清

- **Harness ≠ Prompt**：Prompt 是点菜，Harness 是后厨的整个运营体系
- **Harness ≠ 某个工具**：接一个搜索工具只是工具层的一部分，不是整体
- **Harness ≠ System Prompt**：几百字的角色设定只是约束层最浅的实现

### Claude Code vs Claude Web 的 Harness 差异

同一个模型，两套截然不同的 Harness：
- **Claude Web**：一问一答模式，无文件系统访问，无执行能力，无自动验证 → 反复修改仍报错
- **Claude Code**：读取项目结构，执行修改，跑测试，自动继续修复直到通过 → 一次闭环解决

这不是模型差异，是系统差异。

## 数据与案例

### LangChain 实验（严格变量控制）
- **变量**：模型不变、API不变，只改 Harness 设计
- **改动**：加入自验证循环、优化上下文注入、防死循环、推理预算分配、失败案例分析
- **结果**：任务完成率 **52.8% → 66.5%**（+13.7pp），排名 Top 30 → Top 5

### Vercel 案例（少即是多）
- **操作**：可用工具从 **15个砍到2个**
- **结果**：准确率 **80% → 100%**，Token消耗 **减少37%**，速度 **提升3.5倍**
- **启示**：工具过多增加选择噪声，约束到恰好够用反而更好

### 行业信号
- GitHub Copilot 工程师：产品体验提升主要来自**系统层优化**，而非切换更新模型
- Martin Fowler 提出 **"Harness Engineering"** 概念
- Anthropic 讨论长任务 Agent 时反复强调**如何构建稳定 Harness**
- AI 招聘需求从 "Prompt 工程师" → **"Agent 系统工程师"**

## 启发与思考

- **Harness 框架可以直接指导 AI 产品设计**：作为产品经理，在设计 AI 功能时，不能只关注"用什么模型"，而要从六层结构审视产品的系统完整性。验证层和约束层是最容易被忽视的，也是用户体验差异最大的来源。
- **"少即是多"原则对 AI 产品设计的启示**：Vercel 案例表明，给 AI 过多工具/能力反而降低表现。这与之前 Claude Code Computer Use 中"分层工具优先级"的设计哲学一致——精准设计边界比堆砌能力更重要。
- **Harness 思维可以迁移到个人效能系统**：文章最后的类比非常有价值——个人工作系统也可以用六层框架审视：有没有每日检视循环（Loop）、快速进入状态的机制（Context）、知识沉淀系统（Persistence）、注意力边界控制（Constraints）。
- **与淘宝闪购 Tair 记忆架构的对应**：之前归档的 Tair 短期记忆文章，正是 Harness 六层中 Persistence 层和 Context 层的生产级实现。两篇文章一个讲框架，一个讲落地，互为印证。

## 原文精华

> AI 的上限由模型决定，但下限由 Harness 决定。大多数人正在经历的那些不稳定、不可靠、用起来像"赌运气"的体验，根源不在模型，在 Harness 的缺失。

> 模型 = 马的力量。Harness = 让这股力量变得可控、可用、可交付的完整系统。没有 Harness 的 AI，你会看到这些症状：跑偏、停滞、乱跑、中断。这些问题，换一个更贵的模型都解决不了。因为根子不在马，在没有缰绳。

> Harness 不是堆砌能力，而是精准设计边界。

> Prompt Engineering 解决的是如何更好地问问题。Harness Engineering 解决的是如何构建一个让 AI 可以稳定、持续、可控地完成复杂任务的系统。前者是技巧，后者是工程。技巧可以被模型的进化消化掉，工程能力积累下来就是护城河。

> 提升效率最快的方式，从来不是"变得更聪明"，而是"换一个更好的系统"。
