# Harness 三层结构：知识层、约束与流程层、反馈与运行时层

> 来源：架构师（JiaGouX）- 大家都在讲 Harness，但它到底该怎么理解
> 提炼日期：2026-04-03
> 分类：技术趋势

## 核心观点

1. **Harness 是一个三层结构的工程对象，不是模糊热词**。拆成知识层（模型该读什么）、约束与流程层（任务怎么拆、谁负责什么）、反馈与运行时层（做完之后谁来验），结构才清楚。它管四类事：给模型看什么、允许做什么、怎么判断对不对、出错怎么修。

2. **同一模型同一 Prompt，只换 Harness，产出质量天差地别**。Anthropic 实验：单 Agent 20 分钟/$9 产出不可用；完整 Harness 6 小时/$200 交付了真正可用的应用。差距不在模型，在系统。

3. **Prompt/Context/Harness 是包含关系，不是替代关系**。Prompt 管怎么说，Context 管看到什么，Harness 管整个系统怎么运行。问题升级驱动层次升级。

## 关键知识点

### 第一层：知识层

**解决的问题**：模型该读什么、团队经验放哪里、哪些规则必须显式可见

**OpenAI 的实践**：
- AGENTS.md 不是百科全书，而是地图（约 100 行，只告诉 Agent 去哪里找什么）
- 真正的知识库拆进结构化 docs/ 目录（设计文档、产品规范、执行计划、技术债追踪）
- 代码仓库本身变成记录系统
- 跑 "doc-gardening" 智能体定期扫描过时文档

**核心判断**：对 Agent 来说，看不见的知识就等于不存在。Slack 讨论、Google Docs 共识、资深工程师脑中的隐性判断——没编码进仓库就像从未发生。

**实用原则**：别把确定性逻辑全塞进上下文。能通过 Hooks、代码规则、工具权限表达的，交给外部系统。

### 第二层：约束与流程层

**解决的问题**：任务怎么拆、哪些步骤必须先发生、哪个角色负责什么

**Anthropic 的做法**：Planner / Generator / Evaluator 三角色分离
- Planner：只聚焦产品上下文和高层设计，不规定技术细节（避免错误级联）
- Generator：按 Sprint 逐个实现，每个 Sprint 前谈好"契约"（单 Sprint 27 条验收标准）
- Evaluator：独立验收，用 Playwright 真正跑页面

**Context Reset**：长任务中定期清空上下文窗口，重新启会话，通过结构化工件交接状态。听上去笨，但很工程。

**OpenAI 的路线**：不拆角色，把约束编码成机械规则
- 业务域依赖方向固定：Types -> Config -> Repo -> Service -> Runtime -> UI
- 违反架构约束的 PR 被 linter 自动拦截

### 第三层：反馈与运行时层

**解决的问题**：模型做完之后，谁来告诉它对不对

**Anthropic 的 Evaluator 实战**：
- 不是读代码打分，而是用 Playwright 真正跑页面、点按钮、看行为
- 开箱即用的 Claude 是很差的 QA Agent（会自己说服自己"不是大事"然后通过验收）
- 花了好几轮迭代才把评估质量调到合理水平

**OpenAI 的反馈层**：
- 每次 Codex 改动启动独立实例验证（git worktree）
- Chrome DevTools 协议接入 Agent 运行时（截图、DOM 快照、页面导航）
- 本地可观测性堆栈（LogQL 查日志、PromQL 查指标）

### Harness 的生长规律

- **不是设计出来的，是长出来的**：从第一个犯错开始补，从第一条规则开始写
- **NLAH（清华/哈工大）**：把 Harness 控制逻辑写成接近自然语言但可执行的文件
- **Harness 组件随模型迭代减重**：Sonnet 4.5 需要 context reset，Opus 4.5 不需要了；但 Evaluator 去不掉

### 关键判断

> 评估器的价值取决于任务是否处于当前模型独立可靠完成的边界之外。这个边界会随模型能力移动，但边界本身不会消失。

## 数据与案例

- **Anthropic 对比实验**：单 Agent 20min/$9 vs 完整 Harness 6h/$200，成本 20 倍但前者核心功能是废的
- 规划器把一句话需求展开成 16 个 feature 的产品规格
- 单个 Sprint 27 条验收标准
- OpenAI 单次 Codex 运行经常超过 6 小时（通常在人类睡觉时）
- Anthropic 旧版 Harness 用 Sonnet 4.5 有"上下文焦虑"，换 Opus 4.5 后问题消失

## 启发与思考

1. **Harness 三层框架可以直接用于沃丰 AI 客服产品设计**：知识层 = 客服知识库管理；约束层 = 工单流程编排和角色分离；反馈层 = 客户满意度实时回流和质检系统。

2. **"看不见的知识等于不存在"对企业 AI 落地的启示**：很多企业的核心 know-how 散在老员工脑子里和微信群聊中。AI Agent 的引入倒逼知识显性化——这本身就是巨大的组织价值。

3. **Evaluator 独立于 Generator 的设计，映射到产品领域就是"验收不能自己做"**。产品质量保障需要独立的评估机制，而不是开发自测。

4. **Harness 从脚本走向文件化（NLAH）**：控制逻辑能被表达、复用、迁移和执行，这是 Agent 工程从"手艺活"变成"工程学科"的关键转折。

## 原文精华

> 同一个模型，不改 Prompt，不换数据，只换 Harness：单 Agent 20 分钟花 9 美元产出不能用，完整 Harness 6 小时花 200 美元交付了真正可用的应用。差距不在模型，在系统。

> 在以人为本的工作流中，这些规则可能会让人感到迂腐或束缚。有了智能体，它们就成了倍增器。

> 开箱即用的 Claude 是一个很差的 QA Agent。在早期的运行里，他看着 Evaluator 发现了真正的问题，然后自己说服自己这不是大事，然后通过了验收。

> Harness 的每个组件，本质上都编码了对模型局限性的一个假设。模型变强之后，有些假设不再成立。

> 如果说 Prompt 是在教模型怎么回答，Context 是在决定模型看见什么，那么 Harness 真正关心的，其实是更后一层的事：怎么让一个会生成的模型，慢慢变成一个能在真实系统里稳定交付结果的工作体。
