# Anthropic 的 Agent OS 答案：Meta-Harness 三层架构与停止写框架论

> 来源：微信公众号「Claw团」团爸，基于 Anthropic Lance Martin 的 Claude Managed Agents 工程博客分析
> 提炼日期：2026-04-16
> 分类：AI方法论

## TL;DR

Anthropic 通过 Claude Managed Agents 给出了一个超越"更好框架"的答案：把 Agent 基础设施做成操作系统，核心是 Session（执行事实流）/ Harness（无状态调度脑干）/ Sandbox（可插拔执行环境）三层完全解耦的稳定抽象。核心结论：别再把模型暂时缺陷硬编码成架构设计，那些"聪明"的补丁会随模型进化变成技术债。

## 核心观点

### 1. Agent 框架开发者的共同陷阱：把补丁变成架构

几乎所有 Agent 框架都在犯同一个错误：把模型**当前**的能力缺陷，硬编码成了系统**永久性**的架构设计。

- 模型不会规划 → 框架里加固定 DAG 步骤拆解 → 这是补丁，不是架构
- 模型记不住长上下文 → 加向量记忆检索模块 → 这是对局限的妥协
- 模型调工具容易出错 → 加多层校验重试路由 → 这是保护机制

问题在于，模型能力是指数级进化的：半年前不会正确调用工具，现在已能独立完成完整软件工程任务；一年前算术会出错，现在已能处理超过 10 小时的长时域任务。**你花几个月打磨的"完美框架"，可能一次版本升级就完全过时，反而成为限制模型能力的性能瓶颈。**

### 2. Anthropic 的答案：跳出框架思维，做稳定抽象层

Anthropic 没有给出"更好的 Agent 框架"，而是类比操作系统思路：OS 不关心上层应用怎么写，只提供稳定的进程管理、内存管理、文件系统、设备 I/O 抽象。不管上层应用怎么变，底层抽象是稳定的。

Meta-Harness 的核心承诺：**不承诺 harness 具体长什么样，只承诺几类长期稳定的接口。**

### 3. 未来赌注：harness 会越来越简单直到消失

Anthropic 的产品哲学是："我们不相信今天的 Agent Harness 会是最终形态，所以我们优先投资于稳定接口，而不是一次性的最优实现。" 他们赌的是模型能力持续提升后，harness 逻辑会越来越少，最终由模型自己实现调度，harness 退化为薄薄的一层甚至消失。

## 关键知识点

### Meta-Harness 三层架构

```
应用层：Agent 业务逻辑
    ↓
Session（执行事实流）
Harness（调度脑干）
Sandbox（可插拔执行）
    ↓
基础设施层：工具 / MCP 服务 / Vault
```

#### 1. Session：不是上下文窗口，是可恢复的执行事实流

原话："The session is not Claude's context window."

传统 Agent 的会话 = 聊天记录 = 直接喂给模型的上下文窗口。新架构中，Session 是一个 append-only 的不可变事件日志，记录整个任务执行过程的所有事实，具备四种能力：

| 能力 | 含义 |
|------|------|
| 可查询 | 随时检索执行历史 |
| 可回放 | 重演完整执行过程 |
| 可恢复 | 任务中断可从任意节点恢复 |
| 可重组 | 灵活重构执行路径 |

Session 不是用来直接喂给模型的 Prompt，就算上下文窗口被截断压缩，原始执行事实永远存在。这是比"摘要式记忆"高一个层级的设计——**事实流 vs 摘要流**。

#### 2. Harness：无状态的推理调度脑干，可热替换

早期 Anthropic 把 harness、session、sandbox 塞在同一容器里，遇到：harness 崩了整个会话挂掉、状态丢失、调试困难。

新架构 Harness 的三个关键特性：
- **无状态设计**：不保存任何会话状态，只从 Session 日志查询需要的信息，调度工具调用
- **标准接口**：不假设工具在哪里运行，只需要知道 `execute(name, input) → string`
- **可热替换**：随时替换升级，不需要修改任何会话数据和执行环境

模型变强了 → 直接换一个更简单的 harness → 其他部分完全不用动。

#### 3. Sandbox：可插拔的"手"，与大脑完全解耦

核心设计理念：**decouple the brain from the hands（解耦大脑和手）**

- 大脑 = Claude 模型 + Harness 调度层 → 只负责决策
- 手 = Sandbox + 工具 + MCP 服务 → 只负责执行

Sandbox 的四个设计优势：
1. Sandbox 之间完全独立，可来自不同基础设施
2. 可以共享、可以传递
3. 不需要每个会话都启动完整沙箱
4. 用到了才动态创建（懒加载设计）

### 安全架构：凭证隔离，不依赖模型的"不作恶"

传统思路：给模型有限权限，赌模型不会突破限制 → 随着模型越来越聪明，这种假设越来越脆弱。

Anthropic 方案：**完全不让模型接触到真实凭证**

```
Claude 模型（只看到代理凭证，看不到真实密钥）
    ↓
MCP Proxy（Session Token 验证）
    ↓
Vault（存储真实 Git Token / API Key）
```

Git Token、API 密钥全部存在独立 Vault，模型通过 MCP Proxy 间接调用工具，Proxy 拿 Session Token 去 Vault 取真实凭证执行，模型全程看不到明文密钥。

**核心原则：安全不建立在"模型不会做什么"的假设上，而是建立在系统架构的权限隔离上。**

## 数据与案例

**性能数据（官方，Sandbox 懒加载改造后）：**
- p50 首 Token 延迟降低 **60%**
- p95 延迟降低 **90%+**
- 原来的瓶颈不是模型推理，而是预启动整个执行环境的开销

**模型能力进化时间线（文中描述）：**
- 一年前：模型连简单算术都会错
- 半年前：模型还不会正确调用工具
- 现在：已能独立完成完整软件工程任务、处理 10+ 小时长时域任务

## 启发与思考

**1. 会话设计要从"消息列表"升级到"执行事实流"**
- 当前大多数 Agent 应用把会话存成消息列表，这是消耗品
- 执行事实流应该是不可变的 append-only 日志，支持回溯和恢复
- 对长任务（>30min）这个差异会非常显著

**2. 工具和环境要独立于 Agent 本体**
- 不要把工具环境内化成 Agent 的一部分（比如直接 spawn 子进程、在 Agent 进程内跑工具）
- 工具应该是可插拔的独立服务，通过标准接口调用
- 这和云原生的 sidecar 模式思路类似

**3. 上下文工程逻辑不要硬编码**
- 记忆压缩、检索逻辑、上下文填充策略，全部应该是 harness 层的可替换策略
- 不要为了当前模型缺陷写死逻辑，留好替换接口

**4. 对国内团队的关键问题**
- 当前大多数 Agent 系统 Session = 消息数组，Harness = 主循环，Sandbox = 无概念
- 改造路径可以从最容易的 Sandbox 解耦开始，逐步向 Session 事实流迁移

## 原文精华

> 你花了几个月打磨的"完美框架"，可能模型一次版本升级，就完全过时了，反而成了限制模型能力发挥的性能瓶颈。这就是所有 Agent 框架开发者的共同困境：你永远追不上模型进化的速度。

> 我们不相信今天的 Agent Harness 会是最终形态，所以我们优先投资于稳定接口，而不是一次性的最优实现。

> Agent 的本体不再绑定某个执行壳，而是绑定一组可恢复的状态和可调用的能力。（Many brains, many hands）
