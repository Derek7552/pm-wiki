# OpenClaw 安全框架：十大攻击面与六段攻击链

> 来源：粘贴文本（基于 arxiv.org/pdf/2603.27517 的解读文章）
> 提炼日期：2026-04-14
> 分类：技术趋势

## TL;DR

基于 OpenClaw 190 条安全通告归纳出的 AI Agent 安全分析框架：横向是 10 大攻击面（漏洞出现在哪一层），纵向是 6 段攻击链（攻击如何逐步推进）。核心结论是：单个层面的漏洞未必致命，但跨层串联几乎必然成链——智能体安全最大的危险是每层都"局部自证安全"，却无人为整条链路负责。

## 核心观点

**上下文窗口是智能体安全最被低估的核心边界。** 传统安全盯着"输入"，但智能体的真正风险是：内容进入上下文后的"角色身份混淆"——用户话语、系统规则、工具输出、外部文件、技能说明全部混在一起，模型无法分辨"谁才是真正该服从的对象"。攻击者不需要直接发指令，只需要污染上下文。

**执行策略引擎反复被绕过，根因不是规则不够多，而是设计前提本身不可靠。** 靠文本解析判断命令安全性注定失败：shell 续行语法、busybox 多路复用入口、GNU 长选项缩写，让"看起来安全"的文本在真实执行语义里可能是危险的。字符串理解永远追不上执行语义。

**智能体安全是跨层问题，不是单层问题。** 每一层都"局部合理"，整条链却不再安全——这是智能体系统的结构性困境，因为模型本身就是"把上游信息解释成下游动作"的桥梁，只要桥梁没有严格约束，跨层串联几乎不可避免。

## 关键知识点

### 十大攻击面（从外到内）

| 层级 | 攻击面 | 核心风险 |
|------|--------|---------|
| 1 | 渠道输入接口 | 身份锚点不稳——依赖昵称/展示名而非不可变标识做授权 |
| 2 | 插件与技能分发面 | 上下文供应链风险——技能包直接进入模型上下文，绕过执行策略引擎 |
| 3 | 智能体上下文窗口 | 角色身份混淆——来自不同来源的内容未被区分，模型无法判断谁是可信指令 |
| 4 | 网关 WebSocket 接口 | 连接目的地失控——目标地址可被动态篡改，凭证和控制面被送到攻击者手中 |
| 5 | 工具调度接口 | 语义扩张——模型意图被解释成比预期更宽的工具调用（查询→写入，局部→全局） |
| 6 | 执行策略引擎 | **漏洞最集中层**——字符串理解无法管控真实执行语义，allowlist/正则必然被绕过 |
| 7 | 容器边界 | 隔离退化——容器存在但创建时参数验证不严，危险挂载/宿主网络模式打穿边界 |
| 8 | 宿主机操作系统接口 | 最终影响面——攻击链穿透后的命令执行、文件读写、环境变量访问等 |
| 9 | 大模型服务提供接口 | 前瞻风险——路由错误/响应污染/上下文返回异常可反向影响上游智能体行为 |
| 10 | 智能体间通信 | 污染传播——多 Agent 场景中，一个被污染的 Agent 继续污染下一个 |

### 六段攻击链（Kill Chain）

```
① 初始进入
   └── 入口不止用户消息：文件/渠道/工具输出/插件/远端连接/中间服务

② 上下文操控（Agent 时代新增段）
   └── 先污染模型"怎么看世界"，让其误判指令合法性
   └── 不需要立刻执行，只需改变模型后续决策偏向

③ 执行
   └── 命令执行 / 脚本运行 / 文件操作 / 网络请求
   └── 注意：执行层是中后段，不是起点

④ 凭证获取
   └── Agent 天然具备"代人操作"特征，凭证本来就存在于系统中
   └── 泄露的不只是数据，而是进一步扩张权限的钥匙

⑤ 权限提升
   └── 工具权限→网关控制权限→容器内→宿主机
   └── 路径与工具组合、上下文误导、代理链路结合，比传统提权更隐蔽

⑥ 最终影响
   └── 宿主机控制 / 数据窃取 / 外部服务滥用 / 环境被用于攻击他人
   └── 落地的是系统安全+业务安全+环境安全，不只是模型行为异常
```

### 5 条工程启发

1. **身份锚点绑定不可变标识**：禁止用昵称/展示名做授权，必须使用平台分配的稳定、不可伪造标识。

2. **上下文是安全边界，不是材料池**：每条写入上下文的信息必须标注来源/角色/是否可信/能否参与工具决策，不能全部混入让模型自行判断。

3. **执行策略离开字符串层**：收缩到结构化参数执行，绕开 shell 解释层，权限控制放到工具语义层而非命令字符串层。

4. **容器隔离看创建时验证**：有无容器是起点，真正决定隔离是否成立的是：启动参数/挂载策略/网络模式/安全配置是否严控。

5. **插件/技能市场按供应链治理**：来源审核 + 内容审查 + 权限声明 + 签名校验 + 风险分级 + 上下文来源标注。运行时护栏再厚，也可能被"合法装进来的恶意能力"绕过。

## 启发与思考

**"上下文操控"作为独立攻击阶段是最重要的理论贡献。** 传统 kill chain（MITRE ATT&CK）里没有这一段，但 AI Agent 系统天然需要它——攻击者可以通过污染上下文"预置偏见"，让模型在后续执行阶段主动配合攻击意图，而不需要直接绕过执行层的技术防御。

**插件供应链风险是目前最被低估的攻击面之一。** 技能包进入上下文的路径完全绕过了执行策略引擎——它不需要执行，只需要存在于上下文中就能影响模型行为。这对 Claude Code skills 体系、MCP 插件市场等平台有直接的安全设计启示。

**跨层串联问题的本质是"接口边界的权责真空"。** 每一层都只负责"自己这一层没问题"，但 Agent 的模型推理层横跨所有层，任何一处语义泄漏都可以被模型当成合法指令。解决这个问题需要一个"链路级安全架构师"视角，而不是各层独立做安全审计。

**这套 10+6 框架是安全产品规划的有效工具。** 对于云起无垠这类 AI 安全产品，可以用这套框架做功能覆盖矩阵分析：当前产品覆盖了哪些攻击面？哪些是空白？六段链路中哪些阶段有检测/阻断能力？

## 原文精华

> 这正是智能体安全最麻烦的地方。因为模型本身就承担着"把上游信息解释成下游动作"的桥梁作用，一旦这个桥梁没有被严格约束，跨层串联几乎是必然的。

> 执行策略引擎最大的问题，不是漏了一条规则，而是它一直试图用"字符串理解"去管理"真实执行语义"。

> 智能体安全绝不能只当成模型安全的一个子话题。因为它最终落地的，是系统安全、业务安全和环境安全。
