# Skills 详解：为什么一个 Markdown 文件夹能如此高效

> 来源：ThinkInAI 星科社区文章（直接粘贴），参考 OpenAI Codex Skills 文档
> 提炼日期：2026-04-03
> 分类：AI方法论

## 核心观点

1. **Skill = 提示词工程的正式化与可复用化**：Skill 的本质就是把一组提示词写进 Markdown 文件，附上参考文件或脚本。它与提示词工程并无本质区别——只不过变得可复用、可版本管理、可团队共享了。核心价值不是扩展模型能力，而是让重复任务以可预期的方式执行。

2. **Skill 与可复用提示词的三大差异**：渐进式披露（按需加载而非全量塞入）、结构（文件系统本身传递上下文）、可调用额外资源（脚本/MCP/API，不仅有指令还有工具）。这三个差异让 Skill 从「一段文本」升级为「一个可打包的工作流单元」。

3. **隐性知识编码化是团队价值所在**：组织的「专业知识」往往不是事实，而是一系列习惯和检查动作。Skill 让这些程序性知识被打包成 Agent 能直接遵循的东西，新成员不必从头逆向还原流程。

## 关键知识点

### Skill 与可复用提示词的三大差异

| 差异维度 | 可复用提示词 | Skill |
|---------|------------|-------|
| **加载方式** | 全量粘贴进上下文 | 渐进式披露：先看描述→判断相关性→按需加载完整指令 |
| **组织结构** | 存在文档/笔记里，无结构 | 文件夹结构本身传递上下文，可模块化、版本管理、跨项目分发 |
| **资源边界** | 仅限文本内容 | 可包含附录、示例、脚本、评估标准、MCP 服务器、CLI 脚本 |

### 渐进式披露（Progressive Disclosure）机制
- 每个 Skill 附有简短描述说明用途
- Agent 先读描述，判断是否适用当前任务
- 只有确认相关时，才深入加载 skill.md 完整指令
- 类比：去图书馆先看目录→找到相关书→翻到索引→定位需要的章节
- 实例：ImageGen Skill 区分「生成新图」「编辑现有图」「生成变体」三种子任务，各有独立模板，按需加载

### 文件系统即上下文
- 现代模型擅长遍历文件系统
- 文件存放在特定文件夹这一事实，本身就传达了使用信息
- 一个 Skill = 一个可打包的工作流单元，可复用/共享/版本管理/组合

### Skill 的典型组成结构
```
skill-folder/
├── skill.md          # 核心指令（工作流步骤）
├── reference.md      # 内部规范/参考资料
├── examples/         # 示例输入输出
├── scripts/          # 可执行脚本/CLI 工具
└── templates/        # 输出模板
```

### 提示词依然重要的原因
- 模型无法读取心思，只能根据指令和上下文完成任务
- 不需要「神奇措辞」，但清晰无歧义的指令永远有帮助
- Skill 的意义：把提示词工作做一次，而不是每次都重新摸索

### 提示词版本漂移问题
- 同一流程衍生出三四个版本，没人知道哪个正确
- 某次出问题加了规则，某行多余又删了，别人复制再修改
- Skill 的解决方案：单一来源、统一更新、全员执行相同步骤

## 数据与案例

- **OpenAI Agents SDK**：通过代码审查和 PR 草拟等 Skill，加速了开源开发效率
- **ImageGen Skill 实例**：区分三种任务类型（生成/编辑/变体），各自独立模板按需加载，避免上下文污染

## 启发与思考

1. **我们正在做的就是 Skill 实践**：当前 Claude Code 的 skill 系统（包括这个 knowledge-curator skill）完全符合文章描述的模式——skill.md 定义流程 + 渐进式披露 + 可调用额外资源。这篇文章帮助理解了「为什么这样设计有效」的底层逻辑。

2. **与 Floatboat Combo 的对比**：
   - Skill = 开发者手动编写的显性工作流
   - Combo = 从操作中自动蒸馏的隐性工作流
   - 两者互补：Skill 面向有技术能力的人，Combo 面向普通用户
   - 最终都在做同一件事：将隐性流程转化为显性的、可共享的、可稳定执行的东西

3. **隐性知识编码化对 PM 的启示**：团队中大量「只有老人才知道」的做事方式，本质上是未被 Skill 化的程序性知识。PM 可以主动推动将团队的审核流程、需求评审检查清单、上线前检查步骤等编码为 Skill，降低新人 ramp-up 成本。

4. **渐进式披露是上下文管理的最佳实践**：与之前 Agent Harness 笔记中的 Context 管理层呼应——上下文窗口有限，不能什么都塞进去。先给摘要，按需展开，是平衡信息量与上下文开销的关键策略。

5. **「重复超过几次就做成 Skill」的经验法则**：简洁实用。这也是 DRY 原则在 AI 工作流层面的体现。

## 原文精华

> 技能（Skill），本质上就是将这种习惯正式化的方式。其核心很简单：把一组提示词写进 Markdown 文件，有时再附上模型可以调用的参考文件或脚本。

> 从这个角度看，它与提示词工程并无本质区别——它就是提示词工程，只不过变得可复用了。

> 一个组织内部所谓的"专业知识"，往往不是一套事实，而是一系列习惯和检查动作。

> 技能带来的不是新能力，而是一种方式——将隐性流程转化为显性的、可共享的、可稳定执行的东西。

> 如果你发现自己在重复同一个流程超过几次，这通常就是一个信号：它可以被做成一个技能。
