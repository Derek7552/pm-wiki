# Floatboat：AI 工作环境设计哲学与 Combo 工作流蒸馏

> 来源：APPSO 公众号文章（直接粘贴）
> 提炼日期：2026-04-03
> 分类：产品设计

## 核心观点

1. **「工作环境」而非「AI 助手」**：Floatboat 的核心定位是打破软件之间的墙，让文件管理器+浏览器+AI对话共享同一份上下文。助手是你要求它才动的，工作环境是一直在那里的，你在里面做事，它一边帮你做事一边观察和学习。这个定位差异决定了完全不同的产品设计逻辑。

2. **消灭「前摇」= 消灭上下文搬运**：传统 AI 工具的最大摩擦不是模型能力不够，而是用户需要反复提供上下文（写提示词、上传文件、解释背景）。Floatboat 通过让 AI 感知文件间的关联关系，实现「说前半句，AI 自动推理后半句」——本质上是把上下文管理从用户侧转移到系统侧。

3. **Combo = 从工作中蒸馏可复用工作流**：不需要事先设计工作流，正常干活就行，系统自动从操作中提炼出可复用的 skill 组合。这是「自进化 Agent」的第一步——当 Agent 能感知 80% 的操作时，它就有了自进化能力。

## 关键知识点

### 产品架构：三面板统一上下文
| 面板 | 功能 | 传统方案的痛点 |
|------|------|--------------|
| 文件管理器 | 本地文件浏览/预览/编辑 | 需要单独的 Finder/Explorer |
| 内置浏览器 | 网页浏览 + Agent 操控 | 需要在浏览器和 AI 工具间复制粘贴 |
| AI 对话 | 多模型（Gemini/Claude）| 对话结果无法直接落地为文件 |

核心价值：信息从各个方向流进环境，加工过的内容也能流出去，不会被锁死在某一个面板里。用户不再做「人肉 API」。

### Combo 工作流蒸馏机制
- **生成方式**：完成一轮操作后，系统提示是否保存为 Combo；也可主动要求蒸馏
- **复用方式**：遇到类似任务时自动推荐，一键启动
- **本质**：把隐性的工作方法变成显性的可复用资产（类似 Anthropic 的 skill-creator）
- **冷启动问题**：需要用户先投入时间从零教它，大多数人没有耐心
- **解法**：官方预设 Combo + Combo 市场 + 用户自定义微调

### 两种 AI 产品设计哲学
| 哲学 | 隐喻 | 代表 | 用户信任要求 |
|------|------|------|------------|
| 「你放手，我来」 | Agent 全权接管 | Claude Code / Codex | 高（看不到过程） |
| 「你干活，我在旁边」 | 副驾驶 | Copilot / Cursor | 低（保留控制感） |
| Floatboat 的选择 | 主驾副驾来回切换 | — | 中（看得见过程，可随时纠正） |

**关键洞察**：当前阶段大多数人对 AI 的信任还没到「你尽管干，我不用看」的程度。可见过程 + 可随时纠正 = 安心。

### Selfware 协议：A file is an app
- 核心理念：文件不只携带数据，还携带逻辑和结构（.self 格式）
- 类比：Jupyter Notebook（代码+数据+输出打包）+ Docker（运行环境可分发）
- 与 CLAUDE.md / cursor rules 的区别：后者是指令（告诉 Agent 怎么做），Selfware 是执行单元（文件本身就能运行）
- 现状：主要在 Floatboat 自有生态内运转，离广泛采用标准还很远

### 目标人群策略：OPC（一人公司）
- 绕开了「办公产品需要整个组织一起换」的结构性障碍
- AI 能力跃进让 OPC 从口号变为现实：一个人 + 三五个 Agent ≈ 一个草创团队
- 场景覆盖：内容创作（选题→写稿→排版→分发）、电商（选品→上架→客服→投流）

## 数据与案例

- **邀请函演示**：更新 CSV 表格后，Floatboat 自动推理出需要生成新邀请函——AI 理解了操作间的因果关系
- **性能对比**：Gemini 3.1 Pro 驱动下，Floatboat 文件操作用时约为 Cowork/Claude Code CLI 的 1/3
- **办公产品历史规律**：Email+Office → OA → 钉钉/飞书/Slack，每一代都有产品尝试「用我就够了」，但从未有人真正坐稳
- **IACT 协议**：Inline Action-Clicked Text，在 AI 对话结果中加入可点击的行内按钮，减少交互摩擦

## 启发与思考

1. **「前摇」概念对 PM 的启示**：任何 AI 产品都应该审视用户的「前摇成本」——用户在真正获得 AI 价值之前需要做多少准备工作？减少前摇 = 降低使用门槛。这与之前记录的「AI 产品去界面化」趋势一脉相承。

2. **Combo 蒸馏 vs Claude Code Skills**：两者本质相同——从实际工作中提炼可复用的 Agent 行为模式。区别在于 Combo 面向非技术用户（自动提炼），Skills 面向开发者（手动编写 .md）。未来方向可能是两者融合。

3. **信任梯度决定产品形态**：全托管型（Claude Code）和可见过程型（Floatboat）分别适合不同信任阶段的用户。PM 在设计 AI 产品时，应该先评估目标用户的信任水平，而非一味追求自动化程度。

4. **OPC 作为产品策略的巧妙之处**：绕开组织惯性这个最大障碍，从个人切入。这对所有做 AI 办公产品的团队都有参考价值——不要试图改变组织，先服务好最敏捷的个体。

5. **冷启动困境是所有「越用越懂你」产品的通病**：Combo 市场是一种解法，但更根本的问题是如何在用户第一次使用时就展现足够的价值。

## 原文精华

> 在 Floatboat 出现之前，我们其实一直在做不同软件之间的「人肉 API」，每天按几百次复制粘贴，打开不同的软件或浏览器窗口、编辑不同的文件。在 AI 世代在线办公的我们，成了在窗口与窗口之间疲于奔命的赛博搬运工。

> 助手是你要求它才动的，工作环境是一直在那里的，你在里面做事，它一边帮你做事一边观察和学习。

> 「当 agent 能够感知你 80% 的操作的时候，它就有自进化的能力了」

> 兜售「提示词」的时代，快要结束了。你不再需要像个魔法师一样去背诵枯燥的咒语。通过 Combo，Floatboat 可以让用户把他们每天最经常做的固定动作，提炼成独属于自己的「手艺」和数字资产。

> 一个产品不需要统一所有人的工作方式才算成功。如果它能让一部分人——那些一个人干五个人的活、每天在软件之间当搬运工的「OPC」们，每天省出一个小时来做真正需要动脑子的事，那它就已经值得存在了。
