# Hermes Agent — 多模型兼容架构解析

> 一句话总结：**Hermes 通过"统一内部语言 + 外部翻译官"的设计，让用户无感切换 5+ 家 AI 模型厂商，且单点故障时自动降级到备选模型，实现真正的"模型无关"。**

## 1. 为什么要做多模型兼容？

| 业务痛点 | Hermes 的回应 |
|----------|--------------|
| 被单一厂商锁定，涨价或限流时束手无策 | 支持 Anthropic、OpenAI、Google、AWS、xAI 等 5+ 家，随时切换 |
| 不同厂商 API 格式不同，接入成本高 | 用户只需填 Provider 名或 URL，系统自动识别和适配 |
| 某家模型宕机影响全部用户 | 内置多级降级链，主模型不可用时自动切到备选 |
| 新模型发布快，跟不上适配节奏 | 插件式架构，新增一个模型只需写一个"翻译官"文件 |
| 不同模型能力不同（有的能看图、有的能深度推理） | 内置模型能力数据库，运行时自动判断能用什么功能 |

## 2. 核心设计思路（类比）

可以把 Hermes 想象成一个**多语种会议的同声传译系统**：

```
┌─────────────────────────────────────────────────────────┐
│                    Hermes Agent 核心                      │
│                                                          │
│   内部统一用"普通话"（OpenAI 格式）沟通                    │
│                                                          │
└──────────┬──────────┬──────────┬──────────┬─────────────┘
           │          │          │          │
      翻译官 A    翻译官 B    翻译官 C    翻译官 D
      (Anthropic)  (OpenAI)   (Google)   (AWS)
           │          │          │          │
      Claude系列   GPT系列   Gemini系列  Bedrock
```

**关键原则：**
- Agent 核心逻辑只写一套，不关心底层是哪家模型
- 每家厂商有专属"翻译官"（Transport），负责格式转换
- 新增厂商 = 新增一个翻译官，不改动核心代码

## 3. 当前支持的模型厂商

| 厂商 | 代表模型 | 接入方式 | 特色能力 |
|------|----------|----------|----------|
| **Anthropic** | Claude Opus / Sonnet / Haiku | 官方 API / OAuth | 深度推理（Thinking）、超长上下文 |
| **OpenAI** | GPT-4o / GPT-5 / Codex | 官方 API | 函数调用、Responses API |
| **Google** | Gemini 2.5 Pro / Flash | 原生 REST / Cloud Code | 多模态、超大上下文 |
| **AWS Bedrock** | Claude / Mistral / Llama | AWS SDK (boto3) | 企业级安全、IAM 集成 |
| **xAI** | Grok 系列 | 兼容 OpenAI 格式 | 实时信息获取 |
| **OpenRouter** | 200+ 模型中转 | 统一 API | 一个 Key 访问所有模型 |

此外还兼容国产模型代理（MiniMax、DashScope 等通过 URL 尾缀 `/anthropic` 自动识别）。

## 4. 四大核心机制

### 4.1 零配置自动路由

用户**不需要理解**底层 API 协议差异，系统按以下优先级自动判断：

1. 用户明确指定了协议 → 直接使用
2. 填了 Provider 名（如 `anthropic`）→ 自动匹配
3. 只填了 URL → 通过 URL 模式识别（如含 `bedrock-runtime` 就走 AWS）
4. 什么都没填 → 默认走通用 OpenAI 兼容协议

**对产品的启示：** 降低用户配置门槛 = 降低流失率。用户能用，比用户能配置更重要。

### 4.2 模型能力自动探测

不是所有模型都一样强。Hermes 维护了一个**模型能力数据库**（来自社区 models.dev），记录每个模型：

| 能力维度 | 示例 |
|----------|------|
| 是否支持工具调用 | GPT-4o ✓ / 某些小模型 ✗ |
| 是否支持图片理解 | Claude Opus ✓ / 纯文本模型 ✗ |
| 是否支持深度推理 | Claude 4.6+ ✓ / 旧版 ✗ |
| 上下文窗口大小 | 8K / 128K / 1M |
| 最大输出长度 | 4K / 64K / 128K |

系统会根据当前模型的能力**自动开关功能**：
- 模型支持推理 → 自动启用 Thinking 模式
- 模型支持图片 → 允许用户发送图片
- 模型不支持工具调用 → 走文本提取方案兜底

**对产品的启示：** 功能不应该是"全有或全无"，而是根据底层能力优雅降级。

### 4.3 多级降级容错

当主模型不可用时（欠费、限流、宕机），Hermes 会**自动尝试备选**：

```
主模型（用户配置的）
    ↓ 失败
OpenRouter（200+ 模型中转）
    ↓ 失败
Nous Portal（项目方自有服务）
    ↓ 失败
自定义端点
    ↓ 失败
原生 Anthropic API
    ↓ 失败
国产模型（Kimi / MiniMax / GLM）
```

整个过程对用户**透明无感**，对话不中断。

**对产品的启示：** 对于 AI 产品，"可用性"比"用最好的模型"更重要。用户宁可用稍弱的模型继续工作，也不想看到报错。

### 4.4 插件式扩展

新增一个模型厂商的步骤：

1. 写一个"翻译官"文件（约 300-800 行 Python）
2. 实现 4 个标准方法：消息转换、工具转换、参数构建、响应标准化
3. 注册到系统 → 完成

**不需要改动** Agent 核心逻辑、不需要改动其他厂商的代码、不需要停机部署。

## 5. 工具调用的兼容处理

AI Agent 的核心能力是"调用工具"（搜索、执行代码、读文件等），但各家对工具调用的实现差异巨大：

| 厂商 | 工具定义格式 | 调用返回格式 | 结果回传格式 |
|------|-------------|-------------|-------------|
| Anthropic | `input_schema` | `tool_use` block | `tool_result` block |
| OpenAI | `function.parameters` | `function_call` object | `tool` role message |
| Google | `functionDeclarations` | `functionCall` part | `functionResponse` part |
| AWS | `toolConfig` | Converse API 格式 | `toolResult` 字段 |

Hermes 的解法：**内部统一用一种格式（OpenAI 式），每个翻译官负责双向转换**。

核心代码只需要知道"有一个工具被调用了，名叫 X，参数是 Y"——完全不关心底层是哪家的协议。

## 6. 性能优化细节

| 优化点 | 做法 | 效果 |
|--------|------|------|
| 按需加载 SDK | 不用 Anthropic 就不加载其 SDK | 启动快 220ms |
| 翻译官缓存 | 同一厂商复用翻译官实例 | 减少内存开销 |
| 能力数据本地缓存 | 模型信息缓存 1 小时 | 减少网络请求 |
| 无状态设计 | 翻译官不保存状态 | 可安全并发 |

## 7. 对产品设计的启发

### 7.1 "模型无关"是 AI 产品的护城河之一

- 不绑定单一厂商 → 议价能力更强
- 新模型上线可快速接入 → 永远用最好的
- 用户可选择 → 满足合规/成本/延迟等不同需求

### 7.2 用户体验优先于技术正确

- 自动路由 > 要求用户手动选协议
- 能力降级 > 报错"该功能不支持"
- 静默切换 > 弹窗"主模型不可用，是否切换？"

### 7.3 可扩展性的关键：好的抽象层

Hermes 的成功在于找到了**正确的抽象边界**：
- 统一了"什么是一条消息"
- 统一了"什么是一次工具调用"
- 统一了"什么是一个模型的回复"

这三个统一，让上层业务逻辑完全解耦于底层模型差异。

## 8. 总结

Hermes 的多模型架构可以概括为 **"一个标准、多个翻译官、智能路由、自动降级"**：

| 设计维度 | Hermes 的选择 |
|----------|--------------|
| 内部协议 | OpenAI 格式作为通用语言 |
| 适配策略 | 每家厂商独立翻译官，互不干扰 |
| 路由策略 | 零配置自动识别 + 手动覆盖 |
| 容错策略 | 6 级降级链，对话不中断 |
| 能力适配 | 模型能力数据库驱动功能开关 |
| 扩展成本 | 新增厂商 ≈ 1 个文件，不改核心 |

这套设计让 Hermes 成为目前开源 Agent 中**模型兼容性最强**的项目之一，值得任何做 AI 中间件产品的团队参考。
