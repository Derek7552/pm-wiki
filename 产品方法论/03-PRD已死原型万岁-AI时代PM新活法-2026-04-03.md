# PRD 已死，原型万岁：AI 时代产品经理工作方式变革

> 来源：公众号文章（基于 Anthropic Cat Wu 博客解读）
> 提炼日期：2026-04-03
> 分类：产品方法论

## 核心观点

1. **PM 的核心产出物从文档变成了代码/原型**。Anthropic Claude Code 产品负责人 Cat Wu 团队的实践：写完 spec 直接扔给 Claude Code 看能不能跑通，原型质量好就直接成为上线版本的基础。PRD 在模型能力快速迭代的时代变成了"沙滩上的城堡"。

2. **"建造鸿沟"正在消失，PM 的核心价值从"能不能做"转向"该不该做"**。从想法到原型一个下午就能到达，实现成本不再是瓶颈。但你可以快速抵达一百个方向，只有两三个是对的——判断方向才是新的稀缺能力。

3. **模型能力是一块"正在上升的地面"，复杂 workaround 是注定要被删掉的技术债**。Anthropic 每次新模型发布就翻一遍"做不到清单"再试一次，同时删掉旧模型时代堆的 workaround。Opus 4.6 的 system prompt 比上一代精简了 20%。

## 关键知识点

### PM 工作方式对照表

| 维度 | 传统 PM | AI 时代 PM |
|------|---------|------------|
| **核心产出** | PRD、需求文档、流程图 | 原型、eval 定义、Prompt 集合 |
| **规划方式** | 锁路线图，按计划执行 | 短冲刺 + Side Quest 快速试错 |
| **驱动模式** | 文档驱动 | Prompt 成为新的 PRD |
| **技艺定义** | 提前定义确定性 | 加速发现 |
| **角色边界** | PM/设计/工程分明 | 设计师提交代码，工程师做产品决策，PM 写原型 |

### Anthropic 的 Side Quest 机制

- **规则**：任何人（工程师/设计师/PM）有想法，可以自己花时间验证，不走正式评审流程
- **案例**：Claude Code 的桌面版集成、AskUserQuestion 工具、Todo lists 都是 Side Quest 产出
- **共同点**：没写过 PRD，没走评审，某天下午做了原型，同事觉得好用就上线了

### "Do the Simple Thing" 原则

- 在 Agent 系统里，复杂度是指数级放大失败的
- 多加一层 workaround，失败模式就多一类
- 如果模型今天不够好，先别急着搭复杂脚手架——等下一代模型出来，问题可能就不存在了
- **今天的 hack = 明天的技术债**，在 AI 领域"明天"可能就两三个月

### 新模型发布时的操作清单

1. 翻一遍之前的「做不到清单」，用新模型再试一次
2. 删掉旧模型时代堆的 workaround
3. 评估哪些 system prompt 指令已被模型原生支持，可以精简

## 数据与案例

- **模型能力跃迁**：Excalidraw 加表格功能——Sonnet 3.5（2024 底）完全做不到 → Opus 4（2025 中）偶尔成功 → Opus 4.6（2026 初）能现场演示。不到两年，从"不可能"到"能现场演示"
- **METR 数据**：16 个月内模型能处理的任务时长从 21 分钟涨到 12 小时，41 倍提升
- **System Prompt 精简**：Opus 4.6 比上一代精简 20%，因为很多行为已成为模型原生能力
- **效率悖论**：产品团队和研发团队都用 AI 提效了，但整个产品线的交付效率几乎没变——因为做东西更快了，但"该做什么"的决策反而变慢了

## 启发与思考

1. **PM 面试准备的直接启示**：面试沃丰科技 PM 岗位时，可以主动展示"原型驱动"的工作方式——不只是写 PRD 的人，而是能用 AI 工具快速验证想法的人。这是区分度很高的能力信号。

2. **"建造鸿沟"消失后的组织影响**：当实现成本趋近于零，PM 的核心竞争力从"协调资源把事做出来"转向"在一百个方向中判断哪两三个是对的"。产品直觉、用户理解、在模糊中找方向——这些变成了最稀缺的能力。

3. **Side Quest 机制值得借鉴**：传统组织的创新被流程卡住（评审→排期→开发），Side Quest 把"从想到到做到"的缝抹平了。关键是组织要容忍"不走正式流程"的试错成本。

4. **与之前提炼的知识形成闭环**：
   - Vibe Coding 30 个概念（AI方法论/05）讲的是工具层面的"怎么做"
   - 本文讲的是方法论层面的"为什么要这么做"和"做了之后 PM 的角色怎么变"
   - AI 产品去界面化（产品设计/02）讲的是产品形态层面的变化
   - 三者拼在一起：**工具变了→工作方式变了→产品形态变了→PM 角色变了**

## 原文精华

> 你在一块正在上升的地面上盖房子。以前做产品，有一个默认前提——项目开始时技术能干嘛，结束的时候差不多还能干嘛。现在呢？这前提不存在了。

> 从「想到」到「做到」，中间那条缝，快被抹平了。

> 以前，从想法到产品之间隔着一条河：工程师、设计师、测试、排期、开发、联调。这条河就是实现成本。PM 的核心价值是判断哪些想法值得过河。现在，Claude Code 这类工具把河水抽干了。

> PM 的技艺，从「提前定义确定性」转向了「加速发现」。

> 实现的鸿沟在缩小，但「该做什么」的鸿沟在变大。PM 的核心价值，正在从「能不能做」的把关者，变成「该不该做」的判断者。

> 今天的 hack，就是明天的技术债。但在 AI 产品领域，这个「明天」来得特别快——可能就两三个月。
