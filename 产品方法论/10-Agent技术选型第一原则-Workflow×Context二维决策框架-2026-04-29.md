# Agent 技术选型第一原则：Workflow × Context 二维决策框架

> 一句话总结：**Agent = {Workflow} + {Context}，用流程确定性和上下文确定性两个维度画四象限，每个象限对应一种最优技术选型，这是 AI 产品经理做 Agent 架构决策的第一原则。**

## TL;DR

AI 产品经理容易迷失在单体 Agent、多智能体协作、Workflow 等术语中。达摩院提出的 Agent 二维框架用 **Workflow 确定性**（流程是否固定）和 **Context 确定性**（输入是否标准）两个轴将场景分为四个象限，每个象限有明确的技术选型：RPA+AI → 单Agent+RAG → 端到端RL Agent → 多Agent协作。核心决策原则：**Workflow 是对下限的守卫，Agent 是对上限的探索**。

## 核心框架：四象限决策矩阵

### 公式

```
Agent = {Workflow} + {Context}
```

- **Workflow 确定性**：业务流程是否固定、可预编排
- **Context 确定性**：输入信息是否标准化、可预期

### 四象限技术选型

| | **高 Context 确定性**（输入标准） | **低 Context 确定性**（输入多变） |
|---|---|---|
| **高 Workflow 确定性**（流程固定） | 🟢 **象限一：传统RPA（主）+ AI（辅）** | 🔵 **象限二：单Agent + RAG/知识图谱** |
| **低 Workflow 确定性**（路径不明） | 🟡 **象限三：端到端RL Agent（单Agent）** | 🔴 **象限四：多Agent协作（MAS）** |

## 四象限深度解析

### 象限一：高Workflow + 高Context → RPA 为主，AI 为辅

**场景特征**：流程死板，输入标准
**典型案例**：发票报销、固定表单填报、固定化数据分析

**技术选型**：传统 RPA（主）+ AI（辅）

**核心洞察**：
- 这种场景不需要大脑，只需要按流程执行的手脚
- AI 只是粘合剂，解决传统 RPA 处理不了的一点点非结构化数据
- **反直觉**：过度智能反而是浪费，千万别把 Agent 做得太重

**决策关键词**：`能 RPA 就别 Agent`

---

### 象限二：高Workflow + 低Context → 单Agent + RAG

**场景特征**：流程固定，但输入千变万化
**典型案例**：智能客服、合同关键条款提取/审核

**技术选型**：单 Agent + RAG / 知识图谱

**核心洞察**：
- 核心难点在于**对齐**——如何让大模型理解行业专有知识
- 需要强大的 Embedding 和语义解析能力
- 做到精准检索外部和内部资料并给出结果

**决策关键词**：`流程编排 + 知识注入`

---

### 象限三：低Workflow + 高Context → 端到端 RL Agent

**场景特征**：目标清晰，输入充足，但路径不明
**典型案例**：给定资料生成市场分析报告、个性化推荐

**技术选型**：End-to-End RL Agent（端到端强化学习 Agent），即大多数单 Agent

**核心洞察**：
- 这是当前大模型最性感的领域 — **System 2 思维（推理能力）**
- 输入确定但解题路径呈指数级爆炸
- 不能靠硬编码 Workflow，必须依靠模型内化知识进行自主规划
- OpenAI o1 之后，行业最大范式转移：从 System 1（直觉）→ System 2（推理）
- Ilya Sutskever 一直强调：**token 预测只是手段，Reasoning 才是目的**

**决策关键词**：`让模型自己想，别帮它规划路径`

---

### 象限四：低Workflow + 低Context → 多Agent协作

**场景特征**：既要探索方向，又要搜集信息，又要执行工具
**典型案例**：开放式创新方案设计、跨部门复杂协同（类似 Manus、Flowith）

**技术选型**：多 Agent 协作（Multi-Agent System）

**核心洞察**：
- 最复杂的**熵增场景**
- 单个 Agent 的上下文窗口和推理能力在双重不确定性下迅速崩溃→幻觉
- 解决方案是**分治**：用多 Agent 模拟人类组织架构
  - Planner：负责拆解
  - Executor：负责执行  
  - Critic：负责纠错
- 可以是链式执行（流水线），也可以由总 Agent 负责调度

**决策关键词**：`分治降熵，模拟组织`

## 决策原则金句

> **Workflow 是对下限的守卫，Agent 是对上限的探索。**

- 业务**容错率低、SOP 明确** → 尽可能把逻辑固化在 Workflow 中（成本最低的控制不确定性手段）
- 业务**充满未知和开放性** → 赋予 Agent 更多工具和权限（让智能在交互中涌现）
- **实际业务一定是三者结合使用**，单一技术无法解决复合问题

## 可复用的决策 Checklist

拿到一个 Agent 需求时，依次回答两个问题：

1. **Workflow 确定性**：业务流程能不能画出确定的流程图？
   - 能 → 高 Workflow 确定性
   - 不能 / 只能画出模糊方向 → 低 Workflow 确定性

2. **Context 确定性**：输入数据格式和内容是否标准化？
   - 格式固定、字段明确 → 高 Context 确定性
   - 自然语言输入 / 多模态 / 行业知识 → 低 Context 确定性

然后查四象限矩阵，直接定位技术方案。

## 个人批注

这个框架最大的价值不是"告诉你用什么技术"，而是**给了一个判断坐标系**。实际项目中，大多数需求不会干净地落在某一个象限里，而是在象限之间游走。框架的真正用法是：

1. 先判断需求**主要**落在哪个象限
2. 用该象限的技术选型作为**架构骨架**
3. 对于溢出到其他象限的部分，用对应技术做**补充**

比如智能客服（象限二）遇到完全陌生的问题时会滑向象限四，这时候不是推翻单 Agent 架构，而是在单 Agent 内增加一个"升级协作"的 fallback 机制。
