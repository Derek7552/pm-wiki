> 分类：技术趋势

## 核心观点

**Ontology 的工程本质是"统一语义层"**：把企业分散的数据、系统、对象、关系、动作和权限，用业务语言重新组织——不是让业务人员面对表名和字段，而是面对"飞机、航班、订单、旅客、设备、告警、工单"这些真实业务对象。

**可自研落地**：不必购买 Palantir 产品，用关系型数据库 + 图数据库 + 时序数据库 + Action 封装 + 大模型 Function Calling，就能实现一套可治理、可落地、可闭环的轻量 Ontology 系统。

## 关键知识点

### Ontology 的七类核心组件

| 组件 | 含义 | 备注 |
|------|------|------|
| **Object Types** | 业务对象类型（旅客/订单/设备/航班） | 描述现实实体或事件 |
| **Properties** | 对象属性（订单号/设备温度/优先级） | - |
| **Link Types** | 对象间关系（订单属于旅客/工位绑定设备） | 使系统能看"对象网络"而非单对象 |
| **Actions** | 有业务语义的受控动作（确认订单/停止设备） | 单次事务性操作，不直接��数据库 |
| **Functions** | 复杂业务逻辑层（读取属性/遍历关系/编辑 Ontology） | 受控业务逻辑层 |
| **Interfaces** | 对象通用能力协议（相当于抽象父类，支持多态） | 大型企业做深后非常关键 |
| **Security/Governance** | 对象级/属性级/动作级细粒度权限控制 | 与语义层绑定而非散落各系统 |

> Ontology ≠ 知识图谱，≠ 数据目录升级版，而是"语义 + 逻辑 + 动作 + 安全"的企业级统一运营层。

### 五层工程实现架构

```
1. 对象模型层     → 关系型数据库存储统一业务对象
2. 关系层         → 图数据库存储对象关系（可遍历、可计算）
3. 状态与时序层   → 时序数据库存储动态状态（按需，非必须）
4. Action 层      → 手写代码封装受控业务动作，调用业务系统 API
5. 安全治理层     → 与语义层绑定的统一权限 + 审计
```

**底层技术栈选择原则**：取决于业务需要，不是 Ontology 成败的决定因素。真正决定成败的是能否建成"稳定、清晰、可执行、可治理的统一语义层"。

### AI 接入 Ontology 的三步法

**步骤一：暴露元数据，不暴露底层结构**
- 给 AI 的是：有哪些对象、关键属性、对象关系、可调用动作、动作参数和权限
- 本质：给 AI 一本"业务字典 + 操作手册"

**步骤二：封装工具，不让 AI 直接改数据库**
```
getOrder(orderId)
getMachineTemperature(machineId, timeRange)
stopMachine(machineId)          # 真正执行的是后端受控服务
changeOrderPriority(orderId, priority)
```

**步骤三：按需检索，不全量注入 Prompt**
- 先根据用户问题做检索/路由/场景识别
- 只取与当前问题最相关的一小部分对象和动作
- 再把这一小块上下文喂给大模型

> AI 接入 Ontology = RAG + 工具调用 + 权限控制 + 人机协同，不是"把所有知识塞给大模型然后自由发挥"。

### 存储策略：存"核心表示"，不存"全部明细"

**混合模式**（推荐）：

| 数据类型 | 存储位置 | 原因 |
|----------|----------|------|
| 高频访问的核心业务对象（订单/航班/设备/工单） | Ontology 侧额外存一份 | 支持统一权限控制和快速查询 |
| 大体量原始日志/时序点/历史明细 | 留在源系统或专用存储 | 通过视图/函数/时序属性按需挂接 |

> Ontology 中真正值得保存的：全局对象 ID、对象类型、来源映射、核心主数据属性、关键状态摘要、索引字段和高价值关系。数据量可控，价值极高。

### Realtime Mapping 的正确理解

**误区**：要求所有源系统数据实时全量复制。

**正解**：让统一语义层中的业务对象、对象关系和可执行动作始终"足够新鲜、在线可用"。实时可以是流式/秒级/分钟级/准实时，按业务场景定义，不必毫秒级同步。

**原因**：Ontology 面向的是"看见对象—理解关系—触发动作"的运营型场景，对象不够新鲜就无法支撑业务决策与执行闭环。

### 是否需要微调专用大模型？

通常不需要。更现实的做法：通用基础模型 + 领域元数据 + 工具接入 + 评测/提示词工程/上下文裁剪。

**可能需要微调的条件**：
- 领域术语特别封闭
- 问法高度专业化
- 对固定格式输出/动作选择准确率要求极高
- 已有稳定样本集且微调成本可控

## 完整工程案例：民航登机口设备异常处置系统

### 业务目标
当机场登机口设备出现持续异常时，系统自动识别受影响航班和旅客，给出处置建议，人工确认后执行"切换备用设备 + 通知相关岗位 + 生成保障工单"。

### 九步实现流程

**1. 关系型数据库：统一业务对象**

```sql
-- 设备表
device(device_id, device_name, device_type, gate_id, status)
-- 登机口表
gate(gate_id, terminal, gate_code, current_device_id, standby_device_id)
-- 航班表
flight(flight_id, flight_no, departure_time, gate_id, status)
-- 工单表
work_order(work_order_id, object_type, object_id, priority, status)
```

**2. 图数据库：对象关系网络**

```
Device -> BELONGS_TO -> Gate
Gate -> SERVES -> Flight
Flight -> IMPACTS -> PassengerGroup
Device -> HAS_ALERT -> Alert
Alert -> TRIGGERS -> WorkOrder
```

**3. 时序数据库：动态状态**（设备温度/读卡失败率随时间变化）

**4. Action 层：受控业务动作**

```python
switch_to_standby_device(gate_id, failed_device_id)   # 风险：高
create_maintenance_work_order(device_id, alert_level, reason)  # 风险：中
notify_operation_staff(flight_ids, gate_id, message)  # 风险：低
recommend_gate_change(flight_id, candidate_gate_id)   # 只输出建议，人工确认
```

**5. 本体元数据索引表**（三张表）：
- `ontology_object_meta`：对象类型、描述、主要属性、数据来源
- `ontology_relation_meta`：关系类型、业务语义（图数据库存实例关系，此表存类型定义）
- `ontology_action_meta`：动作名称、描述、输入参数、适用对象、风险级别

**6-7. 用户提问 → 后端 Ontology 检索裁剪**

用户问："A12 登机口设备持续异常，会不会影响接下来两小时的航班？"

后端先识别涉及对象（Device/Gate/Flight）→ 检索相关 Action → 拉取实时数据（关系型 DB + 图 DB + 时序 DB）→ 只把这一小块上下文交给大模型。

**8. 大模型 Function Calling**

大模型输出解释性回答 + 发起工具调用：
```json
{"function": "switch_to_standby_device", "arguments": {"gate_id": "A12", "failed_device_id": "GATE_DEV_01"}}
{"function": "create_maintenance_work_order", "arguments": {"device_id": "GATE_DEV_01", "alert_level": "HIGH", "reason": "..."}}
{"function": "notify_operation_staff", "arguments": {"flight_ids": ["CA123", "MU456"], ...}}
```

**9. 执行、写回、闭环**：Action 执行 → 写回对象层状态 + 更新图数据库关系 → 反馈给用户

**完整闭环**：感知异常 → 识别对象 → 穿透关系 → 判断影响 → 选择 Action → 执行动作 → 写回状态 → 反馈结果

## 启发与思考

1. **大模型只是"智能调度员"，底盘是统一语义层**：这颠覆了很多企业的 AI 落地思路——先买大模型 API，再想怎么接数据。正确顺序应该是：先建统一语义层，再让 AI 在受控范围内做调度。

2. **对安全产品的直接启示**（云起无垠场景）：安全场景的核心对象是"资产/漏洞/告警/工单/责任人/处置动作"。如果把这些建成 Ontology，AI 就能在统一语义上做告警关联、影响分析、处置建议——从"AI 问答工具"跳到"安全决策操作系统"。

3. **Action 层是从"看板"到"执行系统"的关键跨越**：很多企业的 AI 建设卡在"能看到问题"，迈不到"能自动处理问题"。Action 层的工程化封装，是这个跨越的具体实现路径。

4. **元数据索引表是可扩展性的关键**：动作和对象越来越多时，不用重构 AI 提示逻辑，只需维护元数据表——这是良好工程设计的典范。

## 原文精华

> Palantir Ontology 的核心，是把企业底层分散的数据和系统，上升为统一的业务对象、关系、动作和权限，再让分析、应用和 AI 都围绕这同一套业务语义运行。

> 真正决定 Ontology 是否能做成的，不是使用哪种数据库和技术组件，而是你有没有建成一套稳定、清晰、可执行、可治理的统一语义层。

> 大模型只是这个系统的"智能调度员"，而真正的底盘，是你自己构建出来的统一语义层。
