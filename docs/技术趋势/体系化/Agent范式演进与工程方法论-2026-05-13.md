## 方向定位

本方向研究 AI 系统如何从「想得更久」迁移到「为行动而想」，以及由此衍生的整套工程方法论。核心问题：当模型能力从「单轮推理」演化到「跨多轮行动 + 环境闭环」，如何在数据、架构、工具接口、自主性边界、组织协作层面重新做工程？

与相邻方向的边界：
- 与「AI 重塑网络安全」的边界：本方向关心 Agent 通用方法论（任意垂直领域适用），后者关心安全场景中 Agent 落地的具体形态。
- 与「LLM 科学」的边界：本方向关心「模型 + harness + 用户」三方耦合后的工程行为，后者关心模型本身的可解释性与训练机理。
- 与「AI 时代组织与个人」的边界：本方向关心 Agent 的能力与工程化范式，后者关心人在 Agent 时代的角色重构（虽两者在 PM/工程师角色重构上必然产生交叉，本文档保留交叉处的工程视角）。

## 核心概念

### 1. 推理式思考 vs 智能体式思考

- **推理式思考（reasoning thinking）**：让模型「想得更久」，通过更长的 CoT、更深的 RL 训练，提升单轮回答的质量。主战场是数学、代码、逻辑——因为奖励信号确定、稳定、可规模化 [来源 #1]。
- **智能体式思考（agentic thinking，又称「为行动而想」「通过行动来推理 reasons through action」）**：核心问题从「能不能想得够久」变成「能不能用撑得起有效行动的方式来想」。在与环境交互中持续修正计划，关心五件事：何时停止思考开始行动 / 工具选择与排序 / 处理残缺有噪声的环境反馈 / 失败后改计划 / 跨多轮保持思路连贯 [来源 #1]。
- **关键判断**：推理链更长不等于模型更聪明。好的思考是「在真实约束下最能撑起行动的那条轨迹，而非最长或最显眼的那条」[来源 #1]。

### 2. Coding Agent 中心论

- **核心论断**：所有 Agent 本质上都是 Coding Agent。OpenClaw 的核心是一个名为 Pi 的 Coding Agent，其余都是套壳；Coding Agent + Skill 基本可以覆盖大多数垂直场景，垂直 Agent 这条路需要被高度怀疑 [来源 #4]。
- **三个阶段**：
  - 第一阶段 Copilot：代码补全，人写代码。
  - 第二阶段 Claude Code：主动 prompt，人当 AI 指挥者。
  - 第三阶段 AI 管 AI：AI 自己发现需求、调度执行，人当质检员 [来源 #4]。
- **典型旁证**：Claude Code 50%+ 使用场景不是 Coding（数据分析、marketing、文案）——印证 Coding Agent 作为通用底座的可组合性 [来源 #4]。

### 3. 可验证性原则（Verifiability Principle）

- **定义**：一切可验证的领域（有客观指标、可自动评估），要么已经属于机器，要么很快就会；一切不可验证的领域（需要品味、判断、创造性），仍然属于人类 [来源 #3]。
- **关键要素**：
  - 「发现」昂贵 vs 「验证」廉价的不对称性——是 AutoResearch 可去中心化（Folding@home 模式）的根本原因 [来源 #3]。
  - 「参差感」（jaggedness）——模型在训练轨道上比光速还快，偏离轨道就像十岁小孩 [来源 #3]。
- **适用场景**：作为「该不该交给 Agent」的第一性判据。可验证领域可全自动闭环；不可验证领域必须保留人类品味节点。

### 4. Agent 自主性（含部署悬挂与三方共建）

- **部署悬挂（Deployment Overhang）**：模型能力远超用户实际授予的自主权。METR 评估 Claude Opus 4.5 能以 50% 成功率完成人类需 5 小时的任务，但 Claude Code P99.9 单轮时长仅约 42 分钟，中位数仅 45 秒 [来源 #5]。
- **三方共建模型**：自主性 = 模型能力 × 用户信任 × 产品默认。任何单一维度都无法完整刻画 Agent 自主性 [来源 #5]。
- **自主性的可观测信号**：P99.9 turn 时长（头部用户的自主性边界）、auto-approve 率（事前授权）、中断率（事后介入）、自我暂停率（模型自限）。

### 5. Agentic Coding（模型层的工程化能力）

- **定义**：模型层针对 Agent 化编程做的专项后训练能力，区别于通用基模。
- **关键能力维度**：超长上下文、原生工具调用、长程多步骤连贯、仓库级理解、GUI 操作、终端 CLI 操作 [来源 #2]。
- **两条路线**：
  - 全能型（Qwen3.6-Plus）：1M 上下文 + 多模态 + GUI 操控，瞄准「氛围式编程」（Vibe Coding） [来源 #2]。
  - 专精型（GLM-5.1）：744B 总参 / 40B 激活 MoE，200K 上下文，编程评测直逼 Claude Opus，开源 MIT 协议 [来源 #2]。

### 6. 智能体优先开发（Agent-First Engineering）/ 仓库即记录系统

- **定义**：以「Agent 是首要工程参与者」为前提反向设计开发流——人类不直接写代码，而是设计环境、明确意图、构建反馈回路 [来源 #6]。
- **核心命题——仓库即记录系统（System of Record）**：Agent 在运行时情境里看不到的东西等于不存在。Google Docs、Slack 讨论、人脑里的隐性知识对 Agent 都不可见，必须全部版本化到仓库 [来源 #6]。
- **激进推论**：代码库首先为 Agent 可读性服务，人类可读性是次优先级 [来源 #6]。

### 7. CLI/Unix 范式（单 run 工具 vs Function Calling 目录）

- **核心论断**：单个 `run(command="...")` 工具 + Unix 风格命令，全面胜过几十个类型化 function calling 工具的目录式架构 [来源 #7]。
- **历史巧合（同义概念：「文本流偶遇」）**：Unix 1969 决定「一切都是文本流」，LLM 2020 决定「一切都是 token」——两个独立设计决定的同构。LLM 训练数据里 CLI 模式密度最高（GitHub README / CI 脚本 / Stack Overflow），因此「不需要教 LLM 用 CLI，它已经会」 [来源 #7]。
- **认知负担差异**：函数选择 = 不相关 API 之间的上下文切换；命令选择 = 统一命名空间内的字符串组合，后者难度低几个量级 [来源 #7]。

## 方法论与框架

### A. 可验证性原则的应用框架

| 任务类型 | 是否交给 Agent | 工程动作 |
|---------|--------------|---------|
| 有客观自动评估（代码、CUDA 优化、超参搜索、测试通过） | 全交，闭环自迭代 | 构建 AutoResearch 式想法队列 + 工作者池 |
| 半客观（数据分析、bug 验证） | 交，但保留人类抽查 | 输出格式标准化、错误自带导航 |
| 不可验证（产品方向、架构权衡、品味、笑话） | 不交 | 把人类 taste 作为 PR 合并门 |

**关键不对称性**：发现极其昂贵（尝试一万个想法），验证极其廉价（跑一遍训练）——这使 AutoResearch 天然适合去中心化 [来源 #3]。

### B. 智能体优先开发的四大支柱 [来源 #6]

| 支柱 | 传统做法 | 智能体优先做法 |
|------|---------|--------------|
| 知识存储 | Google Docs / Slack / 人脑 | 全部编码进仓库 (.md + 代码 + 计划) |
| 架构规则 | Code review 人工把关 | 自定义 linter + 结构测试机械强制 |
| 代码评审 | 人类评审 PR | Agent-to-Agent 评审循环 (Ralph Wiggum loop) |
| 质量治理 | 周五统一清理 | 后台 Codex 任务持续 GC |

**关键工程技法**：
- **AGENTS.md 是目录不是百科**：约 100 行的地图指向 `docs/` 下的结构化文档，避免「一切都重要 = 一切都不重要」+ context 浪费 + 立即腐烂 + 难以核实四大失败模式 [来源 #6]。
- **渐进式披露**：智能体从小而稳定的切入点开始，被指导「下一步去哪看」，由 doc-gardening 智能体定期扫描过时文档 [来源 #6]。
- **架构必须前置严格化**：分层规则（Types → Config → Repo → Service → Runtime → UI，依赖只能向前）由 Codex 生成的自定义 linter 机械强制；错误信息里直接注入修复指令，让错误进入智能体情境 [来源 #6]。
- **依赖选择新标准**：「智能体可建模性」——枯燥技术更友好（可组合性、API 稳定、训练集覆盖多）；有时重新实现比引入库便宜（OpenAI 自写 p-limit 替代品） [来源 #6]。
- **金原则 + 持续 GC**：带主观意见的机械规则 + 后台 Codex 任务定期扫描偏差并发起重构 PR——人类品味一旦被捕捉就持续应用于每一行代码 [来源 #6]。

### C. CLI 范式两层架构 + 三项启发式 [来源 #7]

**两层架构（逻辑必然，不可合层）**：
- **Layer 1 Unix 执行层**：保留纯 Unix 语义（pipe `|` / and `&&` / or `||` / seq `;` 4 算子 + exit code）。**绝不在 pipe 中间截断或加 metadata**——否则 grep 只能搜前 200 行，`[exit:0]` 会被当成搜索目标污染下游。
- **Layer 2 LLM 表示层**：在 pipe chain 完成、最终结果即将返 LLM 时才做处理。针对 LLM 两条硬约束（context 有限且贵 / 仅能处理文本）做四件事：

| 机制 | 解决约束 | 实现 |
|------|---------|------|
| Binary Guard | 二进制污染上下文 | Null byte 检测 / UTF-8 验证 / 控制字符 >10% → 返回导航式错误 `Use: see photo.png` |
| Overflow Mode | context 经济 | 输出 >200 行或 >50KB → 截断 + 落盘到 `/tmp/cmd-output/cmd-{n}.txt` + 提示 grep/tail/head 探索 |
| Metadata Footer | 通用 | 末尾追加 `[exit:0 \| 1.2s]` |
| stderr Attachment | 通用 | 失败时永远 attach stderr |

**三项启发式设计**：
- **渐进式 --help 发现**：Level 0 工具描述里注入命令清单 → Level 1 命令名无参返回 usage → Level 2 子命令缺参返回具体参数。对应 Agent 三种学习模式：「What can I do?」（主动发现）/「What should I do?」（反应纠错）/「How did it go?」（持续学习）。
- **错误消息即导航**：每条错误必含「做错了什么 + 该改成什么」。Agent 恢复成本通常 1–2 步。
- **输出格式一致**：每条结果末尾 `[exit:N | Xs]`——Agent 看几十次后内化模式（exit:1 检查错误 / 长耗时减少调用）。

### D. Agent 自主性的监督策略演化 [来源 #5]

```
新用户（<50 sessions）：
  └→ 20% auto-approve、5% 中断率 → 事前审批为主
有经验用户（750+ sessions）：
  └→ 40%+ auto-approve、9% 中断率 → 监控+介入为主
本质转变：Pre-approval → Monitor & Intervene
```

**四项工程建议**：
1. 投资部署后监控（pre-deployment 评估覆盖不到的模式）。
2. 训练模型识别自身不确定性——Agent 主动暂停提问是重要安全属性。
3. 为「用户监督」而非「用户审批」设计产品（提供可信可见性 + 简单介入机制）。
4. 不要过早规定交互模式——强制「人类必须批准每个动作」会产生摩擦而不一定带来安全收益。

**Claude 自我暂停 vs 人类中断分布**：复杂任务上 Claude 主动暂停频率是人类中断频率的 2 倍以上——在最需要监督的场景下，模型比人类更积极地寻求对齐 [来源 #5]。

### E. 思考与指令模式的合并困境（Qwen3 四阶段后训练经验）[来源 #1]

- **难点不在架构，在数据**：指令模式要求干脆利落、低延迟；思考模式要求多花 token 探索路径。两种行为画像天然互斥。
- **失败画像**：数据没策展好的话两头平庸——思考模式变啰嗦膨胀不果断，指令模式变不稳定且比客户实际需要的更贵。
- **Qwen 的回退**：Qwen3 走融合，Qwen 2507 回到分离路线（独立 Instruct 30B/235B 和 Thinking 版本）。
- **关键判断**：成功合并的目标是「平滑的推理力度光谱」（模型自适应分配算力，而非二元开关）；Anthropic Claude 3.7/4 是集成路线标杆，GLM-4.5、DeepSeek V3.1 也走混合方向。

## 案例库

### 案例 1：OpenAI Codex 7 人团队百万行代码、0 行人类手写代码 [来源 #6]

- **背景**：2025 年 8 月下旬空仓库起步，构建 Codex 相关产品。
- **做法**：人类掌舵，Agent 执行。AGENTS.md（~100 行）+ docs/exec-plans/active/ + 自定义 linter + Ralph Wiggum 循环 + doc-gardening Agent + 按 git worktree 启动应用 + Chrome DevTools Protocol + Vector→Victoria Logs（LogQL）+ Victoria Metrics（PromQL）+ Victoria Traces（TraceQL）。
- **结果**：5 个月、3→7 人团队、约 100 万行代码、1500 PR、人均 3.5 PR/日、约手工 1/10 时间、单次 Codex 任务经常连续 6 小时以上（人类睡眠时），**人类直接贡献 0 行代码**。
- **启发**：吞吐量重构合并哲学——「在智能体吞吐量远超人类注意力的系统中，纠错成本低，等待成本高」，应最小化阻塞性合并门。测试偶发失败应后续重跑而非无限阻塞——这在低吞吐量环境下不负责任，在高吞吐量环境下通常是正确选择 [来源 #6]。

### 案例 2：Qwen3.6-Plus（全能型）vs GLM-5.1（专精型）两条路线分化 [来源 #2]

- **背景**：2026 年 4 月初一周内阿里 Qwen 与智谱 GLM 接连发布 Agentic Coding 专项优化版本。
- **做法**：
  - Qwen3.6-Plus：1M 上下文 + 原生多模态（Visual Grounding + GUI 操作）+ `enable_thinking` 思维模式，定位「氛围式编程」。
  - GLM-5.1：MoE 744B/40B 激活、200K 上下文、28.5 万亿 Tokens 训练、面向 Coding Plan 用户、4 月 6-7 日 MIT 开源。
- **结果**：SWE-bench、Terminal-Bench 双双杀入第一梯队；开发者实测 GLM-5.1 在 Claude Code / Cline 环境编码质量出色（推理速度待优化）；已验证场景包含「一小时从零搭建内容付费系统」「仓库级 Bug 一键定位修复」「多模态 Agent 直接操作 GUI」。
- **启发**：模型竞争节奏从「年度发布」变为「月度迭代」；多模型切换成为标配，对 Agent 框架提出「模型路由」新需求；GUI 操作正成为 Agent 标准能力层级。

### 案例 3：Karpathy 编码/委托比从 80/20 翻转为 20/80 [来源 #3]

- **背景**：2025 年 12 月起 Karpathy 没有亲手打过一行代码。
- **做法**：转型为「向 Agent 传达意志」连续 16 小时；建立 AutoResearch 想法队列（arXiv 论文 + GitHub 仓库 + 人类想法）→ 自动科学家 → 工作者池 → 有效结果走特性分支 → 人类偶尔监控 → 合并主分支。
- **结果**：一个晚上的 AutoResearch 发现了值嵌入权重衰减和 Adam beta 参数的交互效应——20 年经验研究员遗漏的；Dobby 智能家居用 3 个提示词扫描局域网、找到 Sonos、调用 API、接管整栋房子。
- **启发**：新瓶颈不是算力而是「token 吞吐量」——你能同时管理多少个 Agent 会话。「焦虑 = 订阅额度还有剩余就去睡觉」。

### 案例 4：Manus 前负责人 MorroHsu 弃用 Function Calling 的两次生产事故 [来源 #7]

**事故 A（PNG cat 颠簸）**：
- 背景：用户上传架构图，Agent 用 cat 读取，收到 182KB 原始 PNG 字节流。
- 过程：tokenizer 把字节变成成千上万无意义 token 塞进 context；LLM 看不懂开始尝试 `cat -f` / `cat --format` / `cat --type image`，每次收到同样垃圾。
- 结果：20 轮后被强制终止。
- 修复：isBinary() guard + `Use: see photo.png` 导航式错误。
- 教训：**工具结果是 Agent 的眼睛，返回垃圾 = Agent 失明**。

**事故 B（stderr 静默丢失导致 10 轮包管理器盲猜）**：
- 背景：Agent 尝试 `pip install pymupdf`，收到 exit code 127。stderr 含 `bash: pip: command not found`，但代码因「stdout 非空就丢 stderr」的逻辑把它丢了。
- 过程：Agent 只知失败不知原因，开始盲猜：`pip install` → `python3 -m pip` → `uv pip install` → `pip3 install` → `sudo apt install` → … 第 10 次 `uv run --with pymupdf python3 script.py` 才成功。
- 结果：10 次调用，每次 ~5 秒推理。若 stderr 第一次可见，1 次就够。
- 教训：**stderr 是 Agent 在命令失败时最需要的信息——永远别丢**。

### 案例 5：Claude Code 自主性指标 3 个月演化 [来源 #5]

- **背景**：Anthropic 分析 998,481 个随机 API 工具调用 + 50 万 Claude Code 交互会话。
- **数据**：
  - 中位 turn 时长 ~45 秒（3 个月基本稳定，因新用户涌入拉低分布）。
  - P99.9 turn 时长从 <25 分钟翻倍到 >45 分钟。
  - 内部最难任务成功率翻倍（Aug→Dec），同期人均干预次数从 5.4 降至 3.3。
  - 80% 工具调用有至少一种安全防护；73% 有某种形式的人类参与；仅 0.8% 动作不可逆。
  - 软件工程占近 50% 的 agentic 活动。
- **关键观察**：自主性增长在模型发布间平滑过渡，无阶梯跳变——说明自主性增长不纯粹由模型能力驱动，信任积累和产品改进是重要因素。
- **启发**：印证「部署悬挂」与「三方共建」论断，用数据反驳了「监督就是逐项审批」的产品默认。

### 案例 6：OpenClaw 单日 1600 commit 与「AI 管 AI」 [来源 #4]

- **背景**：OpenClaw 作者展示新一代 Agent 形态。
- **数据**：单日 1600 次 commit ≈ 三四人团队一年工作量；Cursor 一周做出 300 万行代码的浏览器；Anthropic 用 Agent 端到端实现 C 语言编译器；7 人团队 ≈ 过去 30-50 人产出；月消耗 2-3 万美元的 90 分用户与 10 分大多数用户差距可达 1000 倍。
- **启发**：Agent 时代不再按组织人数分层，而是按 Token 消耗分层——日消耗 1 亿 Token 的用户 ≈ SaaS 时代的世界 500 强。

### 案例 7：长程任务从状态机到文件系统 [来源 #4]

- **背景**：去年 Agent = 状态机，状态存内存不可逆；几十步任务就崩塌。
- **做法**：状态落到文件上，做错了可以回溯修复；Agent 自己维护 context（不再由人控制）；最好的 context 就是给模型一台电脑或一个文件夹。
- **结果**：从几十步扩展到几百甚至上千步；错误可见可修复。
- **核心转变**：**Less structure, more intelligence**——与 OpenAI Codex「仓库即记录系统」、MorroHsu「Overflow 模式落盘」三处独立但同构的实践 [来源 #4 + #6 + #7]。

## 关键洞察

> 以下洞察必须依赖多篇互相印证或冲突激发，任一单篇都不能独立给出。

1. **Agent 时代的瓶颈从算力转向「token 吞吐量 + 上下文工程」**：Karpathy [来源 #3] 把它命名为「token 吞吐量瓶颈」（你能管理多少 Agent 会话），OpenAI Codex [来源 #6] 用「6 小时单次任务 + 仓库即记录系统」给出工程化解，MorroHsu [来源 #7] 用「Overflow Mode + Metadata Footer + 落盘提示」给出工具接口层解。三篇分别在「人的工作流 / 知识载体 / 工具表示层」三个层面发现了同一个瓶颈——上下文是新的稀缺资源，工程化它就是新的工程学科。

2. **工具接口的反向收敛：CLI 50 年与 LLM 50 年偶遇于「文本流」**：MorroHsu [来源 #7] 提出「Unix 文本流哲学与 LLM token 流本质同构」，曲凯/文锋 [来源 #4] 在另一个方向上印证——「最好的 context 就是给模型一台电脑或一个文件夹」（即把 Unix 文件系统当 context）；OpenAI Codex [来源 #6] 用 Chrome DevTools / LogQL / PromQL / TraceQL 作为 Agent 工具——这些**全部都是 CLI 风格的字符串查询接口**。三家在毫无协调下殊途同归。Function Calling 目录式架构正在被「单 run + Unix 命名空间」反向收敛。

3. **数据与 harness 工程取代 RL 算法成为新竞争焦点**：林俊旸 [来源 #1] 明确判断「推理时代拼 RL 算法和训练流水线；智能体时代拼环境质量、harness 工程、训练推理紧耦合」；Qwen vs GLM 的路线分化 [来源 #2] 体现「不在模型架构上 PK，在 Coding 后训练数据和场景能力上 PK」；OpenAI Codex [来源 #6] 把「自定义 linter + AGENTS.md + Ralph 循环」做成了完整的 harness 工程范式——这条护城河比任何 RL 算法 trick 都更难复制，因为它沉淀在仓库和组织流程里。

4. **自主性是模型、用户、产品三方共建的涌现属性，非单一维度能力**：Anthropic 实证 [来源 #5] 给出最干净的定义——「The autonomy agents exercise in practice is co-constructed by the model, the user, and the product」；Karpathy [来源 #3] 从用户侧给出案例（80/20 → 20/80 是用户主动让渡）；OpenAI Codex [来源 #6] 从产品侧给出案例（4 个 Codex 任务并行 + git worktree 是产品默认抬升自主性）。三方共建意味着「等模型变强自主性就上来」是空想——必须三方协同演化。

5. **Coding Agent + Skill 的可组合性使垂直 Agent 路线被高度怀疑，但 harness 工程留出了护城河空间**：曲凯/文锋 [来源 #4] 明确说「垂直 Agent 这条路需要被高度怀疑」；但林俊旸 [来源 #1] 强调「环境质量、harness 工程是新护城河」——这两条并非冲突，而是暗示：真正的护城河不是「为某垂直场景训一个专用模型」，而是「为某垂直场景构建优秀的 harness + 知识库 + 工具集」。这意味着 SaaS 公司的转型路径是「从 know-how 卖方变成 harness 提供方」，而非「训一个垂直模型」。

6. **「人类不可被替代的点」越来越窄但越来越清晰**：曲凯/文锋 [来源 #4] 列出两点（taste + 协作默契）；Karpathy [来源 #3] 用可验证性原则给出第一性解释（不可验证 = 人类领地）；OpenAI Codex [来源 #6] 在实践中验证（人类只做意图描述、优先级决策、缺失能力诊断）；Anthropic 实证 [来源 #5] 给出量化（仅 0.8% 动作不可逆）。四篇汇聚到同一个判断：人的价值正在从「执行」迁移到「判断、品味、意图设计」，且这部分工作的产品形态尚未完全成型——这是 PM 和工程师在未来 12-24 个月最重要的能力迁移。

7. **「监督 ≠ 审批」是产品默认设置的关键反转**：Anthropic 数据 [来源 #5] 给出最反直觉的事实——经验丰富用户的 auto-approve 率（40%+）和中断率（9%）**同时**上升，这意味着「让 Agent 跑得更猛 + 出问题更激进介入」并非矛盾；MorroHsu [来源 #7] 在工具层给出对应工程化（exit code + stderr + 错误导航让「介入式监督」变得低成本可行）；OpenAI Codex [来源 #6] 在团队层给出对应实践（Ralph Wiggum 循环 + 后台 doc-gardening 而非周五大扫除）。Pre-approval → Monitor & Intervene 是 Agent 产品默认设置的全行业级反转。

## 观点张力（保留冲突）

### 张力 1：Coding Agent 中心论 vs 环境与 harness 才是护城河

- **正方（曲凯/文锋）[来源 #4]**：「所有 Agent 本质都是 Coding Agent」，垂直 Agent 路线需要被高度怀疑，今年的机会是谁能把 Coding Agent 套壳做得足够好。
- **反方（林俊旸）[来源 #1]**：模型只是新一代护城河的一半——「环境质量、harness 工程、训练推理紧耦合」才是另一半。SFT 时代执着数据多样性，Agent 时代应执着环境质量。
- **不强行调和的张力**：两者并非冲突，但暗示 Coding Agent + 优秀 harness 才是答案。如果只押注 Coding Agent 模型本身，会输给那些把 harness 工程化做到极致的对手；如果只押注 harness 工程，会被更强的底层 Coding Agent 模型反包。

### 张力 2：多 Function Calling 目录 vs 单 run 工具

- **正方（主流 Agent 框架，含 LangChain、OpenAI Function Calling、MCP）**：类型化工具目录提供 schema 验证、可观测性、权限分级——是政企合规友好的选择。
- **反方（MorroHsu）[来源 #7]**：2 年实战后弃用 function calling，单 run + Unix 命令在「认知负担 / 调用次数 / 越用越聪明」三个维度全面胜出；并明确划界——CLI 不适合的只有三类（强类型交互、高安全需求、纯多模态原生）。
- **保留张力**：两种实现各有适用边界。MorroHsu 自己也承认安全场景需要 sandbox 隔离 + API 预算 + 用户取消三重保险。真实产品很可能是混合架构——核心通用工具走单 run，边界场景走 typed function。

### 张力 3：思考/指令模式融合 vs 分离

- **融合派（Anthropic Claude 3.7/4 / GLM-4.5 / DeepSeek V3.1 / Qwen3 初版）[来源 #1]**：成功合并需要「平滑的推理力度光谱」，模型自适应分配算力，而非二元开关。
- **分离派（Qwen 2507 回退后路线）[来源 #1]**：发独立的 Instruct（30B / 235B）和 Thinking 版本——承认融合的数据策展难度，企业客户大批量处理任务要的是指令模式的确定性和低成本。
- **张力本质**：这不是「谁对谁错」，而是「为谁服务」的产品选择。开发者市场拥抱融合（光谱），企业市场偏好分离（确定性）。这条路线分化短期内不会收敛。

### 张力 4：仓库即记录系统的可迁移性

- **激进派（OpenAI Codex）[来源 #6]**：仓库是唯一记录系统，所有决策必须版本化进去。
- **保留派（OpenAI 自己也承认）[来源 #6]**：「此行为高度依赖此代码仓库的具体结构和工具，不应假设泛化」——5 个月、7 人团队、空仓库起步是否只适用于 0→1 新产品场景？老系统、多团队并行、复杂合规约束场景能否工作仍未验证。
- **不强行调和**：当前最稳妥的判断是「新建项目按 Codex 范式做，老系统先做 doc-gardening Agent 把隐性知识抽出来，再逐步迁移」。

## 待探索问题

> 文档继续生长的钩子，每个问题指向未来需要补充的源材料方向。

1. **思考模式与指令模式如何在数据层面真正「融合而非平庸」？** Qwen3 四阶段后训练的具体数据策略尚未公开，需要后续 Anthropic / Qwen / DeepSeek 的技术报告补足。
2. **当 Coding Agent 自身的代码也由 Agent 写，谁来界定「代码 = 仓库 = 记录系统」的边界？** OpenAI Codex 案例中「7 人团队 / 0 行人类代码」长期演化的架构连贯性问题——多年尺度上是否会形成人类无法理解的「暗代码」？
3. **Agent 自主性的法律与商业责任如何随「AI 管 AI」演化？** 当 Proactive Agent 自己提需求、搭临时 Agent、执行后销毁，0.8% 的不可逆动作如何归责？这部分需要补充法律/政策方向的源材料。
4. **可验证领域 vs 不可验证领域之间的「灰色区」如何工程化？** Karpathy 给出二分法，但「半客观」任务（数据分析、bug 验证、合规审计）的工程模板尚未沉淀。
5. **CLI 范式在多 Agent 协作下的表达力？** MorroHsu 是单 Agent + 单 run；当多 Agent 协作时，「agent-to-agent 通信」是否也应 CLI 化（`msg send agent-name "..."`）？这与 Anthropic A2A 协议讨论形成对照点，需后续源材料补充。
6. **「命令清单 Level 0 注入」vs「按需发现」的平衡点？** MorroHsu 自己承认这是开放问题——命令多到 50+ 时，全注入会消耗多少 token？是否需要做「按 session 任务类型动态裁剪命令清单」的中间层？
7. **「部署悬挂」的下一阶段：Agent 自主性 P99.9 突破单日级别后，监督机制如何演化？** Anthropic 报告截止数据点为 Aug-Dec，下一份报告若 P99.9 进入数小时甚至数天，事后监控基础设施要怎么建？
