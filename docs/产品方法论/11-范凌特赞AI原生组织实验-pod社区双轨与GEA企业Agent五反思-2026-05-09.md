# 范凌特赞 AI 原生组织实验——pod+社区双轨与 GEA 企业 Agent 五反思

> 来源：腾讯研究院《AI 跃迁者调研》第一期对谈（用户直接提供正文）；嘉宾：范凌（特赞 Tezign 创始人兼 CEO / 同济大学教授 / 设计人工智能实验室主任）；访谈人：余一（腾讯研究院高级研究员）
> 提炼日期：2026-05-09
> 分类：AI产品方法论

## TL;DR

特赞做了 10 年企业级 Agentic AI、自身又用 AI 重做组织——范凌给出**五个反共识**：① **AI 不是帮研发提效的工具，而是帮"需要研发资源的人"拿到研发资源的 Agent**——团队里把 Cursor 用得最好的不是研发而是产品经理和设计师，由此触发反工业革命的"全能人"判断；② **绝大多数公司还停留在 Copilot 阶段（给原职能加 AI），但 AI 能力已发展到"按 AI 重新设计组织"的地步——AI 原生组织不是人的工作流嵌 AI，而是 AI 的工作流嵌人的 judgement，公司本身可以是一个 Agent**；③ 组织改造两刀：实线作战单元 **pod（3–10 人跨职能闭环）**+ 虚线 **Community（横向社区，专设 Leadership Community）**——AI 时代带 100 个 Agent 比带 10 个人更难，pod leader 需要 P&L + 商业直觉 + 韧性 + 耐心，**这些"虚"的素质在 AI 时代变得最具体**；④ 文化引擎是创始人下场 build——三人小团队（CEO + CTO + 产品负责人）做新产品，用户增长远超其他 7 人团队，午饭咖啡时间秀 demo 形成 dogfooding 文化，传染力远超自上而下推广；⑤ **现在是产品过剩、用户不足的时代，AI 时代要 Focus 在 AI 不能压缩时间的能力积累上**——demo 不难做，evals + 真实物理场景 + 独有数据才是壁垒。具体落地：分层上下文系统（公司 schema.md / pod / 个人三层；曾因实时更新烧掉 10 万）+ GEA（Lead Agent 带 Sub-Agent + Skills + Context 的 7×24 虚拟公司）+ Atypica（100 万真实用户主观世界模型，美国教授用 2 万家庭样本生成 1000 画像研究生育意愿）+ Game Lab（真人与 AI 玩电车难题/最后通牒博弈调优）+ SPIS 方法论（Situation-Pain-Impact-Solution，pod leader 30–40% 时间做场景发现）。**业务结果：团队人数不变，去年增长 60%，今年目标 80%–90%**。

## 核心观点

### 观点一：AI 是反工业革命的——它帮"非研发"拿到研发资源，让人从"千行百业"回到文艺复兴式全能状态

范凌的 Aha Moment 来自一次内部观察：**团队里把 Cursor 用得最有创造力的不是研发，而是产品经理和设计师**——他们过去想实现产品功能必须排期等研发，现在 Cursor 直接帮他们拿到研发资源。**Cursor 对他们而言本身就是研发资源**。由此推出底层判断：**AI 不是让专业分工更卷，而是让一个人跨越多个角色——AI 在底层逻辑上是反工业革命的**。工业革命以来"千行百业 + 一人一岗 + 逐级晋升"的组织假设和教育体系，正被从底层动摇。落到组织上是"高内聚（小组三五人内部闭环交付，分工不需要细）+ 低耦合（部门间不需要太多跨部门协调）"——三年前推过一次 pod 失败了，因为大家心理上没准备好；去年 AI 让团队自发减少横向调配，pod 土壤成熟。**关键论据**：Marketing 团队同学一个人用 AI 做出过去 20 人研发团队都不一定搞得定的官网 CMS 系统。

### 观点二：AI 原生组织不是"给职能加 AI"，而是"把人的 judgement 嵌进 AI 的工作流"——公司本身就是 Agent

范凌区分两年的形态演进：**两年前 AI 的主要产品形态是 Copilot（每个人配助手，给个体提效但组织形态没变）→ 去年推理模型让 AI 开始能规划任务 → 今年是 Agentic 元年，AI 真正开始干活**。但绝大多数公司的组织架构还停留在两年前——给职能部门加 AI 工具来提效。事实上 AI 能力已经发展到"按 AI 来重新设计组织"的地步：**公司本身可以是一个 Agent，人只是在这个 Agent 里提供 judgement 的角色**。具体表现：与一家企业合作做了 7×24 小时后台进行新品研发的 Agent——过去人发起研发项目、AI 在绘图调研环节帮点忙、节奏由人控制；新形态是 AI 24 小时不断做研发，只在适当时候停下来问你想法，10 分钟不回复就自己推进下一件事。这是完全不同的节奏感。**这一观点与 45 号 Claude Code Harness 笔记的 KAIROS 常驻 Agent + 46 号 MorroHsu 单 run + 36 号 Hermes 自进化机制构成"Agent 不是嵌进人 / 而是人嵌进 Agent"的方法论三角**。

### 观点三：在 AI 时代要 Focus 在"AI 不能压缩时间"的能力积累上——产品过剩、用户不足

最深层的张力：**AI 让探索成本变低，也让人更焦虑——产品过剩、用户不足，demo 不难做，增长才是稀缺资源**。范凌的应对哲学是 Build in Public（一边 build 一边拿外界反馈，让探索过程本身成为结果的一部分）+ 不要只盯降本增效（去开拓增量市场）+ AI 转型是一把手工程（创始人必须亲自下场写代码做新产品而非死磕旧代码）。**核心反共识**：如果一家企业还在炫耀"我们是技术公司，研发占比 90%"，那它投入在销售和增长上的精力绝对不够——以前一个创始人自己写代码、带 10 人 Marketing 团队，可能是未来 AI 公司的新常态。在产品壁垒维度，**evals + 真实物理场景 + 独有数据**是必须投入的——宁愿花钱买真实用户访谈数据、训自己的基础模型，也不做三个月后会被通用大模型顺手替代的浅层功能。给老 ToB 公司的建议：留 50 人维护老代码对老用户负责，剩下 950 人去做 10 个新产品（举例：喜马拉雅 1000 人研发团队）。给 20 岁年轻人的建议：**学历的必然性在降低，能力的必然性在升高——不要再执着于做"好学生"，去 Build 吧**。

## 关键知识点

### 1. pod + Community 双轨制结构

**实线作战单元 pod（高内聚 + 低耦合）**：
- 规模：3–10 人，内部闭环交付，不依赖跨部门拉通
- 类型：服务客户的 pod / 钻研技术的 pod
- 范凌偏好：越小越好，能 1 人不上 2 人，不存在魔法数字（不同于安克龚银的"6 人=1 大牛+5 会用 AI"路线）
- 客观现实：胜任 pod leader 的人不够多，目前 pod 偏大约 10 人

**虚线 Community（横向社区，培养而非考核）**：
- 工程师 / 产品经理 / 销售 / 写代码 / 做产品 5 个 Community
- 专设 **Leadership Community**：AI 时代带 100 个 Agent 比带 10 个人更难，需要 P&L、商业直觉、责任心、Resilience（韧性）、耐心
- Community leader 让公司更多人学会销售/写代码/做产品，不再让 title 决定能力边界

**配套培训项目 ABC Plus**（AI Builders and Creators Plus）：外请专业讲师手把手教 Claude Code 和各种 Agent 工具，解决 0→1 认知负担。

### 2. pod 落地的两个副作用与一个原则

**副作用 A：AI 用得越好的人越累**——他们容易并行很多任务 + 不自觉把任务边界向外延展（写前端的顺便做产品 → 顺便做 Go-to-Market）。原来靠自己只能做到 60 分，现在有 AI 努力一下能到 90 分，于是没日没夜卷。**这是范凌承认的最大未解问题**。

**副作用 B：必须扶上马送一程**——不能完全指望大家自发摸索，必须做系统培训（即 ABC Plus 项目）。

**关键原则**：范凌**没有压缩团队的"人天"**——本来需要 100 人天的事不会因为现在 3 个人就逼着用 30 人天做完。**省出来的时间是因为减少了横向拉通和反复对齐这些低效环节，而不是靠压榨个人工作量**。

### 3. 角色边界溶解的具体表现

| 原职能 | AI 时代实质 | 触发动作 |
|-------|-----------|---------|
| Marketing 同学 | 实质上是 Marketing Engineer | 用 Claude Code 写脚本抓 LinkedIn 联系人（1000 个）/ 自己用 AI 搭官网 CMS |
| 产品经理 / 设计师 | 实质上是产品工程师 | 用 Cursor 直接出功能，不再等研发排期 |
| 经典研发 | 绝对比例下降 | 从 50% 研发 + 50% 业务的旧配比向"研发占比下降 + 会写代码的人变多"演进 |

**双向变化**：经典研发比例下降 + 非研发岗位变得越来越像 Engineer。Everyone is an engineer，但经典工程师的比例确实会减少。

### 4. pod leader 的"首席 AI 场景发现官"角色与 SPIS 方法论

**场景发现工作流**：
- 公司有深厚文档文化（连会议录音都转成文档）
- 用 OpenClaw 读海量文档总结客户场景
- 过去 3 个月收到 600+ 客户需求 → AI 提炼出约 100 个共性场景

**SPIS 方法论**：

| 字母 | 含义 |
|------|------|
| **S**ituation | 客户现状描述 |
| **P**ain | 痛点 |
| **I**mpact | 引入 GEA 后的影响 |
| **S**olution | 对应方案 |

**对外打法**：拿"别人的痛点"去和客户聊（不再硬做产品 demo）→ 客户感到"你懂我"分享更多场景 → 再告诉客户 AI 能怎么解决。**pod leader 30–40% 时间在场景收集上**。

**与 43 号 AI 红队笔记的对照**：SPIS 与 OWASP 4 大领域 + NIST 4 RMF Function 同属"客户对话结构化模板"——把客户痛点结构化是 ToB AI 销售的杀手级动作。

### 5. 创始人下场 build 的 dogfooding 文化引擎

**关键案例**：CEO 范凌 + CTO + 产品负责人组成 3 人小团队用 AI 做新产品，**带来的用户数和业务增长远超其他 7 人团队**。

**文化传导路径**：
- 不只靠外部讲师培训，必须自己做出东西直接秀给团队看
- 中饭、晚饭、喝咖啡时间找不同人秀 demo（如范凌秀自己的 Wiki 怎么用 AI 搭的）
- pod leader 也跟着秀给客户搭的 demo / Marketing 老大秀自己搭的 CMS
- 形成"自豪地展示自己 build 的东西"的习惯

**国际同好印证**：去年 Shopify、Intercom、Airtable 创始人发布新产品时都说"从来没有像去年那样写过这么多代码"。

**幸福的烦恼**：现在不是大家做得少，**而是重复建设太多**——比如场景库不同组都想自己搞一个，因为 Coding 成本太低。范凌**选择容忍而非高压管控**——AI 变化太快，自己心里也没底，鼓励不同组试不同方向，过几天有更好工具出来才知道最优解。

### 6. 分层上下文系统的工程实现

**三层立体结构**：

| 层 | 内容 | 更新频率 |
|----|------|---------|
| **公司层** | schema.md 等 Guidance（指导性文件）+ 索引指向几亿个文档 | 一周一次 |
| **pod 层** | 各小队基于公司规则搭建专属上下文，不需要全公司共享 | 自主决定 |
| **个人层** | 每个人自己的对话和偏好管理（如不同模型对话上下文统一管理） | 范凌每天清理一次（文档洁癖） |

**关键经验教训**：曾试过实时更新上下文，**很快烧掉 10 万元** → 现在公司级一周更新一次。

**返璞归真原则**：以前像复杂 Wiki / 知识库 → 现在就是一个简单文件夹系统 + Guidance。**只要上下文系统足够简单清晰，它就能像索引一样指向过去几亿个文档**。

**两个反共识**：
1. **不实时更新** —— 烧钱
2. **更倾向给框架而非塞细节** —— 不希望 AI 太顺着自己走，跟下属聊天若永远顺着也没意义，需要不同角度

**核心难点：权限与保密**——范凌投屏搜公司 Wi-Fi 密码，AI 顺带搜出只有他权限能看的保密密码直接投屏。**核心保密数据范凌倾向于压根不放进上下文系统**。

### 7. GEA（Generative Enterprise Agent）架构

**核心设计哲学**：不执着于单个 Agent，把重点放在 **Context（上下文）+ Orchestration（编排）** 上。

**架构组件**：
```
Lead Agent
  ├─ Sub-Agent 1
  ├─ Sub-Agent 2
  ├─ Sub-Agent N
  + 企业 Skills（SOP/能力固化）
  + 企业 Context（共享世界模型）
```

**应用领域**：用户洞察 / 内容增长 / 产品创新 → 搭出企业专属的 Agent 项目组 = **后台 7×24 小时跑的虚拟公司**。

**理论引用**：Jack Dorsey（Block 创始人）观点——每个企业都有一个自己的"世界模型"，即共享 Context System。

**与 44 号 MCP 笔记的"分层互补"对照**：API + CLI + MCP + Skills + Plugin 五层叠加，GEA 实际就是 Skills + Context + Orchestration 的企业级组合实现。

### 8. Atypica：理解人的 AI（主观世界模型）

**数据基础**：约 100 万个真实用户的表达、故事、认知和行为数据。

**核心使命**：用 AI 模拟人的主观世界（不是替代人，而是模拟人）。

**典型案例 — 美国教授生育研究**：
- **背景**：研究中国年轻家庭为什么不愿意生孩子
- **样本**：2 万个家庭真实数据
- **做法**：AI 采访 2 万样本 → 提炼出 1000 个典型人物画像 → 让代表夫妻和双方父母的 AI Persona 聚在一起开会讨论 → 不断抛政策变量（生育补贴给老人 / 老公工资翻倍）观察行为变化
- **价值**：真人讨论这个话题会非常尴尬 → AI 分身突破社会研究的伦理与场景限制
- **范畴**：把 AI 用在社会科学研究上

### 9. Game Lab：用经济学博弈调优 AI evals

**核心问题**：AI 模拟人的准确性如何评估？

**做法**：让真人和 AI 玩同样的经济学博弈游戏：
- 电车难题（Trolley Problem）
- 最后通牒博弈（Ultimatum Game）

**调优逻辑**：用真人的数据不断调优 AI 的表现，直到 AI 的决策和真人无限接近。

**链接**：game.atypica.ai

**对应通用洞察**：AI 产品从 demo 到 production 之间横着一道 evals 的鸿沟——这是国内 AI 圈讨论最少、海外 AI 市场最大的环节。**绝大多数客户还没在 evals 上做投入**，只是拿工具试了试看到可能性，但要让全公司安全稳定使用产生价值，没 evals 系统做不到。

### 10. AI 时代领导者的硬能力清单

**软能力（共识）**：AI 能力 / 学习能力。

**硬能力（范凌强调，反共识）**：
- **Leadership** — 领导力
- **看 P&L 的能力** — 即使是研发出身也得懂 P&L
- **极致的责任心**
- **耐心** — 当前变化极快的节奏下耐心反而非常重要
- **Resilience（韧性）**

**为什么"虚"的素质变得最具体**：AI 时代一个人可能要带 100 个 Agent 干活，或带领 100 个熟练使用 Agent 的员工——管理一个 100 人产能的小组长本质上是在管理一家中型公司。

**培养路径**：Leadership Community + 商学院课程 + 每个人都值得上一上商业课。

**反差洞察**：技术问题大家每天主动或被动地学习，但**商业思维和管理能力反而成了短板**。

### 11. 封闭问题 vs 开放问题——AI 价值最大的地方

| 类型 | 特征 | AI 主流应用 | 范凌主张 |
|------|------|-----------|---------|
| **封闭问题** | 有明确可衡量标准，可不断逼近完美 | 大多数 AI 助理产品 | 优化导向 |
| **开放问题** | 没有标准答案，靠经验和智慧探索新机会 | 蓝海市场探索 | **AI 在开放问题上价值更大，应勇敢用 AI 探索没有标准答案的市场** |

**反直觉论断**：很多人只把 AI 当作解决封闭问题的助理去追求精准和优化——但 AI 在解决开放问题时价值更大。

## 数据与案例

### 业务结果数据

- **团队人数**：保持不变
- **去年业务增长**：60%
- **今年目标**：80% – 90%
- **过去 3 个月客户需求**：600+ 个
- **AI 提炼共性场景**：约 100 个
- **pod leader 场景收集时间占比**：30% – 40%

### Atypica 案例完整链条

- **背景**：美国教授研究中国年轻家庭为什么不愿意生孩子
- **样本**：2 万个家庭真实数据
- **做法**：AI 采访 2 万样本 → 1000 个典型画像 → 模拟夫妻 + 双方父母 4 类 AI Persona 多方对话 → 政策变量（补贴给老人 / 老公工资翻倍）注入观察
- **价值**：真人讨论极度尴尬，AI 模拟突破伦理与场景限制
- **意义**：AI 用在社会科学研究的早期范式

### 创始人下场 build 案例

- **3 人小团队**：CEO 范凌 + CTO + 产品负责人
- **Vibe Coding 实践**：自己动手写代码做新产品
- **结果**：带来的用户数和业务增长**远超其他 7 人团队**
- **国际同好印证**：Shopify / Intercom / Airtable 创始人都说"从来没有像去年那样写过这么多代码"

### Marketing 同学 CMS 案例

- **场景**：做官网 CMS 系统
- **过去成本**：可能需要 20 人研发团队都不一定搞得定
- **AI 时代成本**：1 个人用 AI 搞定
- **本质**：Marketing 同学实质上已经是 Marketing Engineer

### 上下文系统烧钱事故

- **做法**：曾试过让上下文系统实时更新
- **代价**：很快烧掉 10 万元
- **修复**：公司级改为一周更新一次；个人级范凌自己每天手动清理

### Wi-Fi 密码尴尬事件

- **场景**：范凌投屏现场搜公司 Wi-Fi 密码
- **结果**：AI 顺带搜出一堆只有他权限能看的保密密码直接投屏
- **决策**：核心保密数据压根不放进上下文系统

> ⚠️ 待验证：原文提到 "GPT 是由 35 个 pod 做出来的"——OpenAI 官方未公开此组织数据，需对照公开信息核实；Meta 早期 pod 模式的细节也需核实首创时间点。

## 启发与思考

### 与个人工作的关联（云起无垠产品总监 / 安全 + AI 方向）

1. **范凌的 5 条反共识可作为云起内部 AI 转型 OKR 的"诊断清单"**：① 哪些岗位"非研发但需要研发资源"已经先用 AI 跑起来？（PM/设计/红队/法务）；② 云起的组织还停在 Copilot 阶段还是已开始按 Agent 重新设计？；③ 云起 pod 的组织形态是否成熟？哪些 pod leader 缺 P&L 和领导力？；④ 创始人是否亲自下场写代码？；⑤ 云起在 evals + 真实物理场景 + 独有数据上的投入比例够不够？这 5 条可作为 2026 Q3 OKR 复盘的 5 个硬指标。
2. **pod + Community 双轨制对云起政企客户交付有直接借鉴意义**：客户内部安全部门 + IT 部门 + 业务部门常因协调断裂导致 AI 安全产品落地失败——可以建议客户用"实线 pod（具体业务场景闭环）+ 虚线 Community（培养跨界能力）"的双轨结构改造，云起作为顾问参与设计——这是从"卖产品"升级到"卖组织变革方法论"的蓝海。
3. **GEA 的 Lead Agent + Sub-Agent + Skills + Context 架构可直接套用到云起 SOC Agent 设计**：云起 AI 安全产品如果按 GEA 架构重做：1 个 Lead SOC Agent + 多个 Sub-Agent（漏洞扫描 / 威胁情报 / 事件响应 / 报告撰写）+ 客户专属 Skills（OWASP/NIST/CSA 风险分类规则、政策合规清单）+ Context（客户私有资产清单、历史事件库），就是一个 7×24 小时虚拟 SOC 团队——这与 42 号 GigaOm 雷达里 Exaforce 的 Exabots 架构、45 号 Claude Code 的 Sub-Agent 编排同构。
4. **Atypica 主观世界模型对云起红队 PoC 数据集是直接可借鉴的范式**：44 号笔记 Connor 的 114 个 PoC 数据集是组件中心视角；Atypica 主观世界模型 + Game Lab 博弈调优是行为视角——把"攻击者人格" + "防御者人格" + "中立 IT 人员人格"做成 AI Persona，让它们在虚拟 SOC 场景下博弈，反向产出"攻击者最可能下一步做什么"的高质量训练数据。这是"AI 红队 + AI 蓝队 + AI 中间人"的三方对抗范式。
5. **SPIS 方法论应该立刻成为云起销售对话的标准模板**：拿 600 个客户需求提炼 100 个共性场景 + Situation/Pain/Impact/Solution 四字段结构化——这是 AI 时代 ToB 销售的杀手级动作。云起销售可以立刻把现有客户需求池跑一遍 SPIS 提炼，下个月销售对话就用"行业共性痛点+特赞客户语言"开场而非产品 demo 开场。
6. **Evals 缺位是云起 ToB 客户的隐形需求**：范凌点出"绝大多数客户还没在 evals 上做投入"——云起的 AI 安全产品本质上就是给客户的 AI 系统做 evals（红队评估、漏洞挖掘、行为偏离监控）。可以反向定位云起为"客户的 AI evals 基础设施"而非"AI 安全工具"，借范凌这句话作为 BD 对话的切入点。
7. **范凌"AI 不能压缩时间"的清单是云起壁垒选择的诊断**：哪些云起能力是 AI 不能压缩时间的？真实漏洞挖掘数据 + 客户安全工程师访谈语料 + 历史 PoC 验证报告 + 政企客户合规交付经验——这些都不是模型升级能替代的。云起应该把这 4 类资产作为产品 roadmap 第一优先级，而不是把研发资源投在"提示词优化 / Agent 框架包装"这类三个月被 GPT-5/Claude 5 顺手替代的浅层功能。
8. **创始人下场 build 的标准对云起一号位有挑战**：范凌+CTO+产品负责人 3 人小团队 vs 云起一号位是否每周亲自写代码？这是 AI 转型一把手工程的具体落地标准——范凌在原文里直接说了"AI 转型是一把手工程，创始人必须重新回到一线"。

### 可落地的行动项

- **5 条反共识自检清单**：在云起 2026 Q3 OKR 复盘里增加范凌 5 条反共识对照打分（1–5 分），找出云起当前最薄弱的一条作为下季度焦点。
- **pod 组织变革 PoC**：选择云起一个客户交付场景（如某政企客户的红队评估项目），改成 3–5 人 pod 形态运行 1 个季度，对比"pod 模式 vs 传统职能模式"在交付时长 / 客户满意度 / 团队疲惫度三个维度的差异。
- **SPIS 客户对话模板上线**：48 小时内把云起当前客户需求池跑一遍 SPIS 结构化，整理出"行业共性 20 痛点 + 推荐方案"销售开场材料。
- **评估范凌"封闭 vs 开放问题"路线**：列出云起当前所有 AI 产品功能，按封闭 / 开放问题分类，开放问题占比是多少？是否存在"AI 探索蓝海市场"的产品方向被忽略？
- **GEA 架构应用 PoC**：选云起一个 SOC 模块按 GEA 架构（Lead + Sub + Skills + Context）重构，A/B 对比与现有架构的客户体验和总 token 消耗。
- **创始人下场 build 倡议**：建议云起一号位每周固定半天亲自写代码做新产品 PoC，参考范凌+CTO+产品 3 人组形态。
- **跟踪 Atypica + Game Lab 后续动态**：作为"AI evals 国内最佳实践"持续跟踪——这是云起做 AI 安全产品 evals 模块的直接参考。

### 值得进一步探索

- **"重复建设容忍度"在政企客户场景的可行性**：范凌容忍 pod 间重复建设场景库——这在创业公司 OK，但政企客户对"统一标准 + 唯一数据源"要求极高，云起做政企客户时如何在容忍创新与控制重复之间平衡？
- **AI 的"复利工程"如何设计**：范凌坦言"模型自我学习的问题没完全想透"——云起的安全 Agent 如何把每次客户事故的失败原因沉淀为可跨客户复用的 Skills？这与 33 号 HL（Heuristic Learning）笔记里"经验存进代码而非权重"的思路同构——HL 范式在云起场景的落地路径。
- **企业级 Context 系统的权限管理工程**：范凌投屏密码事故揭示了 RBAC + ABAC + Need-to-Know 在 LLM 上下文层的复杂性——云起做政企产品时这是必经技术挑战，需要在云起的安全 Agent 中先做出标杆方案。
- **pod 内部"重复建设"在客户合规审计场景的边界**：政企合规审计要求"统一审计轨迹 + 唯一权威数据源"——云起的 pod 重复建设若导致客户事件响应过程中出现两份不一致的报告，会构成合规事故。
- **GEA 与 MCP（44 号笔记）的协议层互补**：GEA 是企业级编排层，MCP 是工具调用协议层——这两个层在企业部署时如何协同？是否需要 MCP-over-GEA 的扩展协议？
- **范凌的"耐心是硬能力"判断与 ToB 销售周期的张力**：政企销售周期长（6–18 个月），AI 时代客户期望"立即见效"——耐心既是必需品又是奢侈品，pod leader 的耐心是怎么训练的？

## 原文精华

> "AI 不仅仅是一个帮研发人员提高效率的工具，它更是一个能帮"缺乏研发资源的人"解决问题的 Agent。AI 的出现让我们变得更全能，而不是让原有的专业变得更卷。"

> "虽然很多人说 AI 是下一次工业革命，但我反而觉得，AI 在底层逻辑上是"反工业革命"的。它打破了"千行百业"和严密的职业划分，让我们回到了类似文艺复兴时期的状态——一个人可以是一个"全能的人"。"

> "在 AI 时代，你一个人可能要带 100 个 Agent 干活；或者你作为管理者，你要带领 100 个熟练使用 Agent 的员工。这个时候，像领导力、Ownership、责任心，以及 Resilience，这些过去听上去有点"虚"的素质，在 AI 时代反而变成了最具体、最核心的硬能力。"

> "时至今日，绝大多数公司的组织架构还停留在两年前 Copilot 的形态——就是给各个职能部门加上一点 AI 工具来提效。但事实上，AI 的能力已经达到了"你可以按照 AI 来重新设计组织"的地步。公司本身就可以是一个 Agent，人只是在这个 Agent 里面提供"判断"的角色。"

> "AI 原生组织不等于"每个人都用上了 AI"，而是用 AI 的逻辑重构了组织。"

> "现在的烦恼不是大家做得少，而是重复建设太多了。"

> "如果有企业还在炫耀"我们是技术公司，研发占比 90%"，我心里就会想：那你投入在销售和增长上的精力绝对是不够的。"

> "AI 时代，必须 Focus 在那些"AI 不能压缩时间"的能力积累上。"

> "学历的必然性在降低，能力的必然性在升高。"

> "不要再执着于做一个"好学生"了，去 Build 吧！"
