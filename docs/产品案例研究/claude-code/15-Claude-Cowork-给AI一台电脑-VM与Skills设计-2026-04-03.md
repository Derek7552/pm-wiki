# Claude Cowork：给 AI 一台电脑而非一堆工具

> 来源：Founder Park 访谈 Anthropic Claude Cowork 产品负责人 Felix Rieseberg（Latent Space 采访）
> 提炼日期：2026-04-03
> 分类：技术趋势

## 核心观点

1. **给 AI 一台完整的电脑，比逐个接入工具更高效**。Felix 的比方：如果你雇了一个开发者却告诉他只能通过邮件收发代码，这有多荒谬？Cowork 通过 VM 让 Claude 像真人一样在安全隔离环境中自由安装软件、执行任务，而不是困在输入框里。这解决了"批准疲劳"问题——每步都审批就不是自动化。

2. **执行成本变廉价后，平台基础设施的价值反而上升**。不是所有东西都值得从零重建。Cowork 明确选择在 Claude Code 之上构建，而非重写。变得廉价的是"把乐高积木组合成对用户有意义的东西"，而非重建积木本身。超个性化软件不是未来——你总需要和别人协作。

3. **高度专业化的 AI 应用短期有效，但生命周期不长**。一旦模型泛化能力提升，那些靠精细脚手架补足模型能力的产品就会被淘汰。与其过度投资 workaround，不如给 AI 尽可能多的能力 + 确保安全，然后等下一个模型。

## 关键知识点

### Cowork 架构核心：VM + Claude Code + Skills

| 组件 | 作用 | 设计思考 |
|------|------|----------|
| **VM（虚拟机）** | 安全隔离环境，Claude Code 在其中运行 | 让 AI 自由安装软件（Python/Node.js），严格网络进出控制 |
| **Claude Code** | 底层 Agent 框架 | 不重建，在其上构建。"用户友好版"实际是"超级集合" |
| **Skills** | Markdown 文件描述任务 | 意外诞生：本来要建自定义工具连接数据仓库，发现写个 markdown 让 Claude 自己搞定更好 |
| **Dispatch** | 手机远程指挥电脑上的 Cowork | 复用 Cowork 完整底层能力，本地文件/浏览器/邮箱/网盘/Slack 都可远程调用 |

### Skills 的诞生路径（产品设计教科书级案例）

1. 最初目标：让非编码用户做数据分析
2. 需要连接数据仓库 → 准备构建自定义工具
3. 发现：写个 markdown 文件（"亲爱的 Claude，这是端点和 API"）比建工具更有效
4. 模式复制：所有任务都用 markdown 描述 → 打包成 Skills
5. 特性：**极易创作**（一条短信都可以是 Skill）+ **高度个性化**

### Agent 安全的"瑞士奶酪模型"

- 不需要等 100% 模型对齐，也不需要完美系统
- 多层防护，每层有漏洞，但组合起来足够安全
- VM 隔离 + 网络进出控制 + 自然语言权限声明
- 核心洞察：逐步审批 ≠ 安全，只是把自动化变成了人工操作

### 本地计算 vs 云端的取舍

| 方案 | 优势 | 问题 |
|------|------|------|
| **本地 VM（Cowork 选择）** | 天然拥有用户所有工具和登录态 | 受限于本地硬件 |
| **云端克隆** | 随时可用，不占本地资源 | 银行等网站检测异地登录会锁号；权限管理复杂 |

Felix 的判断：硅谷低估了本地计算的价值。让 Agent 最有效的方式是把它放在你工作的地方。

### 产品方法论变化

| 旧方式 | 新方式（Anthropic 内部） |
|--------|--------------------------|
| 找客户→写 spec→设计→执行 | 把所有候选方案都快速构建出来，选最好的 |
| 在技术 A 和 B 之间做昂贵决策 | 都建出来，小型焦点组测试，选更好的 |
| 10 天从零做产品 | 10 天从大量内部原型中组合成 Cowork |

## 数据与案例

- **Cowork 开发周期**：10 天上线，但并非从零开始——Anthropic 内部有一年半的原型积累，原型和演示优先是内部文化
- **Skills 使用案例**：Felix 每天早上让 Cowork 检查日历冲突，并设置优先级规则（"如果 Dario 安排了会议，不要重新安排"）
- **用户案例**：让 Cowork 从 Zoom 下载录音→压缩→上传 YouTube，全程自动完成，然后打包成 Skill 复用
- **VS Code 类比**：最初定位为"更用户友好的 Visual Studio"，被质疑"不适合严肃开发者"，最终因可扩展性成为主流。Cowork 正经历类似路径
- **Cowork 三大方向**：在电脑上做更多事 / 独立完成更长任务 / Remote 功能即将到来

## 启发与思考

1. **"Factorio 式自动化"是 Agent 产品最佳用户心智模型**。从小处开始自动化→成功后不断扩展范围→随信任增加教会 AI 取代你。这个渐进式信任建立过程，比一上来就宣称"AI 替代一切"有效得多。

2. **Skills = markdown 文件的产品启示**。最终最有效的 Agent 工具接口不是 API、不是 SDK，而是一段自然语言描述。这和 Claude Code 的 CLAUDE.md、Cursor 的 Rules 是同一个模式——**自然语言成为新的配置语言**。

3. **"给 AI 一台电脑" vs "给 AI 配工具"是两种根本不同的产品哲学**：
   - 配工具（MCP 路线）：逐个接入，精确控制，但覆盖面有限
   - 给电脑（VM 路线）：一次性获得所有能力，安全靠隔离而非审批
   - 两者不矛盾但优先级不同——Cowork 选择了后者作为基座

4. **多 Agent 协作的终局可能很简单**：不是构建复杂的 Agent 间通信协议，而是直接给每个 Agent 一个 Slack/Gmail 账号，用人类已有的工具协作。这个思路极其朴素但可能是对的。

## 原文精华

> 如果你雇了一个开发者，却告诉他只能通过邮件收发代码。这有多荒谬？我们对待 AI 就是这么做的。

> 通常我们说想法廉价，执行才是难点。但现在不是了。现在 Anthropic 内部的做法是：不要写备忘录了，直接把所有候选方案都快速构建出来。把它们都造出来，然后选最好的。

> 这有点像玩 Factorio，但是为你自己的生活。你从很小的地方开始，开始自动化一些很小的事情，一旦成功了，你就不断添加到这个自动化帝国中，让你的生活越来越轻松。

> 我越来越倾向于这样的想法：与其过度投资这些脚手架修正，不如给它尽可能多的能力，努力确保安全，这样最坏的情况也不会太糟，然后等下一个模型发布。

> 我通常认为硅谷整体上低估了本地计算机的价值。我的默认论点总是：你们为什么都在用 MacBook 而不是 iPad 或 Chromebook？

> 我们不需要构建完美的系统。不需要等待 100% 的模型对齐。我们可以依靠行业长期使用的「瑞士奶酪模型」，采用多层防护的方式，每层都有一些漏洞，但组合起来就很安全。
