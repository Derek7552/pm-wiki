# Lazada 广告 AI Agent 实战——ChatLSS 与 Paimon 双产品架构深析

> 来源：文章《AI Agent 在国际化广告主业务的探索》（文字输入，无 URL）
> 提炼日期：2026-04-13
> 分类：AI 应用案例

## TL;DR

Lazada 在广告投放操作智能化方向落地了两款 B 端 AI 产品：ChatLSS（对话式 Copilot，解决"人找功能"）和 Paimon（场景预测性干预，实现"功能找人"）。核心架构方法论包括 P-R-C-D-A 五步闭环、四大设计原则、三模式能力中心，以及 PARSER 框架解决 Agent 高 RT 与规划不稳定问题。

## 核心观点

**1. 广告 AI 商业化的三大方向，中国平台与欧美平台打法不同**

行业层面，AI 在广告商业化的落地分三个方向：交互形态重构（对话式广告，重塑流量入口）、创意生产力革命（AIGC 解决素材产能）、投放操作智能化（降低操作门槛和决策成本）。中国平台（字节、阿里）更激进——视频化、虚拟人、托管型投放；欧美平台（Amazon、Google）更保守——辅助修图、赋能型教导商家。Lazada 聚焦于投放操作智能化，采用"赋能+托管"融合路线。

**2. 广告 AI Agent 的三大根本性痛点不是模型不够聪明**

核心矛盾在于：执行鸿沟（自然语言意图→确定性 API 参数，极易引发幻觉与资损）、供需不对称（平台有海量策略但商家视而不见，"人找功能"导致机会损失）、专业度缺失（通用大模型缺乏垂类广告知识，无 RAG 支持的 AI 像"什么都懂但什么都不精的实习生"）。针对这三点，需要构建"知识驱动、主被动一体化"的智能体网络。

**3. ChatBot（被动响应）和 Paimon（主动干预）代表两种不同的 AI 产品形态，各解不同问题**

ChatLSS 解决的是"我有问题找 AI"——降低操作门槛，让 Ads Manager 变成对话框。Paimon 解决的是"AI 主动发现我的机会"——打破商家的"隧道效应"，在不打扰操作流的前提下将 AI 决策"无感"融入工作流。两者共用同一套推理内核（双向同构原则），但产品形态和架构侧重完全不同。

## 关键知识点

### 大模型三大商业化核心特质

| 特质 | 传统模型局限 | 大模型能力 | 商业化价值 |
|------|------------|-----------|------------|
| 深度语义理解（World Model） | 关键词匹配，难捕捉潜台词 | 能推理"海边度假"→"波西米亚风长裙"的潜在需求 | 精准营销基石 |
| 多模态生成（Native Multimodality） | 仅处理文本 | Text-to-Image/Video，商品可实时渲染 | 素材产能革命 |
| 逻辑规划与 Agent 能力（Reasoning & Planning） | 无法拆解复杂任务 | CoT + 工具调用，从"回答问题"到"解决问题" | 投放自动化基础 |

### 四大架构设计原则

**① 双向同构原则**：显式提问（Chat）和隐式行为信号（系统后台），在架构内部统一转化为"意图对象"处理。好处：共用同一套推理大脑、RAG 知识库、原子工具库，极大降低维护复杂度。

**② 检索增强的认知推理（RAG First）**：模型推理前，必须先检索业务知识（流量竞争环境、相似商家成功案例）。知识前置确保建议有据可依，通过将生成内容与 RAG 检索事实严格对齐来抑制幻觉。

**③ 认知与执行协同解耦**：AI 层（概率性）负责意图理解与策略生成；工程层（确定性）负责原子工具调用与状态管理。分层设计保留 AI 灵活性，工程手段保障执行安全。

**④ 分形协作与协议优先**：Multi-Agent 模式各司其职，引入 MCP 作为统一连接标准，实现异构工程系统（广告中台）与 AI 的标准化互联。

### P-R-C-D-A 通用闭环流程

整合显式提问和隐式需求的统一抽象：

| 阶段 | 全称 | 核心动作 |
|------|------|----------|
| P | Perception（感知） | 接收用户自然语言输入 OR 监听行为流/经营数据 |
| R | Retrieval（检索） | RAG 加载知识/记忆，少样本学习，MCP 获取实时账户状态 |
| C | Cognition（认知） | 动态路由分层推理：简单任务→MCP Tools，标准流程→Skills SOP，复杂问题→Sub-Agents |
| D | Decision（决策） | 价值仲裁（建议价值>当前操作价值才透出）；涉资金操作强制 Human-in-the-loop |
| A | Actuation（执行） | 标准化调度，结果反馈回上下文形成记忆闭环 |

### 三模式能力中心

| 模式 | 适用场景 | 实现方式 |
|------|----------|----------|
| Mode A：原子工具（Atomic Tools） | 无状态简单操作（getReport、updateCampaign） | 通过 MCP Server 直接暴露现有 API |
| Mode B：组合技能（Composite Skills） | 固化最佳实践（"新人快速起量"） | 将选品、圈人、做素材封装为 DAG 工作流，触发即自动运行 |
| Mode C：垂类子智能体（Domain Sub-Agents） | 复杂问题（投放诊断） | 路由至拥有独立人设、知识库、状态机的 Specialist Agent |

### ChatLSS：意图识别三级漏斗

从快到慢、从便宜到贵的三道过滤器：

1. **规则识别**：正则/关键词匹配，响应最快，成本零（适用于"查询当日报表"等明确指令）
2. **相似度识别**：RAG 向量化检索 FAQ，相似度 ≥ 0.9 直接返回预设答案，无需调用大模型
3. **大模型识别**：前两道无法处理时，LLM 深度语义理解，决定是否调用工具

### PARSER 框架（解决 Agent 高 RT 与规划不稳定）

**PARSER** = Parallel Agent with Replan, State, Execution and Reflection

**核心创新**：
- **混合规划**：Global Planning（全局高效但纠偏难）+ Next-Step Planning（单步精确但速度慢）的结合
- **外层并行**：Stater（状态记录）与 Planner（规划器）并行运行，节省串行时间
- **内层并行**：单步内支持多个 Actions 并行执行，缩短总任务步数
- **反思融合**：Reflection（反思）与 Observation（观察）同时输入给下一步规划，消除传统 ReAct 的 Step 级别 Rollback

**Stater Agent 双重功能**：
- 生成对用户的自然语言回复（NL）
- 从环境中提取关键信息（KR, Key Results）提供给 Planner

### Agent 多维度评估体系

传统单点问答评估已不够，需要三个维度：

| 维度 | 指标 | 意义 |
|------|------|------|
| IO 维度 | Pass Rate（任务完成率）、QA Score（答案相关性与忠诚度） | 整体输入输出质量 |
| Agent Path 维度 | Reflection Rate（反思次数占比）、Plan Score（Jaccard 相似度计算工具调用准确性） | 规划路径质量 |
| 工程维度 | Time Cost per query | 性能基线 |

### Paimon：基于场景的预测性干预

**核心理念**：从"同步响应"转向"异步预判"，从"人找服务"转为"服务找人"

**两大核心痛点**：
- 用户操作"隧道效应"：固定操作习惯让商家对更好的新功能视而不见
- 静态建议"低信噪比"：基于死规则的硬编码建议缺乏个性化，点击率极低

**决策引擎四步**：召回 → 合并 → 排序 → 过滤

**双流程架构**：
- 离线流程：商家画像 Agent，天级别更新，生成短中长期操作习惯和偏好
- 在线流程：意图理解 Agent + 策略生成 Agent，用户关键动作时触发，端到端实时个性化

**核心设计原则**：嵌入式非侵入性（副驾驶而非夺方向盘）、全链路时空感知（知道用户从哪来在看什么）、概率决策与价值排序（只透出 Top 1 最高价值建议）

## 启发与思考

- **P-R-C-D-A 框架可迁移**：这个五步闭环不只适用于广告，任何需要"感知意图→检索知识→认知推理→决策过滤→执行反馈"的 AI Agent 场景都可以套用

- **三级漏斗意图识别是工程实践中降成本的关键**：规则→RAG→LLM 的阶梯式处理，大量高频问题被前两道拦截，只有复杂问题才消耗 LLM，显著降低成本

- **"陪伴式消息气泡"是 Paimon 的终局形态**：用户操作时 AI 主动在旁边给出建议，而不是等用户主动来问——这个交互范式值得学习，在 ToB 产品设计中有很强的参考价值

- **ChatLSS 的三大设计原则直接可用**：意图解释权前置（先翻译确认再执行）、原子能力工具化（业务逻辑解耦为工具）、可观测与可回滚（AI 操作具备事务性）——这是所有涉及资金/高风险操作的 AI Agent 的必要设计

- **PARSER 框架**：解决的是 ReAct 框架在垂直业务中响应慢、步骤级回滚频繁的问题，内外层并行+反思融合是值得深入研究的架构创新

## 原文精华

> 核心痛点并非模型不够聪明，而是商业经营的严肃性与大语言模型（LLM）的概率性之间的矛盾。

> 我们不推翻原有的广告引擎，而是用大模型构建一个"超级业务员"（Agent）。它听得懂人话（交互层），脑子清楚（编排层），并且手握操作系统的权限（工具层），最终实现从"人找功能"到"功能找人"的跨越。

> AI 应像"副驾驶"一样，仅在关键路口高亮显示最佳路径，而非抢夺方向盘。

> 缺乏 RAG 支持的 AI，就像一个"什么都懂但什么都不精"的实习生，无法给出具备业务深度的建议。
