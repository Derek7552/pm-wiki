# 贾维斯计划：一个安全人用 MCP 串联工具链的自动化渗透测试 Agent 架构

> 来源：粘贴文本（作者 yhy，微信公众号「谁不想当剑仙」）
> 提炼日期：2026-04-14
> 分类：AI 应用案例

## TL;DR

一位有6年经验的安全工程师（yhy）规划用 AI Agent 重构个人渗透测试工作流：以 LLM 核心助手「太阿」为总指挥，通过 MCP 协议串联已有工具链（信息收集/漏洞扫描/流量代理/代码审计），目标是让 Agent 自主完成 SRC 漏洞挖掘全流程。架构清晰且工具链相对完整，但在真实环境对抗层面存在明显挑战。

## 核心观点

**渗透测试全流程天然适合 Agent 编排，每个环节都有明确的输入输出。** 信息收集→资产发现→漏洞扫描→漏洞验证→深度利用→报告提交，每一步都是可工具化的子任务。这与 Agent 的核心优势（拆解任务、调用工具、多步推理）高度吻合，是 AI Agent 渗透安全领域最清晰的落地路径之一。

**MCP 协议是打通孤岛工具链的关键缺口。** 原先各个工具都是独立系统，互相调用需要定制集成。MCP 提供了统一接口规范，让每个工具像 API 一样被任何 Agent 调用，从而使多工具编排成为可能。承影（BurpSuite 类工具）已实现 MCP 接口是整个计划的基础。

**AI 不只是个人能力的放大器，更可以是执行者。** 以往自动化工具只能按预设规则运行，遇到非标场景就失效。LLM 的上下文理解和灵活推理能力让 Agent 可以处理"不在剧本里"的情况，这是 2025-2026 年这波 AI Agent 热潮的真实技术底座。

### 自动化渗透工作流

```
用户指令："帮我挖挖某某 SRC"
         ↓
太阿（评估目标，甚至自主从 HackerOne 选项目）
         ↓
鱼肠（子域名扫描 + 端口扫描 + 资产发现）
         ↓
承影（代理流量捕获 + 分析）
         ↓
承影 Agent（对重点目标自动化渗透）
         ↓
纯钧（若有开源代码，进行代码审计）
         ↓
整理漏洞报告 + 自动提交
```

### 已识别的技术挑战

**环境层面**
- WAF 绕过：真实站点有防火墙，靶场环境无法模拟
- 验证码 / 人机验证：需要额外能力突破（图像识别、行为模拟）

**Agent 层面**
- 模型幻觉：误报（把非漏洞判断为漏洞），需要验证机制
- 上下文丢失：长流程中关键信息在有限 context 里消失
- 多工具结果融合：各工具输出格式不一，需要统一的结果语义理解
- 策略回溯：当某条路走不通时，Agent 需要回溯并换思路

## 数据与案例

- **黄金时代对比**：作者刚入行时用 lijiejie 开源的 BBScan 扫站，拿过多次京东 SRC 月度前十——靠的是时间+执行力，没有自己的工具
- **六年后现状**：有完整工具库（承影/Jie/SScan/承影Agent/纯钧），却没有时间去挖洞——工具建设反而成为"等我写完工具再挖"的借口
- **承影 Agent 验证**：2025年腾讯云黑客松智能体挑战赛 Top9，证明自动化渗透 Agent 的可行性

## 启发与思考

**"工具孤岛 → MCP 互联 → Agent 编排"是安全工具产品演进的清晰路径。** 传统安全工具（Nmap、Burp、Nuclei）之间没有标准通信协议，集成靠脚本和定制化。MCP 作为统一接口规范，可以让这些工具真正被 Agent 编排，这对安全产品的产品化思路有直接参考价值。

**渗透测试全流程的可工具化程度，比大多数人想象的要高。** 每个步骤都有成熟工具对应，缺的只是"有推理能力的大脑"来做决策和串联。这个判断对评估 AI Agent 在其他复杂工作流（如安全运营、情报分析）的落地可行性有参考意义。

**个人工具链的最大瓶颈不是技术，而是时间。** 这个案例反映了一个普遍规律：工程师往往陷入"先把工具做完美再用"的完美主义陷阱，AI Agent 的价值之一是降低"开始行动"的门槛——工具不完美也没关系，在 Agent 迭代中补全。

**自动化渗透的核心难题是处理"不在剧本里的情况"。** 这与所有 AI Agent 工程面临的通用挑战一致：LLM 推理强但幻觉存在，需要验证机制；长流程需要记忆管理；多工具协作需要结果语义对齐。安全场景因为对抗性强，这些问题被放大了。
