## 方向定位

Agent 工程实践与落地研究的核心问题是："Agent 从能跑通到能上生产，差的是什么？"。它的边界包括：

- **包含**：手搓 vs 框架的阶段性取舍、ReAct 最小实现的工程决策、工具失败处理与降级、上下文管理与压缩、Harness Engineering 十二维实践、三大底层短板（记忆/安全/Harness）、拷问对齐方法论、Agent 自主度三件套（跨 session 续跑/任务状态/后台 I/O）
- **不包含**：架构选型与多 Agent 协作模式（→ Agent 架构方向）、记忆系统内部设计（→ Agent 记忆方向）、评测方法论细节（→ Agent 评估方向）、工具协议与 MCP 安全（→ Agent 工具生态方向）
- **与相邻方向的区别**：本方向不讨论"应该用什么形态"，而是"已经选定形态后，怎么把它工程化做扎实"

工程实践是 Agent 从"温室里的花"走到"真生产系统"的护城河——demo 不算落地，进生产系统才算落地。

## 核心概念

### 概念 1：手搓 vs 框架的阶段性取舍

来自美团面试题解析——框架痛点不是一开始暴露，而是随项目推进逐渐浮出。

**框架三大痛点及出现阶段**：

| 阶段 | 痛点 | 具体表现 |
|------|------|---------|
| **排查期** | 抽象层太多，调试穿透困难 | 50 行代码 + 40 层 stack trace，不知道问题在自己代码还是框架内部 |
| **运维期** | 版本升级 breaking change | LangChain 早期频繁改接口，线上稳定性受制于第三方 |
| **规模化期** | 隐性性能开销 | 框架通用性设计：每次调用做不需要的序列化/callback/日志，高流量下累积成真实延迟 |

> 类比：框架是"租房"（方便但改不了结构，房东随时调政策）；手搓是"自建"（建起来慢但住着踏实）。

**手搓三层优势**：链路透明（任意位置插日志/断点）/ 精确裁剪（无通用性包袱）/ 稳定可控（依赖仅底层 LLM SDK）。

**最务实策略**：核心手写，周边借用——
- **手写（Agent 心脏）**：工具调用循环、对话历史管理、错误重试、任务状态维护
- **借用（工具性周边）**：LangSmith tracing、LlamaIndex 文档解析、向量库客户端、可观测性工具

> 类比：自己设计承重墙和布局，但门锁/插座/水龙头买现成的。

**关键判断信号**："框架在某个地方替我做了什么"——有掌控感可以用；完全不知道方法内部发生了什么——黑盒需要警惕。**框架本身不是问题，"不理解就依赖"才是**。

### 概念 2：ReAct 框架三步循环

来自 DeepResearch Agent 实战——ReAct 是 Deep Research Agent 的行动逻辑骨架。

```
① Thought：我现在知道什么？还缺什么？下一步该做什么？
② Action：调用工具（搜索关键词 / 打开网址等）
③ Observation：把工具结果追加进对话历史
→ 回到 ① 继续，直到信息足够或达到 max_steps
```

**ReAct 的核心价值在错误处理，不在主路径**——工具调用在生产环境一定会失败（网络超时/404/API 限流），不捕获错误会导致整个 Agent 崩掉。正确做法是把错误写回 Observation，让模型自己决定换策略继续。

**最小实现的关键设计决定**：

| 决定 | 理由 |
|------|------|
| 保留完整对话历史，不截断 | ReAct 依赖模型看到完整推理链——切掉早期搜索结果会导致重复搜索 |
| try/except 捕获工具失败 | 失败信息写回 Observation 而非抛异常——成功率 78%→91% |
| tool 消息顺序约束 | OpenAI 格式要求 `tool` 消息紧跟触发它的 `assistant` 消息（tool_call_id 配对），顺序错了 API 直接报错 |
| max_steps 超限返回最后一条 assistant 输出 | "用户等了这么久，哪怕不完整的答案也比什么都没有强" |

### 概念 3：Context Rot 与 ReAct 物理天花板

**Context Rot 定义**：无关内容占上下文大头后，Agent 决策质量明显下滑。很多看起来像模型能力不足的问题，追溯后是上下文组织不当。

**ReAct 的天然上限**：
- **膨胀速率**：每步平均 2500 token → 10 步 = 25000 token，15 步 = 37500 token
- **症状**：注意力稀释（模型对早期信息注意力下降，靠后内容权重更高）+ 策略停滞（用几乎相同的关键词反复搜索）
- **真实案例**：新能源政策比较第 7 步 messages 超 28000 token，模型把欧盟补贴数据安到了美国政策里

**关键论断**：**"上下文越长模型越聪明"是错的**——这不是模型 bug，是注意力机制的物理特性。郝建业的判断（Transformer Attention 架构固有瓶颈）与此完全一致——上下文窗口拓展到百万级，窗口长度一旦拉长性能急剧下降。

**IterResearch 解法**：不把原始搜索结果堆进上下文，而是每轮执行完更新一份固定大小的结构化"研究摘要"——记录"目前已知信息"和"还没解决的子问题"。**上下文大小保持常量，无论跑多少轮都不会溢出**。

**监控建议**：Agent 执行完后打印 messages 的 token 总数——当某类问题到第 5 步就超过 2 万 token，就是上 IterResearch 的时机。

### 概念 4：上下文分层管理策略

来自阿里 OpenClaw 工程实践——防 Context Rot 的工程化方案。

| 层 | 内容 | 策略 |
|----|------|------|
| **常驻层** | 身份、约定、禁止项 | 短、硬、可执行 |
| **按需加载** | Skills/领域知识 | 描述符常驻，完整内容触发时注入 |
| **运行时注入** | 时间、渠道 ID、用户偏好 | 每轮按需拼入 |
| **记忆层** | 跨会话经验 | 写入 MEMORY.md，需要时读取 |
| **系统层** | Hooks/代码规则 | 完全不进上下文 |

**压缩时的保留优先级**（极其实用）：
1. 架构决策——不得摘要
2. 已修改文件和关键变更
3. 验证状态 pass/fail
4. 未解决的 TODO 和回滚笔记
5. 工具输出——可删，只保留结论

**压缩的关键陷阱**：**不要改动标识符**（UUID、hash、IP、端口、URL、文件名）——改错一位后续工具调用直接失效。

### 概念 5：郝建业的三大底层短板

来自 FreeBuf 对话——Agent 走不进生产系统的根因不是基座大模型不够强，而是缺少三块底层能力。

| 能力 | 定义 | 当前主流方案的局限 |
|------|------|------------------|
| **记忆** | 处理超长对话上下文、沉淀和复用用户长期交互信息 | 工作流 + 外部数据库 = "翻笔记"，未做到模型内生 |
| **安全（对齐）** | 从技术上约束智能体，避免损害用户利益、行为不可预测/不可控 | 提示词层 safety prompt，覆盖不到工具链层复杂攻击 |
| **Harness 框架** | 任务编排与自主调度——熟练调用各类工具，根据场景上下文自主组合工具、规划完整路径 | 多数 Agent 仅做工具列表罗列，无自主编排策略 |

**三者关系**：记忆是 Harness 长链路任务的支撑（让 Agent 能复盘历史），安全是 Agent 进生产系统的准入门槛，Harness 是把基座能力转化为业务结果的转换器。**三者缺一，Agent 都只能停留在 demo**。

**真正的落地标准**："demo 不算落地，进生产系统才算落地"——智能体必须能真正走进各垂直领域的生产系统，而不是停留在 demo 和演示层面。

> "现在很多 Agent 应用，打个比方，就像把东西摆在自家后花园里做试验，只敢在隔离的仿真环境里跑一跑，却不敢真正放进真实业务、真实生产环境里去做操控和执行。这样的 Agent，做得再漂亮，也是温室里的花。"

### 概念 6：ACI 工具设计三代演进

来自阿里十二维实践——工具设计比提示词设计更重要（与 Anthropic SWE-bench 经验印证）。

| 代际 | 思路 | 问题 |
|------|------|------|
| **第一代 API 封装** | 每个 Endpoint = 一个工具 | 粒度过细，Agent 需协调多工具 |
| **第二代 ACI** | 工具对应 Agent 目标而非 API 操作 | 面向目标，一次把动作说完整 |
| **第三代 Advanced Tool Use** | 动态发现 + 代码编排 + 示例驱动 | 进一步优化发现、调用和描述 |

**第三代的三个方向**：
- **Tool Search**（动态发现）：Agent 按需发现工具定义，上下文保留率 95%，准确率 49%→74%
- **Programmatic Tool Calling**（代码编排）：中间结果不经过 LLM，token 消耗从 ~150,000 降到 ~2,000
- **Tool Use Examples**（示例驱动）：附 1-5 个真实调用示例，准确率 72%→90%

**ACI 好工具 vs 差工具**：

| 维度 | 好工具 | 差工具 |
|------|--------|--------|
| 粒度 | 对应 Agent 目标 | 对应 API 操作 |
| 返回 | 与下一步决策直接相关 | 完整原始数据 |
| 错误 | 结构化 + 修正建议 | 通用字符串 "Error" |
| 描述 | 何时用、何时不用 | 只写功能说明 |

### 概念 7：Agent 自主度三个基础设施

来自阿里 OpenClaw 落地经验——把 Agent 从"指挥者模式"推向"统筹者模式"的工程基础。

**1. 跨 session 续跑**：
- Initializer Agent 只跑一次，生成 feature-list.json + init.sh + claude-progress.txt
- 后续 Coding Agent 从文件系统恢复状态，每次实现一个功能、跑测试、更新 passes、提交代码后退出
- **进度放文件里不放上下文里，功能清单用 JSON 不用 Markdown**

**2. 显式任务状态**：
- 同一时间只能有一个 in_progress
- 每完成一步先更新状态再继续
- 连续多轮未更新时自动注入 reminder

**3. 后台 I/O**：
- 慢速 subprocess 放后台线程
- 通过通知队列在下一轮 LLM 调用前注入结果
- **比把整个 loop 改成复杂 async runtime 更稳**

### 概念 8：Prompt Caching 反直觉结论

来自阿里十二维实践——稳定的大系统提示，比频繁变动的小提示实际成本更低。

- 写入成本只付一次，后续读取折扣可达 90%
- 常驻层越稳定，前缀命中率越高
- Skills 延迟加载不破坏系统提示前缀，追加在稳定前缀之后
- **MCP 工具集频繁变动会导致缓存命中不断失效**

### 概念 9：Skills 按需加载的工程细节

**Skill 描述符的关键**："何时该用我"比"我能做什么"重要得多。

```
# 低效（约 45 tokens）
description: |
  This skill handles the complete deployment process...
# 高效（约 9 tokens）  
description: Use when deploying to production or rolling back.
```

**反例的量化影响**：
- 无反例：准确率从基准 73% 掉到 53%
- 加反例：升到 85%，响应时间还降 18.1%

**三个典型反模式**：
1. 几百行工作手册全塞 Skill 正文，不拆 supporting files
2. 一个 Skill 试图覆盖 review/deploy/debug/incident 五件事
3. 有副作用的 Skill 没有显式限制调用时机

**Skills vs MCP**：MCP 把完整结果返回给模型，更容易吃掉上下文预算；CLI + 单句描述的 Skill 在数据读取任务里更简洁。**MCP 适合需要维护状态的任务**（如 Playwright）。

### 概念 10：拷问对齐方法论（替代 Plan Mode）

来自 Matt Pocock 的 /grill-me 实践——规划的本质不是产出文档，而是建立深度共识。

**Plan Mode 的三个问题**：

| 问题 | 表现 | 后果 |
|------|------|------|
| 太急于生成 | AI 还没理解意图就吐出长文档 | 计划与真实需求脱节 |
| 很少主动提问 | 不问关键假设、不挖需求细节 | 隐藏的复杂度在编码阶段才暴露 |
| 计划变废话 | 又臭又长，扫一眼就跳过 | 双方没有真正对齐，返工不可避免 |

**核心论断**：**糟糕的计划 = 糟糕的输出（Bad plans = Bad outputs）**——前期多花时间"被拷问"，后期编码和调试时间大幅减少。

**三步 Skill 链**：
```
/grill-me（拷问对齐）
  ↓ AI 连续问 20-30 个问题，遍历设计决策树所有分支
  ↓ 逼出模糊想法和隐藏假设
/write-a-prd（生成 PRD）
  ↓ 基于深度对话生成结构化需求文档
  ↓ 此时 AI 已充分理解需求，PRD 质量远超 Plan Mode
/prd-to-issues（拆解任务）
  ↓ 将 PRD 转化为 GitHub Issues
  ↓ 具体、可执行、可跟踪
→ 进入编码阶段（AI 可 AFK 自主推进）
```

**/grill-me 的设计精髓**：
- **极简指令**：只有三句话——指令越短，AI 的自由度越高，提问越深入
- **核心机制**：让 AI "relentlessly"（无情地）采访你关于计划的每一个方面
- **遍历决策树**：走遍设计决策树的所有分支，一步步解决依赖关系
- **终止条件**：直到双方对齐颗粒度才结束

## 方法论与框架

### 方法 1：手搓 vs 框架的选型决策矩阵

**核心思想：** 框架在不同阶段有不同的工程价值——POC 阶段加速，生产阶段制约。

| 场景 | 推荐 | 原因 |
|------|------|------|
| POC/快速验证 | **框架** | 两周工作缩短到两天，跑通即可 |
| 团队刚接触 Agent | **框架** | 少踩基础坑 |
| 周边工具依赖生态 | **框架** | 文档解析/向量检索等 |
| 准备上生产 | **手搓核心** | 稳定性成为核心关切 |
| 流量上来 | **手搓核心** | 性能和成本敏感 |
| 高度定制业务 | **手搓核心** | 和框架通用设计偏差大 |
| 需要高可观测性 | **手搓核心** | 链路随时监控回溯 |

**典型项目演进轨迹**：
```
1. 框架快速跑通 POC → 验证方向
2. 遇到线上 bug → 排查困难的关键部分替换为手写
3. 流量上来 → 性能敏感的核心逻辑全部手写
4. 最终态 → 框架只保留做得好的周边工具
```

### 方法 2：ReAct 工具失败的标准处理流程

**核心思想：** "工具失败时不崩"是 Agent 工程的第一优先级。

**操作步骤：**
1. 工具调用包裹在 try/except 内
2. 异常发生时，把错误信息（含异常类型）格式化为 Observation
3. 写回对话历史的 `tool` 角色消息中
4. 让模型自主决策"换一种方式"或"放弃当前路径"
5. 不返回 None 或抛异常给上层

**核心代码模板**：
```python
try:
    observation = tools[tool_call.function.name](**args)
except Exception as e:
    observation = f"工具调用失败（{type(e).__name__}）：{e}。请尝试换一种方式。"

messages.append(response.to_message())
messages.append({"role": "tool", "content": str(observation), "tool_call_id": tool_call.id})
```

**注意事项：**
- 失败信息要带类型（network/timeout/auth/parse），模型据此判断换什么策略
- `tool` 消息必须紧跟触发它的 `assistant` 消息（OpenAI 格式约束）
- max_steps 超限时返回最后一条 assistant 输出，附超限说明

### 方法 3：系统提示的测试驱动迭代

**核心思想：** 系统提示是测试驱动的，不是一次性设计的——先跑起来让 Agent 出真实问题，再针对性加约束。一开始就写得很长，模型反而"读不进去"。

**系统提示的三个角色**：
1. **定义角色**：研究型助手 vs 对话助手，行为模式完全不同
2. **规定输出格式**：每次调工具前先说 Thought，工具结果回来后先评估质量
3. **设置行为边界**：针对真实测试中发现的问题加约束

**关键边界规则**（DeepResearch 实战验证有效）：
- "如果信息来源之间有矛盾，明确指出，不要自己融合矛盾的信息"
- "如果找不到答案，直接说找不到，不要编造"
- 加这两条后，信息混淆率从 13% 降到 6%

**写法原则**：测试 → 暴露问题 → 针对性补一条 → 再测试。不要一开始就写完整。

### 方法 4：上下文压缩与整合流水线

**核心思想：** 50% token 阈值触发整合，系统只移动指针不删除原始消息——整合失败时原始消息写入 archive/ 保留完整历史。**流程必须可回退**。

**操作步骤：**
1. 监控 messages 的 token 总数
2. 超过 50% 上下文窗口阈值触发整合
3. 用 LLM 生成结构化摘要（按保留优先级处理）
4. 创建新消息列表 = 系统提示 + 摘要 + 最近 N 轮原始消息
5. 旧消息归档到 archive/，不立即删除
6. 切换指针到新消息列表

**保留优先级**（从高到低）：
1. 架构决策——不得摘要
2. 已修改文件和关键变更
3. 验证状态 pass/fail
4. 未解决的 TODO 和回滚笔记
5. 工具输出——可删，只保留结论

**陷阱**：不要改动标识符（UUID、hash、IP、端口、URL、文件名）——改错一位后续工具调用直接失效。

### 方法 5：拷问对齐三步工作流（替代 Plan Mode）

**核心思想：** 反转主动权——不是 AI 生成计划让你审核，而是 AI 疯狂提问逼你把模糊想法想清楚、把隐藏假设暴露出来。

**操作步骤：**
1. **/grill-me 拷问对齐**：AI 连续问 20-30 个问题（约 15-20 分钟），遍历设计决策树所有分支，逼出模糊想法和隐藏假设
2. **/write-a-prd 生成 PRD**：基于深度对话生成结构化需求文档（不是人手写的长文档，而是深度对话后 AI 自动生成的精炼共识记录）
3. **/prd-to-issues 拆解任务**：将 PRD 转化为 GitHub Issues，具体、可执行、可跟踪

**完成后**：AI 已有足够清晰的上下文，可以 AFK 自主推进编码。

**适用场景：**
- 需求模糊、隐藏假设多的复杂功能开发
- PM 自己也没想清楚细节的探索性需求
- 多人协作中需要"对齐颗粒度"的关键决策

### 方法 6：OpenClaw 落地工程顺序

**核心思想：** 工程实现顺序本身就是知识——错的顺序会让后续工作返工。

**正确顺序**：
1. **单渠道先跑通**（不要第一版就抽象多渠道）
2. **安全边界先于功能**（白名单授权 + 工作空间隔离 + 操作审计日志）
3. **记忆整合要早做**（不加整合，第 20 轮对话就垮）
4. **Skills 先于新工具**（先写程序性知识，再扩工具集）
5. **第一个失败就建评测**（每个真实失败案例转测试用例）

**五层架构**：
| 层 | 实现 | 关键设计决策 |
|----|------|-------------|
| Gateway | WebSocket:18789 | Channel 和 Agent 统一走 Gateway |
| Channel 适配器 | 23+ 渠道 | 新增渠道不修改 Agent 代码 |
| Pi Agent | 工具调用支持流式 | Agent 核心循环和渠道完全解耦 |
| 工具集 | shell/fs/web/browser/MCP | ACI 原则，面向目标 |
| 上下文+记忆 | Skills 延迟加载 + MEMORY.md | 50% token 阈值自动整合 |

### 方法 7：Agentic RL 训练专门记忆模型（MemoraX 路线）

**核心思想：** 记忆能力不能用监督学习训练——没有人类专家能写出"如何记住前 3 万字对话"的训练语料。记忆本质是序列决策问题（什么该记/什么该忘/什么时候调取），天然属于强化学习领地。

**操作步骤：**
1. 设定记忆系统的优化目标（不同层面，如长程任务成功率/检索召准率/Token 经济性）
2. 借助强化学习反向迭代优化记忆模型参数
3. 数据驱动而非专家标注驱动
4. 与基础大模型结合，从模型底层实现高效记忆管理

**注意事项：**
- 与主流 RAG 路线不是替代关系，初期更可能是叠加
- 优化目标设定是关键，奖励信号设计决定最终效果
- 公开数据少，工程落地难度高

### 方法 8：反模式速查表

**核心思想：** 这八种反模式覆盖 Agent 工程中最常见的"看似合理实则致命"问题。

| 反模式 | 问题 | 修法 |
|--------|------|------|
| 系统提示当知识库 | 关键规则被忽略 | 约定留系统提示，知识移 Skills |
| 工具数量失控 | 频繁选错工具 | 合并重叠，明确命名空间 |
| 缺少验证 | 说完成但没法验证 | 每类任务绑可执行验收标准 |
| 多 Agent 无边界 | 状态漂移，归因困难 | 角色权限 + worktree 隔离 |
| 记忆不整合 | 第 20 轮后质量下降 | 超阈值自动触发整合 |
| 没有评测 | 不知是否引入回归 | 每个真实失败案例转测试用例 |
| 过早多 Agent | 协调开销超并行收益 | 先验证单 Agent 上限 |
| 约束靠期望 | Agent 选择性遵守 | 编码进 Linter/Hook/工具验证 |

## 案例库

### 案例 1：DeepResearch 工具失败处理 78%→91%

- **背景**：DeepResearch Agent 在生产搜索任务中频繁出现 API 限流/404/网络超时导致 Agent 直接崩溃
- **做法**：把工具调用包在 try/except 内，异常发生时把错误信息（含异常类型）格式化为 Observation 写回，让模型自主决策"换一种方式"
- **结果**：成功率从 78% 提升到 91%，代价只是多写几行代码
- **启发**：**任何外部依赖调用都要有降级处理，失败信息写回 Agent 上下文，让 Agent 自主决策换策略**——这是 Agent 工程的第一优先级。

### 案例 2：新能源政策上下文溢出事故

- **背景**：新能源政策多市场比较问题，第 7 步 messages 超过 28000 token
- **现象**：模型把欧盟补贴数据安到了美国政策里，排查两小时才发现根因
- **根因**：注意力稀释——模型对早期信息注意力下降，靠后内容权重更高
- **解法**：上 IterResearch——每轮更新固定大小研究摘要，上下文保持常量
- **启发**：**"上下文越长模型越聪明"是错的**。当某类问题到第 5 步就超过 2 万 token，就是上 IterResearch 的时机。

### 案例 3：DeepResearch 系统提示边界规则

- **背景**：DeepResearch 在多源信息冲突时存在 13% 信息混淆率
- **做法**：加两条边界规则——"如果信息来源之间有矛盾，明确指出，不要自己融合矛盾的信息"+"如果找不到答案，直接说找不到，不要编造"
- **结果**：信息混淆率从 13% 降到 6%
- **启发**：**好的系统提示不是写越多越好，是每一条都针对你在测试里看到的真实问题写的**。测试驱动的迭代式补充，比一次性完整设计更有效。

### 案例 4：LangChain 早期框架痛点的演进轨迹

- **背景**：LangChain 早期 @tool 装饰器 + AgentExecutor 封装 ReAct loop + 内置 tracing/callback/记忆，POC 阶段极高效
- **痛点演进**：
  - 排查期：50 行代码 + 40 层 stack trace，调试困难
  - 运维期：早期频繁改接口（breaking change），线上稳定性受制于第三方
  - 规模化期：内部序列化/callback/日志累积成可见延迟和费用浪费
- **结果**：很多团队后期把核心 ReAct loop 替换为手写，只保留 LangSmith tracing 等周边工具
- **启发**：**框架的痛点不是一开始暴露的，而是随项目推进逐渐浮出**——POC 用框架快速验证，生产逐步替换核心心脏。

### 案例 5：Matt Pocock /grill-me Skill 实践

- **背景**：Plan Mode 经常产出"伪对齐"——AI 没真正理解意图就吐出长文档
- **做法**：/grill-me 仅三句话指令，让 AI "relentlessly" 采访用户关于计划的每一个方面
- **结果**：前期被拷问 20-30 个问题（约 15-20 分钟），后期编码和调试时间大幅减少
- **安装**：`npx skills@latest add mattpocock/skills/grill-me`
- **启发**：
  - **指令越短，AI 的自由度越高，提问越深入**——验证了"Skill ≠ 长指令"的设计理念
  - PRD 没死，死的是人手写的 PRD——深度对话后 AI 自动生成的 PRD 质量远超 Plan Mode

### 案例 6：郝建业多智能体产业落地

郝建业团队过去十年把多智能体强化学习落地到的工业场景：

| 场景 | 智能体抽象 | 优化目标 |
|------|-----------|---------|
| 阿里广告推荐 | 广告主 / 用户 / 平台三方博弈 | 三方利益均衡最优 |
| 自动驾驶（华为）| 自动驾驶车 / 人类驾驶车 / 行人 | 路口交汇等复杂场景的安全与驾乘体验 |
| 无线网络优化（华为）| 不同基站作为协同智能体 | 整体优化效率 + 减少人工调优成本 |
| 供应链调度（华为）| — | 系统级全局优化 |
| 芯片设计（华为）| — | EDA 流程自动化 |

**沉淀**：一套成熟可复用的"**黑盒优化 + 白盒优化**"通用算法平台。

**关键论断**：停留在虚拟场景里的成功，不是真正的落地。

### 案例 7：OpenClaw 落地五层架构与工程顺序

- **背景**：阿里 OpenClaw 项目从单渠道 POC 走到 23+ 渠道生产部署
- **做法**：严格按工程顺序——单渠道跑通 → 安全边界 → 记忆整合 → Skills → 评测
- **关键决策**：
  - Gateway 统一接入（Channel 和 Agent 都走 Gateway）
  - Channel 适配器使 Agent 核心循环和渠道完全解耦
  - 50% token 阈值自动触发记忆整合
- **启发**：**工程实现顺序本身就是知识**——错的顺序会让后续工作返工。不加整合，第 20 轮对话就垮；没有评测，不知是否引入回归。

## 关键洞察

### 1. "Demo 不算落地，进生产系统才算落地"

来自郝建业 + 阿里 OpenClaw + 多个面试题解析的共同判断：当前 Agent 在记忆、安全、Harness 关键能力上短板明显。**只有把这三块做扎实，企业和用户才敢把 Agent 部署到垂域生产系统真正落地**。"现在很多 Agent 应用，就像把东西摆在自家后花园里做试验"——温室里的花做得再漂亮，也不是真生产。这条标准应作为 Agent 类产品的统一交付标尺。

### 2. 框架不是问题，"不理解就依赖"才是

来自美团面试题解析——框架痛点不是一开始暴露，而是随项目推进在不同阶段逐渐浮出（排查期/运维期/规模化期）。最务实策略是**核心手写，周边借用**——工具调用循环、对话历史管理、错误重试、任务状态维护是 Agent 心脏必须手写掌控；LangSmith tracing、文档解析、向量库客户端是工具性周边可以借用。"完全不知道方法内部发生了什么"是危险信号。

### 3. 工具失败时不崩，是 Agent 工程的第一优先级

来自 DeepResearch 实战 + 阿里十二维指南共同验证：**try/except 把错误写回 Observation 让模型自主决策**，这一改动让成功率从 78% 提升到 91%，代价只是多写几行代码。这一原则可推广到所有 Agent 系统：任何外部依赖调用都要有降级处理，不能让外部失败直接让 Agent 崩。

### 4. 上下文越长越聪明是错的——这是物理天花板

来自 DeepResearch + 郝建业 + 阿里十二维三重印证：超过某个阈值后，模型对早期信息的注意力稀释，靠后内容权重更高。这**不是模型 bug，是注意力机制的物理特性**。解法有两条路线：工程化（IterResearch 固定大小研究摘要）+ 模型路线（MemoraX Agentic RL 训练专门记忆模型）。**当某类问题到第 5 步就超过 2 万 token，就是上 IterResearch 的时机**。

### 5. 调试 Agent 时应优先检查工具定义，而不是先怀疑模型能力

来自阿里十二维 + Anthropic SWE-bench 双重验证：多数工具选择错误出在描述不准确（没说明"何时该用/何时不该用"），而不是模型不够聪明。**ACI 工具设计的"何时用、何时不用"比"我能做什么"重要得多**。这一原则把调试焦点从模型层转到工具描述层，是 Agent 工程师的关键判断习惯。

### 6. 评测系统本身的问题比 Agent 出问题更难发现

来自阿里十二维实践——评测环境资源不足、评分器 bug、测试用例与生产脱节，这些在表现上和模型退化一模一样，但**基于失真信号去改 Agent 会越改越坏**。当 Agent 表现下降时，本能反应是去改 Agent，但正确做法是**先验证评测本身有没有出问题**——这和软件工程里"先检查测试用例再检查代码"的原则完全一致。

### 7. 测试驱动的系统提示迭代，优于一次性完整设计

来自 DeepResearch 实战经验：**一开始就写得很长，模型反而"读不进去"**。正确做法是先跑起来让 Agent 出真实问题，然后针对性地加约束。"如果信息来源之间有矛盾明确指出""如果找不到答案直接说找不到"这两条规则——是测试中发现 13% 混淆率后针对性补的，加完降到 6%。**每一条提示都该有真实数据支撑**。

### 8. 规划的本质不是产出文档，而是建立深度共识

来自 Matt Pocock /grill-me 实践——**糟糕的计划 = 糟糕的输出**。Plan Mode 的根本缺陷是"伪对齐"，AI 既不主动提问关键假设也不深挖需求细节。反转主动权——让 AI 拷问你 20-30 个问题，遍历设计决策树所有分支——前期 15-20 分钟"被拷问"，后期编码和调试时间大幅减少。这一思路与"提问能力>回答能力"在 AI 时代的核心竞争力判断完全一致。

### 9. 工程实现顺序本身就是知识

来自阿里 OpenClaw 实践：单渠道先跑通 → 安全边界先于功能 → 记忆整合要早做 → Skills 先于新工具 → 第一个失败就建评测。**错的顺序会让后续工作返工**——不加整合第 20 轮对话就垮；没有评测不知是否引入回归。这一顺序看似平凡，但实际上是踩过坑后的总结，违反任一条都会付出真金白银的代价。

### 10. "Humans steer. Agents execute."——工程师的工作正在被重新定义

来自 OpenAI Harness Engineering 实践 + 多 Agent 协作范式：用 Codex Agent 构建超百万行代码产品，全程无人类直接写代码。早期进展慢的原因不是 Agent 能力不够，而是**环境没有准备好**（缺少工具、抽象和结构）。工程师的工作从写代码 → **设计环境、定义意图、构建反馈循环**。这是 Agent 时代工程能力的根本性重构。

## 待探索问题

- **手搓 vs 框架的临界点量化**：项目演进到什么规模/流量/复杂度需要从框架切换到手搓？是否有可量化的判断指标（如 QPS、模型调用延迟、调试耗时）？
- **IterResearch 与 MemoraX 路线的关系**：IterResearch 是工程优化（外部摘要），MemoraX 是模型内生记忆——两者是替代还是叠加？混合架构下如何分配职责？
- **拷问对齐的疲劳边界**：用户被 AI 连续拷问 20-30 个问题后会不会疲劳放弃？是否需要分阶段拷问（先粗后细）或者引入用户体验缓冲？
- **测试驱动的系统提示如何沉淀为模板**：每个项目都从零开始迭代 system prompt 太慢——是否有"通用边界规则库"可复用（如"禁止编造""明确标注矛盾"等）？
- **Agent 自主度三件套在客服场景的落地**：跨 session 续跑/任务状态/后台 I/O 这套思路目前主要在编程 Agent 实证——在 AI 客服场景下应做什么改造？
- **Harness Engineering 在中小团队的可负担性**：OpenAI 用 Codex Agent 构建百万行代码的前提是有强大基础设施投入——中小团队的"Harness 最小可用版"应包含哪些必要组件？
- **三大底层短板（记忆/安全/Harness）的优先补齐顺序**：郝建业判断三者缺一不可，但实际产品资源有限——应该先补哪一块？三者之间是否存在依赖关系（如安全要求记忆先完善）？
