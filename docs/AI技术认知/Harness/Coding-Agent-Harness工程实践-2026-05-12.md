## 方向定位

Coding Agent Harness 工程实践研究**"理论概念到生产环境"中间这段最难走的路**。它要回答的核心问题：

1. **从哪里开始**：面对六层架构 / 七大原语 / 六大构件这些理论，最该先做哪个？
2. **用什么杠杆**：在已有产品（Claude Code / Cursor / Codex）之上，团队还能做什么投入？四大杠杆（Custom Rules / MCP / Skills / Spec Driven Development）如何选择优先级？
3. **AGENTS.md 怎么写才不腐烂**：业界踩过的坑（巨型文件、规则冲突、文档腐烂）如何系统规避？
4. **怎么从一个人扩展到组织**：MercadoLibre 在 2 万开发者、数千仓库上的 rollout 经验有哪些可复用？
5. **百万行代码实战的真实路径**：OpenAI / Anthropic / Hashimoto 这三类成功案例分别走的是什么路线？

**不在本方向范围**：Harness 是什么（概念）、各种分层模型（架构）、Context Anxiety 与 GAN 三角色（失效模式与反馈方向）、Harness 减重与 AgentOS（演化方向）。

## 知识图谱

- **四大工程杠杆（16 框架）**
  - Custom Rules（最低门槛、最高 ROI）
  - MCP（接组织知识与系统）
  - Skills（最强单元：知识+脚本+可组合）
  - Spec Driven Development（重写任务输入质量）
  - 外加 Feedback Loop（外部纠偏闭环）
- **行动优先级**
  - P0 立即可做（AGENTS.md + 自定义 Linter + 仓库知识化）
  - P1 反馈回路成熟期（分层上下文 + 进度文件 + 端到端验证 + 上下文利用率监控）
  - P2 规模化能力沉淀期（Agent 专业化分工 + 垃圾回收 + 可观测性）
- **AGENTS.md 设计要点**
  - 不是百科全书，是地图（约 100 行）
  - 每行对应一个历史失败案例（Hashimoto）
  - 按需加载 vs always-on 的边界
  - 不适合放进去的内容
- **实战路线类型**
  - OpenAI 路线：机械约束 + 分层架构 + 仓库即事实源
  - Anthropic 路线：三角色分离 + Sprint Contract + 浏览器自动化验证
  - Hashimoto 路线：单 Agent + 失败案例驱动 + AGENTS.md 持续迭代
- **组织级实践（MercadoLibre）**
  - 近 2 万开发者、数千仓库
  - 9+ 技术栈的标准化规则体系
  - 内部 MCP 接平台/SDK/RAG/应用文档
  - 独立审查 Agent 接 CI
- **常见误区**
  - 把 AGENTS.md 当超级 System Prompt
  - 给 Agent 全量工具
  - 把规则做成"始终注入"
  - 任务描述过粗导致"错得合理"
- **数据基线**
  - OpenAI 3 人 5 月 100 万行代码、1500 PR
  - Vercel 工具 15 → 2，准确率 80% → 100%
  - 单次 Codex 任务可连续 6 小时

## 核心概念

### 概念 1：Coding Agent 内部的统一循环

**定义（16）**：当前主流 coding agents（Claude Code、Cursor、Codex）虽然产品形态不同，内部都共享同一个循环：

```
Read → Plan → Code → Validate → Iterate
```

**关键属性**：
- 循环的每一步都消耗**同一份有限资源**：context window
- 任何与当前任务不相关的内容都在挤压真正有用的信息
- 上下文利用率达到约 **60%** 之后，继续塞信息反而让 agent 表现变差（16 引用研究，未独立核实）

**为什么这个循环对工程实践重要**：
- Harness 设计的核心不是"加更多能力"，而是"在循环的每一步精细化调度信息"
- 这就是为什么 P0 行动里最重要的不是接 MCP，而是先做 AGENTS.md（控制 Read 阶段的输入质量）

**适用场景**：评估 coding agent 性能问题时，先定位是哪一步出问题——Read 阶段缺信息？Plan 阶段方向错？Validate 阶段无外部信号？

---

### 概念 2：Context Window 是稀缺资源

**核心论断**：约束 AI coding agent 的关键资源不是推理能力，而是 context window。

**Context window 的六类主要消耗项**（16）：

| 消耗项 | 典型膨胀点 |
|--------|----------|
| System prompt / custom rules | 每轮都注入，若不相关就是长期浪费 |
| Tool definitions | 每接一个 MCP server 增加 schema 占用 |
| Conversation history | 线性累积 |
| Tool results | 文件内容、终端输出、API 返回可能非常长 |
| Agent output | 模型自己写的代码与解释 |
| 长链迭代堆积 | 每多一次循环窗口进一步膨胀 |

**关键转向（16）**：
> 很多团队以为自己在优化 prompt，实际上更该优化的是"哪些信息不该一直存在上下文里"。

**与 AGENTS.md 设计的直接关联**：
- 巨型 AGENTS.md 的真正问题不是"写得不好"
- 而是它**每轮都注入、挤掉了任务工作空间**
- "Context 是稀缺资源，臃肿文件挤掉任务空间"（12 总结的 OpenAI 教训之一）

**适用场景**：每次有人提议"在 system prompt / AGENTS.md / custom rules 里加点东西"时，先问"这条内容需要每轮都看到吗？如果不需要，怎么改成按需加载？"

---

### 概念 3：AI 是"第一天入职的天才新人"

**定义（16 的经典类比）**：
> 你的 AI coding agent 是入职第一天的天才新人。能力强、反应快，对代码库、规范、架构、流程和组织知识几乎一无所知。

**这个比喻的工程含义**：
- 现实里没有人会让一个刚入职的新同学在没有 onboarding 的前提下直接改复杂系统
- 但很多团队默认 AI 可以在极少背景信息下正确完成任务
- Context engineering 不是"优化 prompt"，而是**像组织 onboarding 一样系统性提供环境、规则、流程、工具和监督**

**实际后果**：
- 不做 onboarding → Agent 凭直觉写代码 → "看似能跑的 MVP" 但违反隐性业务约束
- 例如：用户说"做一个后台新增商品功能"但没说幂等性、权限、API 合约 → Agent 实现一个 MVP → 再次点击按钮重复插入 → 问题不是"AI 写错了"，而是"系统没把隐性业务规则显式化"

**适用场景**：每次想说"AI 太蠢/不靠谱"之前，先自问"我对一个刚入职的天才新人会给的背景信息，给齐了吗？"

---

### 概念 4：Custom Rules vs MCP vs Skills vs Spec 的功能分工

**核心论点（16）**：四大杠杆解决**完全不同**的问题，不是同一类东西。混淆它们的边界会导致选型错位。

| 杠杆 | 解决的问题 | 注入时机 | ROI |
|------|----------|---------|-----|
| **Custom Rules** | 规则约束（怎么写、怎么命名） | 每轮自动注入 | 最低门槛、最高 ROI |
| **MCP** | 外部能力接入（查库、调 API、读 Figma） | 工具调用时 | 让 agent 从"只懂仓库"升级到"懂组织" |
| **Skills** | 按需加载的知识 + 可执行脚本 + 可组合行为 | 简短描述常驻，完整内容按需注入 | 最强单元（兼具知识与可执行性） |
| **Spec Driven Development** | 任务输入质量（功能、边界、验收、测试计划） | 写代码前 | 最彻底，但流程改变最难 |

**关键边界**：
- Custom Rules 是**静态约束**（不能执行）
- Skills 是**动态能力包**（描述 + 脚本 + 模板）
- MCP 是**外部能力入口**（既是信息入口也是验证出口）
- SDD 是**输入质量重构**（解决"agent 不知道要做什么"的根因）

**与 P0/P1/P2 行动清单的衔接**：
- Custom Rules / AGENTS.md → P0（最先做）
- Skills、MCP → P1
- 全组织级 SDD → P2 / 长期文化建设

**适用场景**：当团队问"我们已经有 AGENTS.md 了，下一步做什么"时，按照"规则约束 → 能力接入 → 知识/技能包 → 任务输入质量"的顺序决定下一步。

## 方法论与框架

### 方法 1：四大杠杆 + Feedback Loop（16 完整版）

**核心思想**：真正有效的 harness 不是单一技巧，而是"多杠杆协同 + 反馈闭环"的系统工程。

**结构**：
```
前端（校准方向）                    后端（收紧结果）
Custom Rules ─┐                 ┌─ Tests
MCP ──────────┼─ Agent 工作 ──┼─ Lint / Type check
Skills ───────┤                 ├─ Build scripts
SDD ──────────┘                 ├─ Review Agent
                                  └─ Hooks
```

**关键洞察**：
- Context engineering 是"前端"（让 agent 看到正确的东西）
- Feedback loop 严格来说不是 context engineering，但**决定整个系统能否真正运转**
- 只有两端同时存在，agent 才能从 demo 走向规模化生产

**两个特别有价值的细节**：

1. **Skills 的特殊地位**（16）
   - 只有简短描述常驻上下文，完整内容按需注入
   - 可封装模板、示例输出、参考文档、执行脚本
   - 既可以是 reference skill 也可以是 task skill
   - 可以在独立 subagent 中运行，避免主上下文被重任务污染
   - Skills 的真正价值不是"多一份说明书"，而是把高密度上下文从 always-on 区域迁移到 on-demand 区域

2. **Agent Hooks 把软约束升级为硬约束**（16）
   - 把验证逻辑挂到 Stop hook
   - Agent 完成任务前**必须**通过这些检查
   - "建议你去跑一下测试" → "必须先过门槛"
   - 这是 Custom Rules 的反向补充：Custom Rules 在前端，Hooks 在后端

**注意事项**：
- 不要试图一次上齐四大杠杆，按 P0/P1/P2 节奏推进
- Feedback Loop 比单纯加杠杆更重要——没有 pass/fail 信号，所有杠杆都只是"建议"
- 杠杆是手段，不是目的；目的是"agent 每次都处在对的上下文里、拿到对的工具、接受可执行的反馈"

### 方法 2：P0/P1/P2 行动清单（05 的优先级模型）

**核心思想**：不要一开始搭齐六层。按 ROI 排序：先做"能立竿见影"的，后做"投入大但收益慢"的。

#### P0：立即可做，效果最显著

| 行动 | 为什么 | 参考实践 |
|------|--------|---------|
| 创建 AGENTS.md 并持续维护 | Agent 每次启动自动加载，犯错就更新，形成反馈循环 | Hashimoto：每一行对应一个历史失败案例 |
| 构建自定义 Linter + 修复指令 | 错误消息直接告诉 Agent 怎么改，纠错同时在"教" | OpenAI：Linter 报错自带修复方法 |
| 把团队知识放进仓库 | 写在 Slack/Wiki/Docs 的知识对 Agent 等于不存在 | OpenAI：仓库为唯一事实源 |

> ⚠️ **AGENTS.md 误区警示**：不要当"超级 System Prompt"，不要把所有规则塞一个文件。正确做法：AGENTS.md 只当目录用（约 100 行），详细规则放子文档按需加载。

#### P1：P0 完成后考虑

| 行动 | 为什么 | 参考实践 |
|------|--------|---------|
| 分层管理上下文 | 渐进式披露，不要一次性塞满 | OpenAI AGENTS.md 当目录 |
| 建立进度文件和功能列表 | JSON 格式追踪功能状态，Agent 不太会乱改结构化数据 | Anthropic 两阶段 Agent |
| 给 Agent 端到端验证能力 | 浏览器自动化让 Agent 能像用户一样验证功能 | Anthropic 用 Playwright/Puppeteer MCP |
| 控制上下文利用率 | 尽量不超过 40-60%，增量执行 | Dex Horthy 的 Smart/Dumb Zone |

#### P2：有余力再考虑

| 行动 | 为什么 | 参考实践 |
|------|--------|---------|
| Agent 专业化分工 | 每个 Agent 携带更少无关信息，留在 Smart Zone | Carlini 的去重/优化/文档 Agent |
| 定期垃圾回收 | 确保清理速度跟得上生成速度 | OpenAI 后台清理 Agent |
| 可观测性集成 | 把"性能优化"从玄学变成可度量 | OpenAI 接入 Chrome DevTools |

**适用场景**：评估当前团队所处阶段、规划下一季度 Harness 投入、回答"我们现在该投资什么"时使用。

### 方法 3：AGENTS.md 设计的"地图模式"

**核心思想（综合 05、12、16、18）**：AGENTS.md 不是百科全书，是**地图**。

**为什么巨型 AGENTS.md 必败**（12 总结的 OpenAI 教训）：

| 失败原因 | 机理 |
|---------|------|
| Context 是稀缺资源 | 臃肿文件挤掉任务空间 |
| 指导过多反而失效 | "一切都重要 = 没有重点" |
| 文档快速腐烂 | 旧规则无人更新 |
| 单文件无法机械验证 | linter 检查不到内容偏差 |

**OpenAI 的解法**：
- AGENTS.md 收缩到 **~100 行**
- 只做"地图"指向 `docs/` 分类文档
- linter / CI 验证链接完整性
- 跑 "doc-gardening" 智能体定期扫描过时文档
- 代码仓库本身变成记录系统

**Mitchell Hashimoto 的解法**：
- AGENTS.md 的**每一行**对应一个历史失败案例
- 单 Agent 路线（适合中小规模项目）
- 六步进阶路径：积累失败 → 更新 AGENTS.md → 形成持续反馈循环

**适合放进 AGENTS.md / Custom Rules 的内容**（16）：
- 技术栈与架构模式
- 命名规范与代码风格
- 测试哲学
- 代码库中常见坑点
- 已经多次观察到的 agent 反模式

**不适合放进去的内容**（16）：
- 完整 API 文档
- 空泛废话，如"写干净代码"
- 互相冲突的指令

**实操建议**（16）：
- 保持在 **500 行以内**
- 按架构、测试、安全等主题模块化拆分
- 用少量 few-shot example 提升可执行性
- 不要把所有规则都做成 always-on，应按需加载

**关键认知（16）**：Custom Rules 不是一次性文档，而是随着 agent 犯错**不断迭代的"行为调优资产"**。

### 方法 4：OpenAI 路线——机械约束 + 分层架构

**核心思想（05、12、18、22）**：用"代码即治理"的思路把约束编码成机械规则，避免依赖人工提醒。

**典型场景**：3 人 5 个月 100 万行代码、1500 PR，几乎不手写代码。

**五大方法论**：

1. **地图式文档**：AGENTS.md 只当目录（~100 行），详细规则按需加载（渐进式披露）
2. **机械化约束**：自定义 Linter，**报错信息直接包含修复方法**——"错误消息直接告诉 Agent 怎么改"
3. **可观测性接入**：接入 Chrome DevTools 协议（截图 + 运行时事件）+ LogQL / PromQL 可观测性堆栈 + 明确验证标准（如服务启动 ≤800ms）
4. **熵管理**：专门的后台 Agent 做垃圾回收，确保清理速度 > 生成速度
5. **仓库即事实源**：所有知识必须在代码仓库，Slack / Wiki 对 Agent 不可见

**架构约束（12）**：
- 严格分层架构 + 依赖方向单向验证：**Types → Config → Repo → Service → Runtime → UI**
- 违反架构约束的 PR 被 linter 自动拦截
- 技术债务清理循环：定期 Codex 任务扫描代码偏差 → 发送重构 PR → 大多数一分钟自动合并

**关键洞察（12）**：
> 架构约束在传统公司要等团队扩到数百人才引入，但对编码 Agent 来说是**早期先决条件**——没有约束，速度越快，架构漂移越严重。

**人工时间分配**：最初每周约 20% 时间清理"AI slop"（冗余代码、过时文档、不一致命名），系统化后由 documentation consistency agents、refactor agents 接管。

### 方法 5：Anthropic 路线——三角色分离 + Sprint Contract

**核心思想（05、12、18、22）**：把 Generator 和 Evaluator 彻底拆开（GAN 思路），避免 Agent 自我评估虚高。

**三角色架构（2026.03 升级版）**：

| 角色 | 职责 | 设计要点 |
|------|------|---------|
| **Planner** | 把一两句描述扩展成完整 product spec 和分步 feature list | 只聚焦产品上下文和高层设计，不规定技术细节（避免错误级联） |
| **Generator** | 逐 feature 落地 | 每个 Sprint 前谈好"契约"（单 Sprint 27 条验收标准），每完成一个 commit |
| **Evaluator** | 独立验收 | 用 Playwright 真正跑页面、点按钮、看行为；标记 pass/fail，给出改进建议 |

**关键工程实践**：
- **Initializer agent** 创建 `init.sh`、`claude-progress.txt`、初始 git commit
- 把高层需求展开成 **200+ 条具体 feature**，全部初始标记为 failing
- 每个 coding agent 只做增量推进，session 结束时留 clean state
- 规则：**agent 只能改 feature passes 状态，不能修改测试定义本身**

**Context Reset 策略**：
- 长任务中定期清空上下文窗口，启动全新 agent
- 通过结构化交接文档（JSON 进度 + 功能列表）恢复状态
- 类似"重启进程解决内存泄漏"，而非无限加内存
- ⚠️ 这是失效模式应对，详细机理见「失效模式与反馈机制」方向

**Evaluator 调教难点（18）**：
- 开箱即用的 Claude 是很差的 QA Agent（会自己说服自己"不是大事"然后通过验收）
- 花了好几轮迭代才把评估质量调到合理水平
- → 详细机理见「失效模式与反馈机制」方向

### 方法 6：Mitchell Hashimoto 路线——单 Agent + 失败案例驱动

**核心思想（05）**：适合中小规模项目的最简化路线，靠 AGENTS.md 持续迭代而非多 Agent 编排。

**六步进阶**：
1. 启动单个 Agent + 极简 AGENTS.md
2. 观察 Agent 犯错
3. 把这次错误的根因抽象为一条规则
4. 加入 AGENTS.md
5. 形成持续反馈循环
6. AGENTS.md 的每一行 = 一次历史失败案例

**适用场景**：
- 单人或小团队项目
- 还在探索方向、不确定要哪些工具的早期阶段
- 把 P0 做到极致再考虑 P1/P2

**与 OpenAI / Anthropic 路线的对比**：

| 维度 | Hashimoto | OpenAI | Anthropic |
|------|-----------|--------|-----------|
| Agent 数量 | 单 Agent | 多 Agent + 后台清理 | 三角色（Planner/Generator/Evaluator） |
| 约束方式 | AGENTS.md 文本规则 | 自定义 Linter 机械规则 | Sprint Contract + Evaluator |
| 适用规模 | 个人 / 小团队 | 中等规模工程团队 | 长任务 / 高复杂度 |
| 投入成本 | 最低 | 中 | 最高（Solo 的 20x） |

### 方法 7：MercadoLibre 组织级 Rollout

**核心思想（16）**：把个人级 Harness 实践扩展到 2 万开发者、数千仓库的工程治理方案。

**规模数据**：
- 近 **20,000** 名开发者
- 覆盖**数千个**代码仓库
- Spec Driven Development 已获得约 **4,000 名**开发者采用
- 组织维护 **9+ 技术栈** 的标准化规则体系

**关键实践**：
- 组织维护标准化 rules 覆盖 9+ 技术栈
- 团队在标准化基础上叠加仓库级规则
- 自建内部 MCP，对接内部云平台、SDK、RAG 业务知识和应用文档
- 独立 Code Review Agent 接入 CI，对所有 PR（人写 + AI 写）做自动分析

**Code Review Agent 的组织收益**：
- 人类 reviewer 更少花时间找风格/规范问题
- 更聚焦高层设计与架构决策
- 更早发现 bug，降低进入生产的风险
- review 周期更快
- 质量控制能力随 PR 数量扩展，而非依赖 reviewer 人数扩展

**关键认知**：
- 大规模 AI coding agent 不是"工具部署问题"，而是"工程制度升级问题"
- 规则如何分层治理、工具如何合规接入、反馈如何自动执行、不同技术栈如何建立最低统一标准、组织知识如何通过 MCP/Skill 进入 agent 回路 —— 这是新的开发平台治理工程

**注意事项**：
- 即使 SDD 流程对业务有显著价值，**日常习惯改变仍然困难**（4000/20000 的覆盖率说明问题）
- 不要假设"上工具就能改变行为"，需要持续推动机制

## 案例库

### 案例 1：OpenAI 三人五月百万行零手写（05、12、22）

- **背景**：OpenAI 内部团队 3 人，5 个月，从空仓库开始，全部交给 Codex
- **产出**：约 100 万行代码，约 1,500 个 PR；初始 scaffold 仍由 Codex 在少量模板引导下生成
- **关键做法**：地图式文档 + 机械化约束 + 可观测性 + 熵管理 + 仓库即事实源（见方法 4）
- **辅助数据**：服务启动验证标准 800ms 内完成、单次 Codex 任务可连续工作超过 6 小时（人类睡觉时间）
- **启发**：工程师工作重心转向"设计环境、明确意图、构建反馈回路"，不是写代码

### 案例 2：Anthropic 三角色 GAN 架构 vs Solo Agent（12、18）

- **背景**：Solo Agent 与三 Agent Harness 对比同一编程任务
- **Solo 结果**：游戏能启动但连线断裂、界面无提示
- **三 Agent 结果**：正常游玩 + AI 关卡生成 + 精灵动画 + 音效
- **成本对比**：三 Agent 是 Solo 的 **20 倍**成本
- **本质**：从"不能用"到"能用"的跨越
- **辅助数据**：规划器把一句话需求展开成 16 个 feature 的产品规格、单个 Sprint 27 条验收标准
- **启发**：在长任务、低容错场景，分角色的高成本是值得的——单 Agent 的"看起来 demo 行"在生产环境就是不能用

### 案例 3：MercadoLibre 2 万开发者 rollout（16）

- **背景**：大规模组织级 Harness 实践（见方法 7）
- **关键数据**：Spec Driven Development 已获约 4,000 名开发者采用，但日常习惯改变仍然困难
- **结果**：建立跨技术栈、跨仓库、跨团队复制的工程治理方案，而非单一 prompt 技巧
- **启发**：AI coding agent 规模化的真正瓶颈是组织治理，不是工具选型

### 案例 4：典型失败案例——需求过粗导致"错得合理"（16）

- **场景**：用户说"做一个后台新增商品功能"
- **缺失信息**：技术栈、后台模块、API 合约、存储位置、幂等性、角色权限
- **结果**：Agent 基于现有代码 + MySQL 实现一个看似可工作的 MVP
- **暴露问题**：再次点击按钮出现重复插入
- **根因**：不是 Agent 不会写代码，而是初始输入未说明幂等性与角色权限等隐含业务约束
- **启发**：很多"AI 写错了"的问题本质是系统没有把隐性业务规则显式化——Spec Driven Development 解的正是这个问题

### 案例 5：Vercel 工具精简（与方向 1 共享，但角度不同）

- **背景**：少即是多原则在 Coding Agent 工具设计中的应用
- **做法**：可用工具从 **15 个砍到 2 个**
- **结果**：准确率 80% → 100%，Token 消耗减少 37%，速度提升 3.5 倍
- **工程启示**：在 Tool System 层做减法的 ROI 远高于做加法——不要假设"接更多 MCP 就更强"
- **与 Anthropic Tool Mediation 的呼应**（22）：精简到最小必要工具集比给 agent 全量工具效果更好

### 案例 6：OpenAI 机械约束实战——Linter 报错自带修复（12）

- **背景**：架构约束如何在没有 senior reviewer 把关时仍然成立
- **做法**：自定义 Linter，**报错信息直接嵌修复指令**
- **关键变化**：Linter 不只是"指出错误"，是"教 Agent 怎么改"
- **配套**：业务域依赖方向固定（Types → Config → Repo → Service → Runtime → UI），违反 PR 自动拦截
- **启示**：约束的形式很重要——"禁止 X" 不如 "把 X 改成 Y"，前者要求 Agent 自己悟出修复方法，后者是 in-context learning 的高质量例子

### 案例 7：混合检索基础设施（12）

- **背景**：Agent 在 Harness 中需要同时执行两类检索
- **两类需求**：
  - 语义查询："产品对用户会话的设计原则是什么？"（文档可能用"会话管理""访问控制"等不同措辞）
  - 精确匹配："哪些文档提到 `validateToken` 函数？"（函数名无语义，向量无法匹配）
- **解法（Milvus 2.6 Sparse-BM25）**：
  - 入库时同时生成 dense embedding + TF 编码稀疏向量
  - 全局 IDF 自动维护，随文档增删实时更新
  - 查询时 Reciprocal Rank Fusion 融合排序
- **性能**：BEIR 上吞吐量比 ES 高 3-4x，特定负载 QPS 7x
- **启示**：Coding Agent 的 RAG 不能只做向量检索——代码相关查询大量是精确匹配，混合检索是基础设施

## 关键洞察

> 跨多源融合后提炼的、超越单篇来源的发现。

1. **三种实战路线的"团队规模 × 任务复杂度"决定论**
   - Hashimoto 路线（单 Agent + AGENTS.md）适合个人 / 小团队
   - OpenAI 路线（机械约束 + 分层架构）适合中等规模工程团队
   - Anthropic 路线（三角色 + Sprint Contract）适合长任务 / 高复杂度
   - 选型不是"哪个更对"，而是匹配你的规模和复杂度——20x 成本的三角色架构对小项目是浪费

2. **Custom Rules 是基础投入，不是高级技巧**
   - 16 把它放在"最低门槛、最高 ROI"位置
   - 05 / 12 / 18 把它放在 P0 第一位
   - 即使没有 MCP、没有 Skills、没有 SDD，光把 AGENTS.md 做好（地图模式 + 失败案例驱动）就能解决大部分问题
   - 这就是为什么投入 Harness 从来不是"先平台后规则"，而是"先规则再平台"

3. **AGENTS.md 的失败模式有清晰规律**
   - 巨型化（>500 行） → 挤占 context、文档腐烂
   - 通用化（"写干净代码"） → 等于没说
   - 当超级 system prompt 用 → 与每轮任务上下文耦合
   - 一次性写完，从不更新 → 错过反馈闭环
   - 反过来，成功的 AGENTS.md 都满足三个特性：**短（~100 行）、按需加载（地图模式）、可迭代（每行有出处）**

4. **机械约束 vs 文本规则的"硬度差"**
   - 文本规则（"请遵守命名规范"）→ Agent 可能遵守也可能不
   - Linter 规则（机械检查）→ Agent 必须遵守
   - Hooks（执行拦截）→ Agent 没办法绕过
   - 同样一条规则，工程化为不同硬度，效果差异巨大——把"团队的口头规范"变成 Linter 规则是 P0 中最容易被忽视的高价值动作

5. **组织级 rollout 的"40% 边际效益陷阱"**
   - MercadoLibre 数据：SDD 只覆盖 4000/20000 = 20% 开发者
   - 这不是工具问题，是习惯改变问题
   - 工程化方案做得再好，没有强制机制（CI 拦截、Hooks 阻塞），20% 是 自然天花板
   - 这与"上工具就能改变行为"的常见乐观假设直接冲突

6. **"Code Review Agent" 是被低估的杠杆**
   - 在 P0/P1/P2 行动清单里没有显式出现
   - 但 MercadoLibre 案例显示它能"重构组织审查成本曲线"
   - 它同时是 Feedback Loop 和 Entropy Control 两个层面的实现
   - 任何团队 PR 量超过 reviewer 处理能力时都应该考虑

7. **三角色架构的真正价值不在分工而在"独立性"**
   - 三角色的核心不是"把任务拆三段"
   - 而是 **Evaluator 必须独立于 Generator**（GAN 思路）
   - 让一个 Agent 既写代码又评价代码 → 自评虚高
   - 独立 Evaluator 即使能力较弱，也能发现 Generator 看不见的问题
   - 这同样适用于其他场景：写 PRD 的人不能验收、做需求的人不能写 acceptance criteria

## 待探索问题

- **棕地项目的 Coding Agent Harness 怎么做？**
  - 所有公开案例（OpenAI、Anthropic、Stripe、Hashimoto）全是绿地项目
  - 存量代码库怎么接 AGENTS.md / Linter / 分层架构？
  - 现有架构债如何不被 Agent 复制放大？
  - 目前没有任何公开方法论
- **AGENTS.md 的腐烂速度怎么测量？**
  - "AGENTS.md 一旦超过 500 行就开始腐烂"是经验值
  - 有没有可量化的"AGENTS.md 健康度"指标？
- **Skills 的复用边界**
  - SkillHub 这样的 Skills 流通平台能成立吗？
  - 跨团队 / 跨公司 Skills 复用的实际障碍是什么？
- **Spec Driven Development 的覆盖率上限**
  - 4000/20000 是巧合还是规律？
  - 不靠强制机制（CI 拦截、Hooks），SDD 自然覆盖率天花板在哪？
- **Code Review Agent 的失效模式**
  - Agent 评 Agent 的代码会不会陷入"互相说服没事"的死锁？
  - Review Agent 本身的训练数据如何更新？
- **多 Agent 编排的"成本临界点"**
  - 20x 成本是任意场景都值得的吗？
  - 什么样的任务规模、容错要求下，多 Agent 是必要的而不是奢侈的？
