# Harness 演化与未来形态

> 方向定位：研究 Harness Engineering 的**时间维度**——Harness 不是静态架构，而是随模型能力演化的动态系统。涵盖 Harness 动态演化规律、模型-Harness 协同演化的双刃剑、Anthropic Meta-Harness 的稳定抽象哲学、AgentOS 层级关系、Harness 生态格局、安全维度升级、Skills/SkillHub 跨框架流通、工程师能力转型。
> 不涵盖：Harness 概念定义与分层模型（→ 概念与架构体系）；具体落地杠杆与四大方法（→ 工程实践）；失效模式机理（→ 失效模式与反馈机制）。

## 方向定位

Harness 演化与未来形态聚焦三组互相缠绕的问题：

1. **Harness 自身的演化规律**：Harness 的每个组件都编码了一个"模型做不到"的假设，会随模型升级而失效——哪些组件正在变成冗余？哪些是核心承重墙？
2. **模型-Harness 协同演化的副作用**：模型在特定 Harness 下训练带来正向反馈循环，但也导致泛化下降、Harness 难以解耦——这种耦合的代价与解药？
3. **未来形态预判**：Harness 会被模型吸收变薄、变成"Meta-Harness 稳定抽象层"、还是上升为 AgentOS 内核？Skills/SkillHub 这样的"跨 Harness 能力流通"层会成立吗？

**核心命题**：
- **Harness 是一个会移动的边界，不会消失但会重塑**
- **架构必须建立在稳定假设上，不是当前模型的暂时缺陷上**
- **驾驶员的"目的地"问题永远是人的责任，再强的模型也不能自己生成**

**不在本方向范围**：基础概念（→ 概念与架构体系）、四大杠杆和 AGENTS.md 设计（→ 工程实践）、Anti-pattern 与反馈机制细节（→ 失效模式与反馈机制）。

## 知识图谱

- **演化规律（核心）**
  - "每个组件都编码一个'模型做不到'的假设"
  - 假设失效 → 组件冗余 → 应该删除
  - 移除而非添加是 Harness 工程的常态动作
- **协同演化的双刃剑**
  - 正循环：原语 → 训练数据 → 模型变强 → 发现新原语
  - 副作用：泛化下降、Harness 难解耦
  - 案例：Codex-5.3 换补丁方式就掉分
- **Anthropic Meta-Harness 哲学**
  - 别把模型暂时缺陷硬编码成架构
  - 投资稳定接口而非一次性最优实现
  - Session / Harness / Sandbox 三层完全解耦
  - 安全靠架构隔离，不靠模型不作恶
- **AgentOS 层级关系**
  - Harness = AgentOS 用户态层（shell/daemon）
  - AgentOS = 内核（调度、隔离、资源管理）
  - 两者是上下层，不是竞争
- **Harness 生态格局**
  - 纵深型：Claude Code、Codex
  - 横向型：OpenClaw、Hermes
  - 编码 IDE：Cursor、Windsurf
- **Skills / SkillHub**
  - 跨框架通用的自然语言能力单元
  - 跨 Harness 流通平台
- **模型"长出手脚"趋势**
  - 工具调用、上下文管理、记忆系统正被内化
  - 外面的脚手架变薄
  - 极端情况：Harness 被吸收
- **不变的核心：人的责任**
  - "目的地"永远是人的责任
  - 模型越强，对人的能力要求越高
  - 品味、判断力、创造性
- **安全维度升级**
  - 攻击目标从数据变成 Agency
  - Tool Poisoning Attacks
  - 权限模型从静态升级为动态
- **工程师能力转型**
  - 码农 vs 工程师的分水岭
  - 系统复杂性理解深度
  - 驾驭不确定性的能力
- **Harnessability（可 Harness 度）**
  - 系统天然有多容易被 Agent 驯化
  - 强类型、测试、文档、可观测性 → 高
  - 散落知识、口耳相传 → 低

## 核心概念

### 概念 1：Harness 动态演化规律

**核心论断（12 的关键句）**：

> Harness 里的每个组件，都编码了一个**模型自己做不到的假设**——而这些假设，会随着模型能力的提升逐渐失效。

**演化模型**：

```
新模型发布 → 审视整套 Harness → 找出不再是承重墙的组件 → 删除

组件存在的前提 = 模型做不到的假设
假设失效 → 组件变成冗余 → 增加系统复杂度 → 降低效率
```

**实例（12、18）**：

| 组件 | 历史必要性 | 当前状态 |
|------|-----------|---------|
| Sprint 分解 | Opus 4.5 必要 | Opus 4.6 可删 |
| Context Reset | 早期模型必要 | Opus 4.5+ 可删 |
| Context Slice 精细管理 | 短窗口模型必要 | 长上下文模型可粗放 |
| AGENTS.md 大量约束 | 弱模型必要 | 强模型只需地图模式 |
| Compaction 摘要策略 | 长任务必要 | 大窗口模型可减少 |

**反向不变的组件**（18 的关键提醒）：
- **Evaluator** 永远去不掉——独立评估的需求边界会随模型移动，但边界本身不会消失
- 五大根本挑战（状态持久性、目标一致性、行动可验证性、熵增抑制、人机边界）每一条都不能靠"更强模型"消除

**关键论述（18）**：

> 评估器的价值取决于任务是否处于当前模型独立可靠完成的边界之外。这个边界会随模型能力移动，**但边界本身不会消失**。

**对工程师的工作内容暗示**：
- 新模型发布时，Harness Engineer 的第一件事是**找哪些组件可以删**
- 而不是"加新功能"
- 删除是演化的常态动作

**适用场景**：每次模型版本升级时，做"Harness 减重诊断"——哪些规则、压缩策略、Hook 在新模型下变成负担？

### 概念 2：模型-Harness 协同演化的双刃剑

**核心论断（07）**：

> 有用的框架原语被发现 → 加进框架 → 训练下一代模型 → 模型在这些框架里越来越强 → 发现新原语。但这种协同演化有个副作用：泛化能力会变差。

**正循环（好的一面）**：
- Harness 原语成为训练数据的一部分
- 模型在原语对应的能力上越来越强
- 反过来催生新原语
- 整个生态共同演化

**副作用（坏的一面）**：
- 模型过度拟合训练时用的 Harness
- 换一种工具逻辑就性能下滑
- **最适合你任务的 Harness ≠ 模型后训练时用的 Harness**

**关键案例**：

| 案例 | 现象 | 启示 |
|------|------|------|
| **Codex-5.3 泛化问题（07）** | 换一种补丁应用方式，模型性能直接下滑 | 模型与训练时 Harness 深度耦合 |
| **Opus 在 Claude Code Harness 下表现（05）** | Opus 在 Claude Code Harness 下的得分**远低于**在其他 Harness 下的得分 | "最好的 Harness 不一定是模型默认的那个" |

**对选型的指导（05）**：

> the best harness for your task is not necessarily the one a model was post-trained with——为你的任务选择 Harness 时，不要被模型的默认 Harness 束缚。

**对产品策略的指导**：
- 不要假设"用官方框架就是最优"
- 针对具体任务测试不同框架组合，可能有数量级的提升空间
- Model-Harness 耦合是选型考量，不是技术细节

**深层启发**：
- 这种耦合可能正在让 AI 进入"路径依赖"状态
- 后期解耦的成本可能极高
- 30 提出的 Meta-Harness 三层架构正是对此问题的工程回应

**适用场景**：评估"用 Claude Code 还是自建 Harness"时，要意识到 Claude Code 的能力在 Claude Code Harness 下被放大，但这不等于在你的具体任务下最优。

### 概念 3：Anthropic Meta-Harness 的稳定抽象哲学

**核心论断（30）**：

> 几乎所有 Agent 框架都在犯同一个错误：把模型**当前**的能力缺陷，**硬编码**成了系统**永久性**的架构设计。

**典型的"补丁固化"案例**：

| 模型缺陷 | 框架做法 | 真实身份 |
|---------|---------|---------|
| 模型不会规划 | 加固定 DAG 步骤拆解 | 这是补丁，不是架构 |
| 模型记不住长上下文 | 加向量记忆检索模块 | 这是对局限的妥协 |
| 模型调工具容易出错 | 加多层校验重试路由 | 这是保护机制 |

**风险**：

> 模型能力是指数级进化的：半年前不会正确调用工具，现在已能独立完成完整软件工程任务；一年前算术会出错，现在已能处理超过 10 小时的长时域任务。**你花几个月打磨的"完美框架"，可能一次版本升级就完全过时，反而成为限制模型能力的性能瓶颈。**

**Anthropic 的解答：Meta-Harness 三层架构**：

```
应用层：Agent 业务逻辑
    ↓
Session（执行事实流） ← append-only 不可变事件日志
Harness（调度脑干）   ← 无状态、可热替换、标准接口
Sandbox（可插拔执行） ← 大脑与手解耦
    ↓
基础设施层：工具 / MCP 服务 / Vault
```

**三个关键设计原则（30）**：

1. **Session 不是 context window**：是 append-only 的不可变事件日志，记录整个任务执行过程的所有事实
   - 可查询、可回放、可恢复、可重组
   - 即使上下文窗口被压缩，原始执行事实永远存在
   - 这是"事实流 vs 摘要流"的根本区别

2. **Harness 无状态 + 可热替换**：
   - 不保存任何会话状态，只从 Session 日志查询信息
   - 标准接口：`execute(name, input) → string`
   - 模型变强了 → 直接换一个更简单的 harness → 其他部分完全不用动

3. **Sandbox 与大脑完全解耦**：
   - 大脑（Claude 模型 + Harness 调度层）只负责决策
   - 手（Sandbox + 工具 + MCP 服务）只负责执行
   - Sandbox 独立、可共享、可传递、懒加载

**核心承诺**：
> 不承诺 harness 具体长什么样，只承诺几类长期稳定的接口。

**Anthropic 的赌注**：
> 我们不相信今天的 Agent Harness 会是最终形态，所以我们优先投资于稳定接口，而不是一次性的最优实现。

模型能力持续提升后，harness 逻辑会越来越少，最终由模型自己实现调度，harness 退化为薄薄的一层甚至消失。

**性能数据（30）**：
- Sandbox 懒加载改造后，p50 首 Token 延迟降低 60%、p95 延迟降低 90%+
- 原来的瓶颈不是模型推理，而是预启动整个执行环境的开销

**适用场景**：设计长期演进的 Agent 平台时，决定"哪些是稳定接口、哪些是可替换实现"。

### 概念 4：模型"长出手脚"——Harness 内化趋势

**核心论断（17）**：

> 随着模型能力的持续增强，上下文窗口越来越大，记忆能力不断提升，推理链条越来越长，**模型正在自己长出手脚**。今天需要外部搭建的工具调用、上下文管理、反馈循环、记忆系统，模型正在一项一项地内化。**外面的这套脚手架正在变薄。**

**当前内化进程**：
- 上下文窗口越来越大 → Context Slice 精细管理需求下降
- 记忆能力提升 → 外部记忆系统可减重
- 推理链条变长 → 部分编排逻辑可省略
- 工具调用准确率提升 → 多层校验重试可简化

**类比演进**：

| 阶段 | 汽车 | Agent |
|------|------|-------|
| 早期 | 需要复杂的外部操作机构来转化发动机动力 | 需要复杂的 Harness 转化模型能力 |
| 现代 | 发动机和传动系统高度一体化，线束越来越简单 | Harness 变薄，模型"懂"得越来越多 |
| 极端情况 | 一体化设计 | Harness 被模型完全吸收 |

**与 Meta-Harness 哲学的衔接**：
- 17 描述的"内化趋势"
- 30 给出的"应对方案"（不固化补丁、稳定抽象层）
- 两篇文章共享同一前提：**今天的 Harness 不是最终形态**

**但有些事永远不变**（17 的关键转折）：

> 即便模型吸收了所有的工具和流程，有一件事它永远无法自己生成：**目的地**。去哪里，为什么去，到了之后怎么判断值不值，这些关于方向、意义和价值的问题，永远是人的责任。**模型越强，这个责任越重。**

**"目的地不可吸收"清单**：
- 方向（去哪里）
- 意义（为什么去）
- 价值判断（到了之后怎么判断值不值）
- 品味（什么是好的、对的、值得做的）

**适用场景**：做长期产品路线图时，区分"会被模型吸收的能力"和"永远是人的责任的能力"——前者不应过度投资，后者是核心壁垒。

### 概念 5：AgentOS 与 Harness 的层级关系

**核心论断（22）**：

> Harness 是 AgentOS 的**用户态层**（shell/daemon），AgentOS 是**内核**（调度、隔离、资源管理）。两者是上下层，不是竞争关系。

**三层 Agent 演进路径（22）**：

| 层级 | 时间 | 特征 | 工程抽象 | 天花板 |
|------|------|------|----------|--------|
| Level 1：Chatbot | 2022-2023 | 无状态、单次对话 | Prompt Engineering | 能说不能做 |
| Level 2：Agent IDE | 2024-2025 | 多步任务、工具调用 | Context Engineering + 轻量 Harness | 长任务不稳、多 agent 无标准 |
| **Level 3：AgentOS** | **2026- 萌芽** | **always-on、跨工具、跨身份** | **系统层（调度、内存、隔离）** | 仍在研究阶段 |

**与传统 OS 的类比**：

| 传统软件栈 | Agent 栈 |
|----------|---------|
| OS 内核（调度、内存、文件、I/O） | AgentOS（调度、隔离、资源） |
| 用户态 shell / daemon | Harness（运行时外循环） |
| 应用 | Agent 业务逻辑 |

**关键引用**：ASPLOS 2026 AgenticOS Workshop 论文（agent 工作负载的 OS 原语）

**实用启示**：
- 当一家公司宣称"我们做 AgentOS"时，要追问它是**替代** Harness 还是**承载** Harness
- 这决定了选型逻辑：替代关系下二选一，承载关系下两层都要做
- 短期内 Level 3 仍在萌芽阶段，绝大多数团队的实际工作仍在 Level 2 内

### 概念 6：Harnessability（可 Harness 度）

**定义（22）**：

> 一个系统天然有多容易被 agent 驯化——是 agent 时代的关键变量。

**两端对照**：

| 高 Harnessability | 低 Harnessability |
|------------------|------------------|
| 强类型 | 弱类型 / 动态类型 |
| 测试完备 | 测试覆盖率低 |
| 边界清晰 | 模块耦合严重 |
| 文档版本化 | 文档散落各处 |
| 运行时可观测 | 黑盒难诊断 |
| 知识在仓库 | 知识在人脑/Slack |

**关键判断（22）**：

> 工程基础设施质量（CI 完善度、文档结构化、架构边界清晰）已不只是"工程素养"问题，而是**直接决定 agent 能在系统上走多远**。

**对存量代码库的暗示**：
- 棕地项目的 Harnessability 通常较低
- 这是为什么所有公开成功案例都是绿地项目
- 提升 Harnessability 是改造存量项目接入 Agent 的前置工程

**Harnessability 与"看不见的知识等于不存在"（18）的呼应**：
- 散落在 Slack 讨论、Google Docs 共识、资深工程师脑中的隐性判断
- 对 agent 都不可见
- 这些知识在仓库化之前，Harness 再好也无效

**适用场景**：评估现有项目能否接入 Coding Agent 时，先做 Harnessability 自检——而不是直接接 Claude Code 然后等失败。

### 概念 7：人的责任不可吸收——驾驶员的觉醒

**核心论断（17）**：

> AI 越强大，对人的要求**不是降低了，而是提高了**。

**为什么**（自动驾驶悖论）：
- 表面上自动驾驶让人不用开车
- 但一个能够安全监督自动驾驶系统的人，需要比普通驾驶员**更深刻地理解驾驶本身**

**AI 驾驭需要的能力**：
- 同时理解人类工程实践和 AI 思维方式
- 把错误模式抽象为规则的能力（→ Harness 工程的核心动作）
- "品味"——判断什么是好的、什么是对的、什么是值得做的能力

**双层结构（17）**：

| 群体 | 在 AI 时代的位置 |
|------|----------------|
| 顶尖驾驶员 | 产出最优秀作品；竞争壁垒是品味、判断力、创造性 |
| 大多数人 | 享受 AI 能力提升，不需要成为高阶驾驶者 |

**未来角色转变**：
- 驾驶员角色从"操作者"升级为"委托人"
- 不再告诉 AI 怎么跑
- 而是告诉它要去哪里

**与 Harness 减重的关系**：
- 模型吸收得越多，人需要做的"操作"越少
- 但"决定做什么"的责任反而越重
- 因为当机器什么都能干的时候，**"干什么"变成了唯一重要的问题**

**适用场景**：做组织能力规划时，区分"会被 AI 替代的工种"和"会被 AI 增强的工种"——后者需要的恰恰是"品味"而非"执行力"。

## 方法论与框架

### 框架 1：Harness 减重诊断法

**核心思想**：每次模型版本升级，做一次 Harness 减重诊断。

**诊断流程**：

```
Step 1：列出当前 Harness 的所有组件
    ↓
Step 2：对每个组件回答："它在防止什么模型缺陷？"
    ↓
Step 3：测试新模型是否还有这个缺陷
    ↓
Step 4：如果缺陷消失 → 标记为"待删除"
    ↓
Step 5：分阶段移除，观察是否有回退
    ↓
Step 6：把"曾经必要但已删除"的组件存档（不删代码，备查）
```

**典型可减重项**（12、18）：
- 上下文压缩策略（长窗口模型出现后）
- Sprint 强制分解（推理链变长后）
- Context Reset 触发（Context Anxiety 消失后）
- 工具调用重试机制（工具调用稳定后）
- AGENTS.md 详细规则（in-context learning 改善后）

**不应减重的组件**（22 的"五大根本挑战"）：
- 独立 Evaluator（自评偏差是根本性的）
- 状态持久化（context window 不可能无限大）
- 权限边界与凭证隔离（安全不能依赖模型不作恶）
- 熵增抑制（不会因模型变强而自动消失）
- 人机交接点（"目的地"永远是人的责任）

**注意事项**：
- 减重必须有回退机制
- 一次只减一个组件，不要批量删
- 记录每次减重的前后对照数据

### 框架 2：Anthropic Meta-Harness 三层迁移路径

**核心思想（30）**：把现有"Session = 消息列表"、"Harness = 主循环"、"Sandbox = 无概念"的结构升级到 Meta-Harness 三层。

**当前大多数 Agent 系统的现状**：
- Session = 消息数组（消耗品，无历史回放）
- Harness = 主循环（有状态、与执行环境耦合）
- Sandbox = 无概念（工具直接 spawn 子进程或在 Agent 进程内跑）

**改造路径（30 建议的顺序）**：

```
Step 1（最容易）：Sandbox 解耦
    - 把工具从 Agent 进程内移出
    - 接成独立服务，通过标准接口调用
    - 类比云原生的 sidecar 模式
    ↓
Step 2：Session 升级为执行事实流
    - 改成 append-only 不可变事件日志
    - 实现 query、replay、resume、reorganize 四能力
    - 与 context window 解耦
    ↓
Step 3（最难）：Harness 无状态化
    - 把状态从 Harness 移到 Session
    - 实现 Harness 可热替换
    - 标准化执行接口
```

**关键设计原则**：
1. **会话设计从"消息列表"升级到"执行事实流"** - 长任务（>30min）差异显著
2. **工具和环境独立于 Agent 本体** - 不要内化成 Agent 的一部分
3. **上下文工程逻辑不要硬编码** - 留好替换接口

### 框架 3：凭证隔离架构（安全升级）

**核心思想（30）**：

> 安全不建立在"模型不会做什么"的假设上，而是建立在系统架构的权限隔离上。

**传统思路的失败**：
- 给模型有限权限，赌模型不会突破限制
- 随着模型越来越聪明，这种假设越来越脆弱

**Anthropic 凭证隔离架构**：

```
Claude 模型（只看到代理凭证，看不到真实密钥）
    ↓
MCP Proxy（Session Token 验证）
    ↓
Vault（存储真实 Git Token / API Key）
```

**关键机制**：
- Git Token、API 密钥**全部存在独立 Vault**
- 模型通过 MCP Proxy 间接调用工具
- Proxy 拿 Session Token 去 Vault 取真实凭证执行
- **模型全程看不到明文密钥**

**对新型安全威胁的回应**（22 的 Tool Poisoning Attacks）：

| 威胁 | 机理 | 凭证隔离的应对 |
|------|------|--------------|
| Tool Poisoning Attacks | 恶意指令藏在 MCP 工具描述里（对用户不可见，对模型可见） | 模型即使被诱导调用工具，也拿不到敏感凭证 |
| 跨工具数据外流 | 不可信 MCP server 联动可信 MCP 实现数据外流 | Proxy 层可审计、阻断异常数据流 |

**Harness 权限模型升级（22）**：

> 必须从静态"可以/不可以"升级为**动态**"在什么条件下可以、到什么上限可以、需要人类确认后才可以"。

**对应 26 的 Allow / Deny / Ask 三级权限**：
- Allow：白名单内自动放行
- Deny：黑名单内直接拒绝
- Ask：模糊地带交人确认

### 框架 4：Harness 生态格局判断

**核心思想（26）**：当前 Harness 产品生态分纵深型和横向型，两类各有适用场景。

| 类型 | 代表产品 | 特点 | 适用场景 |
|------|---------|------|---------|
| **纵深型（深度工程开发）** | Claude Code、Codex（OpenAI） | 少而精的工具设计、极强的 Context 压缩能力 | Coding Agent、深度工程任务 |
| **横向型（自动化运营）** | OpenClaw、Hermes | 对接 WhatsApp、飞书等 | 自动运营、信息采集 |
| **编码 IDE** | Cursor、Windsurf | Claude Code 出现前的主流 | 通用代码辅助 |
| **开源平替** | OpenCode | Claude Code 开源平替，配合 DeepSeek 成本低 | 简单任务、成本敏感 |

**典型搭配**：
- Claude Code 编码 + Codex review（互补：前者强生成、后者强审查）

**关键判断**：
- 纵深型 Harness 在特定垂直场景做到极致
- 横向型 Harness 在多平台连接性上做到极致
- 两者**不直接竞争**，是不同的工程权衡

### 框架 5：Skills/SkillHub 的跨 Harness 流通

**核心思想（17）**：基于自然语言描述的 Skills 可以跨框架复用，可能成为 AI 时代的"应用商店"。

**Skills 定义**：
- 大模型能直接读懂的、基于文本描述的能力单元
- 告诉模型"这个工具是什么、能干什么、怎么调用"

**与传统 API 的区别**：

| 维度 | 传统 API | Skills |
|------|---------|--------|
| 描述方式 | 精确参数、严格调用格式 | 自然语言描述 |
| 调用主体 | 编程语言绑定 | 模型可理解 |
| 跨平台 | 需要适配 | 跨框架通用 |

**SkillHub 的产品形态**：
- 为 Harness 时代搭建的 Skills 流通平台
- 让能力可沉淀、可复用、可跨框架共享

**对 16 四大杠杆的呼应**：
- Skills 是 16 提出的四大杠杆中最强的单元
- 但 16 主要讨论团队内 Skills
- 17 把 Skills 升级到**生态层**——跨团队、跨公司流通

**开放性问题**：
- SkillHub 是否能成为现实？取决于标准化程度和激励机制
- 公司是否愿意把内部 Skills 开放？知识产权与竞争优势的权衡
- 跨 Harness 兼容性是否能保持？

### 框架 6：In-the-loop → On-the-loop → Meta-Harness 升级阶梯

**核心思想**：把多个方向的"工程师能力升级路径"放在同一个阶梯上。

| 阶段 | 工程师做什么 | 来源 |
|------|------------|------|
| L0：人工调试 | 看到错就改 agent 输出 | 22（In-the-loop） |
| L1：规则化 | 错了把规则写进 AGENTS.md | 16、Hashimoto |
| L2：机械化 | 错了把规则做成 Linter / Hook | 16、OpenAI 路线 |
| L3：平台化 | 系统化反馈循环（Code Review Agent / Eval 平台） | 22（On-the-loop） |
| **L4：Meta-Harness 化** | **设计稳定抽象层，不固化模型暂时缺陷** | **30** |

**各层级的核心动作**：
- L0 → L1：把一次性错误变成永久规则
- L1 → L2：把文本规则变成机械检测
- L2 → L3：把单点检测变成系统反馈
- L3 → L4：把当前实现变成稳定接口

**为什么 L4 最难**：
- 需要预判哪些是稳定的、哪些是暂时的
- 需要管理层支持长期工程投入
- 需要承认"今天的最优实现是明天的技术债"

**实用判据**：你的团队当前在哪一层？
- L0：每次都人工修
- L1：有 AGENTS.md 但不更新
- L2：有 Linter / Hook
- L3：有独立 Eval / Review Agent
- L4：把核心抽象做成可替换接口

## 案例库

### 案例 1：OpenClaw "第一只爬上岸的龙虾"（17）

- **背景**：2026 年春天 OpenClaw 搅动整个 AI 行业
- **关键事实**：
  - 没有发布任何新模型
  - 没有刷新任何基准测试
  - 没有训练一个新参数
  - 只做了一件事：给大模型搭建了一套完整的工作环境
- **工作环境构成**：文件系统、代码沙箱、工具链、反馈循环、自动验收
- **结果**：同一个模型从只会对话的聊天机器人变成能持续工作、自主解决问题的智能体
- **演化角度的启示**：OpenClaw 是 Harness 作为**独立产品形态**的标志性案例——能力提升完全来自外部脚手架，证明"Harness 即产品"是可行的

### 案例 2：Anthropic Managed Agents 的稳定抽象层（30）

- **背景**：Anthropic 通过 Claude Managed Agents 给出 Meta-Harness 三层架构
- **关键做法**：
  - 早期把 harness、session、sandbox 塞在同一容器里
  - 遇到：harness 崩了整个会话挂掉、状态丢失、调试困难
  - 新架构完全解耦：Session（执行事实流）/ Harness（无状态调度脑干）/ Sandbox（可插拔执行）
- **性能数据**：Sandbox 懒加载改造后，p50 首 Token 延迟降低 60%、p95 延迟降低 90%+
- **核心承诺**："不承诺 harness 具体长什么样，只承诺几类长期稳定的接口"
- **演化角度的启示**：明确把"补丁不该变架构"作为工程哲学落地

### 案例 3：Codex-5.3 泛化问题（07）

- **背景**：模型与训练时 Harness 深度耦合的实际代价
- **现象**：换一种补丁应用方式，模型性能直接下滑
- **演化角度的启示**：协同演化的副作用已经显现——模型在特定 Harness 下训练，换 Harness 就掉分

### 案例 4：Opus 在 Claude Code Harness 下的反常表现（05）

- **现象**：Opus 在 Claude Code Harness 下的得分**远低于**在其他 Harness 下的得分
- **解释**：Claude Code 的模型和 Harness 是一起训练的，但"最好的 Harness 不一定是模型默认的那个"
- **演化角度的启示**：选型时不要假设"用官方框架就是最优"，针对具体任务测试不同 Harness 可能有数量级提升空间

### 案例 5：Anthropic 凭证隔离架构（30）

- **背景**：安全模型从"赌模型不作恶"升级到"系统架构隔离"
- **架构**：Claude 模型 → MCP Proxy → Vault（Git Token / API Key）
- **关键变化**：模型全程看不到明文密钥
- **演化角度的启示**：当模型能力变强，"防御不当行为"必须从训练时约束升级到运行时架构

### 案例 6：Tool Poisoning Attacks（22）

- **背景**：Invariant Labs 在 2025.04 披露的新型攻击
- **机理**：恶意指令可藏在 MCP 工具描述里（对用户不可见，对模型可见），诱导 agent 执行未授权操作
- **进阶威胁**：通过不可信 MCP server 联动可信 WhatsApp MCP 实现数据外流
- **演化角度的启示**：攻击目标从数据变成 Agency；Harness 权限模型必须从静态"可以/不可以"升级为动态条件式判断

### 案例 7：模型能力时间线（30）

- **背景**：Anthropic 给出的模型能力进化时间线
- **数据**：
  - 一年前：模型连简单算术都会错
  - 半年前：模型还不会正确调用工具
  - 现在：已能独立完成完整软件工程任务、处理 10+ 小时长时域任务
- **演化角度的启示**：模型能力的进化速度可能比 Harness 设计的迭代速度更快——这就是为什么要做稳定抽象层而不是"完美框架"

### 案例 8：腾讯内部 Harness 实践（17）

- **背景**：汤道生分享腾讯内部经验
- **发现**：在同样的模型能力下，不同的脚手架设计对实际使用效果与 tokens 成本有很大影响
- **关键变量**：工具调用、分层上下文工程、长记忆管理、工作流设计
- **演化角度的启示**：Harness 不是理论概念，而是已经在大规模生产环境中验证的工程实践

## 关键洞察

> 跨多源融合后的发现。

1. **Harness 是会移动的边界，不会消失但会重塑**
   - 17 说"脚手架变薄"——是 Harness 减重
   - 22 说"五大根本挑战不消失"——是核心承重墙
   - 30 说"投资稳定接口"——是抽象层调整
   - 三个说法不矛盾：**Harness 的具体内容会变，但 Harness 的位置不会消失**
   - 类比：操作系统的具体功能在变（图形界面、虚拟内存、容器），但 OS 这个层级永远存在

2. **协同演化的副作用是"路径依赖"**
   - 模型在特定 Harness 下训练 → 模型与 Harness 绑定
   - 换 Harness 就掉分（Codex-5.3、Opus）
   - 长期看，这可能让 AI 进入路径依赖状态
   - **解药 1**：Anthropic 的 Meta-Harness 三层架构（稳定接口）
   - **解药 2**：Skills/SkillHub 这样的跨框架抽象层
   - 但路径依赖的力量很大，能否真正解耦仍是开放问题

3. **"补丁固化成架构"是 Agent 工程的最大风险**
   - 30 把这个问题点得最透：模型不会规划 → 加 DAG 拆解 → 模型变强了 DAG 反而成为瓶颈
   - 这是为什么 Harness 工程的常态动作是**移除而非添加**
   - 工程师在 Agent 时代的反直觉技能：**学会删自己写过的代码**

4. **Harnessability 把"工程素养"升级为"基础设施差异"**
   - 强类型、测试、文档、可观测性，过去是"加分项"
   - 现在是"决定 Agent 能在系统上走多远"的硬约束
   - 棕地项目接入 Agent 的难度，本质是 Harnessability 不足
   - 这暗示了**未来工程能力的两极分化**：高 Harnessability 项目快速 Agent 化，低 Harnessability 项目被甩开

5. **"模型长出手脚" vs "人的责任不可吸收" 的张力**
   - 17 同一篇文章里两个论点同时存在：
     - 脚手架变薄、模型内化
     - 模型越强、人的责任越重
   - 这不是矛盾，是对"什么会被吸收 / 什么不会"的精准切分
   - 会被吸收：工具调用、上下文管理、反馈循环、记忆系统（"如何做")
   - 不会被吸收：目的地、意义、价值判断、品味（"做什么")
   - 工程上的实用启示：在"如何做"上少投资长期方案，在"做什么"上多投资团队能力

6. **AgentOS 与 Harness 是上下层，不是替代**
   - 22 明确指出层级关系：Harness 是用户态层，AgentOS 是内核
   - 但行业叙事中两者经常混淆
   - "我们做 AgentOS"可能是营销话术，需要追问具体是替代还是承载
   - 短期内（2026-2027），绝大多数团队的实际工作仍在 Level 2（Agent IDE）内

7. **安全模型从训练时约束升级到运行时架构**
   - 传统：训练模型不要做坏事
   - Anthropic Meta-Harness：模型即使想做坏事也拿不到凭证
   - 这是 Agent 时代安全范式的根本转变
   - 类比：早期网络安全靠"用户自律"，现代网络安全靠"系统隔离"
   - 启示：靠模型不作恶的产品架构，在模型变强后会非常脆弱

8. **演化的不可预测性催生"赌哲学差异"**
   - LangChain 等框架的赌注：当前模型缺陷会持续，要先解决眼前问题
   - Anthropic Meta-Harness 的赌注：模型会持续进化，今天的最优实现会变成明天的技术债
   - 两种赌哲学没有绝对对错，看的是时间窗口
   - 短期产品 → 第一种赌注合理
   - 长期平台 → 第二种赌注更安全
