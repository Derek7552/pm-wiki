## 方向定位

Harness Engineering（驾驭工程）研究**模型之外的一切系统组件**——把"会生成的大脑"变成"能稳定交付的工作体"所必需的基础设施。本方向聚焦三类骨架性问题：

1. **定义边界**：Harness 不是 Prompt、不是 System Prompt、不是某个工具、不是 LangChain 这样的框架，而是包覆模型的整套外循环系统。
2. **历史定位**：Harness Engineering 在 Prompt → Context → Harness 这条 AI 工程范式跃迁链上处于何处，与 30 年软件工程史中的复杂性驾驭传统是什么关系。
3. **结构分层**：业界主流的多种"Harness 分层模型"（六层架构、三层结构、七大原语、六大构件、Meta-Harness 三层、六大组件……）如何统一到一张概念地图上。

**不在本方向范围**：具体的工程杠杆（AGENTS.md、Skills、SDD）属于工程实践方向；自我评估偏差、Context Anxiety 等 Anti-pattern 属于失效模式方向；Harness 减重、模型内化、AgentOS 等趋势属于演化方向。

## 知识图谱

- **核心命题**
  - Agent = Model + Harness
  - 模型决定上限，Harness 决定底线（也有反向表述：Harness 决定上限）
  - 同一模型只换 Harness，性能可有数量级差异
- **三层工程演进**
  - Prompt Engineering（怎么问）
  - Context Engineering（看到什么）
  - Harness Engineering（整个系统怎么运转）
  - 关系：嵌套包含 / 范式跃迁 / 同一批工程师的认知升级
- **历史定位**
  - 30 年软件工程驾驭复杂性的延续：对象 → 企业架构 → 分布式 → 数据系统 → 智能体
  - AI 工程五幕：生成 → 连接 → 推理 → 行动 → 治理
  - 术语进入主流的时间线（2026.02 起）
- **Harness 的本体**
  - 外循环系统 vs 模型内循环
  - 它管四类事：给模型看什么、允许做什么、怎么判断对不对、出错怎么修
  - 它不是更长的 system prompt、不是某个框架、是一门工程实践
- **架构分层（多家方案的统一视图）**

  - L6 约束与恢复
- **技术原语（自底向上）**
  - 文件系统（基石）
  - Bash / 代码执行（通用工具）
  - 沙箱
  - 记忆与搜索
  - 上下文管理
  - 长期自主执行
  - 编排逻辑
- **隐喻与类比**
  - 发动机—线束—驾驶员（汤道生）
  - 大脑与手（Anthropic）
  - 马具（Harness 直译）

  - 福特流水线 vs 发动机马力竞赛
- **Harness 与相邻概念的边界**
  - Harness vs Prompt / Context Engineering

  - Harness vs AgentOS（用户态 vs 内核态）
  - agent harness vs evaluation harness

## 核心概念

### 概念 1：Agent = Model + Harness

**定义**：Agent 由两部分组成——Model 提供原始智力（"文本进文本出"的概率函数），Harness 提供把这种智力变成可稳定交付的整套外部基础设施。

**等价表述**：
- "如果你不是模型，你就是框架/Harness。"（Vtrivedy10）
- "模型是 CPU，Harness 是操作系统。CPU 再强，OS 拉胯也白搭。"
- "大模型是发动机，Harness 是线束，使用者是驾驶员。"（汤道生）
- "模型负责提供智能，框架负责把这种智能变成真正有用的东西。"

**关键含义**：
- 评估一个 Agent 效果，真正评估的是 `model + harness` 的组合，不能孤立看模型分数
- Harness 是一个团队/产品的战略级资产，不是一段 prompt
- "你做的事情永远不是在给系统编码，你是在设计并控制这个系统。"

**适用场景**：任何关于 Agent 选型、产品定义、能力归因的讨论，都应回到这条公式去定位变量。

### 概念 2：Harness 的精确定义

**定义（综合 22 / 06 / 17）**：

> Harness 是让模型能够作为 Agent 行动起来的**外循环系统**——决定何时启动模型推理、给它什么上下文、如何验证输出、何时回退、何时停止。

**它包含什么**（跨多源汇总，去重后的最小集合）：

| 组件类别 | 具体内容 |
|---------|---------|
| 系统提示与角色 | system prompt、custom rules、角色设定 |
| 工具与技能 | 文件读写、Bash、Web Search、MCP、Skills |
| 基础设施 | 文件系统、沙箱、浏览器、代码执行环境 |
| 状态与记忆 | 持久状态、跨会话记忆、AGENTS.md、进度文件 |
| 编排逻辑 | 子代理、任务交接、模型路由、Ralph 循环 |
| 反馈与验证 | 自动测试、linter、type check、Evaluator、hooks |
| 约束与安全 | 权限边界、沙箱隔离、操作审计、Vault、人工审批 |
| 观测与审计 | 日志、metrics、traces、可观测性堆栈、审计日志 |

**它不是什么**（三个常见误区，源自 06 / 22）：

- **Harness ≠ Prompt**：Prompt 是点菜，Harness 是后厨整个运营体系
- **Harness ≠ 某个工具**：接一个搜索工具只是工具层的一部分
- **Harness ≠ System Prompt**：把所有规则塞进一个超长 prompt 是当下最常见的失败模式
- **Harness ≠ Framework**：LangChain 是框架，Harness 是工程实践（类似 DevOps 不是工具是文化）

**重要区分**：本方向讨论的 **agent harness（运行时外循环）**，与机器学习社区的 **evaluation harness（评测流水线）** 是两个不同概念。

**适用场景**：当团队争论"我们已经接了 LangChain，Harness 还要再做什么"或"是不是把 system prompt 写细一点就行"时，回到这组边界界定。

### 概念 3：三层工程演进（Prompt → Context → Harness）

**核心论点**：AI 工程方法论经历了三次范式跃迁，三层之间**不是替代关系，而是嵌套包含**——后一层把前一层装进自己内部。

**三层的核心问题对比**：

| 工程层 | 时间 | 核心问题 | 工作对象 | 隐喻（汤道生） |
|--------|------|---------|---------|--------------|
| Prompt Engineering | 2022.11-2024 | 怎么和模型说？ | 单次对话的输出质量 | 给驾驶员一张地图 |
| Context Engineering | 2024-2025 | 给模型看什么？ | 上下文窗口的内容编排 | 给驾驶员一套导航系统 |
| Harness Engineering | 2026- | 给 Agent 造一个什么世界？ | 长任务/复杂场景的稳定自主运行 | 给驾驶员造一辆完整的车 |

**问题升级驱动层次升级**：
- 简单任务 → Prompt 写清楚就行
- 依赖外部知识的任务 → Context 管理是关键
- 长链路、可执行、低容错场景 → 必须有完整的 Harness 保证持续正确和故障恢复

**嵌套关系**（22 给出的最精确表述）：
```
Prompt Engineering  ⊂  Context Engineering  ⊂  Harness Engineering
单次调用如何问           每一步喂什么信息           整条流水线怎么运转
```
- Context Engineering 是 Harness Engineering 的子集
- Harness 还额外包含：多步结构、工具中介、验证门、durable state

**与 Coding Agent 杠杆方向的衔接点（16）**：
> Context engineering 是理论，Agent harness 是把这套理论真正工程化落地的实现。

**适用场景**：判断当前团队的瓶颈在哪一层——只会写 prompt 的团队还在第一层；做了 RAG 但产品 demo 漂亮、生产翻车的团队卡在第二层；要让 Agent 进入企业生产流程的团队必须进入第三层。

### 概念 4：发动机—线束—驾驶员三角

**定义（汤道生）**：用机械工程的隐喻把 Agent 系统的三个角色解耦——

| 角色 | 在 Agent 系统中对应 | 价值定位 |
|------|-------------------|---------|
| 发动机 | 大模型 | 原始动力来源，能力天花板 |
| 线束（Harness） | 模型之外的一切基础设施 | 把动力转化为可控可用 |
| 驾驶员 | 使用者（人） | 决定方向、判断质量、定义"完成" |

**为什么是三角而不是二元**（核心洞察）：
- 单看 Model + Harness 容易陷入"全自动 Agent"的幻觉
- 模型越强，对驾驶员的能力要求**不是降低而是提高**（自动驾驶悖论：监督自动驾驶比开车要求更高）
- 驾驶员的稀缺能力是"品味"——判断什么是好的、什么是对的、什么是值得做的能力

**与其它隐喻的对应关系**：

| 隐喻 | 出处 | 对应映射 |
|------|------|---------|
| 发动机—线束—驾驶员 | 17（汤道生） | 模型—Harness—人 |
| 大脑—手 | 30（Anthropic） | Claude+Harness 调度 vs Sandbox+工具执行 |
| CPU—操作系统 | 05 | 模型—Harness |
| 马—马具—骑手 | 06、17 | 力量—驾驭系统—决策者 |
| 福特流水线 | 17 | Harness 是把贵族玩具变成大众工具的整套系统 |

**适用场景**：向非技术决策者解释 Harness 价值、设计产品时定义"AI 能做到哪、人必须出现在哪"的边界、做组织能力规划时区分"靠模型自动化"与"靠人的品味"的工种。

### 概念 5：30 年软件工程驾驭复杂性的延续

**核心论点（26）**：软件工程的本质是**驾驭复杂性**，核心手段是**抽象 + 结构化**。复杂性的中心每隔数年就发生一次转移，每次转移都催生新的工程范式。Harness Engineering 是这一长期演进的最新一幕。

**演进时间表**：

| 年代 | 代表作 | 驾驭的对象 | 核心问题 |
|------|-------|----------|---------|
| 1994 | GOF《Design Patterns》 | 对象 | 类的生命周期、对象职责与协作 |
| 2002 | Fowler《企业应用架构模式》+ Evans《DDD》 | 企业架构 | 系统分层、业务边界、领域模型 |
| 2010 | 微服务 | 分布式通信 | 几十个微服务间的消息传递、一致性 |
| 2017 | Kleppmann《DDIA》 | 数据系统 | 数据分区、复制、时空维度流动 |
| **2026** | **Harness Engineering** | **智能体（不确定性系统）** | **可控性、自主性与人类监督的平衡** |

**Agent 时代的独特之处**（26 的关键洞察）：
> Agent 是历史上第一次需要驾驭**概率机器**——你给它输入，不一定按你的预期输出。

这意味着：
- 过去的工程经验（DAG、状态机、确定性流程）只能部分迁移
- 必须引入新原语：自我修正、外部验证、约束与回退、熵控
- "工程"这个词的含义被改写：不只是写代码，是设计可控的不确定性系统

### 概念 6：AI 工程五幕演化（治理时代）

**核心论点（22）**：从 ChatGPT 发布到 2026，AI 工程经历了五幕。Harness Engineering 是"治理时代"的工程产物。

| 时间 | 阶段 | 核心矛盾 | 工程产物 | 教训 |
|------|------|----------|----------|------|
| 2022.11-2023 | 生成 | 能说不能做 | Prompt Engineering | 信息密度最大化 |
| 2023-2024 | 连接 | 能连不能治 | Function Calling / LangChain | 连接 ≠ 编排，编排 ≠ 治理 |
| 2024 | 推理 | 单步更强但长任务不稳 | MCP / Context Engineering | 单步质量 ≠ 任务可靠性 |
| 2025 | 行动 | Agent 能跑但频繁崩坏 | Claude Code / Codex / Cursor | 能力超过基础设施，产生断裂 |
| 2026- | 治理 | 从"能干"到"不翻车" | Harness Engineering | 系统设计成为核心竞争力 |

**关键判断**（与"三层工程演进"的关系）：
- 三层演进是**工程认知**的跃迁
- 五幕演化是**矛盾驱动**的视角
- 两者从不同维度刻画同一进程：每一幕的"教训"都是下一幕的"前提"

### 概念 7：术语进入主流的时间线

**Harness Engineering 一词的传播节点**（22）：

- **2026.02.05** — Mitchell Hashimoto 在《My AI Adoption Journey》中写下 "Engineer the Harness"，被认为是术语进入主流的起点。他的实践要点："每当我发现 Agent 犯了一个错误，我就花时间工程化一个解决方案，让它永远不再犯同样的错误。"
- **2026.02.11** — OpenAI 发布工程文章《Harness Engineering: Leveraging Codex in an Agent-First World》，把方法论与百万行代码实战结合。
- **2026.03** — Anthropic 发布《Harness Design for Long-Running Application Development》，把二角色（Generator/Evaluator）升级为三角色（Planner/Generator/Evaluator）架构。
- **2026.04** — Thoughtworks / Martin Fowler 体系把方法论抽象为 **2×2 控制矩阵**（Guides × Sensors × Computational × Inferential），方法论完整成形。

**适用场景**：写综述、做技术汇报、向团队解释"为什么 2026 突然全行业都在谈 Harness"时使用这条时间线。

## 方法论与框架

> 本方向不展开"如何落地"的方法论（→ 工程实践方向），而是把业界主流的**Harness 分层模型**汇总到一张可比较的统一视图里。

### 框架 1：多版本 Harness 分层模型对照表

业界至少有 5 套主流分层方案，它们使用不同的命名和粒度，但底层关心同一组问题。以下用一张对照表统一视图：

| 关心的问题 | 06 六层架构 | 05 六层架构 | 18 三层结构 | 22 六大构件 | 26 六大组件 | 30 Meta-Harness | 07 七大原语 |
|----------|------------|------------|------------|------------|------------|----------------|------------|
| **该知道什么** | Context 上下文层 | L1 信息边界层 | 知识层 | Legibility 感知面 | Memory & Context | Session 事实流 | 记忆与搜索 |
| **能干什么** | Tools 工具层 | L2 工具系统层 | — | Tool Mediation | Tool System | — | 文件系统 + Bash + 沙箱 |
| **怎么一步步推** | Loop 循环层 | L3 执行编排层 | 约束与流程层 | Decomposition & Plans | Agentic Loop | Harness 调度脑干 | 长期自主执行 + 编排 |
| **状态怎么留** | Persistence 持久化层 | L4 记忆与状态层 | — | Durable State Surfaces | Session | Session 事实流 | 文件系统 |
| **怎么判断做对** | Verification 验证层 | L5 评估与观测层 | 反馈与运行时层 | Feedback Loops（2×2） | — | — | — |
| **怎么不出错 / 不越权** | Constraints 约束层 | L6 约束、校验与恢复层 | 约束与流程层 | Entropy Control | Guardrails + Hooks | Sandbox + Vault | 沙箱 |

**关键观察**：
1. **不同模型粒度不同**：18 的三层（知识/约束流程/反馈运行时）是最粗的；07 的七大原语是最细的（按依赖关系自底向上）。
2. **不同模型侧重不同**：22 强调**对抗熵增**（Entropy Control 独立成构件）、30 强调**架构稳定性**（把"调度"剥离成无状态层）、26 强调**事件拦截**（Hooks 独立成组件）。
3. **共性骨架**：每个模型都包含「信息边界 + 工具 + 编排 + 状态 + 验证 + 约束」六个面向，差异在分组和命名。
4. **演进趋势**：06 的"六层"和 18 的"三层"侧重业务层次；22 / 30 把"持久状态"和"事件流"提升为一等公民，标志着 Harness 设计从**业务层抽象** → **基础设施层抽象** 的转变。

### 框架 2：Harness 六层架构（06 + 05 融合版）

把 06 提出的六层（Loop/Tools/Context/Persistence/Verification/Constraints）与 05 的六层（L1-L6）融合后，得到一张更完整的层级表：

| 层级 | 名称 | 解决什么问题 | 关键设计 |
|------|------|------------|---------|
| L1 | **信息边界层** | Agent 该知道/不该知道什么 | 定义角色与目标，裁剪无关信息，结构化任务状态 |
| L2 | **工具系统层** | Agent 怎么跟外部世界交互 | 工具选拔、调用时机、结果提炼与反馈；少即是多 |
| L3 | **执行编排层** | 多步骤任务怎么串起来 | Loop（Observe → Decide → Act → Verify → Repeat）走完完整轨道 |
| L4 | **记忆与状态层** | 长任务中间结果怎么管 | 独立管理任务状态、中间产物和长期记忆 |
| L5 | **评估与观测层** | Agent 怎么知道自己做对了没有 | 建立独立于生成过程的验证机制 |
| L6 | **约束、校验与恢复层** | 出错了怎么办 | 预设规则拦截错误，失败时提供重试或回滚 |

**优先级建议（05）**：
- 一开始不要搭齐六层
- 投入产出比最高的是 **L1（信息边界）和 L6（约束与恢复）**
- L1 决定 Agent 知道该干什么，L6 决定搞砸了能不能拉回来

### 框架 3：七大技术原语（07，按依赖关系自底向上）

把 06 / 05 的"业务层级"换成"实现原语"，得到 07 的版本。它的价值是回答**"具体要做哪些东西"**：

| 原语 | 解决的问题 | 核心机制 |
|------|-----------|----------|
| 文件系统 | 持久存储、信息卸载、跨会话状态 | 文件读写 + Git 版本控制 |
| Bash + 代码执行 | 避免工具集上限卡死能力 | 让模型自己写代码解决问题 |
| 沙箱 | 安全执行 + 横向扩展 | 隔离环境，按需创建/销毁 |
| 记忆与搜索 | 跨会话知识 + 训练截止后信息 | AGENTS.md 记忆文件 + Web 搜索 + MCP |
| 上下文管理 | 对抗上下文衰减 | 压缩机制 + 工具输出卸载 + 技能渐进式披露 |
| 长期自主执行 | 过早停止、跨窗口不一致 | Ralph 循环 + 规划文件 + 自我验证 |
| 编排逻辑 | 复杂任务分解与并行 | 子代理生成、任务交接、模型路由 |

**关键论断（07）**：**"文件系统不只是存储，它实际上是后面所有能力的地基。"** 跨会话持久化、多智能体协作、Git 版本控制全部建立在文件系统之上。

---

### 框架 4：三层结构（18 的最简化视图）

把架构压缩到极简的三层视图，适合向非技术受众讲清楚 Harness 在管什么：

| 层 | 解决的问题 | 案例 |
|----|----------|------|
| **知识层** | 模型该读什么、团队经验放哪里、哪些规则必须显式可见 | AGENTS.md 地图、docs/ 结构化、doc-gardening 智能体 |
| **约束与流程层** | 任务怎么拆、哪些步骤必须先发生、哪个角色负责什么 | Planner/Generator/Evaluator 三角色、机械规则约束、Context Reset |
| **反馈与运行时层** | 模型做完之后，谁来告诉它对不对 | Playwright 跑页面、CDP 截图、git worktree 隔离、可观测性堆栈 |

**核心判断（18）**：**对 Agent 来说，看不见的知识就等于不存在**——Slack 讨论、Google Docs 共识、资深工程师脑中的隐性判断，没编码进仓库就像从未发生。

---

### 框架 5：六大工程构件（22 的工程师视角）

22 在 06/05 的层级模型上做了一次"工程化重写"，把每个层级变成可独立讨论、可独立改进的工程构件：

| 构件 | 关心的问题 | 核心实践 |
|------|----------|---------|
| **Durable State Surfaces** | 状态如何跨 session 续航 | 冷启动 30 秒内能读取的结构化工件 |
| **Decomposition & Plans** | 计划是一次性想法还是一等工件 | PLANS.md、init.sh、200+ feature 列表 |
| **Feedback Loops** | 反馈如何同时管约束（行动前）和反馈（行动后） | Guides × Sensors 2×2 矩阵 |
| **Legibility** | 知识/状态对 agent 是否"可见" | 凡是不在运行时可见就等于不存在 |
| **Tool Mediation** | 工具如何被中介化、不污染上下文 | 让模型写代码调用工具，而非直接调用 |
| **Entropy Control** | 系统噪声如何抑制 | documentation consistency agent、refactor agent、CI 强制 |

**重要洞察（22）**：Harness 不只是"让 agent 跑起来"，还**持续抑制 agent 放大的系统噪声**——这是它与简单框架最本质的区别。

---

### 框架 6：Meta-Harness 三层（30 的稳定抽象层）

Anthropic 在 Claude Managed Agents 中给出的极简抽象，强调**不固化补丁、只承诺接口**：

```
应用层：Agent 业务逻辑
    ↓
Session（执行事实流） ← append-only 不可变事件日志
Harness（调度脑干）   ← 无状态、可热替换、标准接口
Sandbox（可插拔执行） ← 大脑与手解耦，独立、可共享、懒加载
    ↓
基础设施层：工具 / MCP 服务 / Vault
```

**核心承诺**：
- 不承诺 harness 具体长什么样
- 只承诺几类长期稳定的接口
- Session 不是 context window（是事实流，不是消息列表）
- 安全不依赖"模型不作恶"假设，依赖系统级凭证隔离

**与其它模型的差异**：30 是**唯一把"调度逻辑"无状态化**的方案，也是**唯一把模型暂时缺陷与系统架构显式分离**的方案——前提是相信模型会持续进化，今天的最优实现会变成明天的技术债。

---

### 框架 7：六大组件（26 的工程师常用视图）

26 给出的视图最接近今天 Claude Code 这类产品的实际结构，是开发者最容易上手的版本：

| 组件 | 功能 | 类比 |
|------|------|------|
| **Agentic Loop** | 接受输入 → 工具执行循环 → 返回结果（ReAct 论文） | 心脏 |
| **Tool System** | 调用外部工具和 API | 手脚 |
| **Memory & Context Management** | 上下文压缩、记忆管理 | 记忆 |
| **Guardrails** | Allow / Deny / Ask 权限控制 | 刹车 |
| **Hooks** | 事件前后拦截（如防止上传 .env） | 哨兵 |
| **Session** | 会话连续性与运行时机制 | 骨骼 |

**核心隐喻（26）**：**Harness 做的是把大模型的大脑变成 Agent 的身体。**

## 案例库

> 本方向只收录用于**论证"概念定义"或"架构理论"**的案例。涉及具体落地步骤、四大杠杆、AGENTS.md 操作要点的案例归入「工程实践」方向。

### 案例 1：Can.ac 编辑接口实验——同一模型 10 倍差距

- **背景**：测试 Harness 对同一模型性能影响的对照实验
- **做法**：固定模型不变，只改变文件编辑接口的调用方式
- **结果**：编码基准分数从 **6.7% → 68.3%**（约 10 倍）
- **启发**：模型表现的瓶颈是基础设施质量，不是模型智能水平——这是 Harness 概念被广泛接受的"标志性数据"

> ⚠️ 待验证：原文标注实验数据来自第三方，具体实验条件（基准测试名称、模型型号、接口变化细节）未详述，数值合理但无法独立核实。

### 案例 2：LangChain Terminal Bench 2.0——只优化运行环境，排名从 30 跳到 5

- **背景**：保持底层模型不变的 Harness 优化对照实验
- **做法**：只优化运行环境（文档组织 + 验证回路 + 追踪系统）
- **结果**：deepagents-cli 得分 **52.8% → 66.5%**（+13.7 分），排名 **Top 30 外 → Top 5**
- **启发**：当 GPT/Claude/Gemini 在核心能力趋同时，决定产品成败的是 Harness 质量

### 案例 3：Nate B Jones 实验——同一模型，编程成功率从 42% 跳到 78%

- **背景**：广泛被引用的 Harness 对比实验（17 出处）
- **做法**：同一个模型，只换 Harness
- **结果**：编程成功率 **42% → 78%**（提升 86%）
- **启发**：模型能力天花板**不在模型里面，而在模型外面**

### 案例 4：Anthropic 游戏开发实验——同一 prompt，成本 22 倍但产出从无效到可用

- **背景**：同一个模型、同一句提示词，用不同方式驾驭
- **简单方式**：20 分钟、9 美元、核心功能完全无效
- **完整 Harness**：6 小时、200 美元、交付真正可用游戏
- **启发**：变量只有一个——壳（Harness），成本差异 22 倍但产出质量从"完全无效"到"真正可用"

### 案例 5：OpenClaw——第一只"爬上岸"的龙虾

- **背景**：2026 年春天 OpenClaw 搅动整个 AI 行业
- **做法**：没有发布任何新模型，没有刷新任何基准，没训练一个新参数；只给大模型搭建了一套完整工作环境（文件系统 + 代码沙箱 + 工具链 + 反馈循环 + 自动验收）
- **结果**：同一个模型从只会对话的聊天机器人变成能持续工作、自主解决问题的智能体
- **启发**：这是 Harness 作为独立产品形态的标志性案例——能力提升完全来自外部脚手架

### 案例 6：Claude Code vs Claude Web——同模型的两种壳

- **背景**：同一模型在不同 Harness 下的体验对比（06）
- **Claude Web**：一问一答模式，无文件系统访问、无执行能力、无自动验证 → 反复修改仍报错
- **Claude Code**：读取项目结构 → 执行修改 → 跑测试 → 自动继续修复直到通过 → 一次闭环
- **启发**：这不是模型差异，是系统差异；用户体验到的"AI 强不强"很大程度是 Harness 质量

### 案例 7：Vercel——工具从 15 砍到 2，准确率 80% → 100%

- **背景**：少即是多原则的对照实验（06）
- **做法**：可用工具从 15 个砍到 2 个
- **结果**：准确率 80% → 100%，Token 消耗减少 37%，速度提升 3.5 倍
- **启发**：工具过多增加选择噪声，约束到恰好够用反而更好——这是 Harness 设计中"边界比堆砌更重要"的经典数据

### 案例 8：Codex-5.3 泛化问题——换补丁方式就掉分

- **背景**：模型与 Harness 协同演化的副作用（07）
- **现象**：换一种补丁应用方式，模型性能直接下滑
- **启发**：模型过度拟合训练时用的 Harness，"最适合你任务的 Harness ≠ 模型后训练时用的 Harness"——选型时不能假设"用官方框架就是最优"

## 关键洞察

> 跨多源融合后提炼的、超越任一单篇来源的发现。每条洞察标注主要支撑源以便追溯。

1. **"模型决定上限 vs Harness 决定上限"的表述张力**
   - 05 说"模型决定上限，Harness 决定底线"
   - 06 说"AI 的上限由模型决定，但下限由 Harness 决定"
   - 17 说"模型能力的天花板不在模型里面，而在模型外面"
   - 22 说"护城河重心正在上移到 harness 与系统设计"
   - **冲突分析**：这不是矛盾，而是不同视角下的同一现象——
     - 06 / 05 是"理论上限"视角：模型决定一个任务**可达到**的最高分
     - 17 / 22 是"实际可达上限"视角：在当前模型能力区间，可观察到的性能差异主要来自 Harness
   - **统一表述**：在模型边际提升放缓的窗口期，Harness 的工程努力比等下一代模型的回报率更高；但模型一旦跨过新的能力阈值，会把今天的 Harness 复杂度部分"吸收"掉。

2. **Harness 分层模型的"骨架同构"现象**
   - 7 套主流分层方案虽然命名各异，但深层都包含「信息边界 + 工具 + 编排 + 状态 + 验证 + 约束」六个面向
   - 差异不在面向是否存在，而在**粒度**（三层 vs 七层）和**主张**（哪个面向应该独立成构件）
   - 实用启示：选型不在于争论"哪套分层最对"，而是按你的团队成熟度选粒度——早期用 18 的三层做沟通工具，工程化阶段用 06 的六层做架构清单，规模化阶段用 22 的六大构件做改进焦点，平台化阶段用 30 的 Meta-Harness 做接口设计

3. **从"业务层抽象"到"基础设施层抽象"的范式转变**
   - 早期分层（06、05、18）侧重**业务问题分类**：循环、工具、记忆、验证
   - 后期分层（22、30）侧重**基础设施稳定性**：把"调度"剥离成无状态层、把"状态"从消息列表升级为事实流
   - 这一转变与软件工程史上 OSI 七层 → TCP/IP 四层、Monolith → 微服务的转变同构
   - 实用启示：你的 Harness 是否在做"补丁固化"——把"模型当前能力缺陷"硬编码成永久架构？

4. **隐喻的演进暗示了 Harness 在系统中地位的提升**
   - 第一代隐喻（马具、缰绳）—— Harness 是约束物
   - 第二代隐喻（线束、电缆）—— Harness 是连接物
   - 第三代隐喻（操作系统、AgentOS 用户态层）—— Harness 是基础设施
   - 第四代隐喻（大脑—手）—— Harness 是身体本身
   - 实用启示：你的产品文档/团队叙事里 Harness 的位置在哪一代？这暗示了你的成熟度

5. **三层工程演进的"非线性时间窗"**
   - 从行业整体看：Prompt（2022-）→ Context（2024-）→ Harness（2026-）
   - 从个体团队看：任何团队都可能同时处于三个层次——核心任务在 Harness 层，新需求探索还在 Prompt 层
   - 三层并非"过时—新潮"的替代关系，而是"针对不同问题复杂度的工具集"
   - 实用启示：评估团队成熟度不是"你做到第几层"，而是"对每类问题你都能选对工具吗"

6. **Harness vs AgentOS 的层级关系**（22 提出）
   - Harness 是 AgentOS 的**用户态层**（shell/daemon）
   - AgentOS 是**内核**（调度、隔离、资源管理）
   - 两者是上下层关系，不是竞争关系
   - 实用启示：当一家公司宣称"我们做 AgentOS"时，要追问它是替代 Harness 还是承载 Harness——这决定了选型逻辑

7. **"Harness 是长出来的，不是设计出来的"**（18）
   - 没有团队从一张完美架构图开始
   - 都是从第一个犯错开始补，从第一条规则开始写
   - Mitchell Hashimoto：每当我发现 Agent 犯了一个错误，我就花时间工程化一个解决方案，让它永远不再犯同样的错误
   - 实用启示：不要追求"完整六层 Harness 一次到位"，要建立"错误 → 新规则"的反馈循环

## 待探索问题

> 这个方向上还没想清楚、有待进一步研究的开放性问题。是文档继续演进的钩子。

- **统一术语缺失**：Harness 的"层"、"原语"、"构件"、"组件"指代相似但定义不同的事物，业界缺乏统一术语。下一版可尝试给出一个调和后的术语表。
- **分层模型的可量化对照**：现有 7 套分层方案是定性比较，能否给出量化指标（如"Harness 完成度评分"）让团队自测？
- **Harness vs Framework 的边界**：LangChain 的 LangGraph、LlamaIndex 的 Workflows 是否在变成 Harness？是 Harness 的实现还是 Harness 的子集？
- **棕地项目的 Harness 形态**：现有所有架构方案默认是绿地项目；存量代码库的 Harness 该怎么分层？哪些层不能照搬？（→ 与工程实践方向关联）
- **隐喻的天花板**：当模型把工具调用、上下文管理、反馈循环全部内化后，"线束""操作系统"这些隐喻是否会失效？需要新的隐喻吗？（→ 与演化方向关联）
- **三层工程演进之后**：如果 Harness 是 2026 的范式，2028 的范式会是什么？社会化协作工程？多 Agent 治理？
