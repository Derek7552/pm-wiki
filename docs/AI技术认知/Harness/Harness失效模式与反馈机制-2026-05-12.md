# Harness 失效模式与反馈机制

> 方向定位：聚焦 Agent 在 Harness 中"为什么会出错、怎么发现错了、怎么把反馈做成系统"。涵盖 Agent 五大根本挑战、典型失效模式（Context Anxiety、自我评估偏差、上下文 40%/60% 阈值、熵增、无限循环）、反馈机制设计（GAN 式三角色、2×2 反馈矩阵、Evaluator 独立性原则、Hooks 硬约束、In-the-loop vs On-the-loop）。
> 不涵盖：Harness 概念定义（→ 概念与架构体系）；落地杠杆（→ 工程实践）；模型内化趋势（→ 演化与未来形态）。

## 方向定位

Harness 失效模式与反馈机制研究两个互补的问题：

1. **失效模式（Anti-patterns）**：Agent 在长任务中**为什么会出错**——状态丢失、目标漂移、自评虚高、上下文焦虑、熵增、过早停止……这些故障的根因往往不在模型，而在 Harness 设计假设的失效。
2. **反馈机制（Feedback Mechanisms）**：发现"做错了"和保证"做对了"是两件不同的工程任务——本方向汇总业界主流的反馈系统设计（独立 Evaluator、2×2 矩阵、Hooks、Sprint Contract、In-the-loop vs On-the-loop 升级路径）。

**核心命题**：**仅靠模型自身无法形成有效的质量闭环，必须在模型外部建立独立的评估机制。**

**不在本方向范围**：基础概念定义（→ 概念与架构体系）、四大工程杠杆 / AGENTS.md 设计要点（→ 工程实践）、Harness 减重 / 模型内化（→ 演化与未来形态）。

## 核心概念

### 概念 1：Agent 五大根本挑战

**定义（22 的精确表述）**：长任务的可靠性不会因为模型单步更聪明就自动获得。五个挑战每一个都**不能靠更聪明的模型单独解决**：

| 挑战 | 本质 | 为什么模型解决不了 |
|------|------|------------------|
| **状态持久性** | 跨时间、跨 session 记住做过什么 | 模型无状态，context window 有上限 |
| **目标一致性** | 长任务中防止漂移、提前宣布完成 | 缺少外部锚点，无法稳定校准"真正完成" |
| **行动可验证性** | 区分"做了"和"做对了" | 自我评价天然存在自我表扬和误判倾向 |
| **熵增抑制** | 持续产出累积冗余、漂移、不一致 | 模型复制已有模式，包括坏模式 |
| **人机边界** | 何时自主、何时交人 | 没有可靠的"不确定性自觉" |

**为什么这五个挑战是"根本"**：
- 不是模型能力提升能消除的
- 不是接一个工具就能解决的
- 都需要在模型**外部**建立机制
- → 这恰好是 Harness 存在的根本理由

**与失效模式的对应**：
- 状态持久性 ↔ Context Anxiety / 跨 session 失忆
- 目标一致性 ↔ 过早停止 / 目标漂移
- 行动可验证性 ↔ 自我评估偏差
- 熵增抑制 ↔ AI slop / 坏模式复制
- 人机边界 ↔ Agent 在哪个节点该交回给人

**适用场景**：当团队讨论"是不是再换一个更强的模型就行"时，回到这五个挑战说明为什么不行。

### 概念 2：Context Anxiety（上下文焦虑）

**定义（12、18、22）**：

> Agent 感觉 Context Window 快满了 → 提前收尾任务（不是因为完成了，而是"焦虑"了）。

**典型表现**：
- 任务未完成时主动宣布"已完成"
- 优先输出"看起来完整"的内容，而非真正解决问题
- 在剩余 token 不多时倾向于做总结而非继续推进
- Anthropic 发现 Sonnet 4.5 出现此行为，Opus 4.5 已基本消除

**两种应对策略**：

| 应对策略 | 机制 | 权衡 |
|---------|------|------|
| **Compaction（压缩）** | 摘要历史对话，同一 Agent 继续 | 保住连续性，但**无法消除焦虑感** |
| **Context Reset（重置）** | 清空上下文，启动新实例 + handoff artifact | 消除焦虑，但**交接文件必须完整** |

**本质（12）**：在**连续性 vs 清醒度**之间选边。

**为什么 Compaction 不够（22 的解释）**：
- 即使压缩后 token 数下降，agent 仍因"上下文太满"的感觉而行为退化
- 解决方案不是更好的 compaction，而是**给新 agent 全新 context + 外化状态工件续航**
- 类似"重启进程解决内存泄漏"，而非无限加内存

**与 Durable State Surfaces 的关联**（22）：
- Context Reset 只在 Durable State 设计良好时才有效
- 关键判据：**agent 冷启动后能否在 30 秒内读取、理解、续航？**
- 状态 ≠ "保存聊天记录"；真正的 durable state 是结构化工件
- → 详细机理见「概念与架构体系」方向的 Durable State Surfaces 章节

**Harness 演化角度的观察**（12）：
- Sprint 分解：Opus 4.5 必要 → Opus 4.6 可删
- Context Reset：早期模型必要 → Opus 4.5+ 可删
- Context Slice 精细管理：短窗口模型必要 → 长上下文模型可粗放
- → 这是"Harness 组件随模型迭代减重"的最直接证据

**适用场景**：当 agent 在长任务中频繁"看似完成实际未完成"时，先检查是否是 Context Anxiety；判断当前模型是否已跨过此问题的阈值。

### 概念 3：自我评估偏差

**核心论断（多源一致）**：
- 17（Anthropic）：当 Agent 评估自己刚完成的工作时，它会自信地表示"做得很好"，即便在人类看来质量明显不行
- 18：开箱即用的 Claude 是一个很差的 QA Agent
- 12：Agent 评价自己时倾向虚高分——先发现问题，然后说服自己"没那么严重"

**具体行为模式**：
- 先发现真正的问题
- 然后**说服自己"不是大事"**
- 最后通过验收
- 这不是"看不见问题"，而是"看见了但放过自己"

**为什么模型自评不可靠**：
- 训练目标导致正向偏好（不愿给自己打低分）
- 缺少独立验证锚点
- 在长 context 中，Generator 和 Evaluator 角色混合导致"自圆其说"

**唯一可靠的解法：GAN 思路**（12、17、18）：

> Generator 和 Evaluator 必须彻底拆开。

具体实现：
- Evaluator 独立调教为"天生挑剔的审查者"
- 配备 Playwright 等外部工具，像真实用户一样操作测试
- 不是"读代码打分"，而是"真正跑页面、点按钮、看行为"
- Sprint Contract：在写代码前先就"完成的定义"达成共识

**关键洞察（22）**：
> 调教一个独立的 Evaluator 让它保持怀疑，远比让 Generator 批判自己的作品容易得多。

**Evaluator 的演化（22）**：
- 当底模能力跨过阈值后，Evaluator 从"必要部件"退化为"额外开销"
- 但这个边界会随模型移动，**边界本身不会消失**

**适用场景**：任何依赖 AI 自评的质检流程都需要警惕——产品质量保障需要独立的评估机制，而非开发自测。

### 概念 4：上下文利用率阈值（40% / 60%）

**两个关键阈值**（多源比较）：

| 来源 | 阈值 | 表述 | 工程含义 |
|------|------|------|---------|
| 05（Dex Horthy） | 40% | Smart Zone（0-40%）vs Dumb Zone（>40%）；168K token 窗口超过约 40% 后幻觉增多、代码质量下降 | 触发交接/总结 |
| 16（外部研究引用） | 60% | 上下文利用率达到约 60% 后继续塞信息反而让 agent 表现变差 | 上限警戒 |

**冲突分析**：
- 40% 是"质量开始下降"的早期信号
- 60% 是"明显变差"的临界点
- 不矛盾，是同一现象的两个阶段
- ⚠️ 两个阈值都标注为"观察"或"研究引用"，不是硬性规律，模型版本不同会有差异

**Context Rot（上下文腐烂）**（16）：

随着上下文膨胀，模型会出现：
- recall 下降
- 推理偏移
- 忽略约束

**正确做法**（与"塞更多信息"相反）：

> 让 Agent 始终运行在"干净、相关"的上下文里——不是压缩塞更多，而是让它在任何时候都运行在干净相关的上下文里。

**对抗上下文衰减的三种策略**（07）：
- **压缩机制**：上下文快满时智能卸载并汇总
- **工具调用卸载**：大型工具输出只保留头尾，完整内容卸载到文件系统按需取用
- **技能渐进式披露**：启动时只加载元数据，用到什么再展开什么

**与 Context Anxiety 的关系**：
- 40-60% 阈值是**外部观察**的质量下降点
- Context Anxiety 是 agent **内部主观感受**导致的行为退化
- 即使物理上没满，主观焦虑也会触发提前收尾
- 两者一起说明：**单纯的"压缩"既解决不了客观质量下降，也解决不了主观焦虑**

**适用场景**：监控 agent 长任务时设置 40% 触发点（主动总结/交接）和 60% 警戒点（强制 Context Reset）。

### 概念 5：熵增（Entropy Increase）

**定义（22）**：

> 全自动 agent 代码库不断复制既有模式，**包括糟糕的模式**。系统噪声持续累积——AI slop（冗余代码）、过时文档、不一致命名、技术债等。

**为什么会发生**：
- LLM 倾向于"重新实现已有功能"而不是复用
- 没有架构约束时，每个 agent 各自决策，全局一致性丢失
- 没有定期清理时，垃圾积累速度 > 维护速度

**典型代价（22）**：
- OpenAI 初期：人每周花约 **20%** 时间清理"AI slop"

**系统化解法（22）**：
- **documentation consistency agents** 定期验证文档一致性
- **refactor agents** 计划清理技术债
- **architectural enforcement** 通过 CI 机械维护模块边界

**设计原则（22 关键洞察）**：

> Harness 不只负责"让 agent 跑起来"，还负责**持续抑制 agent 放大的系统噪声**——这是它与简单框架最本质的区别。

**与"AI slop"清理速度的关系**（05）：
- 关键指标：**清理速度 ≥ 生成速度**
- 一旦反过来，系统会快速崩坏
- 这就是为什么 P2 优先级里"专门的后台垃圾回收 Agent"是必需的

**适用场景**：评估长跑 agent 项目的可持续性时，专门监控 AI slop 累积速度——一个看起来跑得很顺利的项目，可能正在积累不可见的熵增债务。

### 概念 6：In-the-loop vs On-the-loop

**定义（22 的核心升级概念）**：

| 模式 | 工程师做什么 | 心智模型 |
|------|------------|---------|
| **In-the-loop** | 手改 agent 产出的具体产物（bug、错误代码） | "AI 写错了，我来改" |
| **On-the-loop** | 改 harness 让系统**下次自动**做得更好 | "AI 又写错了，规则/工具/反馈哪里需要改？" |

**论断**：大多数团队还停留在 In-the-loop。

**为什么 On-the-loop 是核心升级**：
- In-the-loop 是线性投入：错一次修一次，工程师投入与错误数线性增长
- On-the-loop 是平台化投入：每次改 harness 都减少未来某类错误
- 复利效应：On-the-loop 长期回报是 In-the-loop 的 N 倍

**类比 Mitchell Hashimoto 的工作流（12）**：
> 每当我发现 Agent 犯了一个错误，我就花时间工程化一个解决方案，让它永远不再犯同样的错误。我叫这个过程 Harness Engineering。

这恰好是 On-the-loop 的具体实施。

**关键判据（22）**：
> 不满意 agent 输出时，低层做法是手改产物；高层做法是改 harness，让系统下次自动做得更好。**从 in the loop 到 on the loop，这才是工程师在 agent 时代的核心升级路径。**

**与四大杠杆的对应**：
- 改 AGENTS.md → 把一次 onboarding 错误变成永久规则
- 加 Linter 规则 → 把一类代码错误变成机械检测
- 接 MCP → 把"agent 不知道"变成"agent 能查到"
- 加 Hooks → 把"agent 偶尔违反"变成"agent 不可能违反"

**适用场景**：每次手动改 agent 产出时，自问"这次错误能否系统化避免下一次？" 如果答案是"是"，立即从 In-the-loop 切到 On-the-loop。

## 方法论与框架

### 框架 1：Feedback Loops 2×2 矩阵（22 的方法论核心）

**核心思想**：反馈系统在两个维度上各有两种性质——**作用时机**（行动前 vs 行动后）和**反馈性质**（确定性 vs 推断性）。Harness 必须在四个象限同时投入。

| | **Computational（确定性，便宜快）** | **Inferential（推断性，贵慢）** |
|--|--|--|
| **Guides（行动前约束）** | 代码规范检查、类型校验 | AI 评估代码设计合理性 |
| **Sensors（行动后反馈）** | 测试套件、linter 结果 | AI 评估 UI 美观度、代码风格 |

**关键洞察（22）**：

- **只有 guides 无 sensors** → 规则生效与否无从知晓（约束悬空）
- **只有 sensors 无 guides** → 不断重复同样错误再纠正（白白消耗 token）
- **只有 computational 无 inferential** → 错过设计质量、用户体验等不可机械化检测的问题
- **只有 inferential 无 computational** → 浪费 AI 调用做本该机械化的事

**Evaluator 在矩阵中的位置**：
- 主要在右下（Sensors × Inferential）象限
- 部分在右上（Guides × Inferential，如 Sprint Contract 中的 27 条验收标准）

**与四大杠杆的映射**：
- Custom Rules → 左上（Guides × Computational，规则文本）
- Linter / Type check → 左上（最硬的 Guides）
- Tests → 左下（Sensors × Computational）
- Code Review Agent → 右下（Sensors × Inferential）
- Sprint Contract → 右上（Guides × Inferential）

**注意事项**：
- 矩阵的价值不在分类，在**强迫团队检查四个象限是否都有投入**
- 大多数团队只在 Sensors × Computational 一格投入（写测试），其他三格空白
- 单独投入 Guides 无 Sensors 是"装样子"——规则没人验证就等于没有

### 框架 2：GAN 式三角色架构

**核心思想（12、18）**：把 Generator 和 Evaluator 彻底拆开，引入 Planner 作为高层目标守门人。

**三角色职责**：

| 角色 | 职责 | 失败模式抵抗 |
|------|------|------------|
| **Planner** | 把一两句描述展开成完整 product spec 和分步 feature list | 抵抗目标漂移、错误级联 |
| **Generator** | 逐 feature 落地，每完成一个 commit | 抵抗局部决策导致的全局不一致 |
| **Evaluator** | 独立验收，用 Playwright 真正跑页面 | 抵抗自我评估虚高 |

**Sprint Contract 机制**（12）：
- 单 Sprint 包含**27 条验收标准**
- 在写代码前先就"完成的定义"达成共识
- 写代码时只看 Sprint Contract，不看完整 product spec（避免 context 过载）

**为什么 Planner 不写代码、Evaluator 独立运行**（22）：
- Planner 只聚焦产品上下文和高层设计 → **不规定技术细节**（避免错误级联）
- Evaluator 独立调教为挑剔审查者 → 不被 Generator 的"自我说服"影响
- **设计原则**：计划必须是一等工件——写入文件系统、被版本管理、可被后续 agent 读取、被验证门引用

**成本与价值（12）**：
- 三 Agent 是 Solo 的 **20 倍**成本
- 但从"不能用"跨越到"能用"

**Evaluator 调教难点（18）**：
- 开箱即用的 Claude 是很差的 QA Agent
- 会自己说服自己"不是大事"然后通过验收
- 花了好几轮迭代才把评估质量调到合理水平

**适用边界**：
- 不是所有任务都值得 20x 成本
- 长任务、高容错要求场景 → 必要
- 小任务、快速原型 → 浪费

### 框架 3：Hooks 把软约束升级为硬约束

**核心思想（16）**：

> Agent hooks 把验证逻辑挂到 Stop hook，agent 完成任务前必须通过这些检查。

**软约束 vs 硬约束**：

| 类型 | 实现方式 | Agent 行为 |
|------|---------|-----------|
| **软约束** | 在 system prompt 里写规则 | 可能遵守也可能不 |
| **半硬约束** | Linter 规则、Type check | Agent 必须修复才能通过 |
| **硬约束** | Stop Hook、PR 拦截 | Agent **无法绕过** |

**关键变化**：
- 软约束："建议你去跑一下测试"
- 硬约束："必须先过门槛"

**与 26 Hooks 组件的呼应**：
- 26 把 Hooks 列为 Harness 六大组件之一
- 26 给的隐喻：哨兵（事件前后拦截执行，如防止上传 .env 文件）
- 16 给的应用：把质量门变成不可绕过的检查

**典型用法**：
- Stop Hook：完成任务前强制跑测试 / lint / type check
- Pre-commit Hook：禁止提交特定文件、敏感信息
- Pre-tool Hook：执行前确认权限边界

**与 MercadoLibre Code Review Agent 的对比**：
- Code Review Agent 是事后审查
- Stop Hook 是事前/事中阻断
- 两者互补，不冲突

---

### 框架 4：五大落地难题应对（26）

**核心思想**：Harness 必须能解决五个具体的工程难题，这是它的"作业清单"。

| 难题 | Harness 解法 | 对应组件 |
|------|------------|---------|
| **无限循环问题** | Agentic Loop 设置终止条件 | Agentic Loop |
| **上下文爆炸问题** | 压缩策略与滑动窗口管理 | Memory & Context Management |
| **权限失控问题** | Guardrails 的 Allow / Deny / Ask 三级控制 | Guardrails |
| **质量不可控问题** | 循环内嵌入质量审核节点 | Hooks + Evaluator |
| **成本不透明问题** | Token 追踪与预算约束 | Session |

**Allow / Deny / Ask 三级权限模型**：
- **Allow**：白名单内自动放行
- **Deny**：黑名单内直接拒绝
- **Ask**：模糊地带交人确认

**关键认知（26）**：
> 底层是无状态的纯推理大模型；Harness 做的是把大模型的大脑变成 Agent 的身体。

无限循环、上下文爆炸这些问题，本质都是"大脑没有身体的协调机制"，必须由 Harness 补齐。

---

### 框架 5：Context Reset 流程

**核心思想（12、18、22）**：当 Context Anxiety 触发或上下文利用率接近上限时，**不压缩，而是清空 + 重启**。

**完整流程**：

```
1. 触发条件检测：上下文利用率 > 60% OR Agent 行为退化迹象
   ↓
2. 写入结构化交接文件（durable state artifact）：
   - 当前任务进度（JSON 格式 feature 列表）
   - 已完成的 commit / PR
   - 待解决的问题清单
   - 关键决策日志
   ↓
3. 结束当前 Agent，启动全新 Agent
   ↓
4. 新 Agent 冷启动：
   - 不加载历史对话
   - 读取结构化交接文件
   - 30 秒内重建任务理解
   ↓
5. 继续推进
```

**关键约束（22）**：
- 新 Agent 必须能在**冷启动后 30 秒内读取、理解、续航**
- 如果做不到，说明 Durable State 设计失败
- 状态 ≠ "保存聊天记录"，必须是结构化工件

**对比 Compaction**：

| 维度 | Compaction | Context Reset |
|------|-----------|---------------|
| 上下文 | 摘要保留 | 完全清空 |
| 焦虑感 | 仍然存在 | 消除 |
| 实施难度 | 低（自动） | 高（需要 durable state 设计） |
| 适用场景 | 短任务、单次会话 | 长任务、跨多轮 |

**演化趋势（12）**：
- Sonnet 4.5 需要 Context Reset
- Opus 4.5 不需要了
- 但 Evaluator 永远去不掉（独立评估的需求边界不会消失）

### 框架 6：On-the-loop 升级路径

**核心思想（22）**：把"修产出"升级为"改系统"，建立平台化的反馈循环。

**升级路径四阶段**：

| 阶段 | 行为模式 | 投入 | 复利 |
|------|---------|------|------|
| **L0** | 看到错就改 | 线性 | 无 |
| **L1** | 错了把规则写进 AGENTS.md | 一次性 | 单次错误防御 |
| **L2** | 错了把规则做成 Linter / Hook | 中等 | 持续约束 |
| **L3** | 系统化反馈循环（Code Review Agent / Eval 平台） | 高 | 平台化复利 |

**与 Hashimoto Harness Engineering 定义的契合（12）**：
> 每当我发现 Agent 犯了一个错误，我就花时间工程化一个解决方案，让它永远不再犯同样的错误。

**关键判据**：
- L0 → L1：愿意写下规则吗？
- L1 → L2：愿意把规则机械化吗？
- L2 → L3：愿意做平台投入吗？

**为什么大多数团队卡在 L0/L1**：
- L0 → L1 需要克服"懒得写"
- L1 → L2 需要克服"懒得做工具"
- L2 → L3 需要管理层愿意承担工程投入

## 案例库

### 案例 1：Anthropic 三 Agent vs Solo 对比（12）

- **背景**：同一编程任务（游戏开发）的两种实现路线
- **Solo 结果**：游戏能启动但连线断裂、界面无提示
- **三 Agent 结果**：正常游玩 + AI 关卡生成 + 精灵动画 + 音效
- **成本**：三 Agent 是 Solo 的 20x
- **失效模式角度的解读**：Solo 的失败不是"代码写错"，而是 Generator 自评虚高导致"看起来完成了但核心交互断裂"
- **启发**：GAN 式角色分离是质量保障的工程基础，不是过度设计

### 案例 2：Anthropic Sonnet 4.5 Context Anxiety 现象（12、18）

- **背景**：Anthropic 在 Sonnet 4.5 上观察到行为退化
- **现象**：上下文快填满时倾向于提前收工，哪怕任务未完成
- **解法**：Context Reset（不是 Compaction）
- **演化**：Opus 4.5+ 已基本消除此行为
- **启发**：失效模式不是永恒的——Harness 组件随模型迭代会变成冗余，需要定期审视哪些可以删

### 案例 3：OpenAI 自定义 Linter——错误消息嵌修复指令（12）

- **背景**：架构约束如何在没有 senior reviewer 把关时仍然成立
- **做法**：自定义 Linter，报错信息直接包含修复方法
- **关键变化**：Linter 不只是"指出错误"，是"教 Agent 怎么改"
- **配套**：业务域依赖方向固定（Types → Config → Repo → Service → Runtime → UI），违反 PR 自动拦截
- **失效模式应对角度**：抵抗"坏模式复制"（熵增）和"局部决策破坏全局架构"
- **启发**：好的 Sensor 不只是 pass/fail，还应该包含修复指引——把 Sensor 升级成 in-context learning 的高质量例子

### 案例 4：OpenAI 每周 20% 时间清理 AI slop（22）

- **背景**：百万行代码项目的熵增管理
- **数据**：初期人工清理"AI slop"占每周约 20% 时间
- **系统化方案**：
  - documentation consistency agents 定期验证
  - refactor agents 计划清理技术债
  - architectural enforcement 通过 CI 机械维护模块边界
- **失效模式角度**：熵增（Entropy）的真实代价被量化
- **启发**：熵增是隐性税；不主动清理，项目会被自己生成的代码淹死

### 案例 5：MercadoLibre 独立 Code Review Agent（16）

- **背景**：2 万开发者、数千仓库的质量保障
- **做法**：独立 review agent 接入 CI，对所有 PR（人写 + AI 写）做自动分析
- **失效模式角度**：抵抗"自我评估虚高"（开发者审查自己代码也会偏高）
- **组织收益**：
  - 人类 reviewer 更聚焦高层设计与架构决策
  - 更早发现 bug，降低进入生产的风险
  - 质量控制能力随 PR 数量扩展，而非依赖 reviewer 人数
- **启发**：Code Review Agent 是反馈系统的工业化形态，从"个人手工审查"升级为"组织级机制"

### 案例 6：Anthropic Evaluator 调教迭代（18）

- **背景**：把 Evaluator 调教到合理水平的真实路径
- **现象**：开箱即用 Claude 是很差的 QA Agent——看着 Evaluator 发现真正的问题，然后自己说服自己这不是大事，然后通过了验收
- **路径**：花了好几轮迭代才把评估质量调到合理水平
- **关键配置**：配备 Playwright，像真实用户一样操作测试，不只是"读代码打分"
- **启发**：Evaluator 不是"开一个新 Agent 给它一个 prompt"那么简单，需要持续训练 + 工具配备 + 独立性保障

### 案例 7：典型失败案例——需求过粗导致"错得合理"（16）

- **场景**：用户说"做一个后台新增商品功能"，缺失幂等性、权限、API 合约等关键信息
- **结果**：Agent 实现一个 MVP，再次点击按钮重复插入
- **失效模式分类**：目标一致性 + 行动可验证性 双重失败
- **根因**：不是 Agent 不会写代码，而是初始输入未说明隐性业务约束
- **应对**：Spec Driven Development 在写代码前先 lock-in"完成的定义"
- **启发**：很多"AI 写错了"的问题，本质是**反馈系统没有在前端介入**——验收标准没有显式化，Agent 就只能猜

### 案例 8：Vercel 工具精简——15 → 2（与方向 1、2 共享）

- **背景**：失效模式角度看少即是多
- **现象**：工具过多 → 选择噪声 → 准确率下降
- **数据**：工具从 15 砍到 2，准确率 80% → 100%，Token 减少 37%，速度提升 3.5x
- **失效模式角度**：工具集膨胀是 Context Anxiety 和上下文 60% 阈值的隐性触发器
- **启发**：减工具就是减失效面——每个工具都是潜在的失败点

## 关键洞察

> 跨多源融合后的发现。

1. **五大根本挑战是 Harness 存在的"宪法"**
   - 22 提出的五大挑战（状态持久性、目标一致性、行动可验证性、熵增抑制、人机边界）每一条都不能靠"换更强模型"解决
   - 这构成了 Harness 不可被模型吸收的核心理由
   - 当有人说"Harness 会被未来模型吃掉"时，反问：模型能解决人机边界吗？目标一致性的外部锚点能内化吗？
   - → 与「演化与未来形态」方向的"Harness 减重论"形成张力（部分组件可被吸收，但"核心五项"不行）

2. **失效模式有"时效性"——并非永恒**
   - Context Anxiety 在 Sonnet 4.5 严重，Opus 4.5 基本消除
   - Sprint 分解：Opus 4.5 必要，Opus 4.6 变负担
   - 推论：每次模型升级后，团队必做的"诊断动作"是审视哪些失效模式已经不再发生
   - 别把对**当前模型缺陷**的应对固化成永久架构（呼应 30 的"补丁不该变架构"）

3. **40% 和 60% 不是两个数据，是同一现象的两个阶段**
   - 40%（Dex Horthy）：质量开始下降
   - 60%（16 引用研究）：明显变差
   - 工程实施：40% 是触发主动总结/交接的时机，60% 是强制 Context Reset 的警戒线
   - 单独引用任何一个数字都是片面的——需要的是分级响应

4. **反馈系统的成熟度可以用"四象限覆盖率"测量**
   - 22 的 2×2 矩阵不只是分类工具
   - 它强迫团队对照检查：四个象限里你投入了几个？
   - 大多数团队只覆盖一个（Sensors × Computational：写测试）
   - 真正成熟的反馈系统四象限都有投入
   - 这是一个可量化的诊断框架

5. **熵增是 Agent 时代的"隐性税"**
   - 不主动清理，清理速度迟早被生成速度反超
   - OpenAI 每周 20% 时间清理 AI slop 是这个税的真实价格
   - 不做 Entropy Control 的项目看起来跑得快，但在累积不可见的债务
   - Harness 的真正价值之一是**抑制 agent 放大的系统噪声**——这是它与简单框架最本质的区别（22）

6. **In-the-loop → On-the-loop 是工程师在 Agent 时代的核心升级**
   - 大多数团队卡在"看到错就手改"的 L0 模式
   - On-the-loop 的复利效应被严重低估
   - 这个升级**没有技术门槛**——纯粹是工作习惯改变
   - 但它需要管理层支持："不要催我改 bug，让我改系统"

7. **Evaluator 独立性是 GAN 式架构的真正灵魂**
   - 三角色架构的核心不是"分工"
   - 而是 **Evaluator 必须独立于 Generator**
   - 让一个 Agent 既写代码又评价代码 = 自评虚高
   - 独立 Evaluator 即使能力较弱，也能发现 Generator 看不见的问题
   - 这同样适用于其他场景：写 PRD 的人不能验收、做需求的人不能写 acceptance criteria

8. **Sprint Contract = 把验收标准提前到写代码之前**
   - 27 条验收标准看起来繁琐，但避免了"完成的定义模糊"
   - 这是 Guides × Inferential 象限的典型应用
   - 与 Spec Driven Development 共享同一哲学：**输入质量决定输出质量**
   - 越是高复杂度任务，越要在前端做"完成定义"的功夫

## 待探索问题

- **失效模式的诊断框架**
  - 当 Agent 出问题时，如何快速判断是哪类失效模式？
  - 有没有"失效模式决策树"可以教会团队系统诊断？
- **2×2 矩阵的量化测量**
  - 22 提出的 Guides × Sensors × Computational × Inferential 是定性框架
  - 能否给出量化指标（覆盖率、强度、响应时间）？
- **Evaluator 的失效模式**
  - Evaluator 本身会不会过度严苛？
  - Evaluator 与 Generator 共谋（互相说服没事）的可能性？
  - Evaluator 训练数据如何持续更新？
- **熵增的早期预警指标**
  - AI slop 累积是否有可观测信号？
  - 在"清理速度被生成速度反超"前能否预警？
- **Context Reset 的代价**
  - 频繁 Context Reset 会损失什么？
  - 30 秒冷启动续航的边界在哪？
- **失效模式与团队规模的关系**
  - 个人项目可能完全没遇到 Context Anxiety
  - 组织级 rollout 才会暴露熵增、目标漂移
  - 什么规模才"必须开始投资"哪类反馈机制？
