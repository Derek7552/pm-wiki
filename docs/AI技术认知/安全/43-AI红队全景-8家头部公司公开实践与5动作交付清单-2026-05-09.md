# AI 红队全景——8 家头部公司公开实践与 5 动作交付清单

> 来源：原文《AI 红队怎么做？8 家头部公司全公开了》（用户直接提供正文）
> 提炼日期：2026-05-09
> 分类：技术趋势

## TL;DR

AI 红队尚无稳定流水线，但 OpenAI、Anthropic、Microsoft、Mozilla、OWASP、NIST、CSA、Google 八家已把流程、字段、案例、工具、bounty 金额、报告格式公开，形成可拼接的方法论地图。从 Web 渗透转 AI 红队的关键差异有三条：合同里必须新增 **Model access** 维度（黑盒/白盒/可否 fine-tune/可否取 system prompt）、报告里必须每条 finding 配**最小可复现 case**（不是 prompt 截图）、风险分类必须对齐 **OWASP Top 10 for LLM + NIST AI 600-1 + CSA Agentic 12 类**而非自造体系。Anthropic × Firefox 用 Claude Opus 4.6 在 2 周扫 6000 个 C++ 文件、产出 22 个 CVE / 14 个高危的实战，验证了 AI 辅助代码审计已到达"产能足以重写交付承诺"的拐点。

## 核心观点

### 观点一：AI 红队没成熟流水线，但已有八张可直接拼接的拼图

Web 渗透有 OWASP WSTG 这种"端口扫描→应用扫描→漏洞验证→报告"的稳定流水线，AI 红队没有，因为被测对象本身在快速变形：去年还是 Chat 接口，今年就是 Agent + RAG + MCP + 多工具调用。但行业不是空白——OpenAI 的 4 决策模型（Scope/Membership/Model access/Final report）、Anthropic 的 AI 辅助代码审计实战、Microsoft 的 PyRIT 6 件套乐高架构、Mozilla 0Din 的两阶段提交流程、OWASP 的 4 大领域 + Top 10 for LLM、NIST AI 600-1 的 12 类风险 + 4 RMF Function、CSA 的 Agent 12 类威胁、Google SAIF 的 6 大要素，全部是公开一手源。乙方应做的是把它们拼成"自家交付套件"，不要自造方法论。

### 观点二：AI 红队相对 Web 渗透多了一项决定性变量——Model access

Web 渗透是"黑盒 / 白盒"，影响发现深度但不影响测试方法本身；AI 红队的 Model access 直接决定能用哪种测试方法：黑盒 API 只能 prompt 探测，白盒可以看 system prompt、看 fine-tune 数据、看 fallback 策略。这一项必须单列在合同附件，OpenAI 把它放进 4 决策模型的核心。把 Model access 没写清楚就开测，等于拿到一张没有比例尺的地图。

### 观点三：AI 辅助审计的产能已经改写客户对"两周能看多少"的预期

Anthropic 用 Claude Opus 4.6 给 Mozilla Firefox 做代码层红队：2 周内扫 6000 个 C++ 文件、提交 112 份独特报告、Mozilla 发出 22 个 CVE（其中 14 个高危）、修复版本 Firefox 148。启动后仅 20 分钟识别出第一个漏洞，两周内发现 50+ 独特崩溃输入。Mozilla 自己博客同步发了致谢文，数据完全对得上，不是 PR 稿。这意味着客户以后不会再接受"两周看 200 个文件"这种交付承诺——AI 辅助审计的标准产能已经换档。

## 关键知识点

### 1. OpenAI Red Teaming Network — 4 个核心决策

| 决策项 | 含义 |
|--------|------|
| **Scope** | 测什么、不测什么（合同附件） |
| **Membership** | 哪些专家进队 |
| **Model access** | 黑盒 API / 白盒 / 可否 fine-tune / 可否取 system prompt / 可否看 fallback 策略 |
| **Final report 格式** | 交付什么样的结构化报告 |

**测试方法两条腿**：
- **人工 prompt**：偏发现新风险/新场景，人在 sandbox 反复试。
- **自动化 + classifier 评分**：偏批量覆盖，几千条 payload 跑分类器，再人工抽查高置信样本。

一手源：
- https://openai.com/index/red-teaming-network/
- https://cdn.openai.com/papers/openais-approach-to-external-red-teaming.pdf

### 3. Microsoft PyRIT — 6 件套乐高架构

PyRIT = Python Risk Identification Tool。官方原话："As much as possible, each component is a Lego-brick of functionality."

| 组件 | 作用 | 关键能力 |
|------|------|---------|
| **Datasets** | 基础数据 | prompts、jailbreak 模板、攻击策略 |
| **Attacks** | 编排层 | single-turn（Role Play / Skeleton Key）、multi-turn（Crescendo / Tree of Attacks） |
| **Converters** | prompt 转换 | 文本改写、文档转换、图片叠加，可堆叠组合 |
| **Targets** | 攻击目标 | 通常是 LLM |
| **Scoring Engine** | 评估攻击结果 | 自动评分 |
| **Memory** | 会话状态 | 追踪对话和分数，存 SQLite 或 Azure SQL |

**乐高化的好处**：每个组件都有 NoOp 版本方便调试；一个攻击的 prompt 可以喂给另一个攻击；一个场景下的攻击可以挂多个 targets。

**Microsoft AIRT 公开案例**：用 PyRIT 跑 100+ 次红队行动，覆盖 Copilot、Phi-3 等产品；某 Copilot 评估中自动生成几千条对抗 prompt + 自动评分 + 人工审查阳性结果，"几小时拿到的结果传统人工方式要几周"。

**仓库迁移**：原本在 Azure 组织下，2026 年 3 月迁移到 Microsoft 主组织。

一手源：
- https://github.com/microsoft/PyRIT
- https://microsoft.github.io/PyRIT/code/framework

### 4. Mozilla 0Din — 独立 GenAI Bug Bounty

**身份**：0Day Investigative Network，Mozilla 自营 GenAI 专项 bug bounty 平台，**独立于 HackerOne**（Mozilla 在 HackerOne 上有传统 bounty，0Din 是单独的 GenAI 项目）。

**Bounty 金额档位**：
- Low：≤ $500
- Medium：≤ $2,500
- High：≤ $5,000
- Severe：≤ $15,000

**Scope 三类**：
- **Guardrail jailbreak** — 绕过模型安全护栏
- **Prompt injection** — 通过外部输入劫持模型行为
- **Training data leakage** — 从模型输出反向提取训练数据

**两阶段提交流程**（关键设计）：
1. **第一阶段**：High-level abstract（风险类别 + 受影响模型）→ Mozilla 在 **3 工作日内**回 scope 评估和 bounty 范围。
2. **第二阶段**：完整技术细节（PoC + 复现步骤 + impact 分析）→ **2 周保密期**内双方复测。

**附加资产**：0Din 旗下开源 **179 个 security probes**，覆盖 35 个漏洞家族——可直接用作 payload set 起点。

一手源：
- https://0din.ai/
- https://github.com/0din-ai/0din.ai

### 5. OWASP — GenAI Red Teaming Guide + Top 10 for LLM

**OWASP GenAI Red Teaming Guide**（2025-01-22 发布）覆盖 4 大领域：

| 领域 | 测试对象 |
|------|---------|
| **Model evaluation** | 对齐、越狱、幻觉、bias |
| **Implementation testing** | prompt 工程、RAG、function calling、输出处理 |
| **Infrastructure assessment** | 部署架构、API 安全、数据流、权限 |
| **Runtime behavior analysis** | 真实流量下的 Agent 链路、tool 调用、memory 使用 |

**OWASP Top 10 for LLM Applications**（报告里写"风险类别"那栏可直接对齐）：

| 编号 | 风险 |
|------|------|
| LLM01 | Prompt Injection |
| LLM02 | Insecure Output Handling |
| LLM03 | Training Data Poisoning |
| LLM04 | Model Denial of Service |
| LLM05 | Supply Chain Vulnerabilities |
| LLM06 | Sensitive Information Disclosure |
| LLM07 | Insecure Plugin Design |
| LLM08 | Excessive Agency |
| LLM09 | Overreliance |
| LLM10 | Model Theft |

一手源：
- https://genai.owasp.org/resource/genai-red-teaming-guide/
- https://genai.owasp.org/llm-top-10/

### 6. NIST AI 600-1 — Generative AI Profile

**身份**：Artificial Intelligence Risk Management Framework: Generative Artificial Intelligence Profile，2024-07-26 发布；是 NIST AI RMF 1.0 的配套生命周期 Profile，响应 Biden 政府 Executive Order 14110（2023-10 签署）。

**12 类生成式 AI 风险**（举例）：
- 数据投毒（Data poisoning）
- 幻觉（Hallucinations）
- CBRN 信息可达（化学/生物/放射/核相关有害信息）
- 有害内容（Harmful content）
- 数据隐私违规（Data privacy violations）

**7 个 Trustworthy AI 特性**：Safe / Explainable / Fair / Accountable / Privacy Enhanced / Secure / Valid and Reliable。

**4 个 RMF 核心 Function**（建议动作映射到这里）：
- **GV** (Govern) — 治理
- **MP** (Map) — 映射
- **MS** (Measure) — 测量
- **MG** (Manage) — 管理

**报告写法**：每条 finding 标注"对应 NIST AI 600-1 的 GV-1.2 / MP-2.3 / MS-3.4 / MG-1.1"——客户合规和审计团队立刻看懂说的是 RMF 哪一块。

一手源：
- https://nvlpubs.nist.gov/nistpubs/ai/NIST.AI.600-1.pdf
- https://www.nist.gov/itl/ai-risk-management-framework

### 7. CSA Agentic AI Red Teaming Guide — 12 类 Agent 威胁

**身份**：CSA × OWASP 联合，2025-05-28 发布，50+ 贡献者。**填补了上面所有框架对 Agent 红队的覆盖空白**。

**12 类威胁完整清单**：

| # | 威胁 | 含义 |
|---|------|------|
| 1 | Agent authorization and control hijacking | 授权与控制劫持，利用权限层与自主 Agent 之间的缝隙 |
| 2 | Checker-out-of-the-loop | 审核者脱环，绕过人在回路审批 |
| 3 | Agent critical system interaction | 与生产关键系统的不安全交互 |
| 4 | Goal and instruction manipulation | 目标和指令链篡改 |
| 5 | Agent hallucination exploitation | 利用幻觉输出作攻击 |
| 6 | Agent impact chain and blast radius | 影响链与爆炸半径 |
| 7 | Agent knowledge base poisoning | 检索/参考库投毒 |
| 8 | Agent memory and context manipulation | 持久记忆被污染或滥用 |
| 9 | Multi-agent exploitation | 多 Agent 间合谋或互相利用 |
| 10 | Resource and service exhaustion | 算力/API 配额耗尽 |
| 11 | Supply chain and dependency attacks | 模型/工具/数据集供应链投毒 |
| 12 | Agent untraceability | 行为不可审计 |

每类都给了**测试方法 + 攻击向量 + 示例 prompt + deliverables**。

**推荐工具**：MAESTRO、Promptfoo 的 LLM Security DB、SplxAI 的 Agentic Radar、Salesforce FuzzAI、Microsoft Foundry red teaming agents。

**与 OWASP Top 10 for LLM 的关系**：不冲突，互补。**单 LLM 应用对齐 OWASP Top 10，Agent 系统额外对齐 CSA 12 类**。

一手源：
- https://cloudsecurityalliance.org/artifacts/agentic-ai-red-teaming-guide

### 8. Google SAIF — Secure AI Framework

**身份**：2023-06 发布，工业界最早把 AI 安全从"研究问题"提到"工程框架"层面的尝试之一。**SAIF 不是红队指南，是端到端工程框架**——覆盖数据、模型训练、部署、监控、事件响应整条链路。红队是其中一个环节，不是孤立活动。

**SAIF 6 大要素**：
1. 把强大的安全基础扩展到 AI 生态
2. 把 AI 检测和响应纳入组织现有安全运营
3. 自动化防御应对存量和新型威胁
4. 协调平台级控制以实现整体一致性
5. 调整控制以加快 AI 部署的反馈循环
6. 把 AI 系统风险纳入更广的业务流程

一手源：
- https://safety.google/intl/en/cybersecurity-advancements/saif/
- https://blog.google/technology/safety-security/introducing-googles-secure-ai-framework/

| 优先级 | 动作 | 落地点 | 来源参考 |
|-------|------|--------|---------|
| **P0** | Scope 多写一项 Model access | 合同阶段 | OpenAI 4 决策模型 |

## 数据与案例

### 关键时间线

- **2023-06**：Google 发布 SAIF。
- **2023-10**：Biden 签署 Executive Order 14110。
- **2024-07-26**：NIST 发布 AI 600-1。
- **2025-01-22**：OWASP 发布 GenAI Red Teaming Guide。
- **2025-05-28**：CSA × OWASP 发布 Agentic AI Red Teaming Guide。
- **2025 年末**：Anthropic × Mozilla Firefox 红队启动。
- **2026-03**：PyRIT 仓库从 Azure 迁移到 Microsoft 主组织。

### Anthropic × Firefox 实战完整链条

- **背景**：Mozilla 想测试 AI 辅助代码审计能否进入 Firefox 的安全保证流水线。
- **做法**：Anthropic 用 Claude Opus 4.6，2 周内扫近 6000 个 C++ 文件，每条 finding 附最小复现 case。
- **结果**：112 份独特报告 → 22 个 CVE（14 个高危）+ 90 个其他 bug，Firefox 148 修复，启动后 20 分钟识别第一个漏洞，2 周发现 50+ 独特崩溃输入。
- **验证**：Mozilla 自家博客同步发文，承认问题、感谢、列修复版本号——双方数据对得上，不是单边宣传。

### Microsoft PyRIT 实战完整链条

- **背景**：Microsoft AIRT 团队需要在 Copilot、Phi-3 等产品发布前完成红队评估，纯人工无法覆盖。
- **做法**：用 PyRIT 自动生成几千条对抗 prompt → Scoring Engine 自动评分 → 人工审查阳性结果。
- **结果**：100+ 次红队行动覆盖多个产品；几小时得到的结果，传统人工要几周。

### Mozilla 0Din 两阶段提交收益

- 研究员先验证"这个范围是不是已经在做了"再投精力。
- Mozilla 先评估"这个有没有 bounty 价值"再要技术细节。
- 双方都被保护，避免一次性把完整技术细节交出去后客户不认 scope。

> ⚠️ 待验证：原文提到 Anthropic 用的是 Claude Opus **4.6**，与 Anthropic 当前最新公开模型版本是否一致需对照官方公告。

## 启发与思考

### 与个人工作的关联（云起无垠产品总监 / 安全 + AI 方向）

1. **AI 红队产品化的差距点已经清晰**：客户问"AI 红队报告长什么样"，答案不是再造方法论，而是把 OpenAI 4 决策 + Anthropic 复现文化 + OWASP/NIST/CSA 三层风险分类 + 0Din 两阶段提交 + PyRIT 自动化 拼成自家交付套件。这是云起 SaaS 化交付物的现成蓝图。
2. **Model access 是产品定价的隐藏维度**：黑盒/白盒/fine-tune 接入是不同 SKU，对应不同发现深度和不同价格。Web 渗透按时间报价，AI 红队应该按 access 层级 + Agent 复杂度报价。
3. **PyRIT 直接可用，不要自造**：评估流水线层面 PyRIT 已经把 Datasets / Attacks / Converters / Targets / Scoring / Memory 拆成乐高，加自家 datasets 和 scoring rules 即可。云起内部 PoC 不应再做底层框架。
4. **Anthropic Firefox 案例改写客户预期**：以前提"两周看 200 个文件"是合理交付，现在客户读过这个数据卡后会问"能不能两周扫 6000 个？"——产品和销售话术要前置准备这个对比。
5. **Agent 红队是新蓝海**：CSA 12 类威胁（多 Agent 合谋、知识库投毒、记忆污染、影响链溢出）是传统 Web 渗透完全覆盖不到的——客户系统从 Chat 升级到 Agent + 多工具调用时，原服务商接不住，乙方有窗口期。

### 可落地的行动项

- **报告模板升级**：在云起当前的渗透/红队报告模板里加三个字段——`Model access`、`OWASP Top 10 for LLM 编号`、`NIST AI 600-1 RMF 编号（GV/MP/MS/MG）`；Agent 类目标加 CSA 12 类编号字段。
- **合同附件升级**：在 SOW 模板里把"Model access 权限矩阵"列为必填项，缺失时不开测。
- **PoC 框架选型**：把内部 AI 红队工具底座切到 PyRIT，按其 6 件套架构组织数据集和评分逻辑，避免重复造轮子。
- **payload 库初始化**：先把 0Din 的 179 个 security probes 跑一遍，过滤掉对国内场景不适用的，作为云起 AI 红队 payload 库的种子。
- **销售话术更新**：把 Anthropic × Firefox 的数据卡（2 周/6000 文件/22 CVE/14 高危/20 分钟首发）做成一页对比图，放进客户提案的开场。

### 值得进一步探索

- **国内 AI 安全标准与 OWASP/NIST 的对照表**：GB/T 系列里哪些与 OWASP Top 10 for LLM 可对齐？哪些是中国独有？这张映射表做出来，能直接进客户报告的"国际对标 + 国内合规"双栏。
- **Agent 级红队工具链**：MAESTRO、Promptfoo LLM Security DB、SplxAI Agentic Radar、Salesforce FuzzAI、Microsoft Foundry red teaming agents 五家工具的 PoC 对比——哪家最适合接入云起现有产品。
- **AI 辅助代码审计的复制路径**：Anthropic Firefox 实战是否可在国内场景（如某 OSS 项目或客户私域代码库）复现？复现的关键变量是什么——模型、scope 设计、还是补丁验证流水线？

## 原文精华

> "Web 渗透那种'端口扫描 + 应用扫描 + 漏洞验证 + 报告'的稳定流水线，在 AI 这边还没成型。但好消息是——国际几家头部团队已经把他们做 AI 红队的实战公开了。"

> "WSTG 里 Scope 是合同附件，AI 红队也一样。但多一项关键的——Model access。Web 渗透是黑盒还是白盒，影响的是发现深度。AI 红队 Model access 影响的是测试方法本身。"

> "扫描了将近 6000 个 C++ 文件 / 提交 112 份独特报告 / Mozilla 最终发出 22 个 CVE，其中 14 个高严重性 / 启动后仅 20 分钟，Claude 就识别出第一个漏洞。"

> "As much as possible, each component is a Lego-brick of functionality."（PyRIT 官方文档）

> "OWASP Top 10 for LLM 之于 AI 红队，相当于 OWASP Top 10 之于 Web 渗透。报告里每条 finding 直接标注 LLM01-10 类别，客户和复测方都能秒懂。不要自己发明风险分类。"
